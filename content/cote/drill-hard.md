# 8.9 실전 풀이 IV — 고난도

::: lead
지금까지의 실전 풀이는 한 가지 도구로 끝났다. 그래프 문제는 BFS/DFS 하나로, DP 문제는 점화식 하나로. 고난도 문제는 다르다. **제약 조건 하나가 알고리즘 하나를 요구하고, 다른 제약 조건이 또 다른 알고리즘을 요구한다.** 두 요구를 동시에 만족시키려면 두 도구를 한 자료구조 위에서 합쳐야 한다. 이 절에서는 "다익스트라인데 상태에 DP 차원이 하나 더 붙는" 문제와 "그리디인데 그 그리디를 빠르게 만드는 게 유니온 파인드인" 문제를 각각 브루트포스부터 실행 검증까지 끝까지 풀어 본다. 둘 다 실제 대회 문제를 그대로 베끼지 않고 같은 유형을 자기완결적으로 재구성한 예제다.
:::

## 왜 복합 유형이 어려운가

단일 유형 문제는 "이건 [7.15 최단 경로](#/shortest-path) 문제다"라고 이름을 붙이는 순간 절반이 끝난다. 복합 유형은 이름을 하나만 붙일 수 없다. 흔히 이런 식으로 나타난다.

- 최단 경로를 구해야 하는데, **경로 위에서 쓸 수 있는 자원(할인권, 통행료 면제, 순간이동)이 제한**되어 있다 → 다익스트라의 상태에 "자원을 몇 개 썼는가"라는 [7.20 DP](#/dp-basics) 차원을 추가해야 한다.
- 그리디로 최적해를 고를 수 있다는 건 증명했는데, **그 그리디를 구현하려면 "지금 쓸 수 있는 가장 늦은 자리를 빠르게 찾는" 연산이 필요**하다 → [7.12 유니온 파인드](#/union-find)가 그 연산을 거의 $O(1)$로 만들어 준다.

두 경우 모두 패턴이 같다. **한 알고리즘이 문제의 "무엇을 최적화할지"를 정하고, 다른 알고리즘/자료구조가 그 최적화를 "충분히 빠르게" 만든다.** 이 구도를 알아채는 게 복합 유형 공략의 핵심이다. 제약 조건을 읽을 때 "이 조건이 상태 공간을 넓히는가, 아니면 연산 속도를 요구하는가"를 나눠서 봐라. [8.4 문제 유형 분류](#/problem-signals)에서 다룬 신호 읽기를 두 번 겹쳐 적용한다고 생각하면 된다.

::: tip 지문을 읽을 때 두 번 밑줄을 그어라
복합 유형 문제를 만나면 지문을 두 번 읽으면서 서로 다른 밑줄을 그어라. 첫 번째 패스에서는 "무엇을 최적화하는가"에 해당하는 문장(최단 비용, 최대 이익 같은 목적 함수)에 밑줄을 긋는다. 두 번째 패스에서는 "그 최적화를 느리게 만드는 조건"(자원 제한, 반복되는 탐색)에 다른 색으로 밑줄을 긋는다. 두 밑줄이 겹치는 문장이 없다면 십중팔구 복합 유형이고, 두 알고리즘을 따로 떠올려야 한다.
:::

## 문제 I: 통행료 할인권 — 다익스트라 + DP

::: cote 문제
$N$개의 도시(1번 ~ $N$번)와 $M$개의 양방향 도로가 있다. 각 도로에는 통행 비용이 있다. 당신은 **통행료 할인권을 최대 $K$장** 가지고 있고, 도로 하나를 지날 때 할인권을 한 장 쓰면 그 도로는 **공짜**가 된다. 1번 도시에서 $N$번 도시까지 가는 데 드는 **최소 비용**을 구하라. 할인권은 안 써도 되고, $K$장을 다 안 써도 된다.
:::

### 왜 순수 다익스트라로는 안 되는가

일반 다익스트라는 "정점마다 지금까지 알아낸 최소 비용 하나"만 들고 다닌다. 그런데 이 문제에서는 **같은 도시에 더 비싸게 도착했더라도 할인권을 덜 썼다면, 그 상태가 나중에 더 유리할 수 있다.** 예를 들어 도시 A에 비용 5로 도착했지만 할인권을 다 썼다면, 그다음 비싼 도로 앞에서 손해를 본다. 반대로 도시 A에 비용 8로 도착했지만 할인권이 남아 있다면, 그다음 비싼 도로를 공짜로 넘을 수 있다. **"도시"만으로는 상태를 구분할 수 없고, "도시 + 지금까지 쓴 할인권 수"까지 있어야 상태가 유일해진다.** 이게 DP 차원이 다익스트라에 얹히는 지점이다.

### 상태 확장: (도시, 사용한 할인권 수)

$dist[v][k]$ = "도시 $v$에 할인권을 정확히 $k$장 써서 도착하는 최소 비용"으로 정의한다. 간선 $(u, v, w)$에서 나가는 전이는 두 가지다.

- 할인권을 안 쓴다: $dist[v][k] = \min(dist[v][k],\ dist[u][k] + w)$
- 할인권을 쓴다 ($k < K$일 때): $dist[v][k+1] = \min(dist[v][k+1],\ dist[u][k])$

이건 정확히 [7.8 힙](#/heap)을 우선순위 큐로 쓰는 다익스트라인데, 힙에 들어가는 노드가 "도시"가 아니라 "(도시, $k$)" 쌍이라는 점만 다르다. 노드 수가 $N$개에서 $N \times (K+1)$개로 늘었을 뿐, 알고리즘의 뼈대(항상 가장 싼 상태부터 확정한다)는 그대로다. 정답은 $\min_k dist[N][k]$다.

```python title="dijkstra_ticket.py — 다익스트라 + DP 상태 확장"
import heapq

INF = float("inf")

def min_cost_with_tickets(n: int, edges: list[tuple[int, int, int]], k: int) -> int:
    adj = [[] for _ in range(n + 1)]
    for u, v, w in edges:
        adj[u].append((v, w))
        adj[v].append((u, w))

    # dist[node][used] = 할인권을 정확히 used장 써서 node에 도착하는 최소 비용
    dist = [[INF] * (k + 1) for _ in range(n + 1)]
    dist[1][0] = 0
    pq = [(0, 1, 0)]          # (비용, 도시, 사용한 할인권 수)

    while pq:
        cost, node, used = heapq.heappop(pq)
        if cost > dist[node][used]:
            continue          # 이미 더 싼 경로로 확정된 상태
        for nxt, w in adj[node]:
            nc = cost + w                       # 할인권 없이 이동
            if nc < dist[nxt][used]:
                dist[nxt][used] = nc
                heapq.heappush(pq, (nc, nxt, used))
            if used < k:                        # 할인권을 써서 이동
                nc2 = cost
                if nc2 < dist[nxt][used + 1]:
                    dist[nxt][used + 1] = nc2
                    heapq.heappush(pq, (nc2, nxt, used + 1))

    return min(dist[n])
```

### 브루트포스와 대조 검증

이 문제의 브루트포스는 독립적인 관점에서 짜야 검증 의미가 있다. **모든 단순 경로(같은 도시를 두 번 지나지 않는 경로)를 전부 나열하고, 각 경로에서 가중치가 가장 큰 간선 $K$개를 공짜로 만든 뒤 남은 비용을 계산한다.** 경로가 고정되면 "그 경로 안에서 어떤 간선에 할인권을 쓸지"는 당연히 가장 비싼 간선부터 골라야 손해가 없으니, 이 브루트포스는 다익스트라 코드와 완전히 다른 경로로 같은 답에 도달한다.

```python title="brute_ticket.py — 모든 단순 경로 나열 후 대조"
def brute_min_cost(n, edges, k):
    adj = [[] for _ in range(n + 1)]
    for u, v, w in edges:
        adj[u].append((v, w))
        adj[v].append((u, w))

    best = [float("inf")]

    def dfs(node, visited, path_weights):
        if node == n:
            weights = sorted(path_weights, reverse=True)
            free = sum(weights[:k])                # 가장 비싼 k개를 공짜로
            cost = sum(path_weights) - free
            best[0] = min(best[0], cost)
            return
        for nxt, w in adj[node]:
            if nxt not in visited:
                visited.add(nxt)
                path_weights.append(w)
                dfs(nxt, visited, path_weights)
                path_weights.pop()
                visited.remove(nxt)

    dfs(1, {1}, [])
    return best[0]
```

정점 4~7개짜리 무작위 그래프 200개, 할인권 수 0~3장을 무작위로 조합해 두 함수를 대조했다.

```text nolines
mismatches: 0 / 200
```

(실제로 실행해서 확인한 결과다.) 구체적인 예시로 확인하면 왜 이 문제가 순수 다익스트라와 다른지 보인다.

```python title="예시 그래프: 짧지만 비싼 직행 vs 길지만 싼 우회"
example_edges = [
    (1, 2, 10), (2, 3, 10), (3, 6, 10),   # 우회로: 총 30
    (1, 4, 1),  (4, 5, 1),  (5, 6, 1),    # 우회로: 총 3 (더 쌈)
    (1, 6, 100),                          # 직행: 100이지만 간선이 하나뿐
]
for k in (0, 1, 2):
    print("k=", k, "최소 비용 =", min_cost_with_tickets(6, example_edges, k))
```

```text nolines
k= 0 최소 비용 = 3
k= 1 최소 비용 = 0
k= 2 최소 비용 = 0
```

$k=0$일 때는 우회로(비용 1짜리 세 간선, 합 3)가 이긴다. 그런데 $k=1$이 되는 순간 답이 **0**으로 뚝 떨어진다. 직행 간선 하나(비용 100)에 할인권을 쓰면 그 간선 자체가 공짜가 되기 때문이다. **경로가 짧을수록 할인권 한 장의 효과가 크다** — 간선 수가 적으면 그중 하나를 공짜로 만들었을 때 비율로 아끼는 비용이 크다는 뜻이다. 이건 순수 최단 경로 사고("무조건 우회로가 싸다")만으로는 나올 수 없는 답이고, 정확히 다익스트라와 DP를 같이 놓아야 잡히는 지점이다.

::: warn 정점이 아니라 상태를 큐에 넣어라
이 유형에서 가장 흔한 실수는 `visited` 배열을 도시 단위로만 두는 것이다. "도시 3을 이미 방문했으니 다시 큐에 안 넣는다"는 순수 다익스트라의 논리를 그대로 가져오면, 할인권을 다르게 쓴 상태를 놓친다. **"도시"가 아니라 "(도시, 사용한 할인권 수)"가 다익스트라의 노드다.** 이 확장 없이 `dist[node]`만 두면 코드는 돌아가지만 답이 틀린다 — 컴파일 에러도 런타임 에러도 없이 조용히 틀리는 가장 위험한 부류다.
:::

### 실행 시간: 상태 공간이 커져도 여전히 빠르다

상태 수가 $N$에서 $N(K+1)$로 늘었으니 다익스트라의 시간복잡도는 $O(NK \log(NK) + MK)$ 정도로 커진다. 정점 수를 50 → 200 → 800으로 키우고 간선을 정점 수의 3배, $K=5$로 고정해서 실제로 재 보면 이렇다.

::: perf 상태가 늘어도 성장 곡선은 그대로다
```text nolines
n= 50   dijkstra+DP 실행시간=0.00031s
n= 200  dijkstra+DP 실행시간=0.00103s
n= 800  dijkstra+DP 실행시간=0.00518s
```

(Python 3.14.5 / Windows 기준 실측.) 정점이 16배(50→800) 늘었는데 시간은 약 17배 늘었다 — $K$가 상수로 고정된 이상 사실상 원래의 $O(M \log N)$ 다익스트라와 같은 성장 곡선을 그린다. **DP 차원을 추가해도 그 차원의 크기($K+1$)가 문제에서 작게 주어지는 한, 알고리즘의 본질적인 확장성은 다익스트라 그대로다.**
:::

$K$ 자체가 정점 수만큼 커지는 문제라면 이야기가 다르지만, 이런 유형에서 $K$는 보통 작게(수십 이하로) 주어진다 — 제약 조건에서 그 신호를 확인하는 습관은 [8.4 문제 유형 분류](#/problem-signals)에서 다룬 것과 같다.

## 문제 II: 마감 있는 작업 스케줄링 — 그리디 + 유니온 파인드

::: cote 문제
작업이 $N$개 있다. 각 작업 $i$는 **마감시간 $d_i$**(정수, $1$부터 $D$까지)와 **이익 $p_i$**를 갖는다. 모든 작업은 정확히 **1단위 시간**이 걸리고, 한 시간에는 작업을 하나만 할 수 있다. 각 작업은 마감시간 $d_i$ **이전에**(즉 시각 $1, 2, \dots, d_i$ 중 한 슬롯에) 끝내야만 이익을 얻는다. 마감을 못 맞추는 작업은 그냥 포기한다. **얻을 수 있는 총 이익의 최댓값**을 구하라.
:::

### 그리디가 맞는 이유, 그리고 그리디만으로는 부족한 이유

직관은 이렇다. **이익이 큰 작업부터 욕심내서 배정하고, 그 작업을 배정할 수 있는 슬롯 중 가장 늦은 자리를 준다.** 이익이 큰 걸 먼저 고르는 건 당연하고("교환 논증"으로 정당화된다 — [7.19 그리디](#/greedy) 참고: 이익이 더 큰 작업을 나중에 고르고 더 작은 작업을 먼저 고른 스케줄이 있다면 그 둘의 순서를 바꿔도 손해가 없다), "가장 늦은 자리"를 주는 이유는 **이른 자리를 아껴 둬야 마감이 빠듯한 다른 작업에 쓸 수 있기 때문**이다.

문제는 "지금 이 작업이 쓸 수 있는 가장 늦은 빈 슬롯"을 어떻게 빨리 찾느냐다. 순진하게 마감시간부터 거꾸로 하나씩 훑으면 슬롯이 이미 다 찬 구간에서 매번 처음부터 다시 훑어야 해서 최악의 경우 $O(D)$가 걸리고, 작업 $N$개에 대해 $O(ND)$가 된다. **여기서 유니온 파인드가 들어온다.** "슬롯 $x$가 이미 찼으면, $x$보다 하나 작은 슬롯을 대표로 삼아라"를 union 연산으로 표현하면, "지금 쓸 수 있는 가장 늦은 빈 슬롯 찾기"가 `find` 한 번으로 끝난다. 이게 [7.12 유니온 파인드](#/union-find)의 경로 압축이 이 그리디를 사실상 $O(N \alpha(N))$으로 만드는 지점이다. MST를 만들 때([7.16 최소 신장 트리](#/mst)) 쓰는 유니온 파인드와 목적은 다르지만("사이클을 막는다" 대신 "빈자리를 찾는다") 도구는 똑같다.

::: note 이 트릭은 "가장 늦은 자리" 전용이 아니다
`parent[slot] = slot - 1`로 "찬 자리 → 그보다 하나 이른 자리"를 잇는 트릭은 방향만 뒤집으면 "가장 이른 빈자리 찾기"에도 그대로 쓸 수 있다(`parent[slot] = slot + 1`). 슬롯이 일렬로 늘어서 있고 "찬 자리는 건너뛰고 가장 가까운 빈자리를 상수 시간에 찾는다"는 요구가 있으면, 마감시간이 아니어도 이 유니온 파인드 패턴을 의심해 봐라.
:::

```python title="job_scheduling.py — 그리디 + 유니온 파인드"
def max_profit_greedy_dsu(jobs: list[tuple[int, int]], max_deadline: int) -> tuple[int, int]:
    # jobs: (마감시간, 이익) 목록. 슬롯은 1..max_deadline, 슬롯 0은 "더 이상 자리 없음" 표시.
    parent = list(range(max_deadline + 1))

    def find(x: int) -> int:
        while parent[x] != x:
            parent[x] = parent[parent[x]]   # 경로 압축
            x = parent[x]
        return x

    total_profit = 0
    scheduled = 0
    for deadline, profit in sorted(jobs, key=lambda job: -job[1]):   # 이익 내림차순
        slot = find(deadline)
        if slot > 0:                         # 쓸 수 있는 자리가 있다
            total_profit += profit
            scheduled += 1
            parent[slot] = slot - 1          # 이 자리는 이제 "다 찼다"
    return total_profit, scheduled
```

### 브루트포스와 대조 검증

브루트포스는 그리디·유니온 파인드와 무관하게 **모든 부분집합**을 시도한다. 부분집합이 실현 가능한지는 잘 알려진 조건 하나로 확인할 수 있다. **마감시간 오름차순으로 정렬했을 때, $i$번째(1부터 시작)로 작은 마감시간이 그 순위 $i$ 이상이어야 한다.** 이건 그리디 알고리즘과 독립적으로 성립하는 조합적 사실이라, 대조군으로 쓰기에 적합하다.

```python title="brute_job.py — 모든 부분집합 전수 조사"
def is_feasible(deadlines: list[int]) -> bool:
    for i, d in enumerate(sorted(deadlines)):
        if i + 1 > d:          # i+1번째로 마감이 빠른 작업인데 자리가 모자란다
            return False
    return True

def brute_max_profit(jobs: list[tuple[int, int]]) -> int:
    n = len(jobs)
    best = 0
    for mask in range(1 << n):
        subset = [jobs[i] for i in range(n) if mask & (1 << i)]
        if is_feasible([d for d, p in subset]):
            best = max(best, sum(p for d, p in subset))
    return best
```

작업 수 1~14개, 마감시간 범위를 무작위로 바꿔 가며 300번 대조했다.

```text nolines
mismatches: 0 / 300
```

(실제로 실행해서 확인한 결과다.) 구체적인 예시도 손으로 따라가 볼 만하다.

```python title="다섯 개 작업, 마감시간 최대 3"
example_jobs = [(2, 100), (1, 19), (2, 27), (1, 25), (3, 15)]
profit, count = max_profit_greedy_dsu(example_jobs, 3)
```

```text nolines
example profit: 142 jobs scheduled: 3
brute check: 142
```

이익 내림차순으로 훑으면 $(2,100) \to (2,27) \to (1,25) \to (1,19) \to (3,15)$ 순서다. $(2,100)$은 슬롯 2에, $(2,27)$은 슬롯 2가 이미 찼으니 `find(2)`가 슬롯 1을 돌려주어 거기에, $(1,25)$는 `find(1)`이 이제 0(자리 없음)을 돌려주니 포기, $(1,19)$도 포기, 마지막 $(3,15)$는 슬롯 3이 비어 있으니 배정된다. 합계 $100+27+15=142$ — 100짜리와 27짜리를 배정하는 대신 25짜리·19짜리를 버린 게 손해처럼 보이지만, **마감이 1인 두 작업(25, 19)은 어차피 슬롯 1 하나를 두고 경쟁할 수밖에 없어 그중 하나는 반드시 버려야 한다.** 이익이 큰 100과 27을 먼저 확보하는 게 전체 합을 최대화한다.

### 실행 시간: 지수 브루트포스 vs 그리디+DSU

작업 수를 10, 16, 20으로 늘리면서 브루트포스($O(2^N \cdot N)$)와 그리디+DSU($O(N \log N + N\alpha(N))$)를 같이 재 봤다.

::: perf $2^N$은 실측으로 봐야 실감 난다
```text nolines
n=10  greedy+dsu=0.000003s  brute=0.000980s
n=16  greedy+dsu=0.000006s  brute=0.080247s
n=20  greedy+dsu=0.000008s  brute=1.321130s
```

(Python 3.14.5 / Windows 기준 실측.) $N$이 10 → 16 → 20으로 2배가 채 안 되게 늘었는데 브루트포스는 0.001초 → 1.3초로 **1,300배 넘게 느려졌다** — $2^N$의 위력이다. $N=30$만 되어도 브루트포스는 시험 시간 안에 끝나지 않는다. 반면 그리디+DSU는 같은 구간에서 시간이 거의 그대로다. 작업 수를 아예 1,000 → 10,000 → 100,000으로 키워서 그리디+DSU만 따로 재 보면 이렇다.

```text nolines
n=1000    greedy+dsu only =0.000248s
n=10000   greedy+dsu only =0.002634s
n=100000  greedy+dsu only =0.030456s
```

$N$이 100배(1,000→100,000) 늘었는데 시간은 약 123배 늘었다 — $O(N \log N)$에 가까운 성장이다(정렬 비용이 지배적이고, `find` 자체는 경로 압축 덕분에 사실상 상수 시간이다). **브루트포스가 20개에서 이미 1.3초를 넘긴 문제를 그리디+DSU는 100,000개에서도 0.03초에 끝낸다.** 이게 "정당성을 증명한 그리디를 올바른 자료구조로 빠르게 만든다"는 이 절의 핵심 메시지다.
:::

::: warn 유니온 파인드의 방향을 헷갈리지 마라
슬롯을 union할 때 `parent[slot] = slot - 1`이지, 그 반대가 아니다. **"이 슬롯은 다 찼으니, 다음에 이 슬롯을 찾으면 그보다 하나 이른 자리를 대신 알려줘라"**는 방향이다. 반대로 `parent[slot - 1] = slot`으로 union하면 `find`가 점점 더 늦은 슬롯을 가리키게 되어 이미 찬 자리를 다시 배정하는 버그가 생긴다. 방향을 헷갈리면 대조 검증에서 반드시 걸린다 — 이런 실수야말로 브루트포스 대조가 잡아 주는 대표적인 버그다.
:::

## 두 문제가 공유하는 교훈

두 풀이를 나란히 놓고 보면 패턴이 보인다.

| | 다익스트라 + DP | 그리디 + 유니온 파인드 |
| --- | --- | --- |
| 무엇을 정하는가 | 다익스트라(최단 비용 순서로 확정) | 그리디(이익 큰 순서로 배정) |
| 무엇을 빠르게 하는가 | DP(상태에 자원 사용량을 얹어 구분) | 유니온 파인드(빈 슬롯 탐색을 상수 시간으로) |
| 상태/자료구조 확장 지점 | 노드를 (도시, k)로 확장 | 슬롯을 트리로 묶어 대표원소로 압축 |
| 브루트포스 검증 방법 | 모든 단순 경로 나열 | 모든 부분집합 전수 조사 |
| 브루트포스가 못 버티는 지점 | 그래프가 조밀해지면 경로 수가 폭발 | $N \geq 25$ 근처부터 $2^N$이 감당 안 됨 |

**복합 유형을 만나면 "이 문제에서 무엇을 최적화하려는가"와 "그 최적화를 느리게 만드는 병목이 무엇인가"를 따로 떼어서 봐라.** 최적화 목표는 보통 익숙한 알고리즘(최단 경로, 그리디, DP) 하나로 잡힌다. 병목은 대개 "같은 계산을 반복한다"거나 "탐색이 선형이다"인데, 그걸 없애는 도구가 힙, 유니온 파인드, 세그먼트 트리([7.25 세그먼트 트리](#/segment-tree)) 같은 자료구조다. 시험장에서 복합 유형을 풀 시간이 부족하다고 느껴진다면, [8.3 시간 초과를 피하는 관용구](#/tle)에서 다룬 것처럼 일단 브루트포스로 부분 점수를 챙기고 남는 시간에 최적화를 얹는 전략도 있다.

## 요약

- 복합 유형은 "무엇을 최적화하는가"를 정하는 알고리즘과, 그 최적화를 "충분히 빠르게" 만드는 알고리즘/자료구조가 따로 있다.
- 다익스트라에 자원 제약(할인권, 순간이동 등)이 붙으면 노드를 "(정점, 자원 사용량)"으로 확장한다. 상태 하나를 정점 하나처럼 취급하면 다익스트라의 뼈대는 그대로 쓸 수 있다.
- 상태를 확장하지 않고 정점 단위로만 방문 여부를 관리하면, 코드는 돌아가지만 조용히 틀린 답을 낸다 — 가장 위험한 종류의 버그다.
- 그리디의 정당성(교환 논증)과 그리디의 구현 속도(자료구조)는 별개 문제다. "가장 늦은 빈 슬롯을 찾는다"는 연산은 유니온 파인드로 경로 압축하면 사실상 상수 시간이 된다.
- 브루트포스 대조 검증은 복합 유형에서 특히 중요하다. 최적화된 코드가 낸 답이 이상해 보여도, 독립적인 방법(모든 경로 나열, 모든 부분집합 전수 조사)으로 짠 브루트포스와 여러 번 맞춰 보면 버그를 잡을 확률이 크게 올라간다.
- 브루트포스는 지수적으로 느려진다는 걸 실측으로 확인했다 — 작업 20개에서 이미 1.3초, 정점이 조금만 늘어도 단순 경로 수가 폭발한다. 최적화 풀이는 같은 구간에서 시간이 거의 그대로다.
