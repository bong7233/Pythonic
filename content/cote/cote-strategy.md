# 8.1 코딩테스트의 구조와 전략

::: lead
Part VII에서 자료구조와 알고리즘을 훑었다면 이제 남은 건 **실전에서 그걸 써먹는 절차**다. 코딩테스트는 알고리즘 지식 시험이 아니라 **제한 시간 안에 점수를 최대화하는 게임**이다. 같은 실력이라도 시간을 어떻게 쓰는지, 막힌 문제를 언제 놓는지, 제출 전에 뭘 확인하는지에 따라 통과 여부가 갈린다. 이 절은 문제를 푸는 법이 아니라 **시험 전체를 운영하는 법**을 다룬다. 여기서 세운 습관은 8.6~8.9의 모든 실전 풀이에서 그대로 쓰인다.
:::

## 코딩테스트라는 게임의 규칙

기술 코딩테스트는 대체로 비슷한 틀을 공유한다. 문제 4~6개, 시간 2~4시간, 온라인 채점 시스템(OJ)이 자동으로 정답을 판정하는 방식이다. 정확한 숫자는 회사·전형마다 다르지만, 이 틀 자체는 대부분의 채용 코테와 백준·프로그래머스 같은 연습 플랫폼에서 공통으로 관찰된다.

중요한 건 **채점 방식이 하나가 아니라는 점**이다. 크게 세 가지가 섞여서 나온다.

| 채점 방식 | 동작 | 전략적 의미 |
| --- | --- | --- |
| **전부 통과 (all-or-nothing)** | 모든 테스트케이스를 통과해야 점수 인정 | 한 케이스라도 틀리면 0점. 엣지케이스 방어가 곧 점수다 |
| **부분 점수 (partial scoring)** | 통과한 테스트케이스 비율만큼 점수 부여 | 완벽한 최적해가 아니어도 브루트포스로 일부 케이스만 맞혀도 점수가 쌓인다 |
| **서브태스크 (subtask)** | 문제가 난이도별 하위 문제로 쪼개져 있고, 하위 문제마다 배점이 있음 (예: $n \le 1000$ 서브태스크는 30점, $n \le 10^6$ 서브태스크는 70점) | 작은 $n$ 서브태스크는 브루트포스로도 만점을 받을 수 있다. 전체를 최적화하지 못해도 절반 이상을 챙길 수 있다 |

::: cote 부분 점수·서브태스크가 있다면 태도를 바꿔라
전부 통과 방식만 겪어 본 사람은 "최적해가 안 나오면 포기"하는 습관이 든다. 부분 점수나 서브태스크가 있는 시험에서는 이게 손해다. $n$ 이 작은 서브태스크만 통과해도 점수가 나오므로, **최적해가 안 떠올라도 일단 브루트포스를 제출해서 확보 가능한 점수를 챙긴 다음** 남은 시간에 최적화를 시도하는 게 기대값이 높다. 프로그래머스와 백준 일부 문제, 그리고 다수의 대회형 코테(삼성 SW 역량테스트 계열 등)가 이런 구조로 알려져 있다.
:::

## 시간 배분: 어려운 문제에 발목 잡히지 않는 법

시험 시간은 유한하고, 문제 난이도는 균등하지 않다. 가장 흔한 실패 패턴은 **첫 문제에서 막혀서 나머지 시간을 거기에 다 쓰는 것**이다. 문제 4개 중 1개를 완벽히 풀고 3개를 못 보는 것보다, 4개 중 3개를 어느 정도 건드리는 게 대부분 더 높은 점수를 준다 — 부분 점수가 있다면 확실히 그렇고, 없더라도 쉬운 문제 3개를 다 맞히는 편이 안전하다.

### 훑기 → 순서 정하기 → 타임박싱

1. **처음 5~10분은 문제를 다 읽는다.** 바로 풀기 시작하지 마라. 제약 조건($n$의 범위, 시간 제한)을 먼저 보고 난이도를 가늠한다. [8.4 문제 유형 분류](#/problem-signals)에서 다루듯, 제약 조건만 봐도 필요한 알고리즘의 복잡도 등급이 추려진다.
2. **쉬운 것부터 순서를 정한다.** 감으로 정하지 말고 기준을 세워라: 문제 길이(짧을수록 대개 쉬움), 제약 조건의 크기($n \le 100$ 같은 작은 수는 브루트포스로도 풀리는 경우가 많다), 이미 아는 패턴인지(익숙한 유형이면 우선).
3. **문제마다 시간 상한을 미리 정한다.** 4문제/3시간이면 문제당 45분이 기본값이고, 쉬운 문제에서 아낀 시간을 어려운 문제로 옮긴다. **상한을 넘기면 일단 넘어간다.** 다시 돌아올 수 있다는 걸 기억하되, 그 순간엔 놓는다.

::: warn "조금만 더 하면 될 것 같다"는 함정
막힌 문제일수록 "5분만 더"가 반복된다. 실제로는 5분이 30분이 된다. **타이머를 눈에 보이게 켜 놓고, 상한을 넘기면 기계적으로 다음 문제로 넘어가라.** 감정이 아니라 규칙으로 끊어야 한다. 넘어간 문제는 다른 문제를 다 처리한 뒤 남은 시간에 돌아온다 — 이때는 처음 봤을 때 놓쳤던 접근이 새로 보이는 경우가 많다.
:::

### 막혔을 때의 판단 트리

막혔다는 건 두 종류로 나뉜다. **접근 자체를 모르는 것**과 **접근은 있는데 구현이 안 되는 것**은 다른 처방이 필요하다.

```text nolines
막혔다
 |
 |-- 접근이 아예 안 보인다
 |      -> 제약 조건을 다시 본다 (8.4). 문제 유형의 신호를 다시 읽는다.
 |      -> 그래도 안 보이면: 브루트포스라도 짜서 부분 점수를 확보하고 넘어간다.
 |
 `-- 접근은 있는데 구현에서 막혔다
        -> 손으로 작은 예시를 하나 그려서 단계별로 따라간다.
        -> 그래도 안 풀리면: 지금 코드를 버리고 브루트포스부터 다시 짜서
           "정답이 뭐가 나와야 하는지"부터 확정한다 (아래 절 참고).
```

이 판단 자체가 시간을 아낀다. "접근을 모르는데 코드를 계속 고치는 것"과 "구현 버그인데 접근을 계속 바꾸는 것"은 둘 다 시간 낭비의 대표적인 형태다.

## 제출 전 체크리스트: 엣지케이스 네 가지

문제를 풀었다고 끝이 아니다. 정답 로직이 맞아도 **경계 조건을 안 챙겨서 틀리는 경우가 실전에서 압도적으로 많다.** 제출 전에 다음 네 가지를 순서대로 확인하는 습관을 들여라.

| 확인 항목 | 왜 자주 놓치는가 | 확인 방법 |
| --- | --- | --- |
| **빈 입력 / 최소 크기** | 배열이 비어 있거나 원소가 1개일 때 인덱싱이나 반복문 경계가 깨진다 | `n = 0`, `n = 1` 을 손으로 넣어 실행해 본다 |
| **최댓값 / 최솟값** | 제약 조건의 상한값에서 오버플로나 시간 초과가 난다 | 문제의 $n_{\max}$, 값의 $\max$ 를 실제로 넣어 시간을 재 본다 |
| **음수** | "당연히 양수겠지"라는 가정이 코드에 숨어 있다 (예: `abs` 빠뜨림, 인덱스로 음수 값을 그대로 씀) | 문제 제약에 음수가 허용되는지 다시 읽고, 허용되면 음수 케이스를 직접 넣는다 |
| **중복** | "원소가 다 다르다"고 가정한 로직(예: `set` 으로 개수를 세는 코드)이 중복이 있으면 깨진다 | 같은 값을 여러 개 넣어 결과가 바뀌는지 확인한다 |

::: cote 이 넷을 코드로 만들어 두면 매번 빠르다
매 문제마다 네 가지를 손으로 떠올리기보다, **입력을 만드는 헬퍼 함수 하나**를 템플릿처럼 갖고 있으면 확인이 빨라진다.

```python title="edge_cases.py — 흔한 엣지케이스 생성기"
def edge_case_inputs(low: int = -(10**9), high: int = 10**9):
    """문제에 넣어볼 전형적인 엣지케이스 입력 목록."""
    return [
        [],                      # 빈 입력
        [0],                     # 원소 1개
        [high, high, high],      # 최댓값, 중복
        [low, low, low],         # 최솟값(음수), 중복
        [low, 0, high],          # 음수/0/최댓값 혼합
    ]
```

이 함수를 실제 풀이 함수에 그대로 넣어서 예외 없이 도는지, 값이 말이 되는지부터 본다. 8.10 [실전 템플릿](#/templates)에 이런 확인용 코드도 함께 정리한다.
:::

## 브루트포스로 먼저 정답을 확정한다

[7.11 트라이](#/trie)에서 트라이 구현을 브루트포스(`str.startswith` 전수 조사)와 대조해서 검증한 것을 기억할 것이다. 이건 그 절만의 특수한 습관이 아니라 **코딩테스트 전체에 적용해야 하는 원칙**이다.

> **최적화된 풀이를 처음부터 한 번에 맞게 짜려고 하지 마라. 먼저 확실히 맞는 느린 풀이를 짜고, 그걸 기준으로 빠른 풀이를 검증하라.**

이유는 단순하다. 최적화된 알고리즘(투 포인터, 이분 탐색, DP)은 로직이 압축돼 있어서 버그가 나도 겉으로 티가 안 난다 — 그냥 조용히 틀린 답을 낸다. 반면 브루트포스는 문제 정의를 그대로 코드로 옮긴 것이라 **틀릴 여지가 거의 없다.** 브루트포스가 먼저 있으면 최적화 버전의 출력을 대조할 기준점이 생긴다.

### 예시: 목표치 이하인 두 수의 합, 몇 쌍인가

정수 배열이 주어지고, 목표치 `target` 이하가 되는 두 원소의 합의 쌍이 몇 개인지 세는 문제로 이 과정을 그대로 밟아 보자. (인덱스 `i < j` 인 쌍만 센다.)

**1단계: 브루트포스로 정답부터 확보한다.** 모든 쌍을 다 본다. $O(n^2)$ 이지만 로직이 문제 정의와 한 줄씩 대응하므로 틀릴 일이 거의 없다.

```python title="brute_pairs.py — 모든 쌍을 전수 조사"
def brute_count_pairs(nums: list[int], target: int) -> int:
    n = len(nums)
    cnt = 0
    for i in range(n):
        for j in range(i + 1, n):
            if nums[i] + nums[j] <= target:
                cnt += 1
    return cnt
```

**2단계: 최적화한다.** 정렬한 뒤 [7.3 투 포인터](#/two-pointers)를 쓴다. 양 끝에서 포인터를 좁혀 가며, `xs[lo] + xs[hi] <= target` 이면 `xs[lo]` 와 `xs[lo+1..hi]` 사이의 모든 조합이 조건을 만족한다는 걸 이용해 한 번에 `hi - lo` 개를 센다.

```python title="fast_pairs.py — 정렬 + 투 포인터"
def fast_count_pairs(nums: list[int], target: int) -> int:
    xs = sorted(nums)
    n = len(xs)
    cnt = 0
    lo, hi = 0, n - 1
    while lo < hi:
        if xs[lo] + xs[hi] <= target:
            cnt += hi - lo     # xs[lo]와 xs[lo+1..hi] 전부 만족
            lo += 1
        else:
            hi -= 1
    return cnt
```

**3단계: 대조 검증한다.** 무작위 배열 2,000개에 대해 두 함수의 결과를 비교했다.

```python title="verify.py — 브루트포스와 대조"
import random

random.seed(0)
mismatches = 0
TRIALS = 2000
for _ in range(TRIALS):
    n = random.randint(0, 12)
    nums = [random.randint(-50, 50) for _ in range(n)]
    target = random.randint(-100, 100)
    if brute_count_pairs(nums, target) != fast_count_pairs(nums, target):
        mismatches += 1
print(f"random mismatches: {mismatches} / {TRIALS}")
```

```text nolines
random mismatches: 0 / 2000
```

**4단계: 위에서 정한 엣지케이스 체크리스트를 그대로 돌린다.**

```python title="edge_check.py"
edge_cases = [
    ([], 10),                       # 빈 입력
    ([5], 10),                      # 원소 1개
    ([10**9, -(10**9), 0], 0),      # 최댓값·최솟값 혼합
    ([-5, -5, -5, -5], -10),        # 전부 같은 음수(중복)
    ([0, 0, 0], 0),                 # 전부 0
    ([2**31, 2**31, 2**31], 2**32), # 아주 큰 값
]
for nums, target in edge_cases:
    a, b = brute_count_pairs(nums, target), fast_count_pairs(nums, target)
    print(f"{'OK' if a == b else 'FAIL'}: nums={nums!r} target={target} brute={a} fast={b}")
```

```text nolines
OK: nums=[] target=10 brute=0 fast=0
OK: nums=[5] target=10 brute=0 fast=0
OK: nums=[1000000000, -1000000000, 0] target=0 brute=2 fast=2
OK: nums=[-5, -5, -5, -5] target=-10 brute=6 fast=6
OK: nums=[0, 0, 0] target=0 brute=3 fast=3
OK: nums=[2147483648, 2147483648, 2147483648] target=4294967296 brute=3 fast=3
```

(실제로 실행해서 확인한 결과다. 파이썬의 정수는 [1.2 숫자](#/numbers)에서 다뤘듯 임의 정밀도라 $2^{31}$ 같은 값에도 오버플로가 없다 — C++이었다면 이 엣지케이스에서 별도로 `long long` 을 챙겨야 했을 자리다.)

**5단계: 시간 제한을 실측으로 확인한다.** "언제부터 브루트포스가 위험한가"를 감으로 말하지 않는다.

```python title="bench_pairs.py"
import time

for n in (500, 2000, 8000):
    nums = [random.randint(-10**6, 10**6) for _ in range(n)]
    t0 = time.perf_counter()
    brute_count_pairs(nums, 0)
    t1 = time.perf_counter()
    fast_count_pairs(nums, 0)
    t2 = time.perf_counter()
    print(f"n={n:>5}: brute={t1 - t0:.4f}s  fast={t2 - t1:.4f}s")
```

```text nolines
n=  500: brute=0.0036s  fast=0.0001s
n= 2000: brute=0.0561s  fast=0.0009s
n= 8000: brute=0.8939s  fast=0.0009s
```

(Python 3.14.5 / Windows 기준 실측. 절대 시간은 기기마다 다르지만 $O(n^2)$ 대 $O(n \log n)$ 의 격차 경향은 어디서나 같다.) $n$ 이 500에서 8,000으로 16배 늘었는데 브루트포스 시간은 약 250배 늘었다 — $O(n^2)$ 이니 $16^2 = 256$ 배와 정확히 맞아떨어진다. 반면 투 포인터는 거의 그대로다. 대회 시간 제한이 보통 1~2초임을 감안하면, $n$ 이 대략 수만을 넘어가는 순간부터 $O(n^2)$ 은 시간 초과가 확실시된다. 이 감각은 [7.1 복잡도](#/complexity)의 표를 실제 숫자로 확인해 주는 것이기도 하다.

::: cote 대조 검증은 실전에서도 시간이 남는다
"검증 코드까지 짤 시간이 있나?"라는 의문이 들 수 있다. 하지만 브루트포스는 보통 3~5줄이면 끝나고, 대조 루프는 반복문 하나다. 반면 최적화 코드를 감으로 제출했다가 틀려서 원인을 못 찾고 시간을 다 쓰는 것보다는 훨씬 싸게 먹힌다. **디버깅 시간을 검증 시간으로 미리 당겨 쓰는 셈이다.** 특히 DP나 그래프처럼 반례를 손으로 찾기 어려운 유형([7.20](#/dp-basics), [7.15](#/shortest-path))에서 이 습관의 효과가 가장 크다.
:::

## 막혔을 때의 디버깅: 8.2/8.3과 이어지는 지점

브루트포스와 대조했는데도 결과가 다르다면, 다음 순서로 좁혀 간다.

1. **작은 입력으로 재현한다.** 실패한 무작위 케이스를 그대로 쓰지 말고, 그중 최소한으로 줄인 예시를 만든다. 원소 2~3개짜리로 줄여도 재현되는지 확인하라.
2. **중간값을 찍어 본다.** 최적화 코드에 `print` 를 임시로 넣어 각 단계의 상태를 본다. 정렬 직후 배열, 포인터 위치, 누적값처럼 핵심 불변식이 깨지는 지점을 좁혀 간다.
3. **입출력 형식 자체를 의심한다.** 로직은 맞는데 틀렸다고 나오면, 실제로는 [8.2 입출력 최적화](#/io-optimize)에서 다루는 **입력 파싱 실수**(공백 처리, 줄 끝 개행, 여러 줄에 걸친 입력)인 경우가 흔하다. 로컬에서는 맞는데 채점 시스템에서만 틀리면 이 가능성부터 봐라.
4. **시간 초과인지 오답인지 구분한다.** 채점 결과가 "틀렸습니다"가 아니라 "시간 초과"라면 로직이 아니라 복잡도 문제다. [8.3 시간 초과를 피하는 관용구](#/tle)로 넘어가라. 로직 버그를 아무리 찾아도 시간 초과는 안 고쳐진다 — 이 구분을 못 하면 엉뚱한 곳에서 시간을 태운다.

::: warn "로컬에서는 통과하는데 채점에서는 틀린다"는 신호
이 증상은 거의 항상 다음 중 하나다. (1) 로컬 테스트 케이스가 채점 시스템의 엣지케이스를 커버하지 못했다 — 위 체크리스트를 다시 돌려라. (2) 입출력 형식이 미묘하게 다르다 — 여러 줄 입력, 공백 구분, 개행 문자. (3) 로컬은 통과 못 할 만큼 느린 하드웨어가 아니라서 시간 초과를 못 알아챘다 — $n_{\max}$ 로 직접 실행 시간을 재 봐라.
:::

## 요약

- 코테는 문제 4~6개/시간 2~4시간대의 자동 채점 시험이며, 채점은 전부 통과·부분 점수·서브태스크 세 방식이 섞여서 나온다. 부분 점수·서브태스크가 있으면 완벽한 최적해가 아니어도 점수를 챙길 수 있다.
- 시간은 문제별 상한을 미리 정해 쓴다. 쉬운 문제부터 처리하고, 상한을 넘기면 기계적으로 다음 문제로 넘어간다.
- 막혔을 때는 "접근을 모르는 것"과 "구현이 안 되는 것"을 구분해서 다른 처방을 쓴다.
- 제출 전 체크리스트는 네 가지: 빈 입력/최소 크기, 최댓값/최솟값, 음수, 중복. 이걸 놓쳐서 틀리는 경우가 실전에서 가장 흔하다.
- 최적화된 풀이를 한 번에 맞게 짜려 하지 말고, **먼저 브루트포스로 정답을 확정한 뒤 그것과 대조해서 최적화 버전을 검증한다.** [7.11 트라이](#/trie)에서 쓴 방법을 모든 문제에 적용하라.
- 로컬은 통과하는데 채점에서 틀리면 입출력 형식([8.2](#/io-optimize))을, 오답이 아니라 시간 초과면 복잡도([8.3](#/tle))를 먼저 의심하라.
