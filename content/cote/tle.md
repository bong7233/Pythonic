# 8.3 시간 초과를 피하는 관용구

::: lead
"이 문제 시간복잡도 $O(n \log n)$ 이니까 통과하겠지" — 코딩테스트에서 가장 흔한 오판이다. 빅오는 **연산 횟수가 입력 크기에 따라 어떻게 늘어나는가**만 말해 줄 뿐, 연산 하나가 몇 나노초 걸리는지는 말해 주지 않는다. C++에서 1억 번 도는 루프는 1초가 안 걸리지만, 파이썬으로 똑같이 짜면 다르다. 이 절은 그 차이를 실제로 재서 숫자로 보여주고, 파이썬 안에서 그 상수를 줄이는 관용구를 정리한다. [7.1 복잡도](#/complexity)에서 배운 "상수의 무게"를 코딩테스트라는 구체적인 압박 속에서 다시 본다.
:::

## 같은 알고리즘, 두 언어 — 실측

이론이 아니라 숫자로 시작하자. 다음 코드는 $10{,}000 \times 10{,}000$ 이중 루프를 돌며 `(i*j) % 7` 을 누적한다. $10^8$ 번의 연산이고, 알고리즘은 당연히 $O(n^2)$ 이다. 이걸 파이썬과 C++로 **한 글자도 다르지 않은 로직**으로 짜서 같은 기기에서 돌렸다.

```python title="bench_nested.py"
import time

N = 10000
t0 = time.perf_counter()
total = 0
for i in range(N):
    for j in range(N):
        total += (i * j) % 7
t1 = time.perf_counter()
print(f"total={total} time={t1 - t0:.4f}s")
```

```cpp title="bench_nested.cpp — g++ -O2 로 컴파일"
#include <cstdio>
#include <chrono>

int main() {
    const long long N = 10000;
    auto t0 = std::chrono::steady_clock::now();
    long long total = 0;
    for (long long i = 0; i < N; i++)
        for (long long j = 0; j < N; j++)
            total += (i * j) % 7;
    auto t1 = std::chrono::steady_clock::now();
    double sec = std::chrono::duration<double>(t1 - t0).count();
    std::printf("total=%lld time=%.4fs\n", total, sec);
}
```

3회씩 돌린 결과다. `total` 값이 두 언어에서 완전히 같다는 것도 확인했다 — `257117141`.

```text nolines
Python  : 6.2643s, 5.6025s, 5.8627s   (평균 약 5.91s)
C++ -O2 : 0.0548s, 0.0607s, 0.0547s   (평균 약 0.057s)
```

::: perf 같은 로직, 같은 결과, 약 104배 차이
(Python 3.14.5 / g++ 13.3 -O2, 동일 기기 실측. 절대 시간은 기기마다 다르지만 **파이썬 순수 루프가 컴파일된 C++ 대비 수십~수백 배 느리다는 자릿수**는 재현된다. 파이썬 쪽 `total=257117141` 과 5초대 시간은 다른 기기에서 재실행해도 그대로 재현됨을 확인했다 — C++ 쪽을 직접 재려면 `g++`/`clang++`/`cl.exe` 같은 컴파일러가 설치돼 있어야 한다. 컴파일러가 없는 환경이라면 파이썬 스크립트만 돌려서 위 표의 왼쪽 열이 자신의 기기에서도 비슷하게 나오는지부터 확인하고, 오른쪽 C++ 수치는 "컴파일된 언어는 이 정도로 빠르다"는 참고치로 받아들여라.)

평균 5.91초 대 0.057초, 약 **104배**다. 이게 왜 벌어지는가. 파이썬의 `for i in range(N)` 한 바퀴는 바이트코드 인터프리터가 매번 `i`, `j` 를 파이썬 `int` 객체로 박싱하고, `*`·`%` 연산마다 해당 타입의 `__mul__`/`__mod__` 를 디스패치하고, 그 결과로 새 정수 객체를 만드는 과정을 거친다. C++는 컴파일 시점에 이 모든 게 기계어 덧셈·곱셈 명령 몇 개로 확정된다. 이 차이는 알고리즘이 아니라 **실행 모델**의 차이이므로 파이썬에서 아무리 관용구를 다듬어도 없앨 수 없다. 다만 뒤에서 볼 것처럼 함수 호출·재귀·문자열 연산 쪽의 상수는 상당 부분 줄일 수는 있다.

::: cote 이게 실전에서 뜻하는 것
백준 같은 저지는 이 격차를 알고 파이썬에 C++보다 2~3배 넉넉한 시간 제한을 준다. 그런데 104배 차이 앞에서 2~3배 여유는 턱없이 부족하다. **제약 조건에 $n \le 10^4$ 같은 숫자가 있고 알고리즘이 $O(n^2)$ 이면, "이 정도면 $10^8$ 번이니 되겠지" 라는 감각 자체를 의심하라.** 이 절 실측 그대로 파이썬에서 6초 가까이 걸리는 연산량이다. [8.4 문제 유형 분류](#/problem-signals)에서 제약 조건만 보고 시간 제한을 가늠하는 법을 더 다룬다.
:::

## 반복문 안 함수 호출은 공짜가 아니다

파이썬 함수 호출은 새 스택 프레임을 만들고, 인자를 바인딩하고, 바이트코드 실행이 끝나면 프레임을 정리하는 일련의 작업이다. 이 비용은 함수 안에서 하는 일이 아무리 간단해도 고정으로 붙는다.

```python title="bench_call.py"
import timeit

N = 5_000_000

def add_one(x):
    return x + 1

def with_call():
    total = 0
    for i in range(N):
        total = add_one(total)
    return total

def without_call():
    total = 0
    for i in range(N):
        total = total + 1
    return total

t_call = timeit.timeit(with_call, number=3)
t_inline = timeit.timeit(without_call, number=3)
print(f"함수 호출 있음: {t_call:.4f}s")
print(f"함수 호출 없음: {t_inline:.4f}s")
print(f"배수: {t_call/t_inline:.2f}x")
print(f"호출당 오버헤드: {(t_call - t_inline)/(N*3)*1e9:.1f} ns")
```

```text nolines
함수 호출 있음: 0.3883s (3회 합)
함수 호출 없음: 0.1989s (3회 합)
배수: 1.95x
호출당 오버헤드: 12.6 ns
```

::: deep 호출 한 번에 12.6 나노초 — 이게 왜 무섭나
(Python 3.14.5 / Windows 기준 실측.) 호출 한 번의 오버헤드는 12.6나노초로 아주 작다. 문제는 **반복 횟수가 곱해진다**는 것이다. 코딩테스트에서 $n = 10^6$ 짜리 배열을 순회하며 매 원소마다 헬퍼 함수를 하나씩 부르면, 그것만으로 $10^6 \times 12.6\text{ns} \approx 12.6\text{ms}$ 가 붙는다. 헬퍼 함수가 재귀 안에 있고 재귀 깊이가 크면 이 오버헤드에 스택 프레임 할당 비용까지 겹친다. `add_one` 처럼 몸통이 한 줄인 함수일수록 오버헤드 비중이 커진다 — 실제로 하는 일보다 호출하는 절차 자체가 더 비싸지는 지점이다.

실전에서 자주 나오는 대응은 다음과 같다.

- 아주 단순한 연산(제곱, 절댓값, 부호 판정 등)은 함수로 감싸지 말고 그 자리에서 인라인으로 쓴다.
- 표준 라이브러리의 C로 구현된 함수(`sum`, `max`, `min`, `sorted`)는 파이썬 함수 호출 오버헤드가 없다. 직접 짠 순수 파이썬 루프보다 대개 빠르다. [5.3 파이썬 레벨 최적화](#/py-optimize)에서 이 차이를 더 다룬다.
- 재귀로 짠 DP를 반복문 기반 타뷸레이션으로 바꾸면 호출 자체를 없앨 수 있다. 바로 다음 절에서 다룬다.
:::

::: note 전역 변수 조회는 이제 옛날만큼 크지 않다
"루프 안에서 전역 변수를 매번 찾지 말고 지역 변수로 캐싱하라"는 조언을 예전 파이썬 최적화 글에서 봤을 것이다. 5,000,000번 순회하는 리스트 합산으로 직접 재 보면 다음과 같다.

```text nolines
전역 변수 순회: 0.7592s, 0.8088s, 0.7635s   (10회 합, 3회 시행)
지역 변수 순회: 0.8102s, 0.7909s, 0.7848s   (10회 합, 3회 시행)
배수: 0.94x, 1.02x, 0.97x
```

(Python 3.14.5 실측, 3회 반복해도 배수가 계속 1.0 근처다.) **차이가 거의 없다.** 3.11부터 들어온 특수화 적응형 인터프리터(specializing adaptive interpreter)가 `LOAD_GLOBAL` 명령어 자체를 인라인 캐시로 최적화해서, 예전만큼 전역·지역 조회 사이의 격차가 크지 않다. 이 최적화는 [3.7 바이트코드와 dis](#/bytecode)에서 더 다룬다. 오래된 최적화 팁을 검증 없이 믿지 마라 — 이 책 전체가 강조하는 원칙이 여기서도 그대로 적용된다.
:::

## 재귀를 반복으로 — 함수 호출을 아예 없앤다

재귀는 매 호출이 스택 프레임 하나를 새로 만든다. 반복문은 프레임 하나로 끝낸다. 같은 $O(n)$ 알고리즘을 두 방식으로 짜서 재 보면 차이가 뚜렷하다.

```python title="bench_recursion.py"
import sys, timeit

sys.setrecursionlimit(2_000_000)
N = 100_000

def sum_recursive(n, acc=0):
    if n == 0:
        return acc
    return sum_recursive(n - 1, acc + n)

def sum_iterative(n):
    acc = 0
    while n > 0:
        acc += n
        n -= 1
    return acc

assert sum_recursive(1000) == sum_iterative(1000)

t_rec = timeit.timeit(lambda: sum_recursive(N), number=5)
t_iter = timeit.timeit(lambda: sum_iterative(N), number=5)
print(f"recursive: {t_rec:.4f}s")
print(f"iterative: {t_iter:.4f}s")
print(f"ratio: {t_rec/t_iter:.2f}x")
```

```text nolines
recursive: 0.0507s
iterative: 0.0133s
ratio: 3.82x
```

::: danger 재귀 깊이 늘리기는 시한폭탄이다
`sys.setrecursionlimit(2_000_000)` 로 파이썬의 재귀 한도는 늘렸지만, 이건 **파이썬 인터프리터가 허용하는 논리적 한도**일 뿐이다. 전통적으로 실제 재귀는 OS가 프로세스에 준 **C 스택**을 그대로 썼고, 이 한도를 아무리 늘려도 C 스택 자체가 넘치면 파이썬 예외가 아니라 **세그멘테이션 폴트로 프로그램이 그냥 죽을 수 있었다.** 코딩테스트 채점 서버에서는 이게 `RecursionError` 도 아니고 원인 불명의 런타임 에러로 찍혀서 디버깅이 특히 괴롭다.

```python
import sys
sys.setrecursionlimit(1_000_000)

def deep(n):
    if n == 0:
        return 0
    return 1 + deep(n - 1)

deep(500_000)
```

(Python 3.14.5 / Windows 실측: 위 코드는 크래시도 `RecursionError` 도 없이 `500000` 을 반환하고 정상 종료한다. 한도를 2천만, 깊이를 1억까지 올려도 마찬가지로 정상 종료됐다. CPython 3.11부터 프레임 스택 구현이 바뀌면서 — 프레임을 매번 별도의 C 스택 위 구조체로 할당하던 방식에서, 파이썬 프레임들을 하나의 연속된 메모리 블록에 쌓는 방식으로 바뀌었다 — 순수 파이썬 함수만 오가는 재귀는 예전만큼 C 스택을 빠르게 소모하지 않는다. 그래서 이 예제만 그대로 돌려서는 "위험"을 체감하기 어렵다.)

그렇다고 이 위험이 사라진 것은 아니다. 다음 조건에서는 여전히 C 스택이 관여하고, 세그멘테이션 폴트 가능성이 남는다.

- 재귀 함수가 **C로 구현된 콜백**(예: `sorted(key=...)` 안에서 재귀 호출, 정규식 엔진, C 확장이 부르는 파이썬 콜백)을 통해 서로를 호출하는 경우 — 이런 경로는 여전히 C 스택 프레임을 하나씩 쌓는다.
- **PyPy나 다른 파이썬 구현**, 혹은 더 오래된 CPython(3.10 이하)에서 실행하는 경우 — 프레임 스택 구현이 다르거나 예전 방식 그대로다.
- 스레드의 **C 스택 크기가 기본값보다 작게 설정된 환경**(임베디드, 일부 채점 서버, `threading.stack_size()` 로 줄인 서브스레드) — 같은 깊이에서도 더 일찍 넘칠 수 있다.

즉 "환경에 따라 다르다"는 헤지는 여전히 유효하지만, 그 방향이 이 책을 쓸 당시 예상과는 반대로 나타났다 — **최신 CPython 단독으로는 실제로 잘 안 죽는다.** 그래도 재귀 깊이가 수만~수십만을 넘는 코드를 채점 서버에 제출하는 것은 여전히 나쁜 습관이다. 채점 서버의 파이썬 버전·스택 설정을 알 수 없고, 위 조건 중 하나에 걸리면 그대로 원인 불명의 크래시로 이어지기 때문이다. 깊이가 수만을 넘는 재귀는 **애초에 반복문으로 바꾸는 게 맞다.** DFS·트리 순회처럼 재귀가 자연스러운 문제도 입력 크기가 크면 명시적 스택을 쓰는 반복 버전으로 바꿔라. [7.13 그래프 표현과 순회](#/graph)의 "재귀 깊이 문제"에서 이 변환을 자세히 다룬다.
:::

::: perf 왜 반복문이 3.8배 빠른가
(Python 3.14.5 실측, $n=100{,}000$ 기준.) 앞 절의 함수 호출 오버헤드가 그대로 원인이다. `sum_recursive` 는 $n$ 번 함수를 호출하고 그때마다 새 프레임을 할당·해제한다. `sum_iterative` 는 프레임 하나로 `while` 루프를 돈다. 재귀 한 번의 비용은 단순 함수 호출보다 크다 — 인자 두 개(`n`, `acc`)를 매번 바인딩하고, 반환값이 호출 스택을 타고 올라와야 하기 때문이다.

같은 문제를 [7.20 동적 계획법 기초](#/dp-basics)에서 메모이제이션(재귀 + 캐시)과 타뷸레이션(반복 + 배열)으로 비교할 때도 이 격차가 그대로 나타난다. **점화식을 세우는 사고 과정은 메모이제이션이 더 자연스럽지만, 제출할 코드는 타뷸레이션으로 바꾸면 상수를 줄일 수 있다.** 단, 점화식이 복잡한 상태 전이(비트마스크 DP, 트리 DP)를 가지면 타뷸레이션으로 옮기는 비용이 더 클 수도 있으니 무조건은 아니다.
:::

## 문자열·리스트 관용구 — 요약과 새로 밝혀진 함정

[1.1 객체·이름·참조](#/objects-names)와 [7.2 파이썬 자료구조의 실제 비용](#/py-ds-cost)에서 이미 다룬 핵심을 시험 직전 체크리스트로 압축하면 이렇다.

| 상황 | 느린 관용구 | 빠른 관용구 |
| --- | --- | --- |
| 문자열 누적 | `s += w` (반복) | `"".join(parts)` |
| 2차원 배열 초기화 | `[[0]*m]*n` | `[[0]*m for _ in range(n)]` |
| 리스트 앞에서 꺼내기 | `lst.pop(0)` | `collections.deque` + `popleft()` |
| 원소 존재 확인 | `x in list_obj` | `x in set_obj` / `x in dict_obj` |
| 리스트 만들기 | `for` + `append` | 리스트 컴프리헨션 |

이 표는 이미 앞 절들에서 실측을 마쳤다. 이 절에서는 문자열 누적 하나를 더 깊이 파 본다 — **함수 안에서 짰는지, 최상위 스코프에서 짰는지에 따라 결과가 완전히 달라지기 때문**이다.

```python title="함수 스코프에서 += vs join"
def bench(N):
    words = [str(i) for i in range(N)]
    s = ""
    for w in words:
        s += w
    return s
```

```text nolines
n=  50000: += 0.0020s  join 0.0002s  배수=13.1x
n= 100000: += 0.0044s  join 0.0003s  배수=13.6x
n= 200000: += 0.0109s  join 0.0006s  배수=18.3x
```

함수 안에서는 `n` 이 4배로 늘어도 배수가 14~21배 사이에서 크게 안 벌어진다. 그런데 **완전히 같은 코드를 함수 없이 모듈 최상위에서** 실행하면 얘기가 달라진다.

```python title="최상위 스코프에서 += vs join — 함수만 뺐다"
words = [str(i) for i in range(N)]
s = ""
for w in words:
    s += w
```

```text nolines
n=  50000: += 0.0466s  join 0.0001s  배수=350.0x
n= 100000: += 0.4011s  join 0.0003s  배수=1213.7x
n= 200000: += 2.6264s  join 0.0008s  배수=3331.7x
```

::: deep 왜 함수 안이냐 밖이냐로 결과가 이렇게 갈리는가
(Python 3.14.5 / Windows 기준 실측. 두 실험 모두 각 $n$ 에서 `assert`로 `+=` 결과와 `join` 결과가 정확히 같음을 확인했다. **절대 배수는 이 절에 실린 값이 아니라 아래 자릿수와 "$n$ 이 커질수록 급격히 벌어진다"는 경향만 믿어라** — 같은 코드를 다른 기기·다른 파이썬 빌드에서 3회 재실행해 보면 자릿수는 그대로지만 배수 자체는 최대 20~30% 정도 흔들렸다.)

CPython은 `s += w` 처럼 문자열에 다른 문자열을 이어 붙이는 패턴을 특별 취급한다. `s` 를 가리키는 참조가 **딱 하나뿐**이면(참조 카운트 1) 새 문자열 객체를 통째로 만드는 대신 **기존 버퍼를 제자리에서 늘려서** 재사용한다(`PyUnicode_Append` 의 최적화 경로). 이 조건이 들어맞으면 `+=` 반복이 이론적인 $O(n^2)$ 보다 훨씬 나아진다.

문제는 이 참조 카운트 조건이 **함수의 지역 변수(`STORE_FAST`)에서는 거의 항상 성립**하지만, **모듈 최상위의 전역 이름(`STORE_NAME`)에서는 깨지기 쉽다**는 것이다. 전역 이름 공간은 딕셔너리이고, 그 딕셔너리를 거쳐 가는 과정에서 참조가 일시적으로 늘어나 "참조 카운트 1" 조건을 놓치는 경우가 많다. 그 결과 최적화 경로를 못 타고 **매번 새 문자열을 통째로 복사하는 진짜 $O(n^2)$** 로 떨어진다. 실측 배수가 $n$ 이 늘수록 350배 → 1214배 → 3332배로 거의 제곱에 비례해서 커지는 것이 그 증거다.

**이 최적화에 의존하지 마라.** CPython의 구현 세부사항이지 언어 명세가 아니고, 이 절에서 봤듯 스코프 하나 차이로 두 자릿수가 갈린다. 실전에서 얻어야 할 결론은 단순하다.

1. 코딩테스트 코드는 어차피 대부분 `def solve(): ...` 형태의 함수 안에서 돌아간다. 무심코 짜도 최상위보다 안전한 편이다.
2. 그래도 문자열을 반복해서 이어 붙일 일이 있으면 **처음부터 `"".join()` 을 써라.** 최적화가 타든 안 타든 무조건 안전하다.
:::

## 리스트 컴프리헨션이 append 루프보다 빠른 이유

```python title="bench_comprehension2.py"
def attr_lookup():
    out = []
    for i in range(N):
        out.append(i * i)
    return out

def comprehension():
    return [i * i for i in range(N)]
```

$N=2{,}000{,}000$, 5회 반복 기준 실측이다.

```text nolines
attr_lookup   : 0.3597s (5회 합)
cached_method : 0.3630s (5회 합)
comprehension : 0.3427s (5회 합)
attr_lookup / comprehension = 1.05x
```

차이가 5% 남짓으로 크진 않지만 방향은 분명하다. `dis` 로 두 함수를 까 보면 이유가 보인다.

```text nolines
append 루프의 핵심 한 줄:
    LOAD_FAST_BORROW  out
    LOAD_ATTR         append + NULL|self
    ... (i*i 계산) ...
    CALL              1
    POP_TOP

컴프리헨션의 핵심 한 줄:
    ... (i*i 계산) ...
    LIST_APPEND       2
```

`out.append(x)` 는 매 반복마다 **속성 조회(`LOAD_ATTR`) + 함수 호출(`CALL`) + 반환값 버리기(`POP_TOP`)** 세 단계를 거친다. 리스트 컴프리헨션은 컴파일러가 "이건 리스트에 계속 추가하는 패턴이다"를 알고 있어서 `LIST_APPEND` 라는 **전용 바이트코드 명령 하나**로 끝낸다. 함수 호출 자체가 사라지는 것이다. `append` 를 지역 변수(`append = out.append`)로 캐싱해도 여전히 `CALL` 이 남아 있어 컴프리헨션만큼 빨라지지 않는다(위 실측에서 `cached_method` 가 오히려 `attr_lookup` 보다 살짝 느리게 나온 것도 이 잡음 범위 안의 차이일 뿐, 캐싱이 `LIST_APPEND` 를 대체하지는 못한다는 뜻이다).

::: tip 언제 컴프리헨션을 쓰고 언제 안 쓰나
"리스트를 새로 만드는 게 목적"이면 컴프리헨션을 써라. 5%든 그 이상이든 공짜로 얻는 이득이고, 코드도 더 짧다. 반대로 루프 안에서 **조건에 따라 여러 자료구조에 나눠 담거나, 부수 효과(로그 출력, 카운터 갱신)가 있으면** 억지로 컴프리헨션에 우겨 넣지 마라. [1.9 컴프리헨션과 제너레이터 표현식](#/comprehensions)에서 "언제 쓰고 언제 쓰지 말아야 하는가"를 더 다룬다.
:::

## PyPy를 쓸 수 있는가부터 확인하라

지금까지 본 모든 관용구는 **CPython 안에서** 상수를 줄이는 방법이다. 더 근본적인 선택지는 인터프리터 자체를 바꾸는 것이다. PyPy는 실행 중인 코드를 분석해서 자주 도는 부분을 기계어로 컴파일하는 JIT(just-in-time) 컴파일러를 쓴다. 순수 파이썬 반복문·산술 연산이 많은 코드에서 이득을 보는 방식으로 알려져 있다.

이 절에서 PyPy 자체의 실측 배수를 제시하지 않는다. 이 책을 쓴 환경에 PyPy가 설치돼 있지 않아서 직접 재지 못했다. 대신 **PyPy를 쓸 수 있는지, 써도 되는지를 판단하는 절차**를 정리한다.

::: cote 시험 시작 전에 확인할 것
1. **채점 환경이 PyPy를 제공하는가.** 백준은 "Python 3"와 별도로 "PyPy3" 언어 선택지를 제공한다. 프로그래머스나 기업 자체 채점 플랫폼은 제공하지 않는 경우가 많다 — 언어 선택 드롭다운을 먼저 확인하라.
2. **표준 라이브러리 의존성을 확인하라.** `numpy`, `pandas` 같은 C 확장 기반 라이브러리는 PyPy에서 지원이 불완전하거나 아예 다른 성능 특성을 보인다. 코딩테스트 코드가 표준 라이브러리(`collections`, `heapq`, `itertools`, `bisect`)만 쓴다면 PyPy로 이식하는 데 문제가 적다.
3. **재귀 깊이 제한이 다를 수 있다.** PyPy는 CPython과 스택 관리 방식이 달라서 `sys.setrecursionlimit` 로 늘릴 수 있는 한도나 실제 터지는 지점이 CPython과 다르게 나타날 수 있다. 재귀가 깊은 코드를 PyPy로 제출할 계획이면 미리 로컬에서 확인하라.
4. **JIT은 예열이 필요하다.** JIT 컴파일은 같은 코드 경로가 일정 횟수 이상 반복돼야 기계어로 바뀐다. 실행 시간이 아주 짧은 문제(반복 횟수가 적은 문제)에서는 JIT 예열 비용 때문에 오히려 CPython보다 이득이 적거나 손해를 볼 수 있다. 반복이 아주 많은 문제일수록 유리해지는 경향이 있다.
5. **지금 실행 중인 인터프리터가 무엇인지는 코드로 확인한다.**

```pyrepl
>>> import sys, platform
>>> platform.python_implementation()
'CPython'
>>> sys.version
'3.14.5 (tags/v3.14.5:5607950, May 10 2026, 10:43:50) [MSC v.1944 64 bit (AMD64)]'
```

로컬에서 PyPy로 미리 검증하고 싶다면 `platform.python_implementation()` 이 `'PyPy'` 를 반환하는지로 실제 어느 인터프리터에서 도는지 스크립트 안에서 확인할 수 있다.
:::

## 요약

- 같은 $O(n^2)$ 알고리즘을 파이썬과 C++로 똑같이 짜서 재면(이 절 실측 기준) 약 104배 차이가 난다. 이건 관용구로 없앨 수 없는 실행 모델의 차이다.
- 반복문 안에서 함수를 부르는 비용은 호출 한 번에 약 12.6나노초지만, 반복 횟수가 곱해지면 무시 못 할 크기가 된다. 몸통이 짧은 함수는 인라인하고, 표준 라이브러리의 C 구현 함수를 우선 써라.
- 3.11+ 의 특수화 적응형 인터프리터 덕분에 "전역 변수를 지역 변수로 캐싱하라"는 예전 조언은 이제 거의 차이가 없다(실측 배수 약 1.0배). 오래된 최적화 팁은 검증 없이 믿지 마라.
- 재귀를 반복으로 바꾸면 함수 호출 오버헤드 자체가 사라진다(이 절 실측 약 3.8배). 단, `sys.setrecursionlimit` 을 무한정 늘리는 습관은 위험하다 — 최신 CPython(3.11+)은 프레임 스택 구현이 바뀌어 순수 파이썬 재귀만으로는 예전만큼 쉽게 세그멘테이션 폴트가 나지 않지만(이 절 실측: 깊이 1억도 정상 종료), C 콜백을 거치는 재귀·다른 파이썬 구현·스택이 작은 환경에서는 여전히 그대로 크래시할 수 있다.
- `s += w` 반복은 함수 지역 스코프에서는 CPython의 제자리 리사이즈 최적화 덕에 상대적으로 덜 나쁘지만, 모듈 최상위 스코프에서는 이 최적화가 깨지며 진짜 $O(n^2)$ 로 떨어진다(이 절 실측 최대 3332배 — 절대 배수는 기기마다 흔들리니 "제곱에 비례해 급격히 벌어진다"는 경향으로 기억하라). 스코프에 의존하지 말고 처음부터 `"".join()` 을 써라.
- 리스트 컴프리헨션은 `append` 함수 호출을 전용 바이트코드(`LIST_APPEND`)로 대체해서 약간이지만 확실하게 빠르다.
- PyPy는 순수 파이썬 반복이 많은 코드에서 유리하다고 알려져 있지만, 채점 환경이 지원하는지, C 확장 라이브러리를 쓰는지, 재귀 깊이 제한이 다른지를 먼저 확인해야 한다.
