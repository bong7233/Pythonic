# 8.5 기업별 출제 경향

::: lead
[7.1](#/complexity)부터 [7.26](#/geometry)까지, 그리고 [8.4 문제 유형 분류](#/problem-signals)까지 왔다면 이제 "무엇을" 풀 줄은 안다. 남은 문제는 "어디에 시간을 더 쓸 것인가"다. 3주 남았는데 자료구조 25개를 전부 같은 비중으로 다시 훑는 건 비효율이다. 기업마다 코딩테스트를 내는 목적이 다르고, 그 목적이 문제 스타일을 결정한다. 이 절은 국내에서 널리 알려진 출제 경향을 정리하고, **어디까지가 알려진 사실이고 어디부터가 추측인지**를 구분해서 말한다. 확신 없는 세부사항은 넣지 않는다.
:::

## 왜 기업마다 다른가 — 시험의 목적이 다르다

코딩테스트는 회사마다 채용 단계에서 하는 역할이 다르다. 이 차이가 문제 스타일을 결정한다.

- **대규모 공채의 1차 필터**로 쓰는 경우: 지원자가 매우 많아서 자동 채점으로 걸러야 한다. 문제가 길고 구현량이 많아진다. 짧고 우아한 풀이보다 "요구사항을 빠짐없이 구현할 수 있는가"를 본다.
- **실무 역량의 대리 지표**로 쓰는 경우: 실제 서비스에서 마주칠 법한 상황을 문제로 각색한다. 알고리즘 지식보다 "문제를 정확히 분해해서 완전탐색이든 최적화든 답을 찾아내는가"를 본다.
- **표준화된 알고리즘 실력 측정**으로 쓰는 경우: 백준·프로그래머스식 순수 알고리즘 문제에 가깝다. [7.13 그래프](#/graph), [7.15 최단 경로](#/shortest-path), [7.20 DP](#/dp-basics) 같은 정석 카테고리가 그대로 나온다.

세 목적 중 어디에 가까운지가 "삼성형"과 "카카오형"이라는, 한국 코딩테스트 커뮤니티에서 널리 쓰이는 두 축으로 갈린다. 이 절 이후로 이 두 단어를 자주 쓸 텐데, 특정 회사의 공식 명칭이 아니라 **문제 스타일을 가리키는 통칭**이라는 점을 먼저 짚어 둔다.

::: note 이 절의 정보 출처와 한계
여기 적힌 경향은 백준 온라인 저지의 기업별 기출 태그, 프로그래머스의 기업 코딩테스트 문제집, 그리고 지원자들의 후기가 오랫동안 쌓이며 커뮤니티에 정착된 통념이다. **회사의 공식 발표가 아니고, 시기·직군·연차에 따라 얼마든지 달라질 수 있다.** 특정 연도의 정확한 문제 수, 제한시간, 합격선 같은 수치는 이 절에서 다루지 않는다. 그런 숫자는 검색하면 바로 나오지만 매년 바뀌므로 책에 박아 두는 순간 낡은 정보가 된다.
:::

## 삼성 SW역량테스트 스타일 — 구현과 시뮬레이션

"삼성형"이라 불리는 스타일은 **긴 지문 + 규칙 기반 시뮬레이션**으로 요약된다.

### 지문이 길다는 것의 의미

문제 하나가 A4 한 장을 넘기는 경우가 흔하다고 알려져 있다. 이런 지문은 알고리즘 지식이 아니라 **읽기 능력과 상태 관리 능력**을 시험한다. "매 턴마다 다음 규칙을 순서대로 적용한다: (1) ... (2) ... (3) ..." 식으로 규칙이 단계별로 나열되고, 그 순서를 하나라도 뒤바꾸면 전혀 다른 결과가 나온다.

::: cote 긴 시뮬레이션 지문을 다루는 순서
1. **상태를 먼저 설계한다.** 격자, 위치, 방향, 점수 같은 걸 어떤 자료구조에 담을지 코드를 짜기 전에 정한다. 2차원 격자면 [1.1 객체·이름·참조](#/objects-names)의 `[[0]*n for _ in range(n)]` 함정부터 피해야 한다.
2. **규칙을 번호 그대로 함수로 쪼갠다.** 지문의 "1번 규칙", "2번 규칙"을 각각 별도 함수로 만들면, 나중에 틀렸을 때 어느 규칙이 잘못됐는지 바로 찾을 수 있다. 한 함수에 다 몰아넣으면 디버깅이 지옥이 된다.
3. **작은 예제로 손으로 먼저 따라간다.** 지문에 주어진 예제 입력을 손으로 3~4턴만 굴려 보고, 코드 결과와 비교한다. 규칙 순서를 잘못 이해했으면 여기서 걸린다.
4. **경계 조건을 지문에서 직접 찾는다.** "격자를 벗어나면", "더 이상 움직일 수 없으면" 같은 예외 규칙이 본문 중간에 슬쩍 끼어 있는 경우가 많다. 규칙 목록을 다 읽은 뒤가 아니라 **읽으면서 바로 체크리스트로 옮겨 적어라.**
:::

### 왜 이 스타일이 자료구조보다 구현력을 재는가

이런 문제는 대개 $O(n)$ 이나 $O(n^2)$ 수준의 단순한 반복이면 시간 제한을 통과한다. 어려운 자료구조나 자료구조 최적화가 필요한 경우는 상대적으로 적다고 알려져 있다. 대신 **방향 벡터 순회, 회전, 확산, 좌표 변환** 같은 구현 패턴이 반복해서 나온다.

```python title="시뮬레이션형 문제의 전형적 뼈대 — 4방향 확산"
DIRS = [(-1, 0), (1, 0), (0, -1), (0, 1)]  # 상하좌우

def spread_one_turn(grid: list[list[int]], n: int) -> list[list[int]]:
    """규칙: 값이 있는 칸은 인접 빈 칸에 자신의 절반을 나눠 준다."""
    new_grid = [[0] * n for _ in range(n)]  # 새 상태를 담을 그릇 — 원본을 직접 고치면 순서가 꼬인다
    for r in range(n):
        for c in range(n):
            if grid[r][c] == 0:
                continue
            share = grid[r][c] // 4
            remain = grid[r][c] - share * 4
            new_grid[r][c] += remain
            for dr, dc in DIRS:
                nr, nc = r + dr, c + dc
                if 0 <= nr < n and 0 <= nc < n:
                    new_grid[nr][nc] += share
                else:
                    new_grid[r][c] += share  # 밖으로 나가면 자기 칸에 남는다는 규칙
    return new_grid
```

```pyrepl
>>> g = [[100, 0], [0, 0]]
>>> spread_one_turn(g, 2)
[[50, 25], [25, 0]]
```

**"한 턴 전체를 동시에 갱신한다"** 는 원본을 즉시 고치지 않고 `new_grid` 라는 별도 그릇에 쌓았다가 한꺼번에 교체하는 패턴을 눈여겨봐라. 원본을 직접 고치면 아직 처리 안 된 칸이 이미 갱신된 값을 읽어 버리는 순서 오류가 생긴다. 이건 삼성형 시뮬레이션에서 가장 흔한 실수 중 하나로 꼽힌다.

::: warn 지문을 다 읽기 전에 코드부터 짜지 마라
긴 지문은 뒤쪽에 앞의 규칙을 뒤집는 예외가 숨어 있는 경우가 많다고 알려져 있다("단, N번째 턴부터는 규칙이 바뀐다" 같은 문장이 지문 맨 아래에 있는 식). 지문 전체를 한 번 다 읽고 규칙을 목록으로 정리한 다음 코드를 시작하는 편이, 짜다가 다시 갈아엎는 것보다 결국 빠르다.
:::

## 카카오 스타일 — 실용적 문제와 완전탐색

"카카오형"은 접근이 다르다. 지문은 상대적으로 짧고 상황 설명이 명확하며, **완전탐색으로 일단 답을 낼 수 있고 그걸 최적화하는 능력**을 함께 본다고 알려져 있다.

### 완전탐색이 출발점이다

문제를 처음 읽었을 때 "모든 경우를 다 따져 보면 어떻게든 답은 나온다"는 감이 오는 경우가 많다. 시험의 핵심은 그 완전탐색이 **왜 느린지 파악하고, 그중 어떤 부분을 가지치기하거나 자료구조로 대체할지 찾아내는 것**이다. [7.18 백트래킹](#/backtracking)의 가지치기 감각, [7.6 해시](#/hashing)로 중복 계산을 없애는 감각이 정확히 여기서 쓰인다.

```python title="완전탐색에서 최적화로 — 전형적인 흐름 예시"
from itertools import combinations

def brute_force(items: list[int], target: int) -> int:
    """모든 부분집합을 다 따져 target 이하로 만들 수 있는 최대 합을 찾는다."""
    best = 0
    n = len(items)
    for r in range(n + 1):
        for combo in combinations(items, r):
            s = sum(combo)
            if s <= target:
                best = max(best, s)
    return best


def optimized(items: list[int], target: int) -> int:
    """부분합 DP — 가능한 합의 집합을 set으로 누적한다."""
    reachable = {0}
    for x in items:
        reachable |= {s + x for s in reachable if s + x <= target}
    return max(reachable)
```

```pyrepl
>>> items = [3, 7, 2, 9, 5, 1, 8, 4]
>>> target = 20
>>> brute_force(items, target)
20
>>> optimized(items, target)
20
```

$n=8$ 에서는 완전탐색이 $2^8=256$ 가지라 눈 깜짝할 사이에 끝나지만, 카카오형 문제는 보통 제약 조건을 $n=20$ 이상으로 슬쩍 올려서 완전탐색만으로는 시간 초과가 나게 만든다고 알려져 있다. 완전탐색으로 정답 로직부터 확인한 뒤, 시간 제한과 제약 조건을 보고 어디를 최적화할지 판단하는 순서가 중요하다.

```python title="n을 키워서 완전탐색이 무너지는 지점을 직접 확인"
import random, time

random.seed(1)
big_items = [random.randint(1, 100) for _ in range(24)]

start = time.perf_counter()
brute_force(big_items, 500)
print(f"n=24 완전탐색: {time.perf_counter() - start:.3f}초")

start = time.perf_counter()
optimized(big_items, 500)
print(f"n=24 최적화: {time.perf_counter() - start:.3f}초")
```

```text nolines
n=24 완전탐색: 1.265초
n=24 최적화: 0.000초
```

(Python 3.14.5 / Windows 기준 실측. 절대 시간은 기기마다 다르지만 $n=24$ 면 $2^{24}$ 약 1,678만 가지를 다 따지느라 최적화 풀이 대비 수천 배 느려지는 자릿수 차이는 어디서나 같다. $n=30$ 만 돼도 시간 제한을 넘기기 십상이다. 부분합 DP는 `reachable` 집합의 크기가 `target` 을 넘지 않으므로 사실상 즉시 끝난다.)

::: cote 카카오형 문제를 알아채는 신호
- 지문에 **캐릭터, 게임, 실제 서비스 상황**(순위표, 채팅, 매칭 등)을 소재로 한 설명이 붙는다. 소재에 현혹되지 말고 결국 무엇을 세거나 찾는 문제인지를 뽑아내라.
- 제약 조건이 "완전탐색은 되는데 딱 아슬아슬하게 안 되는" 크기로 주어진다. $n \le 15$ 정도면 비트마스크나 순열 완전탐색을 그대로 써도 된다는 신호고([7.24 비트마스크](#/bitmask)), $n \le 10^5$ 인데 $O(n^2)$ 로 보이면 해시나 정렬, 투 포인터로 낮출 자리가 있다는 신호다.
- 문제 하나에 **여러 언어의 제출이 허용**되는 경우가 많고, 언어별로 시간 제한이 다르게 걸리는 경우가 있다고 알려져 있다. 파이썬 제출이라면 [8.3 시간 초과를 피하는 관용구](#/tle)의 상수 최적화가 그대로 필요해진다.
:::

## 네이버·라인·쿠팡·토스 — 알려진 경향과 그 한계

여기부터는 확신의 정도를 낮춰서 쓴다. 삼성·카카오만큼 오래, 많이 쌓인 통념이 없기 때문이다.

- **네이버**: 카카오와 마찬가지로 프로그래머스 플랫폼을 채용 코딩테스트에 쓰는 경우가 알려져 있다. 문제 스타일도 실용적 문제 + 완전탐색/최적화 축에 가깝다고 이야기되지만, 삼성-카카오만큼 뚜렷하게 구분되는 고유한 특징이 있다고 단정하긴 어렵다.
- **라인**: 알고리즘 자체보다 **코드 품질, 테스트 가능성, 예외 처리**를 함께 보는 문제가 나온다는 이야기가 있다. 다만 이건 회사·직군·연차에 따라 크게 달라질 수 있는 부분이라 이 절에서 일반화하지 않는다.
- **쿠팡**: 물류·배송 도메인을 소재로 한 문제, 그래프나 최단 경로류가 나온다는 이야기가 있다. 확인된 고정 패턴이라고 말하기는 어렵다.
- **토스**: 실무형 문제(작은 시스템 설계에 가까운 문제)와 순수 알고리즘 문제가 섞여 나온다는 이야기가 있고, 코딩테스트 외에 과제형 전형을 쓰는 경우도 있다고 알려져 있다.

::: warn 여기 적힌 것을 암기하지 마라
네 회사의 세부 경향은 **연도, 채용 공고, 직군(신입/경력, 백엔드/AI 등)에 따라 달라진다.** 이 절이 줄 수 있는 건 "삼성형이냐 카카오형이냐에 가까운 축으로 미리 감을 잡으라"는 것까지다. 실제 지원 직전에는 그 회사·그 해의 채용 공고와 최근 후기를 직접 찾아 확인해라. 이 책이 대신해 줄 수 없는 부분이다.
:::

지원하는 회사의 스타일을 전혀 모르겠으면, [7.13](#/graph)~[7.21](#/dp-advanced)에서 다룬 정석 알고리즘을 기본으로 준비하는 게 안전하다. 표준 알고리즘형 대비는 삼성형·카카오형 어느 쪽으로 문제가 나와도 최소한의 방어력을 준다. 반대로 삼성형에만 특화해서 시뮬레이션만 팠는데 실제로 그래프 최단 경로 문제가 나오면 대비가 전혀 안 된 상태로 시험장에 앉게 된다.
:::

### 자료구조 하나를 완전히 아는 것과 열 개를 얕게 아는 것

시간이 부족할 때 흔히 하는 실수는 25개 알고리즘을 전부 "이름은 안다" 수준으로 훑는 것이다. 그러느니 [7.6 해시](#/hashing), [7.14 BFS/DFS](#/bfs-dfs), [7.20 DP 기초](#/dp-basics) 셋만이라도 **막히지 않고 짤 수 있는 수준**으로 만드는 게 실전에서 더 유효하다. 이 세 가지가 표준 알고리즘형·카카오형 문제 대부분의 뼈대를 이루기 때문이다. 삼성형을 준비한다면 여기에 격자 시뮬레이션 관용구를 더한다.

## 준비 기간별 계획

::: cote 3주 vs 1주 남았을 때의 전략 차이
- **3주 이상**: [7.1](#/complexity)부터 순서대로 다시 훑되, 위 표에서 자신이 준비하는 스타일에 해당하는 항목에 시간을 2배로 배정한다. 매일 1~2문제를 실제로 제한시간 안에 풀어 본다.
- **1주 남았을 때**: 새 개념을 배우지 마라. 이미 아는 것 중 **자주 틀리는 패턴**을 추려서 반복한다. 예를 들어 `[[0]*n]*n` 함정([1.1](#/objects-names)), 문자열 누적 `+=`([1.1](#/objects-names)), 재귀 깊이 초과([7.13](#/graph)) 같은 것들이다. [8.10 실전 템플릿 모음](#/templates)을 미리 손에 익혀 두면 시험장에서 자료구조를 처음부터 짜는 시간을 아낀다.
- **시험 전날**: 새 문제를 풀지 마라. 이미 푼 문제 중 헷갈렸던 것 2~3개를 다시 읽고, [8.2 입출력 최적화](#/io-optimize) 코드 템플릿과 자신의 언어 설정(파이썬 버전, 허용 라이브러리)만 확인한다.
:::

## 이 절이 확실하게 말할 수 없는 것

정직하게 선을 긋는다. 다음은 **이 절이 단정하지 않는 것들**이다.

- 특정 회사의 특정 연도 문제 수, 시험 시간, 커트라인. 매년 바뀌고, 공식 발표 없이 후기로만 도는 수치도 많다.
- "이 회사는 항상 이 알고리즘만 낸다" 같은 단정. 위 표는 **경향**이지 규칙이 아니다. 삼성형 시험에 그래프 문제가 나올 수 있고, 카카오형 시험에 순수 시뮬레이션이 나올 수도 있다.
- 온라인 코딩테스트의 부정행위 감지 방식, 화상 감독 여부 같은 운영상의 세부사항. 회사·시기마다 다르고 이 책이 다룰 영역도 아니다.

::: tip 가장 믿을 수 있는 정보원
회사 스타일에 대한 가장 정확한 정보는 **그 회사가 최근에 공개한 채용 공고와, 그 회사 문제가 실제로 올라와 있는 온라인 저지(백준의 기업 기출 태그, 프로그래머스의 코딩테스트 연습 카테고리)**다. 이 절의 통념보다 실제 기출 문제 몇 개를 직접 풀어 보는 게 훨씬 정확한 감을 준다.
:::

## 요약

- 코딩테스트 스타일은 회사가 시험을 채용 절차의 어떤 역할로 쓰는지에 따라 갈린다. "삼성형"(긴 지문 + 구현/시뮬레이션)과 "카카오형"(실용적 소재 + 완전탐색/최적화)이 커뮤니티에서 널리 쓰이는 두 축이다.
- 삼성형은 지문을 규칙 단위로 쪼개 함수화하고, 상태를 한꺼번에 갱신하는 패턴을 정확히 구현하는 능력을 본다.
- 카카오형은 완전탐색으로 답의 뼈대를 먼저 확인한 뒤, 제약 조건을 보고 어디를 최적화할지(해시, DP, 비트마스크) 판단하는 능력을 본다.
- 네이버·라인·쿠팡·토스 등은 삼성·카카오만큼 뚜렷한 통념이 쌓여 있지 않다. 여기 적힌 것은 참고일 뿐이고, 실제 준비는 최신 채용 공고와 기출 문제로 확인해야 한다.
- 시간이 부족하면 25개 알고리즘을 얕게 훑기보다, 자신이 준비하는 스타일에 맞는 소수의 자료구조를 깊게 파는 게 낫다. 어느 쪽인지 모르겠으면 표준 알고리즘형(그래프, 최단 경로, DP)을 기본값으로 잡아라.
