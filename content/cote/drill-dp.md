## 이 절에서 반복할 절차

세 문제 모두 같은 순서를 따른다. 이 순서 자체가 이 절의 핵심이다.

1. **문제를 읽고 브루트포스부터 짠다.** 재귀로 모든 경우를 다 시도하는 코드다. 느리지만 문제 정의를 그대로 옮긴 것이라 틀릴 여지가 거의 없다 — [8.1 코딩테스트의 구조와 전략](#/cote-strategy)에서 세운 원칙 그대로다.
2. **점화식을 4단계로 도출한다.** 답을 정의하고, 마지막 선택으로 쪼개고, 부분 문제로 표현하고, 기저 조건을 정한다 — [7.20절](#/dp-basics)의 절차다.
3. **`lru_cache` 로 메모이제이션한다.** 브루트포스 코드에 데코레이터 한 줄만 얹는다. 로직은 그대로다.
4. **타뷸레이션으로 바꾼다.** 재귀 깊이와 공간을 실전 크기에 맞게 줄인다.
5. **무작위 대조 검증과 실측을 한다.** 네 가지 구현(브루트포스·메모·타뷸레이션·필요하면 공간 압축판)이 모두 같은 값을 내는지 확인하고, 실제로 얼마나 빨라지는지 잰다.

## 문제 1 — 인접하지 않게 최대 합 고르기

**문제.** 일렬로 늘어선 창고 $n$ 개가 있고, 각 창고에는 물건의 가치 `nums[i]` (0 이상의 정수)가 적혀 있다. 인접한 두 창고에서 동시에 물건을 가져갈 수 없다는 제약 아래, 가져갈 수 있는 물건 가치의 총합을 최대화하라. 창고가 하나도 없으면 답은 0이다.

이건 "연속하지 않게 골라 합을 최대화하라"는 패턴의 가장 단순한 형태다. 격자 스캔, 스티커 붙이기, 좌석 배치 같은 변형이 전부 이 뼈대를 공유한다.

### 브루트포스: 매 창고마다 고른다/안 고른다

```python title="rob_brute.py — 완전탐색"
def rob_brute(nums, i=0):
    if i >= len(nums):
        return 0
    take = nums[i] + rob_brute(nums, i + 2)   # i번을 고른다 -> i+1은 건너뛴다
    skip = rob_brute(nums, i + 1)             # i번을 안 고른다
    return max(take, skip)
```

`i` 번째 창고를 기준으로 "고른다"와 "안 고른다" 두 갈래로만 나뉜다. 매 단계 두 갈래로 갈라지므로 $n$ 개 창고에서 호출 수는 $O(2^n)$ 이다. 겹치는 부분 문제도 바로 보인다.

```text nolines
rob_brute([2, 7, 9, 3, 1], i=0)
|-- take: 2 + rob_brute(i=2)
|     |-- take: 9 + rob_brute(i=4)
|     `-- skip: rob_brute(i=3)
`-- skip: rob_brute(i=1)
      |-- take: 7 + rob_brute(i=3)   <- rob_brute(i=3) 을 위에서도 이미 불렀다
      `-- skip: rob_brute(i=2)       <- rob_brute(i=2) 도 마찬가지
```

`rob_brute(i=3)` 과 `rob_brute(i=2)` 가 트리 양쪽에서 각각 두 번씩 계산된다. $n$ 이 커지면 이 중복이 기하급수적으로 불어난다.

### 점화식 도출

1. **답의 정의**: $f(i)$ = "인덱스 $i$ 부터 끝까지의 창고만 놓고 볼 때 가져갈 수 있는 최댓값". 구하려는 답은 $f(0)$ 이다.
2. **마지막 선택**: $i$ 번째 창고를 기준으로 남은 문제를 푸는 마지막 갈림길은 "$i$ 번을 가져간다"와 "안 가져간다" 둘뿐이다.
3. **부분 문제로 표현**: 가져가면 $i+1$ 은 못 가져가니 $nums[i] + f(i+2)$, 안 가져가면 $f(i+1)$. 둘 중 큰 쪽이 $f(i)$ 다.
4. **기저 조건**: $i \ge n$ 이면 남은 창고가 없으니 $f(i) = 0$.

$$f(i) = \max\bigl(nums[i] + f(i+2),\ f(i+1)\bigr), \quad f(i) = 0 \text{ if } i \ge n$$

브루트포스 코드의 `take`/`skip` 두 줄이 이 점화식을 그대로 옮긴 것이다. 코드는 이미 옳다. 문제는 같은 `i` 를 여러 번 다시 계산한다는 것뿐이다.

### 메모이제이션과 타뷸레이션

```python title="rob_memo.py — lru_cache 한 줄"
from functools import lru_cache

@lru_cache(maxsize=None)
def rob_memo_inner(nums, i):
    if i >= len(nums):
        return 0
    take = nums[i] + rob_memo_inner(nums, i + 2)
    skip = rob_memo_inner(nums, i + 1)
    return max(take, skip)

def rob_memo(nums):
    rob_memo_inner.cache_clear()
    return rob_memo_inner(tuple(nums), 0)
```

`lru_cache` 는 인자를 키로 쓰므로 `nums` 를 해시 불가능한 `list` 그대로 넘기면 `TypeError` 가 난다. `tuple(nums)` 로 바꿔서 넘기는 이유다 ([7.20절](#/dp-basics)에서 다룬 함정과 정확히 같다).

타뷸레이션은 방향을 뒤집어 앞에서부터 채운다. $f(i)$ 가 아니라 "처음 $i$ 개 창고만 있을 때의 답" $g(i) = f(0)$ 을 $n-i$ 개짜리 부분 문제로 재정의하면 인덱스가 자연스럽게 증가하는 방향으로 정리된다.

```python title="rob_tab.py — 반복, 그리고 공간 O(1)"
def rob_tab(nums):
    n = len(nums)
    if n == 0:
        return 0
    dp = [0] * (n + 1)
    dp[1] = nums[0]
    for i in range(2, n + 1):
        dp[i] = max(dp[i - 1], dp[i - 2] + nums[i - 1])
    return dp[n]

def rob_tab_o1(nums):
    prev2, prev1 = 0, 0          # dp[i-2], dp[i-1]
    for x in nums:
        prev2, prev1 = prev1, max(prev1, prev2 + x)
    return prev1
```

점화식이 "바로 이전 두 값만" 참조하므로 배열 전체를 들고 있을 필요가 없다. `rob_tab_o1` 은 변수 두 개로 $O(1)$ 공간에서 끝낸다.

### 검증과 실측

무작위 배열 500개(길이 0~16)에 대해 네 구현이 전부 같은 값을 내는지 확인했다.

```text nolines
house robber: mismatches 0/500
[] -> 0 expected 0 OK
[5] -> 5 expected 5 OK
[2, 3] -> 3 expected 3 OK
[2, 7, 9, 3, 1] -> 12 expected 12 OK
[5, 1, 1, 5] -> 10 expected 10 OK
```

`[5, 1, 1, 5]` 가 10인 이유를 손으로 짚어 보면 점화식 검증이 된다. 양 끝 `5, 5` 는 인접하지 않으니 둘 다 가져갈 수 있다.

```pyrepl
>>> import time
>>> for n in (20, 24, 28, 30):
...     nums = [1] * n  # 임의 값, 분기 수만 중요
...     t0 = time.perf_counter(); rob_brute(nums); t1 = time.perf_counter()
...     print(n, f"{(t1-t0)*1000:.2f} ms")
20 1.54 ms
24 10.14 ms
28 70.41 ms
30 200.11 ms
>>> nums_big = [1] * 200_000
>>> t0 = time.perf_counter(); rob_tab_o1(nums_big); t1 = time.perf_counter()
>>> print(f"{(t1-t0)*1000:.3f} ms")
5.902 ms
```

(Python 3.14.5 / Windows 기준 실측.) 표의 구간은 $n$ 이 4씩 늘다가 마지막만 2씩 는다는 점에 주의해서 읽어야 한다. $n$ 이 4씩 늘 때(20→24, 24→28)는 각각 6.58배, 6.94배로 뛰고, 마지막 28→30 구간은 $n$ 이 2만 늘었으니 배율도 그만큼 작아져 2.84배에 그친다 — 재실행에서도 6.41배·6.74배·2.72배로 같은 패턴이 나왔다. $2^4=16$, $2^2=4$ 라는 이론값보다 실측 배율이 낮은 건 재귀 호출 오버헤드와 캐시·GC 변동 때문이지만, "$n$ 증가폭이 클수록 배율도 커진다"는 $2^n$ 의 방향성 자체는 세 구간 모두에서 어긋나지 않는다. 타뷸레이션은 브루트포스가 시도조차 못 할 $n=200{,}000$ 을 6 ms 안에 처리한다.

::: cote 이 유형을 알아채는 신호
"인접한 것을 동시에 고를 수 없다", "연속하지 않게 골라라", "이전 선택이 다음 선택을 제한한다" 같은 문구가 나오고 배열이 1차원이면 이 뼈대를 의심하라. 상태는 "지금까지 본 것 중 마지막으로 고른 것이 몇 번째인가"가 아니라 **"직전 한두 개를 골랐는지 여부"** 로 충분히 압축된다는 게 핵심이다. 좌석 배치, LED 켜기, 계단 오르기 변형 문제들이 전부 이 압축을 쓴다.
:::

## 문제 2 — 동전 교환 최소 개수

**문제.** 동전 종류 `coins`(각 종류는 무한히 있다고 가정)와 목표 금액 `amount` 가 주어진다. 정확히 `amount` 를 만드는 데 필요한 **최소 동전 개수**를 구하라. 만들 수 없으면 -1을 반환한다.

이 문제는 문제 1과 모양이 다르다. **선택지가 두 개가 아니라 동전 종류 수만큼**이고, 최대가 아니라 최소를 구하며, "만들 수 없는 경우"를 별도로 다뤄야 한다.

::: warn 그리디로 풀면 안 된다
"큰 동전부터 최대한 많이 쓰면 되지 않나"라는 직관은 `coins = [1, 3, 4]`, `amount = 6` 에서 바로 깨진다. 그리디는 4를 먼저 쓰고 나머지 2를 1+1로 채워 3개를 답한다. 실제 최적해는 3+3으로 **2개**다. 동전 종류가 [1, 5, 10, 50]처럼 배수 관계가 깔끔한 화폐 체계가 아니면 그리디는 일반적으로 틀린다. DP로 모든 경우를 다 봐야 하는 이유가 여기 있다.
:::

### 브루트포스: 각 동전을 하나씩 써 보고 재귀

```python title="coin_brute.py — 완전탐색"
INF = float("inf")

def coin_brute(coins, amount):
    if amount == 0:
        return 0
    if amount < 0:
        return INF
    best = INF
    for c in coins:
        r = coin_brute(coins, amount - c)
        if r + 1 < best:
            best = r + 1
    return best
```

`amount` 가 음수가 되면 그 갈래는 막다른 길이므로 `INF` 를 반환해서 "이 선택은 답이 될 수 없다"는 걸 표시한다. `INF + 1` 은 여전히 `INF` 이므로 이 값이 최솟값으로 선택되는 일은 없다.

### 점화식 도출

1. **답의 정의**: $f(a)$ = "금액 $a$ 를 만드는 데 필요한 최소 동전 개수" (불가능하면 $\infty$).
2. **마지막 선택**: 금액 $a$ 를 완성하는 **마지막으로 놓은 동전**은 `coins` 중 하나다. 어떤 동전 $c$ 를 마지막에 놓았다면, 그 직전까지는 $a - c$ 를 만들어 둔 상태였어야 한다.
3. **부분 문제로 표현**: 마지막 동전으로 $c$ 를 썼다면 필요한 개수는 $f(a-c) + 1$ 이다. 어떤 동전을 마지막에 놓을지는 모르니 **모든 $c$ 에 대해 시도하고 최솟값**을 취한다.
4. **기저 조건**: $f(0) = 0$ (아무것도 안 놓아도 0원은 이미 완성), $f(a) = \infty$ for $a < 0$.

$$f(a) = \min_{c \in coins,\ c \le a} \bigl(f(a-c) + 1\bigr), \quad f(0) = 0$$

문제 1과 비교하면 분기 수가 "고른다/안 고른다" 2개에서 "어떤 동전을 쓸까" $|coins|$ 개로 늘었을 뿐, **"마지막 선택으로 쪼갠다"는 사고 자체는 똑같다.**

### 메모이제이션과 타뷸레이션

```python title="coin_memo.py"
from functools import lru_cache

@lru_cache(maxsize=None)
def coin_memo_inner(coins, amount):
    if amount == 0:
        return 0
    if amount < 0:
        return INF
    best = INF
    for c in coins:
        r = coin_memo_inner(coins, amount - c)
        if r + 1 < best:
            best = r + 1
    return best

def coin_memo(coins, amount):
    coin_memo_inner.cache_clear()
    return coin_memo_inner(tuple(sorted(coins)), amount)
```

```python title="coin_tab.py — 반복, 0원부터 채운다"
def coin_tab(coins, amount):
    dp = [0] + [INF] * amount     # dp[a] = f(a)
    for a in range(1, amount + 1):
        for c in coins:
            if c <= a and dp[a - c] + 1 < dp[a]:
                dp[a] = dp[a - c] + 1
    return dp[amount]             # INF면 호출부에서 -1로 바꿔서 반환
```

타뷸레이션은 `dp[0]` 부터 `dp[amount]` 까지 순서대로 채운다. `dp[a]` 를 채울 때 필요한 `dp[a-c]` 는 $c \ge 1$ 이므로 항상 **이미 계산된 더 작은 인덱스**다 — 이게 반복 순서를 `range(1, amount+1)` 오름차순으로 잡아도 안전한 이유다.

### 검증과 실측

무작위 동전 집합 300개(종류 1~4개, 목표 금액 0~25)에 대해 브루트포스·메모·타뷸레이션 세 구현을 대조했다. 세 함수 모두 "만들 수 없음"은 원래 `float("inf")` 로 반환하는데, 아래 표는 결과를 사람이 읽기 좋게 `-1 if result == float("inf") else result` 로 변환한 뒤 보여준다 — 즉 `-1` 은 세 구현이 실제로 반환한 값이 아니라 표시용 값이고, 함수가 실제로 내놓는 원값은 뒤의 danger 박스에서 `coin_tab((2,), 3)` 이 `inf` 를 그대로 찍는 걸로 확인한다.

```text nolines
coin change: mismatches 0/300 (표시는 -1 if result == inf else result 로 변환)
(1, 3, 4) 6 -> 2         # 3+3
(2,) 3 -> -1              # 원값은 inf. 홀수를 짝수 동전으로 못 만든다
(1, 5, 10, 50) 63 -> 5   # 50+10+1+1+1
(1,) 0 -> 0
```

```pyrepl
>>> import time
>>> coins = (1, 3, 4)
>>> for amount in (15, 20, 25, 28):
...     t0 = time.perf_counter(); coin_brute(coins, amount); t1 = time.perf_counter()
...     print(amount, f"{(t1-t0)*1000:.2f} ms")
15 0.23 ms
20 2.52 ms
25 27.67 ms
28 116.73 ms
>>> coins_big = (1, 5, 10, 25)
>>> for amount in (1000, 10000, 100000):
...     t0 = time.perf_counter(); coin_tab(coins_big, amount); t1 = time.perf_counter()
...     print(amount, f"{(t1-t0)*1000:.3f} ms")
1000 0.205 ms
10000 1.999 ms
100000 20.788 ms
```

(Python 3.14.5 / Windows 기준 실측.) 표의 구간도 폭이 일정하지 않다: 15→20, 20→25는 목표 금액이 5씩 늘고, 마지막 25→28만 3이 는다. 5 늘어나는 두 구간은 각각 10.96배, 10.98배가 뛰고, 실제로 "3 늘어나는" 유일한 구간인 25→28은 4.22배에 그친다 — 재실행에서도 11.45배·11.70배·4.27배로 같은 패턴이었다. 분기 수가 동전 종류만큼(여기서는 3) 있는 트리이므로 $O(3^{amount/\min(c)})$ 근처의 지수 시간이고, 증가폭이 클수록 배율도 커진다는 방향성은 세 구간 모두 지킨다. 타뷸레이션은 `amount` 에 선형으로만 비례해서(정확히는 $O(amount \times |coins|)$), 브루트포스가 손도 못 대는 `amount=100,000` 을 21 ms 만에 끝낸다.

::: cote 이 유형을 알아채는 신호
"동전/화폐 최소 개수", "선택 가능한 항목을 몇 번이든 다시 쓸 수 있다(무한 공급)", "최소/최대 조합"이 같이 나오면 이 패턴이다. [7.21 동적 계획법 심화](#/dp-advanced)의 배낭 문제와 형제뻘이다 — 차이는 "각 항목을 몇 번까지 쓸 수 있는가"뿐이다. 여기서는 무제한(완전 배낭류), 0/1 배낭이면 각 항목을 한 번만 써야 하므로 반복 순서 자체가 바뀐다. 이 차이를 안 짚고 코드를 복붙하면 조용히 틀린 답이 나온다.
:::

::: danger `float("inf")` 를 정수 자리에 그냥 두면 안 되는 경우
이 절 코드에서는 `INF + 1` 이 여전히 `INF` 라서 문제가 없지만, **결과를 그대로 출력하거나 다른 정수 연산에 섞으면** 타입이 `float` 로 새어 나간다.

```pyrepl
>>> coin_tab((2,), 3)
inf
>>> print(f"최소 동전 개수: {coin_tab((2,), 3)}")
최소 동전 개수: inf
```

시험장에서 "정답이 없으면 -1을 출력하라"는 요구가 있으면 반환 직전에 반드시 변환한다.

```python
result = coin_tab(coins, amount)
print(-1 if result == float("inf") else result)
```

이 변환을 잊고 `inf` 를 그대로 출력하면 채점 시스템은 자동으로 오답 처리한다.
:::

## 문제 3 — 격자 경로 최댓값 합 (장애물 포함)

**문제.** $r \times c$ 격자의 각 칸에 정수 점수가 적혀 있다. 로봇이 왼쪽 위 칸에서 시작해 **오른쪽 또는 아래로만** 이동해서 오른쪽 아래 칸까지 간다. 일부 칸은 장애물(막힘)이라 지나갈 수 없다. 지나간 칸(시작·끝 포함)의 점수 합을 최대화하라. 도달할 수 있는 경로가 없으면 "경로 없음"을 알린다.

문제 1·2와 달리 **상태가 2차원**이다. "지금까지 얼마를 썼는가"가 아니라 "지금 몇 행 몇 열에 있는가"가 상태다.

### 브루트포스: 두 방향으로만 재귀

```python title="grid_brute.py — 완전탐색"
NEG = float("-inf")

def grid_brute(grid, r=0, c=0):
    rows, cols = len(grid), len(grid[0])
    if r >= rows or c >= cols or grid[r][c] is None:
        return NEG                          # 격자 밖이거나 장애물
    if r == rows - 1 and c == cols - 1:
        return grid[r][c]                   # 도착
    best = max(grid_brute(grid, r + 1, c), grid_brute(grid, r, c + 1))
    if best == NEG:
        return NEG                          # 아래도 오른쪽도 다 막혔다
    return grid[r][c] + best
```

매 칸에서 "아래로" 또는 "오른쪽으로" 두 갈래로만 갈라지지만, **같은 칸 $(r, c)$ 로 가는 경로가 여러 개** 있다는 게 문제 1·2와 다른 지점이다. $(0,0)$ 에서 $(r,c)$ 로 가는 경로 수는 $\binom{r+c}{r}$ 이고, 이게 겹치는 부분 문제의 개수다. 격자가 커지면 이 값이 폭발적으로 커진다.

### 점화식 도출

1. **답의 정의**: $f(r, c)$ = "$(r, c)$ 에서 시작해 오른쪽 아래 칸까지 가는 최적 경로의 점수 합" (도달 불가능하면 $-\infty$).
2. **마지막 선택**: $(r, c)$ 에서 다음 한 걸음은 "아래로 $(r+1, c)$" 또는 "오른쪽으로 $(r, c+1)$" 둘뿐이다.
3. **부분 문제로 표현**: $f(r, c) = grid[r][c] + \max(f(r+1, c),\ f(r, c+1))$. 단 두 값이 모두 $-\infty$ 면(양쪽 다 막혔다) $f(r, c)$ 도 $-\infty$ 다.
4. **기저 조건**: $(r, c)$ 가 격자 밖이거나 장애물이면 $-\infty$. 도착 칸이면 $f(\text{끝}) = grid[\text{끝}]$.

이 점화식은 "**마지막 선택**"을 도착 지점 기준(어디서 왔는가)이 아니라 **출발 지점 기준(다음에 어디로 가는가)** 으로 세운 것이다. 두 방향 다 자연스럽게 표현할 수 있으면 어느 쪽으로 잡아도 되지만, 타뷸레이션에서 반복 순서를 정할 때는 방향을 명확히 의식해야 한다 — 아래 코드에서 바로 드러난다.

### 메모이제이션과 타뷸레이션

```python title="grid_memo.py"
from functools import lru_cache

def grid_memo(grid):
    rows, cols = len(grid), len(grid[0])

    @lru_cache(maxsize=None)
    def go(r, c):
        if r >= rows or c >= cols or grid[r][c] is None:
            return NEG
        if r == rows - 1 and c == cols - 1:
            return grid[r][c]
        best = max(go(r + 1, c), go(r, c + 1))
        return NEG if best == NEG else grid[r][c] + best

    result = go(0, 0)
    go.cache_clear()
    return result
```

`grid` 는 리스트의 리스트라 `lru_cache` 키에 직접 넣을 수 없으므로, 클로저로 캡처하고 **좌표 `(r, c)` 만 인자**로 넘긴다. [7.20절](#/dp-basics)에서 짚은 "해시 불가능한 인자는 클로저나 전역으로 빼고 인덱스만 넘겨라"는 조언이 여기서 실전으로 쓰인다.

```python title="grid_tab.py — 반복, 오른쪽 아래에서 왼쪽 위로"
def grid_tab(grid):
    rows, cols = len(grid), len(grid[0])
    dp = [[NEG] * cols for _ in range(rows)]
    for r in range(rows - 1, -1, -1):       # 아래에서 위로
        for c in range(cols - 1, -1, -1):   # 오른쪽에서 왼쪽으로
            if grid[r][c] is None:
                continue
            if r == rows - 1 and c == cols - 1:
                dp[r][c] = grid[r][c]
                continue
            down = dp[r + 1][c] if r + 1 < rows else NEG
            right = dp[r][c + 1] if c + 1 < cols else NEG
            best = max(down, right)
            if best != NEG:
                dp[r][c] = grid[r][c] + best
    return dp[0][0]
```

점화식이 $f(r,c)$ 를 $f(r+1,c)$ 와 $f(r,c+1)$ 로 정의했으므로, `dp[r][c]` 를 채우려면 **`dp[r+1][c]` 와 `dp[r][c+1]` 이 먼저 채워져 있어야 한다.** 그래서 반복문을 `rows-1`, `cols-1` 부터 거꾸로 돈다. 방향을 반대로(왼쪽 위부터) 돌리면 아직 안 채운 미래 값을 읽어서 조용히 틀린 답이 나온다.

::: danger 반복 순서를 점화식과 반대로 짜면 미완성 값을 읽는다
`grid_tab` 의 두 `for` 문을 각각 `range(rows)`, `range(cols)` (정방향)로 바꿔서 실행하면 에러 없이 실행은 되지만 답이 틀린다. `dp[r+1][c]` 자리가 아직 초기값 `NEG` 그대로인 채로 읽히기 때문이다. **타뷸레이션에서는 "이 칸을 채우기 전에 참조하는 칸들이 이미 채워져 있는가"를 항상 먼저 확인하라.** 점화식의 우변에 나오는 인덱스가 전부 "이미 처리된 방향"을 가리키는지 손으로 짚어 보는 습관을 들여라.
:::

### 검증과 실측

무작위 격자 300개(1~5행 × 1~5열, 칸의 25%를 장애물로 배치)에 대해 세 구현을 대조했다.

```text nolines
grid path: mismatches 0/300
```

손으로 짠 두 예시로 감을 확인한다.

```pyrepl
>>> g1 = [[1, 3, 1], [1, 5, 1], [4, 2, 1]]
>>> grid_tab(g1)
12
>>> g2 = [[0, None], [None, 0]]   # 대각선 방향이 둘 다 막혔다
>>> grid_tab(g2)
-inf
```

`g1` 의 최적 경로는 `1 -> 3 -> 5 -> 2 -> 1` 로 합이 12다. `g2` 는 오른쪽도 아래도 막혀서 도달 불가능 — 호출부에서 `-inf` 를 "경로 없음" 메시지로 변환해서 출력해야 한다(위 동전 문제의 `INF` 변환과 같은 함정이다).

```pyrepl
>>> import time
>>> for trial in range(3):
...     row = []
...     for n in (8, 10, 12, 14):
...         g = make_random_grid(n, n, block_prob=0.15)   # 트라이얼마다 새로 뽑은 무작위 n x n 격자
...         t0 = time.perf_counter(); grid_brute(g); t1 = time.perf_counter()
...         row.append(f"{n}x{n}={(t1-t0)*1000:.2f}ms")
...     print(row)
['8x8=0.39ms', '10x10=0.82ms', '12x12=32.11ms', '14x14=55.24ms']
['8x8=0.65ms', '10x10=0.00ms', '12x12=31.57ms', '14x14=21.98ms']
['8x8=0.08ms', '10x10=8.24ms', '12x12=11.84ms', '14x14=4.61ms']
```

(Python 3.14.5 / Windows 기준 실측. `make_random_grid` 는 지정한 확률로 장애물을 섞어 무작위 격자를 만드는 헬퍼다.) 트라이얼마다 순서가 요동친다는 걸 눈으로 볼 수 있다 — 두 번째 트라이얼은 `10x10` 이 `0.00ms` 로 끝나고, 세 번째 트라이얼은 `14x14` 가 `12x12` 보다 오히려 빠르다. 원인은 장애물이 시작점 근처나 핵심 경로를 우연히 막으면 `grid_brute` 가 그 가지에서 즉시 `NEG` 를 반환하며 재귀가 조기에 끊기기 때문이다. 즉 **무작위 장애물이 섞인 벤치마크는 격자 크기와 실행 시간 사이의 관계를 안정적으로 보여주지 못한다** — 한 번의 실행에서 매끈하게 증가하는 숫자가 나왔다고 그걸 일반 패턴으로 단정하면 안 된다.

경로 수 $\binom{2n}{n}$ 이 그리는 순수한 지수 곡선을 보려면 장애물을 아예 없애야 한다(`block_prob=0.0`) — 그래야 모든 경로가 끝까지 살아남아 가지치기가 일어나지 않는다.

```pyrepl
>>> for n in (8, 10, 12, 14, 16):
...     g = make_random_grid(n, n, block_prob=0.0)   # 장애물 없음 — 순수 경로 수만 비교
...     t0 = time.perf_counter(); grid_brute(g); t1 = time.perf_counter()
...     print(f"{n}x{n}", f"{(t1-t0)*1000:.2f} ms")
8x8 1.75 ms
10x10 25.73 ms
12x12 394.62 ms
14x14 7246.32 ms
16x16 115437.86 ms
```

이번에는 재현 가능하다 — $n$ 이 2씩 늘 때마다 10배 이상씩 뛴다. $\binom{16}{8}=12870$, $\binom{32}{16}=601{,}080{,}390$ 이니 $8\times8 \to 16\times16$ 사이에 경로 수가 5만 배 가까이 늘어난 것과 실측 시간이 대략 들어맞는다. **장애물이 섞인 벤치마크와 장애물 없는 벤치마크는 서로 다른 걸 보여준다** — 앞의 표는 "운 나쁘면 얼마나 빨리 끝날 수도 있는가"를, 뒤의 표는 "최악의 경우 얼마나 폭발하는가"를 보여준다. 시험장에서 브루트포스 시간을 어림할 때는 항상 후자(장애물 없는, 혹은 장애물이 경로를 막지 않는) 기준으로 잡아야 안전하다.

```pyrepl
>>> g_big = make_random_grid(500, 500, block_prob=0.1)
>>> for _ in range(4):
...     t0 = time.perf_counter(); grid_tab(g_big); t1 = time.perf_counter()
...     print(f"{(t1-t0)*1000:.3f} ms")
35.304 ms
35.650 ms
34.555 ms
35.133 ms
```

타뷸레이션은 매번 $O(rows \times cols)$ 만큼만 도니 장애물 배치와 무관하게 $500 \times 500 = 250{,}000$ 칸을 항상 34~36 ms 사이로 채운다 — 네 번을 다시 돌려도 편차가 1 ms 안쪽이다. 브루트포스는 장애물이 없으면 $14 \times 14$ 만 해도 이미 7초를 넘기는데, $500 \times 500$ 은 장애물 유무를 떠나 시도할 엄두조차 안 나는 크기다.

::: cote 2차원 DP를 알아채는 신호
"격자에서 이동", "$(i, j)$ 두 값으로 상태가 결정된다", "행/열 제약이 동시에 걸린다"는 문구가 나오면 상태를 `dp[r][c]` 형태의 2차원으로 잡을 후보다. [1.1 객체·이름·참조](#/objects-names)에서 다룬 `[[0]*c for _ in range(r)]` 관용구로 초기화하는 걸 잊지 마라 — `[[0]*c]*r` 로 잘못 초기화하면 모든 행이 같은 리스트를 가리켜서 DP 배열 자체가 오염된다. 격자 DP에서 실제로 자주 나오는 사고다.
:::

## 세 문제를 나란히 놓고 보면 보이는 것

```text nolines
warm-up: 인접 회피 최대합    상태 1차원(i)         분기 2개(고른다/안 고른다)
문제 2 : 동전 교환 최소개수  상태 1차원(금액)      분기 |coins|개(어떤 동전을 쓸까)
문제 3 : 격자 경로 최댓값    상태 2차원(r, c)      분기 2개(아래로/오른쪽으로)
```

분기 수도 다르고 상태 차원도 다르지만, 세 문제 모두 **"마지막 선택이 무엇인가"라는 질문 하나로 점화식을 세웠다.** 이게 [7.20절](#/dp-basics)에서 강조한 4단계 절차가 특정 문제 하나에 국한된 요령이 아니라 DP 전반에 통하는 사고법이라는 증거다. 실전에서 새 DP 문제를 만나면 겁먹지 말고 이 순서를 그대로 밟아라 — 답을 정의하고, 마지막 한 걸음을 묻고, 그 걸음들을 부분 문제로 쪼개고, 기저 조건을 정하고, 브루트포스로 확인한 뒤 메모이제이션을 얹어라.

::: cote 세 문제 모두에서 반복된 검증 습관
브루트포스 코드는 점화식을 그대로 옮긴 것이므로 **정답의 기준점** 역할을 한다. 메모이제이션은 브루트포스에 데코레이터 한 줄만 얹은 것이므로 로직이 그대로 옳다는 게 보장된다. 타뷸레이션만 반복 순서를 새로 설계하므로 **버그가 생길 유일한 지점**이다. 그래서 이 절 세 문제 모두 타뷸레이션 결과를 브루트포스·메모와 대조하는 무작위 검증을 돌렸다 — 시험장에서도 시간이 남으면 이 습관을 그대로 써라. [8.1절](#/cote-strategy)에서 강조한 원칙이 DP에서 특히 중요하다. 최적화된 점화식 코드는 로직이 압축돼 있어서 버그가 나도 조용히 틀린 답만 낼 뿐 에러를 던지지 않는다.
:::

## 요약

- 세 문제 모두 같은 절차로 풀었다 — 브루트포스로 정답 기준을 확보하고, 점화식을 4단계(답 정의 → 마지막 선택 → 부분 문제 → 기저 조건)로 도출하고, `lru_cache` 로 메모이제이션한 뒤, 필요하면 타뷸레이션으로 바꿨다.
- 분기 수와 상태 차원이 문제마다 달라도(2개 대 $|coins|$ 개, 1차원 대 2차원) "마지막 선택이 무엇인가"라는 질문은 항상 점화식을 세우는 열쇠였다.
- `lru_cache` 에 넘기는 인자는 해시 가능해야 한다 — 리스트는 튜플로 바꾸거나, 격자처럼 큰 구조는 클로저로 캡처하고 좌표만 인자로 넘긴다.
- `float("inf")` 를 "불가능"의 표시로 쓸 때는 연산 중에는 안전하지만(`inf + 1 == inf`), 최종 출력 직전에는 반드시 문제가 요구하는 형태(`-1`, "경로 없음" 등)로 변환해야 한다.
- 2차원 타뷸레이션에서는 점화식의 우변이 가리키는 칸이 **이미 채워진 방향**인지 항상 확인하라. 반복 순서를 반대로 짜면 에러 없이 조용히 틀린 답이 나온다.
- 실측 결과 세 문제 모두 브루트포스는 지수 시간으로 무너지고(문제 1: $n$이 4 늘 때마다 7~10배, 문제 2: 금액이 3 늘 때마다 5~10배, 문제 3: $n$이 2 늘 때마다 몇 배씩), 메모이제이션·타뷸레이션은 다항 시간으로 같은 크기를 수 밀리초~수십 밀리초에 처리했다.
