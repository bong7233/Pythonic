# 8.4 문제 유형 분류와 신호 읽기

::: lead
문제를 읽고 5분 안에 "이건 다익스트라다", "이건 비트마스크 DP다"라고 판단하지 못하면 시험 시간의 절반을 헤매다 날린다. 그런데 그 판단은 감이 아니다. 문제 지문에는 항상 두 종류의 신호가 박혀 있다 — **제약 조건**(N이 얼마까지인가)과 **키워드**(무엇을 묻는가). 이 절은 그 두 신호를 읽어서 알고리즘을 역추론하는 절차를 세운다. [7.1 복잡도](#/complexity)에서 빅오를 이해했다면, 이제 그 빅오를 거꾸로 뒤집어 "이 문제는 어떤 빅오까지 통과하는가"를 계산할 차례다.
:::

## 왜 신호부터 읽어야 하는가

흔한 실패 패턴은 이렇다. 문제를 읽자마자 익숙한 접근이 떠오르고, 그대로 코드를 짜기 시작한다. 절반쯤 짰을 때 제약 조건을 다시 보니 $N \le 100{,}000$ 인데 지금 짜고 있는 게 $O(N^2)$ 풀이다. 이미 20분을 썼다. 다시 설계해야 한다.

반대 방향의 실패도 있다. $N \le 12$ 인 문제를 보고 "제약이 작으니 뭐든 되겠지" 하고 지문을 대충 읽은 채 코드를 짜다가, 사실은 순열이 아니라 조합을 요구하는 문제였다는 걸 제출 후에 깨닫는다.

두 실패 모두 같은 원인에서 온다. **제약 조건과 지문을 코드로 옮기기 전에 먼저 "어떤 종류의 문제인가"를 확정하지 않았다.** 이 절차는 순서가 있다.

1. 제약 조건에서 **허용되는 시간복잡도의 상한**을 계산한다.
2. 지문의 키워드에서 **후보 알고리즘**을 뽑는다.
3. 1번과 2번이 맞아떨어지는지 대조한다. 안 맞으면 둘 중 하나를 잘못 읽은 것이다.

이 순서를 거꾸로 하면 안 된다. 키워드만 보고 알고리즘을 정하면, 그 알고리즘이 실제로는 시간 초과인 경우를 놓친다.

## 제약 조건에서 시간복잡도 상한 역산하기

### 감으로 어림잡지 마라 — 실제로 재본 숫자

"1초에 파이썬은 대략 1천만\~1억 번 연산한다"는 말을 들어봤을 것이다. 이게 맞는지, 정확히 어느 쪽인지 실측해보자. 순수한 `for` 루프만 돌리면 이렇다.

```python title="bench_loop.py — 아무 일도 안 하는 루프"
import timeit

for n in (10**6, 10**7, 10**8):
    t = timeit.timeit("for i in range(n): pass", globals={"n": n}, number=1)
    print(f"range({n}) pass만: {t:.4f}s")
```

```text nolines
range(1000000) pass만: 0.0060s
range(10000000) pass만: 0.0585s
range(100000000) pass만: 0.5865s
```

`pass` 만 있는 빈 루프는 1억 번에 0.58초다. 그런데 실제 문제 풀이는 루프 안에서 뭔가를 한다. 덧셈이라도 들어가면 얼마나 느려지는지 $O(N^2)$ 중첩 루프로 재보자.

```python title="bench_n2.py — 실제 연산이 있는 O(N^2)"
for n in (2000, 5000, 10000, 20000):
    code = "total=0\nfor i in range(n):\n    for j in range(n):\n        total += 1"
    t = timeit.timeit(code, globals={"n": n}, number=1)
    print(f"n={n}: n^2={n*n:,} 연산, {t:.4f}s")
```

```text nolines
n=2000: n^2=4,000,000 연산, 0.0503s
n=5000: n^2=25,000,000 연산, 0.3195s
n=10000: n^2=100,000,000 연산, 1.2813s
n=20000: n^2=400,000,000 연산, 5.5266s
```

(Python 3.14.5 / Windows 기준 실측. 각 $N$ 마다 4회 반복 실행한 뒤 평균 낸 값이다 — `timeit(..., number=1)` 은 초 단위 단발 실행이라 특히 $N$ 이 큰 구간에서 백그라운드 부하에 따라 회차마다 5\~10% 정도 흔들린다. 절대값은 채점 서버·기기마다 다르지만 **자릿수 차이**는 어디서나 같다.)

여기서 실전 감각이 나온다. 위 표의 네 구간을 각각 "연산 수 ÷ 걸린 시간"으로 역산하면 $N=2{,}000$ 은 약 79.5M/s, $N=5{,}000$ 은 약 78.2M/s, $N=10{,}000$ 은 약 78.0M/s, $N=20{,}000$ 은 약 72.4M/s — 네 구간 중 8천만(80M)을 넘는 경우는 하나도 없다. **의미 있는 연산(덧셈 포함) 기준으로 파이썬은 초당 대략 7천만\~8천만 번을 처리한다**고 보는 게 실측에 맞다. "1억 번"은 `pass` 뿐인 빈 루프에서나 나오는 숫자지, 덧셈이 들어간 순간부터는 8천만도 못 채운다. $N=10{,}000$ 의 $O(N^2)$ 은 1억 번이라 1.3초 — 시간 제한이 보통 1\~2초인 걸 감안하면 **아슬아슬하게 위험한 크기**다. $N=20{,}000$ 은 4억 번이라 5.5초, 명백한 시간 초과다.

정렬($O(N \log N)$)도 재보면 감이 잡힌다.

```python title="bench_sort.py"
import random
for n in (10**5, 10**6, 5*10**6, 10**7):
    random.seed(0)
    data = [random.random() for _ in range(n)]
    t = timeit.timeit("sorted(data)", globals={"data": data}, number=1)
    print(f"n={n:,}: {t:.4f}s")
```

```text nolines
n=100,000: 0.0091s
n=1,000,000: 0.1164s
n=5,000,000: 0.7405s
n=10,000,000: 1.6409s
```

$N=10^6$ 정렬은 0.12초로 여유롭다. $N=10^7$ 도 1.64초라 정렬 자체는 문제없다 — 정렬 뒤에 $O(N)$ 후처리를 더 붙여도 감당된다는 뜻이다.

지수·팩토리얼 폭발도 실측으로 보면 왜 "지수는 N이 작을 때만"인지 체감된다.

```python title="bench_exp.py — 2^N 부분집합 순회 vs N! 순열 생성"
for n in (15, 20, 22, 25):
    code = 'c=0\nfor mask in range(1<<n):\n    c += bin(mask).count("1")'
    t = timeit.timeit(code, globals={"n": n}, number=1)
    print(f"n={n}: 2^n={1<<n:,}, {t:.4f}s")

import itertools, math
for n in (8, 9, 10, 11):
    t = timeit.timeit(
        "sum(1 for _ in itertools.permutations(range(n)))",
        globals={"n": n, "itertools": itertools}, number=1,
    )
    print(f"n={n}: n!={math.factorial(n):,}, {t:.4f}s")
```

```text nolines
n=15: 2^n=32,768, 0.0019s
n=20: 2^n=1,048,576, 0.0678s
n=22: 2^n=4,194,304, 0.2716s
n=25: 2^n=33,554,432, 2.2469s

n=8: n!=40,320, 0.0013s
n=9: n!=362,880, 0.0122s
n=10: n!=3,628,800, 0.1254s
n=11: n!=39,916,800, 1.5044s
```

(위 두 벤치마크도 각각 3회 반복 평균이다. $11!$ 은 회차마다 1.50\~1.51초 사이로 비교적 안정적으로 재현됐다.)

$2^{20}$ 은 0.07초로 안전하지만 $2^{25}$ 은 2.2초로 넘어간다. $10!$ 순열은 0.13초인데 $11!$ 은 1.5초 — **팩토리얼은 N이 1만 늘어도 완전히 다른 세계**로 간다. 이게 순열 기반 완전 탐색을 $N \le 10$ 근방에서만 쓰는 이유다.

### N에서 알고리즘으로 — 역산표

위 네 벤치마크는 $O(N)$, $O(N^2)$, $O(N \log N)$, $O(2^N)$, $O(N!)$ 구간을 다뤘다. 아래 역산표에는 $O(N^3)$, $O(N^4)$ 행도 들어가는데, 이 둘은 코테에서 흔히 등장하는 상한(플로이드-워셜의 $N \le 500$, 4중 루프 DP의 $N \le 100$)이므로 같은 방식으로 직접 재봐야 표에 넣을 자격이 생긴다.

```python title="bench_n3n4.py — O(N^3), O(N^4) 삼중·사중 루프"
for n in (500,):
    code = "total=0\nfor i in range(n):\n    for j in range(n):\n        for k in range(n):\n            total += 1"
    t = timeit.timeit(code, globals={"n": n}, number=1)
    print(f"n={n}: n^3={n**3:,} 연산, {t:.4f}s")

for n in (100,):
    code = ("total=0\nfor i in range(n):\n    for j in range(n):\n"
            "        for k in range(n):\n            for l in range(n):\n                total += 1")
    t = timeit.timeit(code, globals={"n": n}, number=1)
    print(f"n={n}: n^4={n**4:,} 연산, {t:.4f}s")
```

```text nolines
n=500: n^3=125,000,000 연산, 1.4532s
n=100: n^4=100,000,000 연산, 1.0591s
```

($O(N^2)$ 구간의 실측과 같은 초당 약 8천만 번 처리량으로 설명된다 — $N^3=1.25$ 억, $N^4=1$ 억이니 시간도 그 비율로 나온다.) 이제 이 둘을 포함한 다섯 종류의 실측을 종합하면, 문제 지문의 $N$ 만 보고도 "이 정도 복잡도까지는 통과, 이 이상은 시간 초과"를 계산할 수 있다.

| $N$ 의 크기 | 허용 복잡도 | 대표 알고리즘 |
| --- | --- | --- |
| $N \le 11$ | $O(N!)$, $O(N^N)$ | 순열 완전 탐색, 외판원 문제류 완전 탐색 |
| $N \le 20 \sim 22$ | $O(2^N \cdot N)$ | 비트마스크 DP ([7.24 비트마스크](#/bitmask)), 부분집합 완전 탐색 |
| $N \le 100$ | $O(N^4)$, $O(N^3 \log N)$ | 플로이드-워셜([7.15 최단 경로](#/shortest-path)) + 후처리, 4중 루프 DP |
| $N \le 500$ | $O(N^3)$ | 플로이드-워셜, 구간 DP([7.21 DP 심화](#/dp-advanced)) |
| $N \le 2{,}000 \sim 5{,}000$ | $O(N^2)$, $O(N^2 \log N)$ | 단순 DP 전이, 이중 루프 그리디, 크루스칼의 나쁜 구현 |
| $N \le 100{,}000$ | $O(N \log N)$ | 정렬 기반 그리디([7.19 그리디](#/greedy)), 힙 기반 다익스트라([7.15 최단 경로](#/shortest-path)), 세그먼트 트리([7.25 세그먼트 트리](#/segment-tree)) |
| $N \le 1{,}000{,}000$ | $O(N \log N)$, $O(N)$ | 위상 정렬([7.17 위상 정렬](#/topological)), BFS/DFS([7.14 BFS/DFS](#/bfs-dfs)), 투 포인터([7.3 투 포인터](#/two-pointers)) |
| $N \le 10^7 \sim 10^8$ | $O(N)$, 아주 작은 상수 | 누적합, 카운팅, 단순 순회 — 파이썬이면 이 구간부터 상수 자체가 위협이 된다 |

::: warn 표는 "허용되는 상한"이지 "이 크기면 이 알고리즘을 써도 안전하다"가 아니다
$N \le 100{,}000$ 이라고 $O(N \log N)$ 이 항상 통과하는 게 아니다. 위 실측에서 봤듯 **파이썬의 $O(N \log N)$ 상수는 C++보다 몇 배 크다.** [7.1 복잡도](#/complexity)와 [8.3 시간 초과](#/tle)에서 다뤘듯, 표는 "이 이상 복잡도는 확실히 죽는다"는 하한선으로 쓰고, 경계에 걸리면 반드시 직접 실측하거나 상수를 줄이는 관용구([8.3](#/tle))를 적용하라.
:::

::: cote 시간 제한이 명시적으로 크면 파이썬 페널티를 의심하라
어떤 채점 시스템은 파이썬에 시간 제한을 2\~3배 더 준다. 문제의 시간 제한이 유독 후하게 느껴지면(예: 다른 언어 기준 1초인데 파이썬만 3초) 이건 우연이 아니라 그 채점 시스템의 정책일 가능성이 높다. 다만 **모든 채점 시스템이 이렇게 해주는 건 아니고, 배율도 시스템마다 다르다.** 확실하지 않으면 "시간 제한이 넉넉하니 $O(N^2)$ 도 되겠지"라고 단정하지 말고, 표의 상한대로 설계한 뒤 여유가 있으면 넘어가라.
:::

## 키워드에서 알고리즘으로 — 지문이 흘리는 힌트

제약 조건이 "얼마나 빨라야 하는가"를 알려준다면, 키워드는 "어떤 구조의 문제인가"를 알려준다. 아래는 자주 나오는 매칭이다.

| 지문의 표현 | 신호가 가리키는 알고리즘 | 왜 그런가 |
| --- | --- | --- |
| "최단 거리", "최소 비용으로 도달" (가중치 있음, 음수 없음) | 다익스트라 ([7.15 최단 경로](#/shortest-path)) | 간선 가중치가 있는 그래프의 단일 시작점 최단 경로 = 다익스트라의 정의 그 자체 |
| "최단 거리" + 음수 가중치 언급 | 벨만-포드 ([7.15 최단 경로](#/shortest-path)) | 다익스트라는 음수 간선에서 정당성이 깨진다 |
| "모든 정점 쌍 사이의 최단 거리" + $N$ 작음 | 플로이드-워셜 ([7.15 최단 경로](#/shortest-path)) | 전체 쌍이 필요하면 $N$ 번의 다익스트라보다 $O(N^3)$ 하나가 단순하다 |
| "최댓값을 최소화", "~개를 골라서 최적화", "일부만 선택해서 합/개수 최적화" | DP 또는 그리디 ([7.20 DP 기초](#/dp-basics), [7.19 그리디](#/greedy)) | "선택 + 최적화"는 부분 문제로 쪼개지는 구조를 의심하게 만든다. 그리디가 통하려면 **교환 논증으로 정당성을 증명**할 수 있어야 한다 — 증명이 안 되면 DP로 간다 |
| "같은 그룹인가", "네트워크가 연결되어 있는가", "합치기/분리하기 질의가 반복" | 유니온 파인드 ([7.12 유니온 파인드](#/union-find)) | "그룹핑" + "합치기 연산이 여러 번 반복"이 핵심이다. 그룹을 한 번만 나누면 그냥 BFS/DFS 연결 요소로 충분할 수도 있다 |
| "인접한 것들을 묶어라", "영역의 개수", "퍼진 범위" | BFS/DFS 플러드 필 ([7.14 BFS/DFS](#/bfs-dfs)) | 그래프/격자 위의 연결 요소 탐색 |
| "시작하는 단어를 찾아라", "자동완성", "접두사로 그룹핑" | 트라이 ([7.11 트라이](#/trie)) | [7.11](#/trie)에서 다룬 것처럼 `set` 은 $O(N \cdot L)$, 트라이는 $O(L)$ |
| "순서를 정하라", "선행 조건이 있다", "먼저 해야 하는 작업" | 위상 정렬 ([7.17 위상 정렬](#/topological)) | 선후 관계 = DAG의 순서, 위상 정렬의 정의와 정확히 일치 |
| "$k$ 번째로 작은/큰 것", "최댓값을 계속 꺼내야 한다" | 힙 ([7.8 힙](#/heap)) | 매번 최댓값/최솟값을 꺼내는 반복 연산 = 우선순위 큐의 존재 이유 |
| "구간 합", "구간 최댓값", "갱신하면서 질의도 반복" | 세그먼트 트리 / 펜윅 트리 ([7.25 세그먼트 트리](#/segment-tree)) | 갱신과 질의가 섞여서 반복되면 누적합만으로는 매 갱신마다 $O(N)$ 이 든다 |
| "부분 문자열 검색", "패턴이 몇 번 나오는가" | KMP / 라빈-카프 ([7.22 문자열 알고리즘](#/string-algo)) | 단순 매칭은 $O(N \cdot M)$, 이 알고리즘들은 $O(N+M)$ |
| "모든 경우의 수", "가능한 조합을 다 나열" + $N$ 작음 | 백트래킹 ([7.18 백트래킹](#/backtracking)) | "다 나열"이 명시적이면 완전 탐색이 정답 그 자체다. 가지치기로 상수를 줄인다 |
| "최소 비용으로 모두 연결" | 최소 신장 트리 ([7.16 최소 신장 트리](#/mst)) | "모두 연결" + "비용 최소"는 MST 문제의 정의 |
| "점들의 위치", "선분이 만나는가", "영역의 넓이" | 기하 알고리즘 ([7.26 기하](#/geometry)) | 좌표가 등장하고 교차·포함·넓이를 묻는 순간 CCW/볼록 껍질을 의심한다 |

::: note 이 표는 "1차 후보"를 좁히는 도구다
지문에 "최단 거리"가 있다고 무조건 다익스트라는 아니다. 격자에서 모든 간선 가중치가 1이면 그냥 BFS로 충분하고 다익스트라보다 빠르다([7.14](#/bfs-dfs)의 0-1 BFS도 그 사이 어딘가다). 표는 후보를 줄여줄 뿐, 최종 확정은 다음 절에서 다룰 "충돌 해소"까지 거쳐야 한다.
:::

## 여러 힌트가 충돌할 때 — 우선순위

실전 문제는 이 표 하나로 안 끝난다. 지문 하나에서 여러 키워드가 동시에 잡히거나, 키워드와 제약 조건이 서로 다른 답을 가리킨다. 이럴 때 순서가 있다.

### 원칙 1: 제약 조건이 키워드를 이긴다

"최적화"라는 단어를 보고 그리디를 떠올렸는데 $N \le 20$ 이면, 그리디의 정당성을 굳이 증명하려 애쓰지 마라. 그 크기는 **비트마스크 DP나 완전 탐색으로 확실하게 풀라는 뜻**이다. 반대로 "최단 거리"를 보고 플로이드-워셜을 쓰려는데 $N \le 100{,}000$ 이면, $O(N^3)$ 은 절대 못 돈다 — 다익스트라로 가야 한다. **제약 조건은 거짓말을 하지 않는다.** 키워드 해석이 여러 갈래로 갈릴 때는 항상 제약 조건이 캐스팅보트를 쥔다.

### 원칙 2: "그룹핑"은 union-find와 BFS/DFS 중 무엇이 반복 합치기를 요구하는지로 가른다

지문이 "이미 나눠진 그룹들을 보고 답을 구하라"면 BFS/DFS 연결 요소 한 번으로 끝난다. 지문이 "질의가 여러 번 들어오는데 그때그때 두 원소를 합쳐야 한다"거나 "간선을 하나씩 추가하면서 사이클을 판정해야 한다"(크루스칼이 바로 이 패턴)면 유니온 파인드가 필요하다. 판별 기준은 **"합치기 연산이 여러 번, 순차적으로 일어나는가"** 다.

### 원칙 3: "최적화"라는 단어는 DP인지 그리디인지 스스로 말해주지 않는다

그리디가 통하려면 **국소 최적 선택이 전역 최적을 보장한다는 걸 증명**할 수 있어야 한다([7.19 그리디](#/greedy)의 교환 논증). 증명이 막히거나, 반례가 하나라도 떠오르면 즉시 DP로 넘어가라. 시험장에서 이 판단에 오래 걸리면, **일단 DP로 접근하는 게 안전한 기본값**이다. DP는 그리디보다 항상 느리지만 항상 맞다. 반대로 그리디가 맞다는 확신이 있고 제약 조건이 $O(N \log N)$ 이상을 요구하면(예: $N \le 10^6$) 그리디 쪽이 유일한 답일 가능성이 높다 — DP로는 그 크기를 감당 못 하기 때문이다.

### 원칙 4: 지문에 나온 것보다 뒤에 숨은 제약이 진짜다

"좌표는 정수이고 $-10^9 \le x, y \le 10^9$" 같은 조항은 좌표 압축이나 오버플로 방지가 필요하다는 힌트다. "간선에 가중치가 없다"는 조항이 안 보이면 다익스트라를, 명시적으로 "모든 가중치가 1"이라고 되어 있으면 BFS로 좁혀야 한다. 제약 조건 섹션을 표面적인 $N$ 범위만 보지 말고 **자료형 범위, 특수 케이스 언급, 입력 형식**까지 전부 훑어라.

```text nolines
문제 지문을 읽는 순서
  |
  |-- 1. N의 범위를 본다        --> 허용 복잡도의 상한을 계산한다
  |
  |-- 2. 키워드를 뽑는다         --> 후보 알고리즘 목록을 만든다
  |
  |-- 3. 1과 2를 대조한다
  |      |-- 맞아떨어진다        --> 확정, 코드 작성 시작
  |      `-- 안 맞는다           --> 둘 중 하나를 잘못 읽었다. 다시 읽어라
  |
  `-- 4. 자료형 범위·특수 케이스·입력 형식을 마지막으로 훑는다
         --> 오버플로, 좌표 압축, 예외 케이스를 놓치지 않는다
```

## 종합 사례: 신호를 조합해서 유형을 확정하기

실제로 이 절차를 밟아보자. 아래는 자기완결적으로 구성한 예제 지문이다(특정 대회 문제를 그대로 옮긴 게 아니라, 흔한 유형을 재구성한 것이다).

::: cote 예제 지문
> $N$ 개의 도시가 있다 ($2 \le N \le 100{,}000$). 이 중 일부 쌍 사이에는 양방향 도로가 있고, 도로마다 통행 비용이 든다(비용은 항상 양수). 질의가 $Q$ 개 주어진다 ($1 \le Q \le 1{,}000$). 각 질의는 두 도시 $a$, $b$ 로 이루어지며, $a$ 에서 $b$ 까지 가는 데 드는 최소 비용을 출력하라. 모든 질의는 같은 시작 도시 $s$ 에서 시작한다.
:::

이 지문을 절차대로 분해한다.

1. **N의 범위**: $N \le 100{,}000$. 표에서 이 구간은 $O(N \log N)$ 이 상한이다. $O(N^2)$ 이나 $O(N^3)$ 은 즉시 배제된다 — 플로이드-워셜은 $10^{15}$ 번 연산이라 논외다.
2. **키워드**: "최소 비용", "양방향", "비용은 항상 양수"(음수 간선 없음), "질의가 여러 개지만 시작 도시가 같다". 위 표에서 "최단 거리 + 음수 없음"은 다익스트라를 가리킨다.
3. **대조**: 다익스트라는 $O((V+E) \log V)$ 다. $N \le 100{,}000$ 구간과 맞아떨어진다. **게다가 "모든 질의가 같은 시작 도시"라는 조건이 결정적이다.** 다익스트라를 시작 도시 $s$ 에 대해 **딱 한 번만** 돌리면 $s$ 에서 모든 도시까지의 최단 거리가 한 번에 나온다. $Q=1{,}000$ 번의 질의는 그 결과 배열을 $O(1)$ 씩 조회하는 것으로 끝난다. 만약 질의마다 시작 도시가 달랐다면 다익스트라를 $Q$ 번 돌려야 했을 것이고, 그러면 $O(Q \cdot (V+E)\log V)$ 를 다시 계산해야 한다.
4. **숨은 제약**: 도로 개수(간선 수)가 지문에 명시되지 않았다면, 완전 그래프 최악의 경우($O(N^2)$ 개 간선)까지 고려해야 하는지 반드시 확인한다. 여기서는 생략했지만 실제 문제라면 간선 수의 상한이 항상 따로 주어진다 — 그게 없으면 인접 행렬 자체가 메모리 초과다.

이 네 단계를 거치면 "다익스트라를 한 번만 돌리고 결과를 재사용한다"는 확정된 설계가 나온다. 이 설계를 검증하는 완결된 풀이는 [7.15 최단 경로](#/shortest-path)에서 이미 다뤘다 — 여기서는 **그 알고리즘을 선택하기까지의 추론 과정**만 반복했다.

::: warn "질의가 여러 개"라는 문구를 놓치면 상수를 10배 이상 낭비한다
같은 시작점에서 반복되는 질의를 매번 새로 계산하는 것은 흔한 낭비다. 지문에서 "질의", "여러 번 묻는다", "재사용"같은 단어가 나오면 **"한 번 전처리해서 여러 번 조회"** 패턴을 우선 의심하라. 세그먼트 트리, 다익스트라의 전체 최단 거리 배열, 유니온 파인드가 전부 이 패턴이다.
:::

## 요약

- 문제를 코드로 옮기기 전에 **제약 조건 → 시간복잡도 상한**, **키워드 → 알고리즘 후보** 를 각각 뽑고 대조한다.
- 실측 기준(3.14.5 / Windows), 파이썬은 의미 있는 연산 기준 초당 약 7천만\~8천만 번 처리한다(빈 루프만의 "1억 번"과는 다른 숫자다). $O(N^2)$ 은 $N \approx 10{,}000$ 부터 위험해지고, $2^N$ 은 $N \approx 22$, $N!$ 은 $N \approx 10\sim11$ 에서 각각 한계에 부딪힌다.
- $N$ 의 크기 하나만으로 대략적인 알고리즘 계열이 정해진다 — $N \le 20$ 지수/완전탐색, $N \le 5{,}000$ 은 $O(N^2)$, $N \le 10^5\sim10^6$ 은 $O(N \log N)$.
- 키워드는 "최단거리→다익스트라", "그룹핑+반복합치기→유니온 파인드", "선택+최적화→DP/그리디" 처럼 후보를 좁혀주지만, 세부 조건(음수 가중치 유무, 가중치 균일 여부, 반복 질의 유무)까지 봐야 확정된다.
- 힌트가 충돌하면 **제약 조건이 이긴다.** 그리디처럼 보여도 $N$ 이 작으면 완전 탐색/DP로 가는 게 안전하고, 다익스트라가 맞아도 $N$ 이 매우 크면 세부 구현(우선순위 큐, 인접 리스트)의 상수까지 신경 써야 한다.
- 지문 표면의 $N$ 범위만 보지 말고 자료형 범위, 특수 케이스 조항, "같은 시작점에서 반복 질의" 같은 문구까지 훑어야 최적 설계가 나온다.

::: quiz 연습문제
1. $N \le 15$ 인 문제에서 "물건을 골라서 무게 합이 특정 값 이하가 되도록 가치의 합을 최대화하라"는 지문을 봤다. 이 크기라면 배낭 DP([7.21](#/dp-advanced))와 비트마스크 완전 탐색 중 어느 쪽이 더 안전한 선택인지, 이 절의 실측 표를 근거로 답하라.
2. $N \le 300{,}000$ 인 그래프 문제에서 "모든 정점 쌍의 최단 거리를 구하라"는 요구가 나왔다면, 이 요구 자체가 이상하다고 의심해야 하는 이유를 이 절의 역산표로 설명하라. (힌트: $N^2$ 개의 답을 출력하는 것 자체가 가능한 크기인가?)
3. 지문에 "집합을 합치는 질의"와 "이미 두 원소가 같은 집합인지 묻는 질의"가 섞여서 총 $10^5$ 번 반복된다고 한다. 왜 이게 BFS/DFS가 아니라 유니온 파인드를 가리키는 신호인지 설명하라.
4. 직접 `timeit` 으로 $O(N \log N)$ 정렬과 $O(N^2)$ 이중 루프를 $N=8{,}000$ 에서 각각 실행 시간을 재고, 시간 제한이 1초라면 어느 쪽이 통과하는지 판단하라.
5. "최적화"라는 단어가 들어간 문제에서 그리디의 정당성을 증명하지 못했을 때, 시험 시간이 얼마 안 남았다면 어떤 대안을 택해야 하는지 이 절의 원칙 3을 근거로 답하라.
:::
