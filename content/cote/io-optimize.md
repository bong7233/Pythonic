# 8.2 파이썬 입출력 최적화

::: lead
알고리즘은 $O(n \log n)$ 으로 맞게 짰는데 시간 초과가 난다면, 범인은 입출력일 확률이 높다. 파이썬의 `input()` 과 `print()` 는 C++의 `cin`/`cout` 이나 Java의 `Scanner` 보다 원래 느리게 설계돼 있다. 문제는 이게 알고리즘 복잡도와 무관하게 **상수 배로 시간을 깎아 먹는다**는 점이다. 이 절에서는 `input()` 이 왜 느린지 실제로 측정하고, `sys.stdin`/`sys.stdout` 으로 그 상수를 얼마나 줄일 수 있는지 숫자로 보여준다. [7.1 복잡도](#/complexity)에서 배운 "상수의 무게"가 입출력에서 가장 노골적으로 드러나는 지점이다.
:::

## `input()` 이 정말 느린가 — 실측

먼저 확인하자. 30만 줄짜리 정수 파일을 만들고, 한 줄씩 정수로 읽어 합을 구하는 똑같은 작업을 세 가지 방식으로 해 봤다.

```python title="bench_input.py — input()"
import sys, time

N = 300_000
t0 = time.perf_counter()
total = 0
for _ in range(N):
    total += int(input())
t1 = time.perf_counter()
print(f"time={t1 - t0:.4f}s", file=sys.stderr)
```

```python title="bench_readline.py — sys.stdin.readline()"
import sys, time

N = 300_000
t0 = time.perf_counter()
readline = sys.stdin.readline
total = 0
for _ in range(N):
    total += int(readline())
t1 = time.perf_counter()
print(f"time={t1 - t0:.4f}s", file=sys.stderr)
```

```python title="bench_readall.py — sys.stdin.read().split()"
import sys, time

t0 = time.perf_counter()
data = sys.stdin.read().split()
total = sum(int(x) for x in data)
t1 = time.perf_counter()
print(f"n={len(data)} time={t1 - t0:.4f}s", file=sys.stderr)
```

세 스크립트 모두 같은 30만 줄 입력 파일을 표준 입력으로 리다이렉트해서 3회씩 돌렸다. 세 방식의 `total` 값이 전부 정확히 일치한다는 것도 확인했다 — 속도만 다르고 결과는 같다는 뜻이다.

```text nolines
input()                 : 0.086s, 0.097s, 0.095s   (평균 약 0.092s)
sys.stdin.readline()    : 0.043s, 0.035s, 0.033s   (평균 약 0.037s)
sys.stdin.read().split(): 0.022s, 0.023s, 0.023s   (평균 약 0.022s)
```

::: perf 30만 줄 기준 — input() 대비 4배 이상
(Python 3.14.5 / Windows 기준 실측. 절대 시간은 기기·환경마다 다르지만 **`input()` → `readline()` → `read().split()` 순으로 빨라지는 순서와 대략적인 배수**는 어디서나 재현된다.)

`readline()` 은 `input()` 보다 약 2.5배, `read().split()` 은 `input()` 보다 약 4.2배 빠르다. 10만~100만 줄짜리 입력이 걸린 문제에서 이 차이는 그대로 몇 초 단위로 불어난다. 백준 기준 파이썬 시간 제한이 보통 C++의 2~3배로 넉넉히 주어지는 것도 이런 상수 차이를 감안한 것이지, 입출력을 아무렇게나 짜도 된다는 뜻은 아니다.
:::

### `input()` 이 왜 느린가

`input()` 은 그냥 한 줄 읽는 함수가 아니다. 내장 함수 `input()` 은 다음을 매번 한다.

1. 인자로 받은 프롬프트 문자열이 있으면 출력한다(없어도 이 분기를 탄다).
2. `sys.stdin` 이 진짜 터미널인지 검사한다.
3. 터미널이면 GNU readline 같은 라인 편집 기능(방향키 히스토리 등)을 켠 상태로 입력을 받는다.
4. 끝의 개행 문자를 제거해서 반환한다.

코딩테스트 채점 환경에서는 표준 입력이 터미널이 아니라 파일이나 파이프이므로 3번은 실질적으로 안 쓰인다. 그런데도 `input()` 은 매 호출마다 이 판단 자체를 반복한다. 반면 `sys.stdin.readline()` 은 그런 판단 없이 **버퍼에서 다음 개행 문자까지를 그대로 반환하는** 훨씬 얇은 연산이다.

```text nolines
표준 입력이 파일일 때 실제로 지나가는 경로

OS 파일 버퍼
   │
   ▼
BufferedReader (바이트 단위, 128KB 안팎 블록으로 미리 읽어 둔다)
   │
   ▼
TextIOWrapper (바이트 → 문자 디코딩, 개행 정규화)
   │
   ├─▶ sys.stdin.readline()   : 여기서 끝. 한 줄만 반환한다
   └─▶ input()                : 위 과정 + 프롬프트/터미널 판단 한 겹 더
```

이 경로 자체는 `input()` 이든 `readline()` 이든 공유한다. 차이는 `input()` 이 그 위에 얹는 **한 겹**이고, 그 한 겹이 300,000번 반복되면 무시 못 할 크기로 쌓인다.

::: note 128KB라는 숫자, 어디서 나왔나
`python -c "import io; print(io.DEFAULT_BUFFER_SIZE)"` 를 돌려서 직접 확인한 값이다(Python 3.14.5 기준 131072바이트 = 128KB). 옛날 CPython에서는 이 값이 8192바이트(8KB)였던 시절이 있어서 지금도 "표준입력은 8KB 단위로 버퍼링된다"는 설명이 인터넷에 남아 있는데, 지금 버전에서는 틀린 수치다. 버퍼 크기 자체가 커진다고 `readline()` 의 상대적 우위가 사라지는 건 아니다 — 어차피 `input()` 과 `readline()` 은 같은 버퍼를 공유하므로, 이 절에서 비교하는 속도 차이의 원인이 아니다.
:::

::: note `int()` 는 공백과 개행을 알아서 무시한다
`readline()` 은 줄 끝의 `\n` 을 제거하지 않고 그대로 돌려준다. 그런데 `int("123\n")` 은 정상적으로 `123` 을 반환한다. `int()` 가 앞뒤 공백류 문자를 무시하고 파싱하기 때문이다. 그래서 정수 하나만 읽을 때는 `.rstrip()` 을 따로 안 해도 된다. 다만 **문자열 자체**를 읽어서 비교해야 하는 문제라면 개행이 남아 있으면 비교가 깨지므로 `.strip()` 이 필요하다.
:::

## 한 줄씩 읽지 말고 한 번에 다 읽어라

`readline()` 도 충분히 빠르지만, 더 빠른 방법은 **한 줄씩 읽는 행위 자체를 없애는 것**이다. `sys.stdin.read()` 는 EOF까지 표준 입력 전체를 문자열 하나로 가져온다. 여기에 `.split()` 을 걸면 공백(스페이스와 개행 모두)을 기준으로 쪼갠 토큰 리스트가 나온다 — **줄 구분을 완전히 무시한다.**

이게 왜 유용하냐면, 입력 형식이 "첫 줄에 $N$, 둘째 줄에 정수 $N$개, 그다음 $N-1$줄에 간선"처럼 줄마다 토큰 개수가 다를 때도 신경 쓸 필요가 없어지기 때문이다. 전체를 토큰 스트림으로 보고 필요한 만큼만 순서대로 꺼내 쓰면 된다.

```python title="read_all_pattern.py — 이터레이터로 토큰을 순서대로 소비"
import sys

data = iter(sys.stdin.read().split())
n = int(next(data))
arr = [int(next(data)) for _ in range(n)]
edges = [(int(next(data)), int(next(data))) for _ in range(n - 1)]
```

다음 입력으로 실제 돌려서 확인했다.

```text nolines
5
3 1 4 1 5
1 2
2 3
3 4
4 5
```

```pyrepl
>>> n
5
>>> arr
[3, 1, 4, 1, 5]
>>> edges
[(1, 2), (2, 3), (3, 4), (4, 5)]
```

줄바꿈 위치와 무관하게 정확히 파싱된다. `next(data)` 를 원하는 순서로 호출하기만 하면 되므로, 입력 형식이 복잡한 문제일수록 이 패턴이 `readline()` 을 줄 단위로 세는 것보다 실수가 적다.

::: cote 실전에서 가장 많이 쓰는 템플릿
```python
import sys
input = sys.stdin.readline
```
파일 맨 위에 이 한 줄을 넣고 이후 코드는 그냥 `input()` 을 쓰는 관용구가 백준 파이썬 제출 코드의 사실상 표준이다. 전역 이름 `input` 을 `sys.stdin.readline` 로 재바인딩하면, 기존에 `input()` 으로 짜 둔 코드를 고치지 않고도 속도만 얻는다.

직접 측정한 결과 이 방식은 30만 줄 기준 평균 0.037초로, `readline` 을 지역 변수로 캐싱했을 때(0.037초)와 거의 같고 `input()` 원본(0.092초)보다 뚜렷이 빠르다. **함수 자체가 얇아진 것이지 지역/전역 변수 조회 차이는 이 정도 규모에서 무시할 만하다.**

단, 이 관용구를 쓰면 문자열 끝에 `\n` 이 남는다는 것과, `input()` 과 달리 EOF에서 예외 대신 빈 문자열 `''` 을 반환한다는 것을 반드시 알아야 한다. 아래 danger 박스에서 다룬다.
:::

::: danger `input()` 과 `readline()` 은 EOF에서 다르게 행동한다
줄 수를 모르고 "입력이 끝날 때까지" 읽어야 하는 문제에서 이 차이를 모르면 그대로 예외가 난다.

```pyrepl
>>> import sys
>>> sys.stdin.readline()   # 데이터가 더 없을 때
''                          # 예외 없이 빈 문자열
>>> input()                 # 데이터가 더 없을 때
Traceback (most recent call last):
  ...
EOFError: EOF when reading a line
```

`input()` 은 EOF에서 `EOFError` 를 던진다. `readline()` 은 예외 없이 빈 문자열 `''` 을 돌려준다(주의: 빈 줄 `'\n'` 과는 다르다 — 빈 줄은 최소한 개행 문자 하나가 있는 non-empty 문자열이다). 그래서 줄 수를 모르는 입력을 반복문으로 읽을 때는 이렇게 써야 한다.

```python
import sys

for line in sys.stdin:          # ✅ EOF에서 자동으로 반복이 끝난다
    ...

# 또는
while True:
    line = sys.stdin.readline()
    if not line:                 # ✅ EOF는 '' 이므로 False
        break
```

`input = sys.stdin.readline` 로 재바인딩해 놓은 상태에서 옛날 습관대로 `try/except EOFError` 로 종료 조건을 짜면 절대 걸리지 않는다. 재바인딩을 쓸 거면 종료 조건도 같이 바꿔야 한다.
:::

## 출력도 똑같은 문제를 가진다

읽기에서 본 패턴이 쓰기에도 그대로 반복된다. `print()` 를 반복 호출하는 것과, 한 번에 모아서 쓰는 것 사이에 같은 크기의 격차가 있다.

```python title="bench_output.py — 세 가지 출력 방식"
import sys, time

N = 300_000
nums = list(range(N))

# 방식 1: print() 반복 호출
for x in nums:
    print(x)

# 방식 2: sys.stdout.write() 반복 호출
w = sys.stdout.write
for x in nums:
    w(str(x))
    w("\n")

# 방식 3: 리스트를 모아서 한 번에 join → 단일 write
sys.stdout.write("\n".join(map(str, nums)) + "\n")
```

세 방식 모두 표준 출력을 파일로 리다이렉트해서 3회씩 측정했고, 나온 출력 파일이 바이트 단위로 동일함을 `diff` 로 확인했다.

```text nolines
print() 반복             : 0.104s, 0.091s, 0.095s   (평균 약 0.097s)
sys.stdout.write() 반복  : 0.081s, 0.083s, 0.074s   (평균 약 0.079s)
join 후 단일 write       : 0.0215s, 0.0210s, 0.0207s (평균 약 0.021s)
```

::: perf print() 반복 대비 join은 약 4.6배
(Python 3.14.5 / Windows 기준 실측.) `print()` 를 30만 번 호출하는 것과 `join` 으로 만든 문자열 하나를 한 번에 쓰는 것 사이에 4배 넘는 차이가 난다. `sys.stdout.write()` 를 직접 반복 호출해도 `print()` 보다는 빠르지만(약 1.2배) `join` 에는 크게 못 미친다 — 진짜 병목은 "함수 이름이 `print` 냐 `write` 냐"가 아니라 **호출 횟수 자체**이기 때문이다. `print()` 는 매 호출마다 인자 처리, `sep`/`end` 키워드 해석, 개행 추가를 다시 하고, `write()` 를 직접 불러도 여전히 30만 번의 파이썬 함수 호출 오버헤드가 남는다. `join` 은 그 호출을 **한 번**으로 줄인다.
:::

절충안도 있다. 리스트를 명시적으로 `join` 하기 귀찮으면 `print()` 의 `sep` 인자를 활용해 한 번에 넘기는 방법이다.

```python
print(*nums, sep="\n")
```

측정해 보면 평균 0.065초 정도로, `print()` 반복보다는 빠르지만(호출이 한 번이라서) `join` + 단일 `write` 보다는 느리다 — `print` 내부에서 여전히 `sep` 을 문자열들 사이에 끼워 넣는 처리를 한 번 더 거치기 때문이다. **결과가 정말 크면(수만 줄 이상) `join` 이 가장 안전한 선택이고, 그 정도가 아니면 `print(*nums, sep="\n")` 로 충분하다.**

::: tip 대용량 출력에서 실제로 쓰는 패턴
```python
import sys

out = []
for ...:
    out.append(계산결과)

sys.stdout.write("\n".join(map(str, out)) + "\n")
```
결과를 즉시 출력하지 않고 리스트에 모아 뒀다가 마지막에 한 번에 내보내는 패턴이다. 중간에 리스트에 넣는 비용은 `append` 가 상수 시간이라 무시할 만하고([7.2 파이썬 자료구조의 실제 비용](#/py-ds-cost) 참고), 대신 시스템 호출 수준의 출력 횟수를 극적으로 줄인다. `sys.stdout` 도 결국 버퍼링되는 스트림이라 매번 강제로 내려가는 게 아니지만, 그렇더라도 파이썬 함수 호출 오버헤드 자체가 반복 횟수에 비례해 쌓인다는 사실은 변하지 않는다.
:::

## 백준과 프로그래머스는 입출력 형식이 다르다

지금까지 다룬 최적화는 전부 **표준 입출력을 직접 다루는 채점 환경**을 전제로 한다. 그런데 국내에서 가장 많이 쓰는 두 플랫폼의 입출력 모델 자체가 다르다.

| | 백준(BOJ) | 프로그래머스 |
| --- | --- | --- |
| 입력 받는 방식 | `input()`/`sys.stdin` 으로 **직접** 읽는다 | 채점기가 **함수 인자**로 값을 넘겨준다 |
| 출력 방식 | `print()`/`sys.stdout` 으로 **직접** 쓴다 | 함수의 **반환값**으로 채점한다 |
| 이 절 최적화가 적용되나 | 그대로 적용된다 | **입출력 자체가 없다** — 적용할 대상이 없다 |
| 개행 처리 | 직접 신경 써야 한다(트레일링 개행, 줄 수 등) | 신경 쓸 필요 없다 |

::: cote 프로그래머스에서 `sys.stdin` 최적화를 쓰려는 것은 헛수고다
프로그래머스는 함수 시그니처(`def solution(arr, n): ...`)로 이미 파싱된 값을 넘겨준다. `input()` 이나 `sys.stdin` 을 쓸 일이 애초에 없다. 이 절의 내용은 **백준, 코드포스, SWEA 처럼 표준 입출력을 직접 다루는 플랫폼**에서만 의미가 있다. 프로그래머스에서 시간 초과가 난다면 원인은 입출력이 아니라 거의 항상 알고리즘 복잡도다 — [8.3 시간 초과를 피하는 관용구](#/tle)에서 그 경우를 다룬다.

다만 프로그래머스에서도 디버깅용 `print()` 를 함수 안에 남겨 두면 실행 시간에 반영된다. 반복문 안에 `print(i, arr)` 같은 디버그 출력을 넣고 제출하면 제출용 코드가 아닌데도 시간 제한에 걸릴 수 있다. **제출 전에 디버그 `print` 를 지워라.**
:::

::: warn 백준에서 개행 처리를 틀리는 흔한 경우
1. **출력 끝에 개행이 없어서 틀리는 경우는 거의 없다.** 대부분의 채점기는 트레일링 개행 유무에 관대하다. 그보다 흔한 실수는 **줄 사이에 불필요한 빈 줄을 넣는 것**이다. `print()` 를 여러 번 겹쳐 부르다가 `"\n"` 을 이중으로 넣는 실수가 잦다.
2. `readline()` 으로 문자열을 읽고 그대로 비교하면 끝에 `\n` 이 남아서 값이 같아 보여도 `==` 비교가 실패한다. **문자열을 값으로 비교할 때는 `.rstrip()` 을 잊지 마라.** 정수로 바로 변환하는 경우는 `int()` 가 알아서 처리해 주므로 해당 없다.
3. 여러 테스트 케이스를 한 번에 처리하는 문제에서 **각 케이스 사이에 결과를 개별 `print()` 로 내보내지 말고 리스트에 모았다가 마지막에 한 번에 출력하라.** 케이스 수가 많을수록 이 절에서 실측한 4배 이상의 차이가 그대로 시간 제한에 영향을 준다.
:::

## 요약

- `input()` 은 프롬프트·터미널 판단 로직을 매번 거치는 무거운 함수다. `sys.stdin.readline()` 은 약 2.5배, `sys.stdin.read().split()` 은 약 4배 빠르다(30만 줄 실측).
- `input = sys.stdin.readline` 으로 전역 재바인딩하면 기존 코드를 거의 안 고치고 속도를 얻는다. 단 EOF에서 `EOFError` 대신 빈 문자열 `''` 을 반환한다는 점이 다르다.
- 입력 형식이 복잡하면(줄마다 토큰 수가 다르면) `sys.stdin.read().split()` 로 전체를 토큰 스트림으로 만들고 `iter()` + `next()` 로 순서대로 소비하는 패턴이 줄 단위로 세는 것보다 안전하다.
- `print()` 를 반복 호출하는 것과 결과를 리스트에 모았다가 `"\n".join()` 으로 한 번에 `sys.stdout.write()` 하는 것 사이에 약 4.6배 차이가 난다(30만 줄 실측). 진짜 병목은 함수 호출 횟수다.
- `print(*결과, sep="\n")` 은 반복 `print()` 보다는 빠르고 `join` 보다는 느린 절충안이다.
- 프로그래머스는 함수 인자/반환값으로 채점하므로 이 절의 입출력 최적화가 적용될 대상 자체가 없다. 대신 디버그용 `print()` 를 지우고 제출해야 한다.
- 백준류 플랫폼에서는 문자열 비교 시 `.rstrip()` 을 잊지 말고, 결과를 케이스마다 즉시 출력하지 말고 모았다가 한 번에 출력하라.

5. 다음 코드를 두고 "백준 채점에서 종종 무한 루프에 빠진다"는 말이 인터넷 어딘가에 돌아다닌다고 하자. 실행해서 이 말이 사실인지 검증하라.

   ```python
   import sys
   input = sys.stdin.readline

   while True:
       line = input()
       if line == "":
           break
       print(line.strip())
   ```

   파일 리다이렉트와 파이프 양쪽으로, 다음 입력들을 각각 만들어 직접 돌려 봐라: 중간에 빈 줄이 섞인 입력, 마지막 줄에 개행이 없는 입력, 완전히 빈 입력, 공백만 있는 줄. 무한 루프가 실제로 재현되는가?

   (힌트: `sys.stdin.readline()` 은 EOF에서 정확히 `''` 을 반환하고, 내용만 있는 빈 줄은 최소한 개행 문자 하나가 있는 `'\n'` 이라 `line == ""` 검사로 둘이 구별된다. 즉 표준적인 리다이렉트/파이프 입력에서는 이 코드가 멈추지 않을 이유가 없다 — 직접 돌려서 그 사실을 확인하는 게 이 문제의 핵심이다. 그렇다면 "무한 루프에 빠진다"는 소문이 어떤 상황을 가리키는 것일 수 있는지 생각해 보라. 예를 들어 리다이렉트 없이 대화형 터미널에서 직접 타이핑하며 EOF 신호(Windows는 Ctrl-Z + Enter, Unix 계열은 Ctrl-D)를 보내야 하는 경우, 사용자가 그 키 조합을 모르거나 잘못 입력하면 프로그램이 계속 다음 줄을 기다리는 것처럼 보일 수 있다. 이건 코드의 버그가 아니라 표준 입출력 채점 환경(리다이렉트/파이프)과 다른 실행 방식을 썼을 때 생기는 오해에 가깝다.)
