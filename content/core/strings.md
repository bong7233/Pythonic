# 1.4 문자열과 유니코드

::: lead
[1.1](#/objects-names)에서 문자열이 불변이라고 배웠다. 그런데 불변이라는 성질 하나가 왜 `+=` 를 시한폭탄으로 만드는지, `len("한글")` 이 왜 당신이 기대한 답을 주지 않을 수도 있는지, macOS에서 만든 파일이 왜 Windows에서 "다른 이름"으로 보이는지는 아직 설명하지 않았다. 이 절은 파이썬 문자열의 내부 표현부터 시작해서, 그 표현이 만드는 함정들을 하나씩 파헤친다. 코드포인트, 정규화, f-string의 전체 문법, 그리고 3.14에 새로 들어온 t-string까지 다룬다.
:::

## 문자열은 왜 "글자 배열"이 아닌가

C에서 문자열은 `char` 배열이다. 한 글자 = 한 바이트. 인덱싱은 포인터 연산 한 번이다. 파이썬도 옛날에는(파이썬 2, 그리고 3.2 이전) 비슷했다 — 다만 인터프리터를 빌드할 때 "narrow build"(문자당 2바이트)와 "wide build"(문자당 4바이트) 중 하나를 고정으로 선택해야 했다. narrow build는 이모지 같은 문자를 서로게이트 쌍으로 쪼개서 다뤄야 했고(그러면 `len()`이 실제 글자 수와 안 맞는다), wide build는 항상 4바이트를 쓰니 영어 텍스트만 다뤄도 메모리를 4배 먹었다.

둘 다 나쁜 선택이었다. 그래서 **PEP 393**(Python 3.3, 2013)이 이 문제를 근본적으로 다시 설계했다.

### PEP 393: 유연한 문자열 표현

핵심 아이디어는 간단하다. **문자열마다 자신에게 필요한 최소 바이트 폭을 그때그때 정한다.** 문자열을 만들 때 그 안에 있는 코드포인트 중 가장 큰 값을 보고, 세 가지 저장 방식(kind) 중 하나를 고른다.

| kind | 저장 폭 | 담을 수 있는 최대 코드포인트 | 예시 |
| --- | --- | --- | --- |
| 1바이트 (Latin-1) | 1바이트/문자 | U+00FF까지 | `"hello"`, `"café"` |
| 2바이트 (UCS-2) | 2바이트/문자 | U+FFFF까지 | `"한글"`, 대부분의 한자 |
| 4바이트 (UCS-4) | 4바이트/문자 | U+10FFFF까지 (전체) | `"😀"`, 이모지 대부분 |

중요한 건 이게 **문자열 전체에 적용되는 하나의 폭**이라는 점이다. 문자열 안에 이모지가 단 하나만 섞여 있어도 그 문자열 전체가 4바이트 폭으로 저장된다. 폭은 섞이지 않는다 — 균일해야 인덱싱이 $O(1)$ 을 유지하기 때문이다. 만약 문자마다 폭이 다르면 `s[100]` 에 접근하기 위해 앞의 99개 문자를 순서대로 디코딩해야 한다. PEP 393은 그 대가로 "가장 무거운 문자 하나가 전체 폭을 결정한다"를 받아들였다.

### 실측: `sys.getsizeof`

말로만 하지 말고 실제로 재보자.

```pyrepl
>>> import sys
>>> sys.getsizeof('')
41
>>> sys.getsizeof('a')
42
>>> sys.getsizeof('a' * 5)
46
```

빈 문자열도 41바이트다. 이게 `PyASCIIObject` 구조체(참조 카운트, 타입 포인터, 길이, 캐시된 해시 등)의 고정 오버헤드고, 여기에 코드포인트 하나당 1바이트씩 늘어난다. 이 형태는 **문자열 안의 모든 코드포인트가 ASCII 범위(0~127)일 때만** 쓰인다. ASCII는 UTF-8 인코딩과 완전히 같은 바이트열이라서, 별도의 UTF-8 캐시 포인터를 둘 필요가 없다. 그만큼 구조체가 작다.

코드포인트가 128을 넘는 순간(ASCII를 벗어나는 순간) 더 큰 구조체(`PyCompactUnicodeObject`)를 쓴다. UTF-8 인코딩 결과를 캐시할 포인터와 길이를 추가로 들고 있어야 하기 때문이다.

```pyrepl
>>> sys.getsizeof('é' * 2)      # Latin-1 범위(≤U+00FF)지만 ASCII는 아니다
59
>>> sys.getsizeof('é' * 3)
60
>>> sys.getsizeof('한' * 2)      # 2바이트 폭 (BMP)
62
>>> sys.getsizeof('한' * 3)
64
>>> sys.getsizeof('😀' * 2)      # 4바이트 폭 (astral)
68
>>> sys.getsizeof('😀' * 3)
72
```

정리하면 (Python 3.14.5 / Windows 기준 실측, 오버헤드는 기기·빌드에 따라 다를 수 있으나 **1글자당 증가폭**은 어디서나 같다):

```text nolines
kind                 header   1글자 늘 때마다
ASCII (1B, ascii=1)    41B     +1B
Latin-1 (1B, ascii=0)  57B     +1B
BMP (2B)               58B     +2B
Astral (4B)            60B     +4B
```

::: warn 문자 하나짜리 문자열로 재지 마라
방금 표를 만들 때 일부러 두 글자 이상으로 쟀다. 이유가 있다.

```pyrepl
>>> sys.getsizeof('é')      # 한 글자인데
61
>>> sys.getsizeof('é' * 2)  # 두 글자인데 오히려 더 작다?!
59
```

$1$글자가 $2$글자보다 크다. 공식이 깨진 게 아니라, **길이 1인 Latin-1 문자열은 인터프리터가 시작할 때 미리 만들어 캐싱해 둔 256개 싱글턴 중 하나**이기 때문이다.

```pyrepl
>>> id('é') == id('é')
True
```

같은 코드에서 매번 새로 만드는 것 같지만 실제로는 항상 같은 객체를 돌려받는다. 싱글턴은 일반적인 압축 문자열 생성 경로를 안 타기 때문에 크기 공식에서 벗어난다. 이 책이 이걸 언급하는 이유는 딱 하나다 — **`sys.getsizeof` 로 "1글자당 비용"을 잴 때는 반드시 2글자 이상으로 재라.** 1글자 결과를 일반 공식으로 오해하면 다음 사람에게 틀린 숫자를 퍼뜨리게 된다.
:::

::: perf 왜 신경 써야 하는가
수백만 개의 짧은 문자열을 다루는 코드(로그 파싱, NLP 전처리, 대량 CSV 로딩)에서는 이 배수가 실제 메모리 사용량을 좌우한다. 한글 텍스트 100만 줄을 다루면 같은 글자 수의 영어 텍스트보다 **최소 2배** 메모리를 먹는다는 뜻이다. 이모지가 섞인 SNS 텍스트라면 4배다. `sys.getsizeof` 로 실측하는 습관을 들이면 "왜 우리 서버 메모리가 예상보다 많이 먹지?"라는 질문에 감이 아니라 숫자로 답할 수 있다.
:::

## 코드포인트는 "문자"가 아니다

지금까지 "문자"라고 뭉뚱그려 썼지만, 정확히는 **코드포인트**(code point)다. `len()` 이 세는 것도, 인덱싱이 다루는 것도 전부 코드포인트다. 그런데 사람이 화면에서 "한 글자"로 인식하는 단위 — **문자소 클러스터**(grapheme cluster) — 는 코드포인트 하나가 아닐 수 있다.

### 한글: 조합형 vs 완성형

한글 음절 "한"은 유니코드에서 두 가지 방식으로 표현될 수 있다.

- **완성형(NFC)**: 완성된 음절 하나가 코드포인트 하나. `U+D55C`.
- **조합형(NFD)**: 초성·중성·종성을 별개의 코드포인트로 나눠 쓴다.

```pyrepl
>>> import unicodedata
>>> han = '한'
>>> hex(ord(han))
'0xd55c'
>>> nfd = unicodedata.normalize('NFD', han)
>>> len(nfd)
3
>>> [hex(ord(c)) for c in nfd]
['0x1112', '0x1161', '0x11ab']
>>> han == nfd
False
```

화면에는 똑같이 "한"으로 보이지만, 파이썬 입장에서는 **길이가 다르고 값도 다른 별개의 문자열**이다. `==` 는 코드포인트 시퀀스를 비교하지, 화면에 그려지는 모양을 비교하지 않는다.

### 이모지: 하나처럼 보이는 여러 개

가족 이모지나 피부톤이 있는 이모지는 여러 코드포인트가 **ZWJ**(Zero Width Joiner, `U+200D`)나 변경자(modifier)로 묶여 하나의 그림으로 렌더링된다.

```pyrepl
>>> family = '\U0001F468' + '‍' + '\U0001F469' + '‍' + '\U0001F467'
>>> len(family)
5
>>> waving_hand = '\U0001F44B' + '\U0001F3FD'   # 손 + 피부톤 변경자
>>> len(waving_hand)
2
```

화면에는 각각 "한 가족", "손 흔드는 이모지 하나"로 보인다. 하지만 `len()` 은 5와 2를 반환한다. `family[2:]` 로 슬라이싱하면 가족 이모지 그림이 깨진 조각(엄마+ZWJ+딸)이 나온다 — 시각적으로 완전한 단위를 자른 게 아니라 코드포인트 시퀀스 중간을 자른 것이기 때문이다.

::: warn `len()` 은 "글자 수"가 아니라 "코드포인트 수"다
사용자에게 "입력은 10자 이내"라고 안내하는 폼을 만든다면, `len(text) <= 10` 은 한글이나 이모지가 섞인 입력에서 사용자가 체감하는 글자 수와 어긋난다. **표준 라이브러리에는 문자소 클러스터를 세는 함수가 없다.** 정확히 세려면 서드파티 `grapheme` 패키지나 `regex` 패키지의 `\X` 패턴이 필요하다. 이 책에서는 존재를 아는 것으로 충분하다 — 코딩테스트에서는 대부분 "코드포인트 수 = 문제에서 말하는 글자 수"로 통일해서 낸다.
:::

## 정규화: 같은 글자, 다른 바이트

방금 본 "한" 하나가 NFC냐 NFD냐에 따라 다른 문자열이라는 사실은 실전에서 진짜 버그를 만든다.

```pyrepl
>>> import unicodedata
>>> nfc = 'café'
>>> nfd = unicodedata.normalize('NFD', nfc)
>>> len(nfc), len(nfd)
(4, 5)
>>> nfc == nfd
False
>>> s = {nfc}
>>> nfd in s
False
>>> unicodedata.normalize('NFC', nfd) == nfc
True
```

`café` 를 é 한 글자로 쓰면 NFC(길이 4), `e` + 결합 악센트 두 코드포인트로 쓰면 NFD(길이 5)다. 둘 다 화면에는 "café"로 똑같이 보이고, 둘 다 "올바른 유니코드"다. 그런데 `==` 도, `in` 도, 딕셔너리 키 조회도 전부 실패한다.

::: hist 왜 macOS에서 파일 이름이 이상해지는가
이건 이론이 아니라 실제로 macOS 개발자들이 두고두고 밟는 함정이다. **macOS의 파일시스템(HFS+, 그리고 APFS의 일부 설정)은 파일 이름을 저장할 때 NFD로 정규화한다.** 반면 Windows와 Linux는 앱이 넘긴 바이트를 그대로 저장한다(정규화하지 않는다).

그래서 이런 일이 벌어진다.

1. Windows에서 `"카페.txt"` 라는 파일을 NFC로 만들어 커밋한다.
2. macOS에서 그 저장소를 체크아웃하면, 파일시스템이 이름을 NFD로 바꿔 저장한다.
3. 파이썬 코드에서 `"카페.txt" in os.listdir(".")` 를 실행하면 — 리터럴은 NFC, `listdir` 이 돌려주는 이름은 NFD라서 — **파일이 분명히 있는데 `False`가 나온다.**

macOS를 다루는 코드에서 파일 이름을 비교하거나 딕셔너리 키로 쓸 때는 항상 정규화를 통일해야 한다.

```python
import os
import unicodedata

def norm(s: str) -> str:
    return unicodedata.normalize('NFC', s)

files = {norm(name) for name in os.listdir(".")}
target = norm("카페.txt")
found = target in files      # 이제야 안전하다
```

`unicodedata.normalize` 는 네 가지 형식을 지원한다. `NFC`(정규 결합, 대부분의 경우 기본으로 써야 함), `NFD`(정규 분해), `NFKC`/`NFKD`(호환성 분해까지 포함 — 예를 들어 전각 숫자 "１"을 반각 "1"로 바꾸는 것까지 한다). 사용자 입력을 비교·검색·중복 제거할 때는 **입력받는 시점에 한 가지 형식으로 통일**하는 게 유일하게 안전한 방법이다.
:::

## 불변성의 진짜 대가: `+=` 는 정말 $O(n^2)$ 인가

[1.1](#/objects-names)에서 "문자열 누적에 `+=` 를 쓰면 $O(n^2)$" 라고 배웠다. 이 절에서 이 이야기를 끝까지 파헤친다 — 왜냐하면 통념과 실제가 미묘하게 다르기 때문이다.

문자열은 불변이니, `result += word` 는 이론적으로 *"기존 내용 전체를 새 버퍼에 복사하고 뒤에 word를 붙인 새 객체를 만드는"* 것이어야 한다. $n$번 반복하면 $1+2+\cdots+n = O(n^2)$ 번의 바이트 복사가 일어난다는 게 통념이다.

실제로 재보자.

```python title="세 가지 시나리오"
import time

def simple_concat(words):
    """가장 흔한 패턴: 지역 변수에 그냥 += 한다."""
    result = ""
    for w in words:
        result += w
    return result

def aliased_concat(words):
    """다른 이름이 같은 문자열 객체를 붙잡고 있는 상태에서 += 한다."""
    result = ""
    alias = result
    for w in words:
        alias = result      # result가 가리키던 객체에 이름표를 하나 더 건다
        result += w
    return result
```

```python title="측정 결과 (words 개수를 2배씩 늘리며, time.perf_counter)"
# n       simple_concat   aliased_concat
# 25000   0.0011s         0.0137s
# 50000   0.0022s         0.0431s
# 100000  0.0041s         0.3329s
# 200000  0.0084s         1.1857s
```

(Python 3.14.5 / Windows 기준 실측. 절대 시간은 기기마다 다르지만 **두 함수의 증가 패턴 차이**는 어디서나 같다.)

`simple_concat` 은 입력이 2배가 되면 시간도 정확히 2배가 된다 — 선형이다. `aliased_concat` 은 입력이 2배가 될 때마다 시간이 4배 가까이 뛴다 — 제곱이다. **똑같이 생긴 `+=` 인데 결과가 다르다.**

::: deep CPython의 몰래 하는 최적화 — 그리고 그게 깨지는 조건
CPython은 `result += word` 같은 패턴을 컴파일할 때 특별 취급을 한다. 덧셈의 결과를 곧바로 같은 지역 변수에 다시 저장하는 패턴을 감지하면, 실행 시점에 이렇게 확인한다.

> **지금 `result` 가 가리키는 문자열 객체를 참조하는 이름표가 이거 하나뿐인가?** (참조 카운트 == 1)

하나뿐이면 — 즉 아무도 이 문자열의 "이전 상태"를 붙잡고 있지 않으면 — 파이썬은 **새 객체를 만들지 않고, 기존 버퍼를 확장(realloc)해서 그 자리에 이어 붙인다.** 이러면 매 반복이 상수 시간에 가까워지고, 전체는 (동적 배열의 증폭과 비슷한 방식으로) 상환 $O(n)$ 이 된다.

`aliased_concat` 에서는 매 반복마다 `alias` 라는 두 번째 이름표가 `result` 가 가리키던 객체를 붙잡고 있다가 놓는다. 그 순간 참조 카운트가 1이 아니게 되므로, CPython은 안전을 위해 **진짜로 새 문자열을 통째로 복사해서 만든다.** 통념이 말하던 $O(n^2)$ 이 여기서 그대로 나온다.

이것이 왜 함정인가: **이 최적화는 언어 명세가 아니라 CPython 구현의 우연한 세부사항이다.** 문자열을 리스트나 클래스 속성에 담아 놓고 그 안에서 `+=` 를 하면(`self.buf += word`, `items[i] += word`) 참조 카운트가 1로 유지된다는 보장이 없다. 코드를 조금만 리팩터링해도 조용히 $O(n)$ 에서 $O(n^2)$ 로 퇴화할 수 있다는 뜻이다.
:::

::: cote 그래서 코딩테스트에서는 무조건 `join`
```python
# 성능이 좋을 수도, 나쁠 수도 있다 — 구현 세부사항에 기댄다
result = ""
for word in words:
    result += word

# 항상 O(n) — 언어 차원에서 보장된다
result = "".join(words)
```

`"".join(words)` 은 먼저 `words` 전체를 훑어 최종 길이를 계산하고, **정확히 그 크기만큼 한 번만 할당한** 다음 각 조각을 그 자리에 복사한다. 재할당도, 참조 카운트 눈치 볼 일도 없다. `+=` 최적화가 켜지는 상황인지 매번 따지느니, 처음부터 `join` 을 쓰는 게 맞다. 시험장에서 리스트에 문자열 조각을 모았다가 마지막에 한 번 `join` 하는 패턴을 기본값으로 삼아라. 입출력 최적화는 [8.2 파이썬 입출력 최적화](#/io-optimize) 와 [8.3 시간 초과를 피하는 관용구](#/tle)에서 더 다룬다.
:::

## f-string 심화

기본형은 이미 알고 있을 것이다. `f"{name}은 {age}살이다"`. 여기서는 잘 안 쓰이지만 알아두면 디버깅과 출력 포맷팅에서 코드 줄 수를 확 줄여주는 문법을 정리한다.

### `=` — 디버그 연산자 (3.8+)

표현식과 그 값을 한 번에 찍는다.

```pyrepl
>>> x = 3.14159
>>> f'{x=}'
'x=3.14159'
>>> f'{x=:.2f}'
'x=3.14'
```

`print(f"{x=}")` 는 `print("x =", x)` 보다 짧고, 무엇보다 **변수 이름과 값이 항상 같이 나와서 로그를 뒤섞을 일이 없다.** `pdb` 나 임시 디버깅 print에 최적이다.

### 변환 지정자 — `!r`, `!s`, `!a`

중괄호 안 표현식 뒤에 `!문자` 를 붙이면 포맷 스펙 전에 변환 함수를 먼저 적용한다.

| 지정자 | 호출하는 것 | 언제 쓰나 |
| --- | --- | --- |
| `!s` | `str()` | 기본값. 생략해도 같다 |
| `!r` | `repr()` | 문자열임을 명확히 보이고 싶을 때, 디버깅 |
| `!a` | `ascii()` | 비-ASCII를 이스케이프해서 보고 싶을 때 |

```pyrepl
>>> name = "한글"
>>> f'{name!r}'
"'한글'"
```

`!r` 은 특히 로그에서 "빈 문자열인지 공백 문자열인지" 같은 걸 구분할 때 유용하다. `f"입력: {name}"` 은 빈 문자열이든 공백이든 똑같이 안 보이지만, `f"입력: {name!r}"` 은 `''` 와 `' '` 를 구분해서 보여준다.

### 정렬과 폭 — `:>10`, `:^10`, 중첩 필드

콜론 뒤 포맷 스펙은 표준 포맷 미니 언어를 그대로 따른다. 자주 쓰는 것만 정리한다.

```pyrepl
>>> name = "한글"
>>> f'{name:>10}'      # 오른쪽 정렬, 총 폭 10
'        한글'
>>> len(f'{name:>10}')
10
```

**폭은 코드포인트 개수 기준이다.** `"한글"` 은 `len()` 이 2를 반환하니, 폭 10에 맞추려면 공백 8개가 붙는다. 그런데 한글은 터미널에서 **시각적으로 2칸씩** 차지하므로, 화면에 그려지는 총 너비는 $8 + 2\times2 = 12$칸이다. 영어 문자열과 나란히 표로 늘어놓으면 코드포인트 기준 정렬이 시각적으로는 안 맞는다 — 이게 이 책의 아스키 다이어그램에서 한글을 선 오른쪽에만 쓰는 이유이기도 하다.

폭이나 정밀도를 변수로 넘기고 싶으면 중첩 필드를 쓴다.

```pyrepl
>>> width = 10
>>> f'{name:^{width}}'          # 가운데 정렬, 폭을 변수로
'    한글    '
>>> f'{3.14159:{width}.3f}'     # 폭도 정밀도도 변수로
'     3.142'
```

`{}` 안에 또 `{}` 를 넣을 수 있다는 것만 기억하면 된다 — 포맷 스펙 자체도 f-string 표현식이다.

## `str.translate` — 여러 글자를 한 번에 바꾸거나 지운다

`replace()` 를 여러 번 체이닝하는 대신, 대응표 하나로 여러 문자를 한 번에 치환·삭제할 수 있다.

```pyrepl
>>> table = str.maketrans({'a': '@', 'e': '3', 'i': '1'})
>>> 'leetspeak'.translate(table)
'l33tsp3@k'
```

`str.maketrans` 는 세 가지 형태로 호출할 수 있다.

```pyrepl
>>> # (문자열, 문자열, 삭제할 문자열) — 위치별로 대응, 세 번째는 통째로 삭제
>>> table2 = str.maketrans('abc', 'xyz', 'd')
>>> 'abcd'.translate(table2)
'xyz'
```

`'abcd'` 에서 `a→x`, `b→y`, `c→z` 로 바꾸고 `d` 는 통째로 지웠다. 문자 단위 치환과 삭제를 **한 번의 순회로** 끝낸다는 게 핵심이다. `replace()` 를 세 번 체이닝하면 문자열을 세 번 새로 만들지만, `translate` 는 한 번만 만든다.

::: cote 구두점 제거, 흔한 실전 패턴
```python
import string

table = str.maketrans('', '', string.punctuation)
clean = "hello, world!".translate(table)
print(clean)          # 'hello world'
```

문자열 처리 문제(회문 판정, 단어 빈도수 세기)에서 구두점을 지울 때 정규식보다 빠르고 간단하다.
:::

## 3.14의 새 문법: t-string

f-string은 만드는 즉시 문자열로 평가된다. 그런데 SQL 쿼리나 HTML을 f-string으로 조립하면, 사용자 입력이 그대로 꽂혀 들어가는 인젝션 위험을 문법 차원에서 막을 방법이 없었다. **3.14부터 추가된 t-string**은 이 문제를 풀기 위해, f-string과 똑같이 생겼지만 **문자열이 아니라 구조화된 템플릿 객체**를 반환한다.

```pyrepl
>>> name = "world"
>>> t = t'hello {name}'
>>> type(t)
<class 'string.templatelib.Template'>
>>> t.strings
('hello ', '')
>>> t.interpolations
(Interpolation('world', 'name', None, ''),)
```

`t'...'` 는 즉시 문자열로 합쳐지지 않는다. 대신 리터럴 조각들(`t.strings`)과, 채워 넣을 자리마다 **원본 표현식·값·변환·포맷 스펙을 그대로 들고 있는** `Interpolation` 객체들(`t.interpolations`)을 분리해서 담은 `Template` 객체를 만든다. 최종적으로 문자열로 만들지 말지, 어떻게 이스케이프할지는 **템플릿을 받는 쪽(라이브러리)이 결정한다.**

```python title="왜 안전한가 — 개념 예시"
def safe_sql(template):
    parts = []
    params = []
    for item in template:
        if isinstance(item, str):
            parts.append(item)          # 리터럴 조각은 그대로
        else:
            parts.append("?")           # 값이 들어갈 자리는 플레이스홀더로
            params.append(item.value)   # 실제 값은 파라미터 목록으로 분리
    return "".join(parts), params

user_input = "'; DROP TABLE users; --"
query, params = safe_sql(t"SELECT * FROM t WHERE name = {user_input}")
print(query)    # 'SELECT * FROM t WHERE name = ?'
print(params)   # ["'; DROP TABLE users; --"]
```

f-string이었다면 `user_input` 이 쿼리 문자열에 그대로 녹아들어갔을 것이다. t-string은 값과 리터럴을 끝까지 분리해서 들고 있으므로, 라이브러리가 값 쪽만 따로 이스케이프하거나 파라미터 바인딩으로 돌릴 수 있다. 이 절 시점에서는 **존재를 아는 것으로 충분하다.** SQL/HTML 이스케이핑을 다루는 라이브러리들이 3.14 이후로 t-string을 받아들이기 시작하는 게 눈에 띄면, 그게 이 기능이다.

## 요약

- 파이썬 문자열은 PEP 393에 따라 **필요한 만큼만** 바이트 폭을 쓴다(1/2/4바이트). 폭은 문자열 전체에 균일하게 적용된다.
- `sys.getsizeof` 로 실측할 때 **길이 1인 문자열은 싱글턴 캐시라 공식이 안 맞을 수 있다.** 2글자 이상으로 재라.
- `len()` 은 코드포인트 개수를 센다. 사람이 보는 "글자"(문자소 클러스터)와 다를 수 있다 — 조합형 한글, ZWJ로 묶인 이모지가 대표적이다.
- 같은 글자도 NFC/NFD에 따라 다른 코드포인트 시퀀스가 된다. `==` 와 딕셔너리/집합 조회가 실패할 수 있다. macOS 파일 이름이 대표 사례다.
- `+=` 로 문자열을 누적하는 게 항상 $O(n^2)$ 인 건 아니다 — CPython이 참조 카운트 1일 때 제자리 확장을 시도한다. 하지만 이건 **구현 세부사항**이라 언제든 깨질 수 있다. 항상 `join` 을 써라.
- f-string은 `=`(디버그), `!r`/`!s`/`!a`(변환), `:>10`/`:^{width}`(정렬·중첩 필드)까지 전부 하나의 문법 안에 있다.
- `str.translate` 는 여러 문자의 치환·삭제를 한 번의 순회로 끝낸다.
- 3.14의 t-string은 f-string과 문법은 같지만 **문자열이 아니라 구조화된 템플릿**을 만들어, 인젝션에 안전한 조립을 가능하게 한다.
