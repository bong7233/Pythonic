# 1.1 객체, 이름, 참조 — 실행 모델

::: lead
이 절은 이 책 전체에서 가장 중요하다. 파이썬에서 벌어지는 이상한 일의 절반은 **"변수는 값을 담는 상자"라는 잘못된 모델** 때문에 생긴다. 그 모델을 지우고 올바른 것으로 갈아 끼우는 게 이 절의 유일한 목표다. 여기를 제대로 넘기면 앞으로의 모든 파트가 쉬워진다.
:::

## 잘못된 모델

C나 Java를 배웠다면 머릿속에 이런 그림이 있을 것이다.

```text nolines
int x = 5;        메모리 어딘가에 'x'라는 상자가 있고, 그 안에 5가 들어 있다.

     x
   ┌─────┐
   │  5  │
   └─────┘
```

C에서는 이게 정확하다. `x` 는 메모리 주소에 붙은 이름이고, `x = 6` 은 **그 주소에 있는 값을 덮어쓴다.**

파이썬은 다르다. 이 그림을 지워야 한다.

## 올바른 모델

파이썬에서는 이렇게 생각한다.

> **객체는 저 어딘가에 떠 있다. 이름은 객체에 붙이는 이름표다. 대입은 이름표를 붙이는 행위다.**

```text nolines
x = 5

   x ──────▶ ┌──────────────┐
             │ int object   │
             │ value: 5     │
             └──────────────┘

x = 6      이름표를 뗐다가 다른 객체에 붙인다. 5라는 객체는 그대로 있다.

   x ──────▶ ┌──────────────┐
             │ int object   │
             │ value: 6     │
             └──────────────┘
```

차이가 사소해 보이지만 결과는 크게 다르다. **이름 여러 개가 같은 객체를 가리킬 수 있기 때문**이다.

```text nolines
a = [1, 2, 3]
b = a        상자를 복사한 게 아니라, 같은 객체에 이름표를 하나 더 붙였다.

   a ──────▶ ┌──────────────┐
             │ list object  │  ◀────── b
             │ [1, 2, 3]    │
             └──────────────┘
```

이제 이 절 첫머리에서 낸 문제의 답이 보인다.

## 객체의 세 가지 속성

파이썬의 모든 객체는 세 가지를 가진다. 예외 없이 전부 — 정수도, 함수도, 클래스도, 모듈도.

| 속성 | 의미 | 확인 방법 | 바뀌나 |
| --- | --- | --- | --- |
| **정체성**(identity) | 이 객체가 다른 객체와 구별되는 근거 | `id(x)` | 절대 안 바뀜 |
| **타입**(type) | 무엇을 할 수 있는지 | `type(x)` | 사실상 안 바뀜 |
| **값**(value) | 담고 있는 내용 | `x` | 객체에 따라 다름 |

```pyrepl
>>> x = [1, 2, 3]
>>> id(x)
2103428839296
>>> type(x)
<class 'list'>
>>> x
[1, 2, 3]
```

::: deep id() 가 실제로 반환하는 것
언어 명세는 `id()` 에 대해 *"객체의 생애 동안 유일하고 불변인 정수"* 라고만 정한다. 구현 방법은 자유다.

**CPython에서는 그 객체의 메모리 주소**다. 그래서 실행할 때마다 값이 다르고, 큰 수가 나온다.

```pyrepl
>>> x = [1, 2, 3]
>>> id(x)
2103428839296
>>> hex(id(x))
'0x1e9b2c3f800'
```

여기서 중요한 함정: **"생애 동안" 유일하다는 것은, 객체가 죽으면 그 id가 재사용될 수 있다는 뜻이다.**

```pyrepl
>>> id([1, 2, 3])
2103428839296
>>> id([4, 5, 6])          # 다른 리스트인데 같은 id!
2103428839296
```

이건 버그가 아니다. 첫 번째 리스트는 `id()` 가 값을 반환한 직후 아무도 참조하지 않아 즉시 소멸했고, 두 번째 리스트가 **같은 메모리 자리를 재사용**한 것이다. 그래서 `id()` 값을 어딘가에 저장해 두고 나중에 비교하는 코드는 위험하다. `id()` 는 **디버깅 중 지금 이 순간**을 들여다볼 때만 쓰는 도구다.
:::

## `is` 와 `==` 는 완전히 다른 질문이다

이걸 헷갈리면 반드시 사고가 난다.

- **`is`** — "같은 객체인가?" → `id(a) == id(b)` 와 동치다.
- **`==`** — "같은 값인가?" → `a.__eq__(b)` 를 호출한다.

```pyrepl
>>> a = [1, 2, 3]
>>> b = [1, 2, 3]
>>> a == b          # 값이 같은가? 그렇다.
True
>>> a is b          # 같은 객체인가? 아니다. 리스트를 두 번 만들었다.
False
>>> c = a
>>> c is a          # 같은 객체다.
True
```

::: warn is 를 써야 하는 곳은 딱 정해져 있다
**`None`, `True`, `False` 와 비교할 때만 `is` 를 쓴다.** 이 셋은 인터프리터 전체에 단 하나씩만 존재하는 싱글턴이라 정체성 비교가 정확하고 빠르다.

```python
if x is None:        # ✅ 올바름
if x == None:        # ❌ 동작은 하지만 틀린 습관
if x is not None:    # ✅
```

왜 `== None` 이 나쁜가? `__eq__` 를 오버라이드한 클래스는 `None` 과의 비교에서 `True` 를 반환하게 만들 수 있다. 실제로 NumPy 배열은 `arr == None` 에 대해 **불리언 배열**을 반환한다 — `if` 에 넣으면 예외가 난다.

```pyrepl
>>> import numpy as np
>>> arr = np.array([1, 2, 3])
>>> arr == None
array([False, False, False])
>>> arr is None
False
```

ruff의 `E711` 규칙이 이걸 잡아 준다. ([0.4 도구](#/tooling))
:::

## 캐싱이라는 함정

여기서 초심자를 혼란에 빠뜨리는 현상이 나온다.

```pyrepl
>>> a = 256
>>> b = 256
>>> a is b
True                # 어? 다른 객체여야 하는 거 아닌가?
>>> a = 257
>>> b = 257
>>> a is b
False               # 이번엔 다르다?!
```

::: deep 작은 정수 캐싱
CPython은 시작할 때 **-5부터 256까지의 정수 객체를 미리 만들어 둔다.** 이 범위의 정수가 필요하면 새로 만들지 않고 캐시된 것을 재사용한다.

이유는 성능이다. 이 범위의 정수는 압도적으로 자주 쓰인다(반복문 인덱스, 카운터, 작은 계산). 매번 새 객체를 할당하고 해제하는 비용이 아깝다.

문자열도 비슷하게 **인터닝(interning)** 된다. 식별자처럼 생긴 문자열(글자·숫자·밑줄로만 이뤄진 것)은 하나로 합쳐진다.

```pyrepl
>>> s1 = "hello"
>>> s2 = "hello"
>>> s1 is s2
True
>>> s3 = "hello world!"      # 공백과 느낌표가 있다 → 식별자 모양이 아니다
>>> s4 = "hello world!"
>>> s3 is s4
False
```

### 여기서 진짜 함정이 나온다

방금 본 것은 **REPL에서 한 줄씩** 친 결과다. 같은 코드를 파일에 담아 실행하면 **결과가 달라진다.**

```python title="caching.py — 위 REPL과 같은 코드"
a = 257
b = 257
print(a is b)                    # True  ← REPL에서는 False였다!

s3 = "hello world!"
s4 = "hello world!"
print(s3 is s4)                  # True  ← REPL에서는 False였다!
```

::: deep 왜 REPL과 파일이 다른가 — 코드 객체가 경계다
파이썬 컴파일러는 **하나의 코드 객체**(code object) 안에서 같은 상수가 여러 번 나오면 **하나로 합친다**(constant folding / deduplication). 그 상수들은 `co_consts` 라는 튜플에 딱 한 벌만 저장된다.

- **파일**: 모듈 전체가 코드 객체 하나 → `257` 두 개가 합쳐진다 → `is` 가 `True`.
- **REPL**: 입력한 **줄마다 별개의 코드 객체**로 컴파일된다 → 합쳐질 기회가 없다 → `is` 가 `False`.

그래서 REPL에서도 **한 줄 안에 넣으면** 다시 `True` 가 된다.

```pyrepl
>>> a, b = 257, 257     # 한 문장 = 한 코드 객체
>>> a is b
True
```

같은 이유로, 런타임에 만들어진 문자열은 합쳐지지 않는다. 컴파일러가 볼 수 없기 때문이다.

```pyrepl
>>> s5 = "hello world"
>>> s6 = "hello" + " world"
>>> s5 is s6
False
```

**이 동작에 절대 의존하지 마라.** 언어 명세가 아니라 CPython의 최적화 세부사항이다. 버전마다 다르고, 방금 봤듯 **같은 코드가 REPL이냐 파일이냐에 따라 정반대 결과**를 낸다.

알아 둬야 하는 이유는 딱 하나다. **`is` 를 값 비교에 잘못 썼을 때, 왜 내 테스트에서는 통과하고 실제 환경에서는 실패하는지** 설명할 수 있어야 하기 때문이다. 이건 실제로 일어나는 사고다.
:::
:::

::: danger 정수·문자열 비교에 is 를 쓰면
```python
if n is 256:      # 어쩌다 동작한다
if n is 257:      # 조용히 항상 False
```

3.8부터 파이썬이 이걸 경고해 준다.

```pyrepl
>>> n = 257
>>> n is 257
<stdin>:1: SyntaxWarning: "is" with 'int' literal. Did you mean "=="?
False
```

**값 비교는 항상 `==`.** 예외는 `None`/`True`/`False` 뿐이다.
:::

## 가변과 불변

이제 파이썬 자료형의 가장 중요한 분류가 나온다.

| 불변(immutable) | 가변(mutable) |
| --- | --- |
| `int`, `float`, `complex`, `bool` | `list` |
| `str` | `dict` |
| `tuple` | `set` |
| `frozenset` | `bytearray` |
| `bytes` | 대부분의 사용자 정의 클래스 |
| `range` | |

**불변 객체는 만들어진 뒤 값이 절대 바뀌지 않는다.** 바꾸는 것처럼 보이는 모든 연산은 사실 **새 객체를 만든다.**

```pyrepl
>>> s = "hello"
>>> id(s)
2103425551344
>>> s = s.upper()          # s를 바꾼 게 아니라, 새 문자열을 만들어 이름을 옮겼다
>>> s
'HELLO'
>>> id(s)
2103428838128              # 다른 객체다
```

가변 객체는 제자리에서 바뀐다.

```pyrepl
>>> lst = [1, 2, 3]
>>> id(lst)
2103428839296
>>> lst.append(4)          # 같은 객체를 수정했다
>>> lst
[1, 2, 3, 4]
>>> id(lst)
2103428839296              # id가 그대로다
```

::: cote 코딩테스트 포인트
**문자열 누적은 파이썬에서 시간 초과의 단골 원인이다.**

```python
# ❌ O(n²) — 매 반복마다 새 문자열을 통째로 만든다
result = ""
for word in words:
    result += word

# ✅ O(n) — 리스트에 모았다가 한 번에 합친다
result = "".join(words)
```

문자열은 불변이므로 `result += word` 는 *"기존 문자열 전체 + 새 단어"* 를 담을 **새 객체를 만들고 전부 복사한다.** 길이 $n$ 인 문자열을 $n$ 번 붙이면 $1+2+3+\dots+n = O(n^2)$ 번의 복사가 일어난다.

`"".join()` 은 전체 길이를 먼저 계산하고 한 번만 할당한다. 10만 개 단어에서 수십 초 vs 0.01초 수준의 차이다. [1.4 문자열](#/strings)에서 실측한다.
:::

## 대입은 절대 복사하지 않는다

이것이 이 절의 핵심 문장이다. 밑줄을 그어라.

> **파이썬에서 `=` 는 값을 복사하지 않는다. 이름표를 붙일 뿐이다.**

불변 객체에서는 이 사실이 **드러나지 않는다.** 값을 바꿀 수 없으니 공유하든 말든 티가 안 난다.

```pyrepl
>>> a = 5
>>> b = a          # 같은 int 객체를 가리킨다
>>> b = 6          # b를 다른 객체로 옮긴다. a는 영향 없음
>>> a
5
```

가변 객체에서 드러난다.

```pyrepl
>>> a = [1, 2, 3]
>>> b = a          # 같은 리스트 객체를 가리킨다
>>> b.append(4)    # 그 객체를 수정한다
>>> a
[1, 2, 3, 4]       # a로 봐도 당연히 바뀌어 있다
```

`b` 는 `a` 의 사본이 아니다. **같은 객체의 다른 이름**이다. 이것을 **별칭**(aliasing)이라고 한다.

### `+=` 와 `+` 의 차이

이제 [0.1절](#/how-to-read)의 워밍업 문제를 정확히 설명할 수 있다.

```pyrepl
>>> a = [1, 2, 3]
>>> b = a
>>> b += [4]           # 리스트의 __iadd__ → 제자리에서 확장
>>> a
[1, 2, 3, 4]           # a도 바뀐다 (같은 객체니까)
>>> b is a
True

>>> c = [1, 2, 3]
>>> d = c
>>> d = d + [4]        # __add__ → 새 리스트를 만들고 d를 거기 붙인다
>>> c
[1, 2, 3]              # c는 그대로
>>> d is c
False
```

::: deep += 는 언제 제자리 수정이고 언제 새 객체인가
`x += y` 는 이렇게 동작한다.

1. `x` 의 타입에 `__iadd__` 가 있으면 그것을 호출한다. 반환값을 `x` 에 다시 대입한다.
2. 없으면 `x = x + y` 로 처리한다 (`__add__`).

**가변 타입**(list)은 `__iadd__` 를 구현하고 **자기 자신을 수정한 뒤 self를 반환**한다. → 제자리 수정.
**불변 타입**(int, str, tuple)은 `__iadd__` 가 없다. → 새 객체 생성.

그래서 같은 `+=` 기호가 타입에 따라 정반대로 동작한다.

```pyrepl
>>> t = (1, 2)
>>> u = t
>>> t += (3,)          # 튜플은 불변 → 새 객체
>>> t
(1, 2, 3)
>>> u
(1, 2)                 # u는 그대로
```

이 지식이 필요한 실전 상황: **여러 곳에서 공유되는 리스트에 `+=` 를 하면 모든 곳에 영향이 간다.** 의도한 것이면 좋지만, 대부분은 의도한 게 아니다.

가장 악명 높은 예:

```pyrepl
>>> t = ([1, 2], 3)      # 튜플 안에 리스트
>>> t[0] += [9]
Traceback (most recent call last):
  ...
TypeError: 'tuple' object does not support item assignment
>>> t                     # 그런데?!
([1, 2, 9], 3)
```

**에러가 났는데 값은 바뀌었다.** `t[0] += [9]` 는 (1) `t[0].__iadd__([9])` 로 리스트를 제자리 확장하고 → 성공, (2) 그 결과를 `t[0] = ...` 로 다시 대입 → 튜플이라 실패. 1번은 이미 일어난 뒤다. 이건 [1.14 특수 메서드](#/dunder)에서 다시 만난다.
:::

## 함수 인자 전달

"파이썬은 값 전달인가 참조 전달인가?"는 잘못된 질문이다. 둘 다 아니다. 정확한 이름은 **객체 참조에 의한 전달**(call by object reference), 또는 **공유에 의한 전달**(call by sharing)이다.

규칙은 하나뿐이다. **인자를 넘기는 것은 대입과 똑같다.** 매개변수라는 이름표가 같은 객체에 하나 더 붙는 것이다.

```python title="인자 전달의 두 얼굴"
def modify(lst):
    lst.append(4)        # 객체를 수정한다 → 호출자에게 보인다


def rebind(lst):
    lst = [9, 9, 9]      # 지역 이름표를 옮길 뿐 → 호출자에게 안 보인다


data = [1, 2, 3]

modify(data)
print(data)              # [1, 2, 3, 4]   ← 바뀌었다

rebind(data)
print(data)              # [1, 2, 3, 4]   ← 그대로다
```

`modify` 안에서는 두 이름이 **한 객체**를 가리키므로, 함수가 그 객체를 고치면 밖에서도 보인다.

```text nolines
  data ──┐
         ├──▶ [1, 2, 3, 4]     <- lst.append(4) 가 이 객체를 고쳤다
  lst  ──┘
```

`rebind` 안에서는 `lst` 라는 **지역 이름표만** 다른 객체로 옮겨 간다. `data` 는 원래 객체를 계속 가리킨다.

```text nolines
  data ──────▶ [1, 2, 3, 4]    <- 아무 일도 없었다

  lst  ──────▶ [9, 9, 9]       <- 함수가 끝나면 이 이름표는 사라진다
```

::: warn 함수가 인자를 수정하는 것은 설계 결정이다
파이썬은 이걸 막지 않는다. 그래서 **함수가 받은 객체를 조용히 수정하면 호출한 쪽에서 예상 못 한 버그가 생긴다.**

```python
def normalize(scores):
    scores.sort()             # 호출자의 리스트를 바꿔 버린다!
    return scores


data = [3, 1, 2]
normalize(data)
print(data)                   # [1, 2, 3] — 원본이 정렬됐다
```

원본을 지키려면 명시적으로 복사한다.

```python
def normalize(scores):
    return sorted(scores)     # 새 리스트를 반환. 원본 불변.
```

**경험칙**: 함수 이름이 동사면(`sort`, `append`, `update`) 제자리 수정, 명사나 형용사형이면(`sorted`, `reversed`, `copy`) 새 객체. 파이썬 표준 라이브러리가 이 규칙을 지킨다. 당신의 함수도 지켜라.

| 제자리 수정 (`None` 반환) | 새 객체 반환 |
| --- | --- |
| `lst.sort()` | `sorted(lst)` |
| `lst.reverse()` | `reversed(lst)` |
| `lst.append(x)` | `lst + [x]` |
| `dict.update(d)` | `dict | d` (3.9+) |
:::

::: danger 가변 기본값 — 파이썬 최악의 함정
```python
def add_item(item, bucket=[]):     # ❌
    bucket.append(item)
    return bucket
```

```pyrepl
>>> add_item("사과")
['사과']
>>> add_item("배")
['사과', '배']           # ?! 새 리스트가 아니다
>>> add_item("포도")
['사과', '배', '포도']
```

**기본값은 함수가 정의될 때 딱 한 번 평가된다.** 호출할 때마다가 아니다. 그 리스트 객체 하나가 함수의 속성으로 붙어서 영원히 살아 있다.

직접 확인할 수 있다.

```pyrepl
>>> add_item.__defaults__
(['사과', '배', '포도'],)
```

올바른 관용구:

```python
def add_item(item, bucket=None):    # ✅
    if bucket is None:
        bucket = []                 # 호출할 때마다 새로 만든다
    bucket.append(item)
    return bucket
```

ruff의 `B006` 규칙이 이걸 잡는다. 반드시 켜 둬라. 자세한 배경은 [1.10 함수](#/functions)에서.
:::

## 복사: 얕은가 깊은가

별칭이 싫으면 복사해야 한다. 그런데 복사에는 두 종류가 있다.

### 얕은 복사 (shallow copy)

**바깥 컨테이너만 새로 만들고, 안의 원소는 그대로 공유한다.**

```pyrepl
>>> a = [1, 2, 3]
>>> b = a[:]              # 또는 a.copy(), list(a), copy.copy(a)
>>> b is a
False                     # 다른 리스트다
>>> b.append(4)
>>> a
[1, 2, 3]                 # a는 안전하다
```

원소가 불변이면 여기서 끝난다. 문제는 원소가 가변일 때다.

```pyrepl
>>> a = [[1, 2], [3, 4]]
>>> b = a[:]              # 얕은 복사
>>> b is a
False                     # 바깥 리스트는 다르다
>>> b[0] is a[0]
True                      # 하지만 안쪽 리스트는 같은 객체!
>>> b[0].append(99)
>>> a
[[1, 2, 99], [3, 4]]      # a가 바뀌었다
```

```text nolines
        a ──▶ [ • , • ]
                │   │
                ▼   ▼
             [1,2] [3,4]      ← 이 안쪽 리스트들은 공유된다
                ▲   ▲
                │   │
        b ──▶ [ • , • ]
```

### 깊은 복사 (deep copy)

**중첩된 것까지 전부 새로 만든다.**

```pyrepl
>>> import copy
>>> a = [[1, 2], [3, 4]]
>>> b = copy.deepcopy(a)
>>> b[0] is a[0]
False                     # 안쪽까지 별개다
>>> b[0].append(99)
>>> a
[[1, 2], [3, 4]]          # 완전히 안전하다
```

::: perf deepcopy 는 비싸다
`deepcopy` 는 객체 그래프 전체를 순회하며 재귀적으로 복사한다. 큰 구조에서는 느리고 메모리를 두 배 쓴다. 또 순환 참조를 추적하기 위해 내부적으로 방문 기록(memo dict)을 유지한다.

```python title="직접 측정해 보라"
import copy, timeit

data = [[i, i + 1] for i in range(10_000)]

timeit.timeit(lambda: data[:], number=1000)              # 0.010초
timeit.timeit(lambda: copy.deepcopy(data), number=1000)  # 4.001초  → 약 400배
```

(Python 3.14.5 / Windows 기준 실측. 절대값은 기기마다 다르지만 **자릿수 차이**는 어디서나 같다.)

**deepcopy를 습관적으로 쓰지 마라.** 대부분의 경우 더 나은 선택지가 있다.

- 원소가 불변이면 → 얕은 복사로 충분하다
- 애초에 **불변 자료구조를 쓴다** (tuple, frozenset, frozen dataclass)
- 수정하지 말고 **새 객체를 만들어 반환**하는 스타일로 설계한다
:::

::: cote 코딩테스트 포인트
**2차원 배열 초기화의 함정.** 이건 시험장에서 실제로 많이 당한다.

```python
# ❌ 모든 행이 같은 리스트다!
board = [[0] * 3] * 3
board[0][0] = 1
print(board)          # [[1, 0, 0], [1, 0, 0], [1, 0, 0]]  ← 전부 바뀜

# ✅ 행마다 새 리스트
board = [[0] * 3 for _ in range(3)]
board[0][0] = 1
print(board)          # [[1, 0, 0], [0, 0, 0], [0, 0, 0]]
```

`[x] * 3` 은 **`x` 를 3번 복사하는 게 아니라, `x` 를 가리키는 참조를 3개 만든다.** 안쪽 `[0] * 3` 은 원소가 불변(int)이라 괜찮지만, 바깥의 `* 3` 은 **리스트 객체 하나**를 3번 참조한다.

깊이 3짜리도 마찬가지다.

```python
# ✅ 3차원
dp = [[[0] * c for _ in range(b)] for _ in range(a)]
```
:::

## 참조 카운팅과 객체의 죽음

객체는 언제 사라질까? CPython은 **참조 카운팅**(reference counting)을 쓴다. 각 객체는 *"나를 가리키는 이름표가 몇 개인가"* 를 세고 있고, **0이 되는 즉시 소멸한다.**

```pyrepl
>>> import sys
>>> a = [1, 2, 3]
>>> sys.getrefcount(a)
2
```

::: note 왜 1이 아니라 2인가
`sys.getrefcount(a)` 를 호출하는 순간, `a` 를 **인자로 넘기는 것 자체가 참조를 하나 더 만든다.** 그래서 항상 실제보다 1 크게 나온다. 3.14 기준으로도 이 동작은 그대로다.
:::

```pyrepl
>>> b = a
>>> sys.getrefcount(a)
3                          # a, b, 그리고 인자
>>> del b                  # del은 객체를 지우는 게 아니라 이름표를 뗀다
>>> sys.getrefcount(a)
2
```

::: warn del 은 삭제가 아니다
`del x` 는 **객체를 지우지 않는다.** 이름 `x` 를 현재 이름 공간에서 제거할 뿐이다. 그 결과로 참조 카운트가 0이 되면 그때 객체가 소멸하는 것이다.

```pyrepl
>>> a = [1, 2, 3]
>>> b = a
>>> del a
>>> b            # 객체는 멀쩡히 살아 있다
[1, 2, 3]
```
:::

객체가 죽는 순간을 직접 볼 수 있다.

```python title="소멸 시점 관찰"
class Noisy:
    def __init__(self, name):
        self.name = name

    def __del__(self):
        print(f"{self.name} 소멸")


x = Noisy("A")
y = x                 # 참조 2개
del x                 # 아직 안 죽음 (y가 남았다)
print("del x 이후")
del y                 # 여기서 카운트 0 → 소멸
print("del y 이후")
```

```text nolines
del x 이후
A 소멸
del y 이후
```

::: deep 참조 카운팅의 장점과 한계
**장점**: 결정적(deterministic)이다. 마지막 참조가 사라지는 **바로 그 순간** 메모리가 해제된다. 그래서 파일 핸들 같은 자원이 예측 가능한 시점에 닫힌다.

**한계**: **순환 참조를 못 잡는다.**

```python
a = []
b = []
a.append(b)      # a가 b를 참조
b.append(a)      # b가 a를 참조
del a, b         # 이름표는 사라졌지만...
```

이제 이 두 리스트는 아무도 접근할 수 없다. 그런데 서로가 서로를 참조하므로 **카운트가 0이 되지 않는다.** 참조 카운팅만으로는 영원히 누수다.

그래서 CPython에는 **순환 참조 검출기**(cyclic garbage collector)가 추가로 돈다. 주기적으로 객체 그래프를 훑어서 "서로만 참조하는 고립된 섬"을 찾아 회수한다.

```pyrepl
>>> import gc
>>> gc.collect()          # 수동으로 한 번 돌리기
>>> gc.get_count()        # 세대별 추적 객체 수
(391, 3, 1)
```

이 GC는 **세대별**(generational)로 동작한다 — "오래 살아남은 객체는 앞으로도 살아남을 확률이 높다"는 가설에 따라, 젊은 세대를 더 자주 검사한다. 3.12부터는 **증분(incremental) GC**가 도입돼 일시 정지 시간이 줄었다.

실전에서 이게 중요해지는 지점: **`__del__` 을 정의한 객체가 순환에 끼면 회수가 까다로워진다.** 그래서 `__del__` 에 의존하지 말고 [1.17 컨텍스트 매니저](#/context-managers)의 `with` 를 써야 한다. 자세한 건 [5.2 메모리 모델](#/memory)에서.
:::

## 종합: 실전에서 이게 어떻게 나타나는가

지금까지의 지식을 모아 실제 버그를 읽어 보자.

```python title="어디가 잘못됐을까?"
class ShoppingCart:
    def __init__(self, items=[]):          # ①
        self.items = items

    def add(self, item):
        self.items.append(item)

    def copy(self):
        new = ShoppingCart()
        new.items = self.items             # ②
        return new


cart_a = ShoppingCart()
cart_a.add("사과")

cart_b = ShoppingCart()                     # 새 카트인데...
print(cart_b.items)                         # ['사과']  ← ①의 결과

cart_c = cart_a.copy()
cart_c.add("배")
print(cart_a.items)                         # ['사과', '배']  ← ②의 결과
```

**①** 가변 기본값. 모든 `ShoppingCart` 인스턴스가 **같은 리스트 하나**를 공유한다.
**②** 대입은 복사가 아니다. `new.items` 와 `self.items` 는 같은 객체다.

고친 버전:

```python title="고친 버전"
class ShoppingCart:
    def __init__(self, items=None):
        self.items = list(items) if items else []   # 항상 새 리스트

    def add(self, item):
        self.items.append(item)

    def copy(self):
        return ShoppingCart(self.items)             # 생성자가 복사해 준다
```

::: tip 애초에 이런 버그를 막는 법
1. **가변 기본값을 쓰지 마라.** `None` 을 기본값으로 두고 안에서 만든다.
2. **경계에서 복사하라.** 외부에서 받은 가변 객체를 그대로 저장하지 말고 복사본을 저장한다.
3. **가능하면 불변으로 설계하라.** `tuple`, `frozenset`, `@dataclass(frozen=True)`. 별칭 버그는 원천적으로 불가능해진다. ([2.6 dataclasses](#/dataclasses))
4. **린터를 켜라.** `B006` 이 1번을 자동으로 잡는다.
:::

## 요약

이 절의 내용을 한 문단으로 압축하면 이렇다.

> 파이썬에서 **변수는 상자가 아니라 객체에 붙은 이름표**다. `=` 는 복사가 아니라 이름표를 붙이는 행위이고, 함수 인자 전달도 똑같다. 그래서 여러 이름이 한 객체를 가리킬 수 있다(**별칭**). 객체가 **불변**이면 이 사실이 드러나지 않지만, **가변**이면 한쪽에서의 수정이 다른 쪽에 보인다. 별칭이 싫으면 복사해야 하는데, **얕은 복사**는 바깥만, **깊은 복사**는 안쪽까지 새로 만든다. 객체는 **참조 카운트가 0이 되면** 즉시 죽고, 순환 참조는 별도의 GC가 처리한다.

체크리스트:

- [ ] `is` 는 정체성, `==` 는 값. `is` 는 `None`/`True`/`False` 에만.
- [ ] `a = b` 는 절대 복사하지 않는다.
- [ ] 함수에 넘긴 가변 객체는 함수가 바꿀 수 있다.
- [ ] 가변 기본값(`def f(x=[])`)은 금지.
- [ ] `[[0]*3]*3` 은 같은 행 3개. `[[0]*3 for _ in range(3)]` 을 써라.
- [ ] `deepcopy` 는 비싸다. 불변 설계가 낫다.
