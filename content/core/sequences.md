# 1.3 시퀀스: list, tuple, range

::: lead
파이썬에서 가장 많이 쓰는 자료구조는 리스트다. 그리고 가장 많이 오해받는 것도 리스트다. `append` 는 빠른데 `insert(0, x)` 는 왜 느린가, `getsizeof` 로 재 보면 왜 원소 수와 안 맞는가, 슬라이싱은 복사인가 뷰인가, 튜플은 정말 "빠른 리스트"인가. 이 절은 이 질문들에 **측정치로** 답한다. 여기를 넘기면 자료구조 선택이 감이 아니라 계산이 된다.
:::

## 같은 O(n)번 반복인데 300배 차이가 난다

리스트에 10만 개를 넣는 두 가지 방법이다. 하는 일은 똑같다.

```python title="같은 일, 다른 방향"
l = []
for i in range(100_000):
    l.append(i)          # 뒤에 넣기

l = []
for i in range(100_000):
    l.insert(0, i)       # 앞에 넣기
```

실측 결과다.

::: perf 방향이 바꾸는 것
```text nolines
   append 100k     : 0.0021 s
   insert(0) 100k  : 1.0032 s        <- 약 480배
```

이건 상수 차이가 아니다. 입력 크기를 두 배씩 늘려 보면 정체가 드러난다.

```text nolines
   n         insert(0)     append
   10,000    0.0102 s      0.0002 s
   20,000    0.0406 s      0.0003 s      <- insert 는 4배씩
   40,000    0.1597 s      0.0011 s
   80,000    0.6263 s      0.0014 s
```

`insert(0)` 는 n이 2배가 될 때 시간이 **4배**가 된다. $O(n^2)$ 이다. `append` 는 2배씩 늘어난다. $O(n)$ 이다.

(Python 3.14.5 / Windows 기준 실측. 절대값은 기기마다 다르지만 **증가율**은 어디서나 같다.)
:::

이 차이는 리스트의 내부 구조에서 나온다. 거기부터 보자.

## list는 "포인터의 동적 배열"이다

리스트가 원소를 담고 있다는 말은 틀렸다. **리스트는 원소를 가리키는 포인터를 담고 있다.**

```text nolines
lst = [10, 20, 30]

   lst ──▶ ┌─────────────────┐
           │ PyListObject    │
           │ ob_refcnt       │
           │ ob_type         │
           │ ob_size    = 3  │        <- 지금 들어 있는 원소 수
           │ allocated  = 4  │        <- 미리 확보해 둔 칸 수
           │ ob_item    ──┐  │
           └──────────────┼──┘
                          │
                          ▼
           ┌────┬────┬────┬────┐
           │ p0 │ p1 │ p2 │ -- │      <- 포인터 배열. 마지막 칸은 여유분
           └─┬──┴─┬──┴─┬──┴────┘
             │    │    │
             ▼    ▼    ▼
           [10]  [20]  [30]           <- 각각 따로 떠 있는 int 객체
```

세 가지가 여기서 바로 따라 나온다.

1. **인덱싱은 진짜 $O(1)$ 이다.** `lst[i]` 는 `ob_item[i]` 를 읽는 배열 접근이다. 링크드 리스트가 아니다.
2. **한 리스트에 아무 타입이나 섞을 수 있다.** 칸에 들어 있는 건 전부 그냥 포인터다. 타입이 다르다고 칸 크기가 달라지지 않는다.
3. **리스트는 원소 자체의 메모리를 갖고 있지 않다.** `sys.getsizeof(lst)` 가 원소의 크기를 세지 않는 이유다.

```pyrepl
>>> import sys
>>> sys.getsizeof([1, 2, 3])
88
>>> sys.getsizeof([[0] * 1000, [0] * 1000, [0] * 1000])
80
```

세 원소짜리 리스트가 88, 24KB짜리 데이터를 붙들고 있는 세 원소짜리 리스트가 **80**이다. 오히려 더 작다(만드는 경로가 달라서인데, 곧 설명한다). `getsizeof` 는 **껍데기만** 잰다. 이건 [5.2 메모리 모델](#/memory)에서 `tracemalloc` 으로 제대로 재는 법을 다룬다.

::: deep 56이라는 숫자의 정체
빈 리스트는 56바이트다.

```pyrepl
>>> sys.getsizeof([])
56
>>> [].__sizeof__()
40
```

`__sizeof__` 는 40이고 `getsizeof` 는 56이다. 차이 16바이트는 **GC 헤더**다. 리스트는 순환 참조를 만들 수 있으므로 순환 검출기의 추적 대상이고, `getsizeof` 는 그 헤더까지 더해 준다. ([1.1](#/objects-names)에서 본 그 GC다.)

이 절의 모든 `getsizeof` 수치는 **64비트 CPython(포인터 8바이트) 기준**이다(Python 3.14.5, Windows, win-amd64에서 실측). 필드 하나가 8바이트가 아닌 32비트 빌드나 다른 포인터 크기의 플랫폼에서는 40, 56, 88 같은 절대값이 전부 달라진다 — 다만 "포인터 개수 × 포인터 크기 + 고정 오버헤드"라는 **구조**는 그대로다.

40바이트는 정확히 이렇게 쪼개진다.

```text nolines
   ob_refcnt     8
   ob_type       8
   ob_size       8      <- len(lst)
   ob_item       8      <- 포인터 배열의 주소
   allocated     8      <- 확보한 칸 수
   -------------------
                40
```

그래서 **리스트 크기 = 56 + 8 × allocated** 다. `len` 이 아니라 `allocated` 인 게 핵심이다.
:::

## 증폭: allocated는 len보다 크다

빈 리스트에 하나씩 넣으면서 크기를 재 보자.

```python title="증폭 패턴 실측"
import sys

lst = []
prev = -1
for i in range(200):
    size = sys.getsizeof(lst)
    if size != prev:
        print(f"len={i:3d}  getsizeof={size:4d}  allocated={(size - 56) // 8}")
        prev = size
    lst.append(i)
```

```text nolines
   len=  0  getsizeof=  56  allocated= 0
   len=  1  getsizeof=  88  allocated= 4
   len=  5  getsizeof= 120  allocated= 8
   len=  9  getsizeof= 184  allocated=16
   len= 17  getsizeof= 248  allocated=24
   len= 25  getsizeof= 312  allocated=32
   len= 33  getsizeof= 376  allocated=40
   len= 41  getsizeof= 472  allocated=52
```

`allocated` 수열은 이렇게 이어진다.

```text nolines
   0, 4, 8, 16, 24, 32, 40, 52, 64, 76, 92, 108, 128, 148, 172, 200,
   232, 268, 308, 352, 400, 456, 520, 592, 672, 760, 860, 972, 1100, ...
```

**칸이 꽉 차면 더 큰 배열을 새로 잡고, 기존 포인터를 통째로 옮긴다.** 그런데 딱 필요한 만큼만 잡지 않고 **여유분을 붙여서** 잡는다. 그래야 다음 `append` 들이 공짜가 된다.

::: deep CPython의 증폭 공식
`Objects/listobject.c` 의 `list_resize()` 가 새 용량을 이렇게 정한다.

```c
new_allocated = ((size_t)newsize + (newsize >> 3) + 6) & ~(size_t)3;
```

- `newsize >> 3` — 현재 크기의 1/8을 더한다. **증폭 계수 1.125**다.
- `+ 6` — 작은 리스트에서 1/8이 0이 되는 걸 막는 최소 여유분.
- `& ~3` — 4의 배수로 내림. 정렬을 맞춘다.

파이썬으로 그대로 옮겨 확인할 수 있다.

```python title="공식이 측정과 맞는지 확인"
def cap(n):
    return 0 if n == 0 else (n + (n >> 3) + 6) & ~3


for n in (1, 5, 9, 17, 25, 33, 41):
    print(n, cap(n))
```

```text nolines
   1 4
   5 8
   9 16
   17 24
   25 32
   33 40
   41 52
```

위에서 실측한 수열과 정확히 일치한다.

C++ `std::vector` 나 Java `ArrayList` 는 보통 **2배**로 늘린다. CPython이 1.125를 고른 건 **메모리 낭비를 줄이려는 선택**이다. 파이썬 프로그램은 리스트를 수천 개씩 들고 다니는데, 전부 최대 2배씩 부풀어 있으면 곤란하다. 대신 재할당이 더 자주 일어난다. 이 트레이드오프의 대가는 잠시 뒤에 숫자로 나온다.
:::

::: warn getsizeof 는 만드는 방법에 따라 달라진다
같은 `[0, 1, 2]` 인데 만든 경로에 따라 `allocated` 가 다르다.

```pyrepl
>>> import sys
>>> a = []
>>> for i in range(3): a.append(i)
...
>>> sys.getsizeof(a)
88
>>> sys.getsizeof([0] * 3)
80
>>> sys.getsizeof(list(range(3)))
88
>>> sys.getsizeof([i for i in range(3)])
88
```

`[0] * 3` 은 **80**이다. 곱셈은 결과 길이를 미리 정확히 알기 때문에 증폭 공식을 아예 타지 않고 딱 3칸만 잡는다.

`list(range(3))` 도 길이를 미리 안다(`__length_hint__`). 이 경로는 증폭 공식 대신 짝수로만 맞춰 잡는다 — 100개면 100칸, 101개면 102칸이다.

컴프리헨션은 다르다. 길이를 미리 알 수 없으니 `append` 와 똑같이 증폭 공식을 탄다. 그래서 3개짜리인데 4칸이다.

리터럴은 **원소가 상수냐 아니냐**에 따라 갈린다. 이게 앞에서 본 이상한 결과의 정체다.

```pyrepl
>>> sys.getsizeof([1, 2, 3])                  # 전부 상수
88
>>> x = 1
>>> sys.getsizeof([x, x, x])                  # 변수가 섞였다
80
```

전부 상수면 컴파일러가 `LIST_EXTEND` 경로를 쓰고(4칸), 아니면 `BUILD_LIST 3` 으로 딱 3칸을 잡는다. 바이트코드는 잠시 뒤 `dis` 로 직접 본다.

**교훈: 크기를 아는 데이터는 `list(iterable)` 이나 `[x] * n` 으로 만들어라.** 재할당이 한 번도 안 일어나고 메모리도 안 뜬다.
:::

## append가 상환 $O(1)$ 인 이유, 그리고 그 상수

`append` 는 대부분 그냥 빈 칸에 포인터 하나 쓰고 `ob_size` 를 1 늘리는 게 전부다. 진짜 $O(1)$ 이다. 가끔씩만 재할당 + 복사가 일어나 $O(n)$ 이 된다.

**상환 분석**(amortized analysis)은 이 "가끔"의 비용을 전체에 나눠 붙인다. 100만 번 `append` 하는 동안 재할당이 몇 번 일어나는지 세 보자.

```python title="상환 비용을 직접 세어 본다"
def cap(n):
    return 0 if n == 0 else (n + (n >> 3) + 6) & ~3


allocated = 0
moved = 0
resizes = 0
for size in range(1, 1_000_001):
    if size > allocated:
        moved += size - 1        # 최악의 경우: 기존 원소를 전부 옮긴다
        allocated = cap(size)
        resizes += 1

print("재할당 횟수:", resizes)
print("총 이동 원소 수:", moved)
print("원소당 평균 이동:", round(moved / 1_000_000, 2))
```

```text nolines
   재할당 횟수: 86
   총 이동 원소 수: 8445096
   원소당 평균 이동: 8.45
```

100만 번 중 재할당은 **86번**뿐이다. 나머지 999,914번은 공짜다. 그래서 상환 $O(1)$ 이다.

::: perf 1.125라는 증폭 계수의 청구서
원소당 평균 이동이 **8.45번**이다. 이게 우연이 아니다.

증폭 계수 $r$ 로 늘리는 동적 배열에서, 원소 $n$ 개를 넣는 동안 일어나는 총 이동 횟수는 등비급수의 합이다.

$$\frac{n}{r-1} = \frac{n}{1.125 - 1} = 8n$$

계수가 2였다면 $\frac{n}{2-1} = n$ — 원소당 **1번**이다. CPython은 메모리를 아끼는 대가로 **이동 횟수를 8배 물고 있다.**

그런데도 실측에서 `append` 100k가 0.0021초로 충분히 빠르다. 이유는 `realloc` 이다. 뒤에 여유 공간이 있으면 OS/할당자가 **복사 없이 제자리에서 늘려 준다.** 위의 8.45는 realloc이 매번 복사한다고 가정한 **상한**이다.

실전 교훈: 100만 원소 리스트를 `append` 로 쌓는 건 괜찮다. 다만 **크기를 미리 안다면** `[None] * n` 으로 잡아 놓고 인덱스로 채우는 게 재할당을 0번으로 만든다.
:::

빈 칸이 남아도는 것도 공짜는 아니다.

```pyrepl
>>> import sys
>>> a = []
>>> for i in range(1_000_000): a.append(i)
...
>>> sys.getsizeof(a)
8448728
>>> sys.getsizeof(list(range(1_000_000)))
8000056
```

`append` 로 쌓은 100만짜리 리스트는 정확히 잡은 것보다 **약 449KB(5.6%)** 더 쓴다. `allocated` 는 1,056,084 — 5만 6천 칸이 비어 있다.

### 리스트는 줄어들기도 한다

`list_resize` 는 **크기가 allocated의 절반 아래로 떨어지면** 배열을 줄인다. 그래서 다 비우면 다시 56바이트로 돌아간다.

```pyrepl
>>> import sys
>>> l = list(range(100))
>>> sys.getsizeof(l)
856
>>> l.clear()
>>> sys.getsizeof(l)
56
```

`del l[:]` 도 같다. 다만 `l = []` 는 전혀 다른 일이다 — 이건 새 객체를 만들어 이름표를 옮기는 것이고, 원래 리스트를 공유하던 다른 이름은 옛 객체를 그대로 본다. ([1.1](#/objects-names))

## 앞쪽 연산이 $O(n)$ 인 이유

이제 도입부의 480배가 설명된다.

```text nolines
   lst.append(40)       ┌────┬────┬────┬────┐
                        │ 10 │ 20 │ 30 │ 40 │     <- 빈 칸에 그냥 쓴다. O(1)
                        └────┴────┴────┴────┘

   lst.insert(0, 99)    ┌────┬────┬────┬────┐
                        │ 10 │ 20 │ 30 │ -- │
                        └────┴────┴────┴────┘
                          └────┴────┴───▶            <- 전부 한 칸씩 민다
                        ┌────┬────┬────┬────┐
                        │ 99 │ 10 │ 20 │ 30 │        <- memmove n개. O(n)
                        └────┴────┴────┴────┘
```

배열이니까 0번 칸을 비우려면 **나머지 전부를 한 칸씩 밀어야 한다.** `pop(0)` 도 똑같이 반대 방향으로 민다.

| 연산 | 복잡도 | 이유 |
| --- | --- | --- |
| `lst[i]` | $O(1)$ | 배열 접근 |
| `lst.append(x)` | 상환 $O(1)$ | 뒤 빈 칸에 쓰기 |
| `lst.pop()` | $O(1)$ | 뒤에서 빼기 |
| `lst.insert(0, x)` | $O(n)$ | 전부 밀기 |
| `lst.pop(0)` | $O(n)$ | 전부 당기기 |
| `del lst[i]` | $O(n-i)$ | 뒤쪽만 당기기 |
| `x in lst` | $O(n)$ | 처음부터 훑기 |
| `lst.index(x)` | $O(n)$ | 처음부터 훑기 |

`memmove` 자체는 C 수준의 블록 복사라 **원소 하나당 상수가 아주 작다.** 그래서 리스트가 수백 개 수준이면 `pop(0)` 도 체감이 안 된다. 문제는 이게 반복문 안에 들어가는 순간 $O(n^2)$ 이 된다는 것이다. 10만에서 터진다.

### 그래서 deque

양쪽 끝에서 넣고 빼야 하면 `collections.deque` 를 쓴다. deque는 **이중 연결 리스트인데, 노드 하나가 원소 64개짜리 블록**이다. 그래서 양 끝 연산이 진짜 $O(1)$ 이다.

::: perf deque가 이기는 곳과 지는 곳
```text nolines
   10만 번 앞에서 빼기
     list.pop(0)      : 0.2308 s
     deque.popleft()  : 0.0023 s        <- 100배
     (참고) list.pop(): 0.0023 s        <- 뒤에서 빼면 list도 같다
```

대신 **가운데 인덱싱은 deque가 $O(n)$ 이다.** 블록을 따라가야 하기 때문이다.

```text nolines
   중앙 원소 20만 번 인덱싱
   n           deque[n//2]    list[n//2]
   1,000       0.0070 s       0.0054 s        <- 1.3배
   10,000      0.0204 s       0.0055 s        <- 3.7배
   100,000     0.3432 s       0.0057 s        <- 60배
   1,000,000   4.1900 s       0.0055 s        <- 약 760배
```

list는 n이 커져도 그대로다. deque는 정직하게 선형으로 늘어난다. `n=1,000,000` 에서 list 쪽 절대 시간이 5밀리초 안팎이라 재실행할 때마다 배율이 700~800배 사이로 흔들린다 — list의 측정 잡음이 배율에 그대로 곱해지기 때문이다. 그래도 경향은 분명하다. **deque는 "빠른 리스트"가 아니다. 양 끝만 빠른 자료구조다.**
:::

::: cote 코딩테스트 포인트
**BFS 큐를 `list` 로 만들면 시간 초과가 난다.** 이건 가장 흔한 파이썬 TLE 원인이다.

```python
# ❌ pop(0) 이 O(n) → 전체 O(V²)
queue = [start]
while queue:
    node = queue.pop(0)
    ...

# ✅ popleft() 가 O(1) → 전체 O(V+E)
from collections import deque
queue = deque([start])
while queue:
    node = queue.popleft()
    ...
```

노드가 10만 개면 이 한 글자 차이가 0.002초와 수십 초를 가른다.

같은 이유로 **슬라이딩 윈도우에서 왼쪽 원소를 버릴 때**, **회전(rotate)이 필요할 때**도 deque다. `deque(maxlen=k)` 는 넘치면 반대쪽을 자동으로 버려서 "최근 k개" 유지에 딱 맞는다.

자세한 건 [7.7 스택과 큐](#/stack-queue), [7.14 BFS/DFS 응용](#/bfs-dfs)에서.
:::

## 슬라이싱은 언제나 복사본이다

이게 NumPy를 하다 온 사람이 가장 많이 넘어지는 지점이다.

```pyrepl
>>> a = [1, 2, 3, 4, 5]
>>> b = a[1:4]
>>> b[0] = 99
>>> a
[1, 2, 3, 4, 5]
```

`b` 는 뷰(view)가 아니라 **새 리스트**다. NumPy의 `arr[1:4]` 는 뷰라서 원본이 바뀐다. 파이썬 리스트는 아니다. ([9.1 NumPy](#/numpy-basics)에서 이 차이를 다시 다룬다.)

```text nolines
   a = [10, 20, 30, 40, 50]
   b = a[1:4]

   a ──▶ ┌────┬────┬────┬────┬────┐
         │ p0 │ p1 │ p2 │ p3 │ p4 │
         └────┴─┬──┴─┬──┴─┬──┴────┘
                │    │    │
                │    │    │                <- 포인터 3개만 복사한다
                ▼    ▼    ▼
   b ──▶ ┌────┬────┬────┐
         │ p1 │ p2 │ p3 │                  <- 새 PyListObject
         └────┴────┴────┘
```

**복사되는 건 포인터뿐이다.** 가리키는 객체는 그대로 공유된다. 그래서 슬라이싱은 **얕은 복사**다 — `a[:]` 가 `list(a)`, `a.copy()`, `copy.copy(a)` 와 정확히 같은 일을 하는 이유다. 중첩 리스트에서 무슨 일이 나는지는 [1.1](#/objects-names)에서 봤다.

비용은 $O(k)$ 다. $k$ 는 **잘라 내는 길이**지 원본 길이가 아니다.

::: perf 슬라이스 비용은 결과 길이에 비례한다
```text nolines
   100만짜리 리스트에서
     l[:]        : 0.00373 s      <- 100만 개 포인터 복사
     l[:10]      : 0.00000006 s   <- 10개만 복사. 6만 배 차이
     l[::-1]     : 0.00355 s      <- 뒤집기도 복사다
     reversed(l) : 0.00000008 s   <- 이터레이터. 복사 안 한다
```

`l[::-1]` 과 `reversed(l)` 은 같은 것이 아니다. 앞은 리스트를 통째로 만들고, 뒤는 **게으른 이터레이터**를 반환한다. 그냥 순회할 거면 `reversed` 를 써라. ([1.18 이터레이터](#/iterators))
:::

::: danger 반복문 안의 슬라이싱은 조용히 $O(n \times k)$ 다
```python
# ❌ 매 반복마다 1000개를 복사한다
for i in range(len(data) - 1000):
    total = sum(data[i:i + 1000])
```

슬라이싱이 짧은 한 줄이라 싸 보인다. 아니다. 위 코드는 10만짜리 리스트에서 **1억 번**의 포인터 복사를 한다.

```text nolines
   100k 리스트, 폭 1000 윈도우 합
     슬라이스 방식 : 0.29 s
     누적합 방식   : 0.006 s        <- 약 50배
```

```python
# ✅ 누적합을 한 번 만들고 뺄셈만
from itertools import accumulate

p = [0] + list(accumulate(data))
for i in range(len(data) - 1000):
    total = p[i + 1000] - p[i]
```

**슬라이싱은 값이 아니라 연산이다.** 반복문 안에 있으면 항상 의심하라. [7.3 투 포인터](#/two-pointers)의 핵심 동기가 바로 이것이다.
:::

### 슬라이싱은 범위를 벗어나도 예외를 내지 않는다

```pyrepl
>>> l = [1, 2, 3]
>>> l[10:20]
[]
>>> l[3:1]
[]
>>> l[-100:2]
[1, 2]
>>> l[5]
Traceback (most recent call last):
  ...
IndexError: list index out of range
```

인덱싱은 `IndexError` 를 내고 슬라이싱은 조용히 자른다. **이 관대함이 버그를 숨긴다.** 오프바이원 실수가 예외 대신 빈 리스트로 나타나면 훨씬 늦게 발견된다.

::: deep slice 는 객체다
`a[1:5:2]` 의 `1:5:2` 는 문법 설탕이 아니라 진짜 객체를 만든다. `a.__getitem__(slice(1, 5, 2))` 로 번역된다.

```python title="대괄호 안에 무엇이 오는지 직접 보기"
class Probe:
    def __getitem__(self, key):
        print(repr(key))


p = Probe()
p[1:5]
p[1:5:2]
p[::-1]
p[1:5, ::2]
p[...]
```

```text nolines
   slice(1, 5, None)
   slice(1, 5, 2)
   slice(None, None, -1)
   (slice(1, 5, None), slice(None, None, 2))
   Ellipsis
```

`p[1:5, ::2]` 가 **슬라이스 두 개짜리 튜플**로 오는 것을 보라. NumPy의 `arr[1:5, ::2]` 가 동작하는 원리가 정확히 이거다. `Ellipsis` 도 마찬가지다 — 파이썬 문법은 이미 다차원 인덱싱을 지원하고, 리스트가 그걸 안 받을 뿐이다.

`slice` 객체에는 유용한 메서드가 하나 있다.

```pyrepl
>>> slice(1, 10, 2).indices(5)
(1, 5, 2)
>>> list(range(*slice(1, 10, 2).indices(5)))
[1, 3]
```

`indices(len)` 이 음수 인덱스와 범위 초과를 전부 정규화해 준다. 직접 시퀀스를 만들 때 이걸 쓰면 경계 조건을 손으로 짤 필요가 없다. ([1.14 특수 메서드](#/dunder))
:::

## 슬라이스 대입: 리스트의 만능 도구

슬라이스는 대입의 **왼쪽**에도 올 수 있다. 이게 되면 리스트 조작의 거의 전부가 한 줄이 된다.

```pyrepl
>>> c = [1, 2, 3, 4, 5]
>>> c[1:3] = [10, 20, 30, 40]     # 길이가 달라도 된다
>>> c
[1, 10, 20, 30, 40, 4, 5]

>>> g = [1, 2, 3, 4, 5]
>>> g[2:2] = [99, 98]             # 빈 슬라이스에 대입 = 삽입
>>> g
[1, 2, 99, 98, 3, 4, 5]

>>> f = [1, 2, 3, 4, 5]
>>> del f[1:4]                    # 구간 삭제
>>> f
[1, 5]

>>> h = [1, 2, 3]
>>> h[len(h):] = [4]              # h.append(4) 와 같다
>>> h
[1, 2, 3, 4]
```

**확장 슬라이스**(step이 있는 것)에 대입할 때는 규칙이 하나 더 붙는다. 길이가 정확히 맞아야 한다.

```pyrepl
>>> d = [1, 2, 3, 4, 5]
>>> d[::2] = [0, 0, 0]
>>> d
[0, 2, 0, 4, 0]
>>> d[::2] = [0, 0]
Traceback (most recent call last):
  ...
ValueError: attempt to assign sequence of size 2 to extended slice of size 3
```

당연하다. `d[1:3] = [...]` 는 "구간을 통째로 갈아 끼우기"라 길이가 달라도 되지만, `d[::2] = [...]` 는 "흩어진 칸들에 하나씩 넣기"라 개수가 맞아야 한다.

::: tip a[:] = ... 는 "내용만 갈아 끼우기"다
이 관용구는 알아 둘 값어치가 있다.

```pyrepl
>>> a = [1, 2, 3, 4, 5]
>>> b = a
>>> a[:] = [9, 9]        # 이름표를 옮기지 않는다. 객체 내용을 바꾼다.
>>> b
[9, 9]
>>> b is a
True
```

`a = [9, 9]` 였다면 `b` 는 `[1, 2, 3, 4, 5]` 를 계속 봤을 것이다. `a[:] = [9, 9]` 는 **모든 참조자에게 보이는 변경**이다.

이게 필요한 실전 상황: 리스트를 여러 곳에서 공유하고 있는데(설정 리스트, ROS 노드의 버퍼, 콜백에 넘긴 리스트) 내용을 전부 교체해야 할 때. `list.clear()` + `extend()` 두 줄을 한 줄로 쓴 것이다.
:::

## tuple은 왜 더 작고 빠른가

튜플은 "수정 못 하는 리스트"가 아니다. **구조 자체가 다르다.**

```text nolines
   list    obj ──▶ ptr array ──▶ items      <- 할당 2번, 포인터 두 번 따라감
   tuple   obj ──▶ items                    <- 할당 1번
```

리스트는 껍데기와 포인터 배열이 **따로** 할당된다. 튜플은 포인터가 **객체 안에 바로 붙어 있다.** 크기가 절대 안 변하니까 그럴 수 있다.

```pyrepl
>>> import sys
>>> sys.getsizeof(())
48
>>> sys.getsizeof([])
56
>>> sys.getsizeof((1, 2, 3))
72
>>> sys.getsizeof([1, 2, 3])
88
```

세 가지가 따라온다.

1. **할당이 1번**이다. 리스트는 2번. 만들고 버리는 게 잦으면 이 차이가 쌓인다.
2. **`allocated` 필드가 없다.** 여유분도 없다. 딱 필요한 만큼만 쓴다.
3. **간접 참조가 한 단계 적다.** 캐시 지역성이 좋다.

::: perf 그런데 인덱싱은 안 빠르다
정직하게 재 보면 이렇다.

```text nolines
   1000개 시퀀스에서 중앙 원소 1000만 번 읽기
     t[500]     : 0.0879 s
     data[500]  : 0.0834 s        <- 리스트가 오히려 근소하게 빠르다
```

**"튜플이 리스트보다 빠르다"는 인덱싱에 대해서는 거짓이다.** 둘 다 배열 접근이고, 포인터를 한 번 더 따라가는 비용은 CPU 캐시에 묻힌다.

튜플이 이기는 건 **생성**이다.

```text nolines
   1000만 번 생성 (a, b, c 는 변수)
     (a, b, c)  : 0.1285 s
     [a, b, c]  : 0.1815 s        <- 리스트가 1.4배 느리다
```

차이가 나는 이유: 리스트는 매번 두 번 할당하고, 튜플은 CPython의 **작은 튜플 free list**에서 재활용된다.

그리고 상수 튜플은 아예 런타임 비용이 0이다.

```text nolines
   1000만 번
     (1, 2, 3)  : 0.039 s        <- LOAD_CONST 한 방
     [1, 2, 3]  : 0.2002 s       <- 매번 새로 만든다. 5배
```
:::

::: deep 상수 튜플은 컴파일 타임에 만들어진다
```pyrepl
>>> import dis
>>> dis.dis(compile("x = (1, 2, 3)", "<s>", "exec"))
  0           RESUME                   0

  1           LOAD_CONST               2 ((1, 2, 3))
              STORE_NAME               0 (x)
              LOAD_CONST               1 (None)
              RETURN_VALUE
```

튜플이 통째로 `co_consts` 에 상수로 들어간다. 실행할 때는 그냥 꺼내 쓴다.

리스트는 그럴 수 없다. **가변이니까** 같은 객체를 재사용하면 안 된다. 그래서 매번 새로 만든다.

```pyrepl
>>> dis.dis(compile("x = [1, 2, 3]", "<s>", "exec"))
  0           RESUME                   0

  1           BUILD_LIST               0
              LOAD_CONST               2 ((1, 2, 3))
              LIST_EXTEND              1
              STORE_NAME               0 (x)
              LOAD_CONST               1 (None)
              RETURN_VALUE
```

재미있는 절충안이 보인다. 원소를 하나씩 스택에 올리는 대신, **상수 튜플 하나를 통째로 만들어 놓고 `LIST_EXTEND` 로 붓는다.** 스택 조작을 줄이는 최적화다. [3.7 바이트코드](#/bytecode)에서 이런 걸 읽는 법을 다룬다.
:::

::: warn tuple(t) 는 복사하지 않는다
불변이라는 사실이 만드는 최적화다.

```pyrepl
>>> t = (1, 2, 3)
>>> tuple(t) is t
True
>>> t[:] is t
True
>>> data = [1, 2, 3]
>>> list(data) is data
False
>>> data[:] is data
False
```

**튜플은 자기 자신을 그대로 반환한다.** 어차피 못 바꾸니 복사할 이유가 없다.

```text nolines
   1000개 시퀀스, 1000만 번
     tuple(t)     : 0.0086 s
     list(data)   : 0.9133 s        <- 106배
```

106배는 알고리즘 차이가 아니다. 하나는 $O(1)$ (참조 카운트만 올림), 하나는 $O(n)$ (진짜 복사)다.

**함정**: 방어적 복사를 하려고 `list(data)` 를 쓰던 습관으로 `tuple(t)` 를 쓰면 복사가 안 된다. 튜플에서는 그게 맞는 동작이지만, "복사했다"고 착각하면 안 된다.
:::

### 튜플을 써야 하는 진짜 이유

성능이 아니다. **해시 가능성**과 **의미**다.

```pyrepl
>>> hash((1, 2, 3))
529344067295497451
>>> hash((1, [2, 3]))
Traceback (most recent call last):
  ...
TypeError: unhashable type: 'list'
```

튜플은 원소가 전부 해시 가능하면 자기도 해시 가능하다. 그래서 **dict의 키**, **set의 원소**가 될 수 있다. 리스트는 절대 못 된다.

```python title="튜플 키는 파이썬에서 가장 자주 쓰는 관용구 중 하나"
visited = set()
visited.add((r, c))                 # 좌표를 집합에 넣는다

dp = {}
dp[(idx, remaining)] = value        # 다차원 메모이제이션

grid = {(0, 0): "S", (2, 3): "G"}   # 희소 격자
```

`(1, [2, 3])` 이 해시 불가능한 이유가 중요하다. **튜플의 불변성은 얕다.** 튜플은 "내가 가리키는 포인터를 안 바꾼다"고 약속할 뿐, 그 포인터가 가리키는 객체까지 지켜 주지 않는다. 그래서 안에 리스트가 있으면 해시가 시간에 따라 변할 수 있고, CPython은 그걸 아예 금지한다. ([1.7 set](#/sets), [1.6 dict](#/dict))

::: tip 의미로 고르는 기준
- **튜플** — 위치마다 뜻이 다른 **레코드**. `(x, y)`, `(name, age, email)`. 길이가 타입의 일부다.
- **리스트** — 같은 종류가 여러 개 있는 **컬렉션**. 길이가 몇이든 상관없다.

`for x, y in points:` 가 자연스러운 이유, `points` 자체는 리스트인 이유다. 이름을 붙이고 싶어지면 `NamedTuple` 이나 `dataclass` 로 간다. ([2.5](#/typed-containers), [2.6](#/dataclasses))
:::

## range: 게으른 시퀀스

`range` 는 이터레이터가 아니다. **시퀀스**다. 그런데 원소를 하나도 갖고 있지 않다.

```pyrepl
>>> import sys
>>> sys.getsizeof(range(10 ** 18))
48
>>> sys.getsizeof(list(range(1_000_000)))
8000056
```

$10^{18}$ 개짜리 range가 **48바이트**다. `start`, `stop`, `step` 세 개만 저장하고 나머지는 **필요할 때 계산한다.**

$$\text{r}[i] = \text{start} + i \times \text{step}$$

이 한 줄이 range의 전부다. 그래서 인덱싱이 $O(1)$ 이고, `len` 도 $O(1)$ 이고, 슬라이싱은 또 다른 range를 만든다.

```pyrepl
>>> r = range(10 ** 18)
>>> r[999_999_999]
999999999
>>> len(range(10))
10
>>> range(0, 100, 3)[2:5]
range(6, 15, 3)
>>> r.index(500)
500
```

`range(0, 100, 3)[2:5]` 가 **리스트가 아니라 range**를 반환하는 것을 보라. 슬라이싱해도 메모리를 안 쓴다.

::: perf `in` 이 $O(1)$ 이다
가장 놀라운 건 이거다.

```pyrepl
>>> 10 ** 18 - 1 in range(10 ** 18)
True
```

즉시 나온다. range는 `__contains__` 를 **산술로** 구현한다 — 범위 안인지 보고, `(x - start) % step == 0` 인지 본다. 훑지 않는다.

```text nolines
     99999999 in range(10**8)  : 0.0305 s  / 100만 회     <- O(1)
     99999    in list(10**5)   : 0.3747 s  / 1천 회       <- O(n)
```

호출 횟수가 1000배 다른데도 리스트가 12배 느리다. 원소당으로는 **1만 2천 배**다.

**단, 정수일 때만이다.** 정수가 아니면 진짜로 훑는다.

```pyrepl
>>> 1.0 in range(10)
True
>>> "a" in range(10)
False
```

`1.0 in range(10)` 이 `True` 인 건 `1.0 == 1` 이기 때문이다. 이건 선형 탐색 경로를 탄다.
:::

::: hist range는 원래 리스트였다
파이썬 2에서 `range(10)` 은 **리스트를 만들어 반환했다.** `range(10**8)` 을 치면 메모리가 터졌다. 그래서 `xrange` 라는 게으른 버전이 따로 있었다.

파이썬 3에서 `xrange` 가 `range` 를 대체했다. 이름을 물려받은 것이다. 파이썬 2 코드를 3으로 옮길 때 `range(n)` 이 리스트를 준다고 가정한 코드가 전부 깨진 이유다.

```python
# Python 2 스타일 — 3에서는 range 객체다
r = range(5)
r.append(6)      # AttributeError
```

같은 이유로 `map`, `filter`, `zip`, `dict.keys()` 도 3에서 전부 게을러졌다. **"만들어 두지 말고 요청받을 때 계산한다"** 는 파이썬 3의 일관된 방향이다. [1.18 이터레이터](#/iterators)에서 이 철학을 끝까지 밀어붙인다.
:::

::: cote 코딩테스트 포인트
**`for i in range(n)` 앞에 `list()` 를 붙이지 마라.** 하나도 얻는 게 없고 메모리만 쓴다.

```python
for i in list(range(1_000_000)):    # ❌ 8MB 낭비
for i in range(1_000_000):          # ✅ 48바이트
```

`sum(range(10**7))` 은 0.05초다. 1천만 개 리스트를 안 만들기 때문이다.

그리고 **역순 순회**는 이렇게 한다.

```python
for i in range(n - 1, -1, -1):      # ✅ range 그대로
for i in reversed(range(n)):        # ✅ 이것도 range의 __reversed__ → O(1)
for i in range(n)[::-1]:            # ✅ 이것도 range를 반환한다
for i in list(range(n))[::-1]:      # ❌ 리스트를 두 번 만든다
```

`reversed(range(n))` 이 가장 읽기 쉽다. 세 번째까지는 전부 메모리를 안 쓴다.
:::

## 요약

- **리스트는 포인터의 동적 배열**이다. 원소를 담고 있는 게 아니라 가리키고 있다. 그래서 인덱싱이 $O(1)$ 이고, 타입을 섞을 수 있고, `getsizeof` 가 원소 크기를 안 센다.
- **크기 = 56 + 8 × allocated.** `len` 이 아니라 `allocated` 다. CPython은 `(n + n//8 + 6) & ~3` 만큼 미리 잡는다 — **증폭 계수 1.125**.
- **`append` 는 상환 $O(1)$.** 100만 번에 재할당 86번. 대신 증폭 계수가 작아서 원소당 이동 상한이 8.45번이다. 크기를 알면 `[None] * n` 이나 `list(iterable)` 로 만들어 재할당을 0번으로 만들어라.
- **앞쪽 연산은 $O(n)$.** `insert(0)`/`pop(0)` 은 배열을 통째로 민다. 10만에서 480배 차이가 난다. 양 끝을 쓰면 **deque**. 단 deque의 중앙 인덱싱은 $O(n)$ 이다.
- **슬라이싱은 항상 얕은 복사**다. NumPy와 다르다. 비용은 결과 길이에 비례한다. **반복문 안의 슬라이싱을 의심하라** — 조용히 $O(n \times k)$ 가 된다.
- **슬라이스 대입**으로 삽입·삭제·교체를 한 줄에 한다. `a[:] = [...]` 는 이름표를 옮기지 않고 내용을 갈아 끼운다.
- **튜플은 인덱싱이 빠른 게 아니라 생성이 싸다.** 진짜 이유는 **해시 가능성**이다 — dict 키, set 원소. 불변성은 **얕다**.
- **range는 원소를 저장하지 않는다.** 48바이트, 인덱싱 $O(1)$, 슬라이싱은 range 반환, 정수 `in` 은 $O(1)$.
