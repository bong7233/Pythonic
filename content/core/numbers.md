# 1.2 숫자와 수치 연산

::: lead
C에서 `int`에 20억을 넘게 넣으면 값이 음수로 뒤집힌다. 파이썬에서는 그런 일이 없다. 대신 다른 대가를 치른다. 그리고 정수에서 안전하다고 방심한 순간, `float`가 `0.1 + 0.2 != 0.3`으로 뒤통수를 친다. 이 절은 파이썬의 숫자가 **어떤 대가를 치르고 무엇을 사는지**를 다룬다. 숫자를 정확히 다루지 못하면 금액 계산도, 물리 시뮬레이션도, 코딩테스트의 모듈러 연산도 조용히 틀린다.
:::

## 정수는 어디까지 커지는가

C에서 `int32_t`에 2,147,483,647을 넣고 1을 더하면 -2,147,483,648이 된다. 오버플로다. 경고도 없다. 파이썬에서 같은 걸 해 보라.

```pyrepl
>>> 2**31 - 1 + 1
2147483648
>>> 2**1000
10715086071862673209484250490600018105614048117055336074437503883703510511249361224931983788156958581275946729175531468251871452856923140435984577574698574803934567774824230985421074605062371141877954182153046474983581941267398767559165543946077062914571196477686542167660429831652624386837205668069376
>>> len(str(2**1000))
302
>>> (2**100000).bit_length()

```

파이썬 `int`는 **임의 정밀도**(arbitrary precision)다. 메모리가 허락하는 한 커진다. 오버플로라는 개념 자체가 없다. 이게 공짜일 리 없다. 대가가 무엇인지 보자.

### int 객체의 실제 모습

```pyrepl
>>> import sys
>>> sys.int_info
sys.int_info(bits_per_digit=30, sizeof_digit=4, default_max_str_digits=4300, str_digits_check_threshold=640)
```

`bits_per_digit=30`. CPython은 정수를 **30비트짜리 조각(digit)의 배열**로 저장한다. 우리가 10진법으로 자릿수를 쓰듯, CPython은 $2^{30}$진법으로 자릿수를 쓴다.

```text nolines
   PyLongObject  (CPython 3.12+)
   ┌───────────────────┬──────────┐
   │ ob_refcnt         │  8 bytes │   <- 참조 카운트 (1.1절)
   │ ob_type           │  8 bytes │   <- 타입 포인터
   │ lv_tag            │  8 bytes │   <- 부호 + digit 개수
   ├───────────────────┼──────────┤
   │ ob_digit[0]       │  4 bytes │   <- 하위 30비트
   │ ob_digit[1]       │  4 bytes │   <- 그 위 30비트
   │ ob_digit[2]       │  4 bytes │
   │ ...               │          │
   └───────────────────┴──────────┘
   header = 24 bytes,  digit 하나당 4 bytes
```

`sizeof_digit=4`인데 `bits_per_digit=30`이다. **4바이트(32비트) 중 30비트만 쓴다.** 2비트를 버리는 셈이다. 왜?

::: deep 왜 32비트가 아니라 30비트인가
곱셈 때문이다. digit 두 개를 곱하면 중간 결과를 담을 그릇이 필요하다. 30비트 × 30비트 = 최대 60비트 → `uint64_t` 하나에 들어가고 **자리올림을 누적할 여유 4비트까지 남는다.** 32비트를 다 썼다면 32×32=64비트라 자리올림에 별도 코드가 필요하고, 128비트 정수가 없는 컴파일러에서는 이식이 깨진다.

즉 30은 *"메모리를 6% 낭비하는 대신 곱셈 루프를 단순하게 만든다"* 는 거래의 결과다. 임의로 고른 값이 아니라 **곱셈 알고리즘이 요구한 값**이다. 그러니 `sys.int_info`를 하드코딩하지 말고 읽어라 — 64비트 산술이 없는 플랫폼용 빌드는 같은 논리로 15비트 digit을 쓴다.
:::

메모리 비용을 직접 확인하자.

```pyrepl
>>> import sys
>>> sys.getsizeof(0)
28
>>> sys.getsizeof(2**30 - 1)          # digit 1개
28
>>> sys.getsizeof(2**30)              # digit 2개
32
>>> sys.getsizeof(2**60)              # digit 3개
36
>>> sys.getsizeof(10**100)
72
>>> sys.getsizeof(2**1000)
160
```

정확히 $24 + 4 \times (\text{digit 개수})$ 다. `0`조차 28바이트다. C의 `int` 4바이트와 비교하면 **7배**다. 이것이 임의 정밀도의 첫 번째 청구서다.

::: perf 큰 수 곱셈은 언제부터 비싸지는가
초등학교에서 배운 곱셈은 자릿수의 제곱에 비례한다. CPython은 일정 크기를 넘으면 **카라추바(Karatsuba)** 로 갈아탄다. $O(n^2)$ 대신 $O(n^{1.585})$ 다.

```text nolines
    10 digits:      0.023 us/op
   100 digits:      0.116 us/op
  1000 digits:      5.990 us/op
 10000 digits:    242.860 us/op
100000 digits:   8479.030 us/op
```

표의 배수를 직접 계산하면 구간마다 다르다: 10→100자리는 약 5배, 100→1000자리는 약 52배, 1000→10000자리는 약 41배, 10000→100000자리는 약 35배. **뒤쪽 두 구간만** $10^{1.585} \approx 38$ 에 수렴한다 — 카라추바가 실제로 돌고 있다는 증거다.

앞쪽 구간(10→100자리)이 벗어나는 이유는 자릿수가 작을 때는 곱셈 자체보다 **함수 호출·객체 생성 오버헤드**가 시간을 지배하기 때문이다. 10자리 곱셈은 어차피 30비트 digit 1~2개짜리라 CPU 명령 몇 개면 끝나고, 파이썬 인터프리터가 정수 객체를 만들고 참조를 세는 비용이 오히려 더 크다. 자릿수가 충분히 커져서 곱셈 자체가 지배 비용이 될 때(대략 1000자리 이상)부터 $O(n^{1.585})$ 이론치가 드러난다. 순진한 $O(n^2)$이었다면 그 구간에서도 100배가 나왔어야 한다.

(Python 3.14.5 / Windows 기준 실측. 절대값은 기기마다 다르지만, 큰 자릿수 구간의 **증가율**은 어디서나 비슷하다. 작은 자릿수 구간은 인터프리터 오버헤드 비중에 따라 기기·버전마다 달라질 수 있다.)

10만 자리 곱셈 한 번이 **8.5밀리초**다. 반복문 안에서 1000번 하면 8.5초다. int가 "무한"하다는 말은 "공짜"라는 뜻이 절대 아니다.
:::

::: danger 큰 정수를 문자열로 바꾸면 터진다
```pyrepl
>>> n = 10**5000
>>> str(n)
Traceback (most recent call last):
  ...
ValueError: Exceeds the limit (4300 digits) for integer string conversion; use sys.set_int_max_str_digits() to increase the limit
```

버그가 아니다. **일부러 넣은 안전장치**다. 10진수 ↔ 2진수 변환은 자릿수에 대해 초선형이다.

```text nolines
str()     100 digits:         0.20 us
str()    1000 digits:        11.29 us
str()   10000 digits:       313.99 us
str()  100000 digits:      7576.66 us
```

`int(s)` 도 마찬가지다(10만 자리에 9.1ms). 웹 서버가 사용자 입력으로 `int("9"*10000000)` 을 받으면 그 스레드는 몇 분간 멈춘다. 실제 DoS 취약점이었고(CVE-2020-10735), 그래서 3.11부터 기본 상한 **4300자리**가 생겼다. 정말 필요하면 `sys.set_int_max_str_digits(2_000_000)` 으로 명시적으로 푼다.

**상한은 10진 변환에만 걸린다.** `hex()`, `bin()`, `oct()`, `int.to_bytes()` 는 2의 거듭제곱 진법이라 비트를 잘라 옮기기만 하면 된다 — 선형 시간이고 제한이 없다.
:::

::: cote 코딩테스트에서 큰 수
임의 정밀도는 코테에서 **압도적인 무기**다. C++ 참가자가 `__int128`로 싸울 때 당신은 그냥 `*` 를 쓴다. 이분 탐색의 고전적 오버플로 버그도 없다.

```python
mid = (lo + hi) // 2      # C/Java에서는 오버플로. 파이썬은 안전.
```

하지만 **모듈러 거듭제곱은 반드시 `pow`의 3인자 형태를 써라.**

```python
# ❌ 269만 비트짜리 정수를 통째로 만든 뒤에야 나머지를 구한다
a ** b % m

# ✅ 매 단계 나머지를 취한다 (제곱-곱셈, O(log b))
pow(a, b, m)
```

`a=123456789, b=100000, m=1_000_000_007` 실측:

```text nolines
pow(a, b, m)  :       0.38 us
a**b % m      :   94504.27 us      -> 250874x
```

**25만 배**다. 상수 차이가 아니라 알고리즘이 다르다. `a**b` 는 **약 81만 자리**(2,687,944비트) 정수를 실제로 메모리에 만든 다음에야 나머지를 구한다. `pow(a, b, m)` 은 매 단계 `m` 미만으로 줄여서 정수가 30비트를 넘지 않는다.
:::

## float는 왜 0.1 + 0.2를 틀리는가

이제 반대편이다. 정수는 정확했다. 실수는 그렇지 않다.

```pyrepl
>>> 0.1 + 0.2
0.30000000000000004
>>> 0.1 + 0.2 == 0.3
False
```

**파이썬의 버그가 아니다.** C, Java, JavaScript, Rust 전부 똑같다. 하드웨어가 그렇게 생겼다. 파이썬 `float`는 C의 `double`, 즉 **IEEE 754 binary64**다.

```text nolines
   IEEE 754 binary64
   ┌─┬───────────┬──────────────────────────────────────────────┐
   │S│  exponent │                  mantissa                    │
   │1│    11 bit │                    52 bit                    │
   └─┴───────────┴──────────────────────────────────────────────┘

   value = (-1)^S x 1.mantissa x 2^(exponent - 1023)
```

핵심은 `2^`다. **밑이 2다.** 유한한 2진 소수로 쓸 수 있는 수는 분모가 2의 거듭제곱인 분수뿐이다. $0.5 = 1/2$, $0.25 = 1/4$ 는 정확하다. 그런데 $0.1 = 1/10$ 이고, 10은 5라는 인수를 갖는다. **2진법으로 $1/10$은 순환소수다.** 10진법으로 $1/3 = 0.3333\ldots$ 을 정확히 못 쓰는 것과 완전히 같은 일이다.

```pyrepl
>>> f"{0.1:.55f}"
'0.1000000000000000055511151231257827021181583404541015625'
>>> (0.1).as_integer_ratio()
(3602879701896397, 36028797018963968)
```

`0.1`이라고 쓰는 순간 **당신이 얻는 것은 0.1이 아니다.** 0.1에 가장 가까운 binary64 값이다. 위의 55자리 숫자가 그 값의 정체다. `as_integer_ratio()`가 보여주듯 실제로는 $3602879701896397 / 2^{55}$ 다.

::: deep 마지막 비트 하나
왜 하필 `0.30000000000000004` 인가. 비트를 직접 까 보면 답이 나온다.

```text nolines
                        S    exponent     mantissa
   0.1              :   0  01111111011  1001...1001 1010
   0.2              :   0  01111111100  1001...1001 1010
   0.3              :   0  01111111101  0011...0011 0011
   0.1 + 0.2        :   0  01111111101  0011...0011 0100
                                                    ^^^^
```

`0.3`과 `0.1 + 0.2`는 **부호도 지수도 같고 마지막 비트 하나만 다르다.** 정확히 1 ULP(Unit in the Last Place) 차이다.

`0.1`도 `0.2`도 이미 각각 반올림된 값이고, 그 둘을 더한 정확한 합도 binary64가 아니라 또 반올림된다. 이 세 번의 반올림이 우연히 `0.3` 자신의 반올림과 **다른 방향으로 떨어졌다.** 그뿐이다.

```pyrepl
>>> f"{0.3:.55f}"
'0.2999999999999999888977697537484345957636833190917968750'
>>> f"{0.1 + 0.2:.55f}"
'0.3000000000000000444089209850062616169452667236328125000'
```

`0.3`은 진짜 0.3보다 **아래**로, `0.1+0.2`는 **위**로 떨어졌다. 둘 다 0.3에서 $10^{-17}$ 남짓 떨어져 있다. 오차가 큰 게 아니라 **`==`가 그 차이를 인정하지 않을 뿐**이다.

그러면 REPL은 왜 `0.1`을 `0.1`로 보여 주는가? `repr(float)`은 **다시 읽었을 때 같은 값이 되는 가장 짧은 10진 표현**을 출력한다(3.1부터). **화면에 보이는 것은 저장된 값이 아니라 그 값의 가장 짧은 이름표다.**
:::

::: warn float 비교에 == 를 쓰지 마라
계산으로 만들어진 두 실수를 `==` 로 비교하는 코드는 언젠가 틀린다. `math.isclose`를 써라.

```pyrepl
>>> import math
>>> math.isclose(0.1 + 0.2, 0.3)
True
```

`isclose(a, b, *, rel_tol=1e-09, abs_tol=0.0)` 는 이렇게 판정한다.

$$|a - b| \le \max(\text{rel\_tol} \times \max(|a|, |b|),\ \text{abs\_tol})$$

**기본값은 상대 오차 $10^{-9}$ 이고 절대 오차는 0이다.** 이 설계에는 함정이 하나 있다.

```pyrepl
>>> math.isclose(1e-10, 0.0)
False
>>> math.isclose(1e-10, 0.0, abs_tol=1e-9)
True
```

**0과의 비교에서는 상대 오차가 무의미하다.** $|0|$ 에 뭘 곱해도 0이니 허용 범위가 0이 된다. 0 근처를 비교할 때는 반드시 `abs_tol`을 직접 줘라. 이건 `isclose`의 결함이 아니라, "0에 가깝다"의 기준이 문제 도메인에만 있기 때문이다. 언어가 정해 줄 수 없다.
:::

## sum()은 이미 당신보다 똑똑하다

오차는 쌓인다. `0.1`을 열 번 더해 보자.

```pyrepl
>>> total = 0.0
>>> for _ in range(10):
...     total += 0.1
...
>>> total
0.9999999999999999
```

그런데 이걸 보라.

```pyrepl
>>> sum([0.1] * 10)
1.0
>>> sum([1e100, 1.0, -1e100])
1.0
```

두 번째가 특히 놀랍다. `1e100`에 `1.0`을 더하면 흡수돼서 그대로 `1e100`이어야 하고, 거기서 `1e100`을 빼면 `0.0`이어야 한다. 손으로 하면 실제로 그렇다.

```pyrepl
>>> t = 0.0
>>> for x in [1e100, 1.0, -1e100]:
...     t += x
...
>>> t
0.0
```

**같은 덧셈인데 `sum()`은 1.0, 반복문은 0.0이다.**

::: deep sum()은 3.12부터 보상 합산을 한다
CPython 3.12부터 내장 `sum()`은 float 항목에 대해 **노이마이어(Neumaier) 보상 합산**을 쓴다. 매 덧셈에서 *"버려진 하위 비트"* 를 별도 변수에 모아 뒀다가 마지막에 되돌려 준다. 반복문 `t += x`는 그냥 덧셈이라 그 정보를 그대로 버린다.

```text nolines
naive([0.1]*10)          -> 0.9999999999999999
sum([0.1]*10)            -> 1.0
naive([1e100, 1.0, -1e100]) -> 0.0
sum([1e100, 1.0, -1e100])   -> 1.0
```

그래서 **float 합계에는 손으로 만든 누적 루프보다 `sum()`이 낫다.** 더 짧고, 더 빠르고, 더 정확하다. 세 마리 토끼가 같은 방향인 드문 경우다.

다만 `sum()`도 **정확히 반올림된 값을 보장하지는 않는다.** 보상 항은 하나뿐이라, 그 항 자체가 정밀도를 잃는 입력이 존재한다.

```pyrepl
>>> import math
>>> v = [-8.182709887544512e+16, -1479006824457416.0, -9.350030450290902e-18]
>>> sum(v)
-8.330610569990253e+16
>>> math.fsum(v)
-8.330610569990254e+16
```

`math.fsum`은 **완전한 정확도**를 보장한다. 여러 개의 부분합을 유지해서 무한 정밀도로 더한 뒤 딱 한 번만 반올림한다. 그 대가는 속도다.

정리하면 이렇다.

| 방법 | 정확도 | 언제 |
| --- | --- | --- |
| `t += x` 루프 | 최악 | 쓰지 마라 |
| `sum(v)` | 매우 좋음 (3.12+) | 기본값 |
| `math.fsum(v)` | 정확히 반올림 보장 | 과학 계산, 검증 코드 |
| `sum(decimals)` | 십진 정확 | 금액 |
:::

## round()가 2.5를 2로 만드는 이유

```pyrepl
>>> round(0.5)
0
>>> round(1.5)
2
>>> round(2.5)
2
>>> round(3.5)
4
>>> round(-0.5)
0
```

버그처럼 보인다. 학교에서 배운 반올림이라면 `round(2.5)`는 3이어야 한다. 이건 **은행가 반올림**(banker's rounding), 정식 명칭 **round-half-to-even**이다. 정확히 중간값(`.5`)일 때 **짝수 쪽으로** 간다.

::: hist 왜 짝수인가 — 편향을 없애기 위해
`.5`를 항상 올리면(`ROUND_HALF_UP`) 결과가 **체계적으로 커진다.** 0.5, 1.5, 2.5, 3.5를 반올림해 더해 보라.

- 항상 올림: $1 + 2 + 3 + 4 = 10$
- 짝수로: $0 + 2 + 2 + 4 = 8$
- 반올림 전 실제 합: $0.5 + 1.5 + 2.5 + 3.5 = 8$

**짝수 반올림이 원래 합을 맞춘다.** `.5`를 위아래로 절반씩 나눠 보내기 때문이다. 데이터가 많을수록 차이는 누적된다. 그래서 IEEE 754는 이것을 **기본 반올림 모드**로 정했고 `round()`와 `Decimal`의 기본값(`ROUND_HALF_EVEN`)이 따른다.

"학교에서 배운 방식"이 편향된 규칙이다. 사람 손으로 몇 개 계산할 때는 티가 안 났을 뿐이다.
:::

여기까지는 설계다. 진짜 함정은 다음이다.

```pyrepl
>>> round(2.675, 2)
2.67
```

`2.675`는 짝수 반올림으로도 `2.68`이 나와야 한다(7이 홀수니까). 왜 2.67인가?

```pyrepl
>>> from decimal import Decimal
>>> Decimal(2.675)
Decimal('2.67499999999999982236431605997495353221893310546875')
```

**애초에 2.675가 아니었다.** 저장된 값은 2.675보다 아주 조금 **작다.** 그러니 `round()`는 아무 잘못이 없다. 정직하게 내림한 것이다. 여기에는 중간값이 존재조차 하지 않았다.

::: warn round()의 두 함정은 별개다
1. **은행가 반올림** — 진짜 `.5`일 때 짝수로 간다. 설계된 동작. `round(2.5) == 2`.
2. **표현 불가** — `.5`처럼 보이는 값이 실은 `.5`가 아니다. float의 한계. `round(2.675, 2) == 2.67`.

둘을 뭉뚱그려 "파이썬 round는 이상하다"고 외우면 아무것도 고칠 수 없다. 1번은 의도된 것이고, 2번은 `round`가 아니라 리터럴 `2.675`의 문제다.

"학교식 반올림"이 정말 필요하면 `Decimal`로 명시한다.

```pyrepl
>>> from decimal import Decimal, ROUND_HALF_UP
>>> Decimal('2.5').quantize(Decimal('1'), rounding=ROUND_HALF_UP)
Decimal('3')
>>> round(Decimal('2.675'), 2)
Decimal('2.68')
```

`Decimal('2.675')`는 **진짜 2.675**이므로 반올림이 예상대로 동작한다.
:::

## // 와 % 는 C와 다르다

C, C++, Java, Rust에서 `-7 / 2`는 `-3`이고 `-7 % 2`는 `-1`이다. 파이썬은 다르다.

```pyrepl
>>> -7 // 2
-4
>>> -7 % 2
1
>>> 7 // -2
-4
>>> 7 % -2
-1
```

**파이썬의 `//`는 0 방향이 아니라 $-\infty$ 방향으로 내린다**(floor division). C는 0 방향으로 자른다(truncation).

```text nolines
   -7 / 2 = -3.5

   C      :  trunc(-3.5) = -3      0 쪽으로 자른다
   Python :  floor(-3.5) = -4      아래로 내린다

           -4        -3.5      -3          0
   ---------|----------x--------|----------|--->
         Python                 C
```

`%`는 항등식 `a == (a // b) * b + a % b` 를 지키도록 따라 정해진다. 그래서 **파이썬의 `%` 결과는 항상 나누는 수(`b`)와 같은 부호**다.

::: hist 왜 파이썬은 floor를 골랐나
귀도 반 로섬이 직접 쓴 이유가 있다. **`%`가 순환·주기 계산에서 항상 유용한 값을 내게 하려고**다.

```pyrepl
>>> [i % 3 for i in range(-5, 6)]
[1, 2, 0, 1, 2, 0, 1, 2, 0, 1, 2]
```

음수에서도 결과가 `0, 1, 2` 안에서 깔끔하게 순환한다. C의 규칙이었다면 `-2, -1, 0, -2, -1, 0, ...` 이 나와서 배열 인덱스로 쓸 수 없다. 그 덕에 이런 코드가 그냥 동작한다.

```python
next_idx = (i - 1) % len(buf)        # 원형 버퍼 — 음수 걱정이 없다
x = (a - b) % 1_000_000_007          # 모듈러 — 항상 [0, MOD)
```

C에서 같은 걸 하려면 `((x % m) + m) % m` 이라고 써야 한다. 파이썬 코드에서 이 관용구를 보면 **C 습관을 그대로 옮긴 것**이다. 지워라.
:::

C의 동작을 재현해야 하면 `math.fmod`가 있다. 부호가 **피제수**를 따른다.

```pyrepl
>>> -7 % 2
1
>>> import math
>>> math.fmod(-7, 2)
-1.0
>>> int(-3.5), math.floor(-3.5), math.trunc(-3.5)
(-3, -4, -3)
```

`int()` 도 truncation이라 `math.floor` 와 다르다는 점에 주의하라. 음수에서만 갈라지므로 테스트가 양수만 쓰면 통과한다.

::: danger float에 // 를 쓰면
```pyrepl
>>> 0.7 / 0.1
6.999999999999999
>>> 0.7 // 0.1
6.0
```

`0.7 / 0.1`의 참값은 7이다. 그런데 float 오차 때문에 몫이 7보다 **아주 조금 작게** 나오고, `//`가 그걸 내려서 **6**을 만든다. 오차가 $10^{-15}$ 인데 결과는 1만큼 틀렸다.

**float에 `//` 를 쓰는 코드는 경계에서 반드시 사고를 낸다.** 정수 나눗셈이 필요하면 정수로 만들어서 해라.
:::

### divmod

몫과 나머지를 동시에 원하면 `divmod`가 있다.

```pyrepl
>>> divmod(-7, 2)
(-4, 1)
>>> divmod(7, -2)
(-4, -1)
```

::: perf divmod는 언제 이득인가
"함수 하나로 두 번 계산할 걸 한 번에 하니 빠르다"고 흔히 말한다. 측정해 보면 반만 맞다.

```text nolines
작은 int (a=1234567, b=997)
  divmod(a, b)     :  28.7 ns
  (a // b, a % b)  :  26.9 ns      <- divmod 가 오히려 느리다

1000자리 int
  divmod(a, b)     :  3.17 us
  (a // b, a % b)  :  6.37 us      <- divmod 가 2.01배 빠르다
```

작은 정수에서는 나눗셈이 CPU 명령 하나라 거의 공짜다. 남는 건 **함수 호출 오버헤드**뿐이고 그건 `divmod` 쪽이 크다. 큰 정수에서는 나눗셈이 진짜 비싼 알고리즘이라 한 번만 하는 `divmod`가 정확히 2배 이긴다.

**작은 수에서 `divmod`를 쓰는 이유는 성능이 아니라 의도 표현이다.** 두 값이 한 연산의 결과임을 드러낸다. 그것으로 충분한 이유다.
:::

## Decimal과 Fraction — 정확성을 사는 값

float가 못 하는 걸 하는 두 타입이 표준 라이브러리에 있다.

### Decimal — 사람의 10진법

`Decimal`은 **10진법으로** 수를 저장한다. 그래서 0.1이 정확히 0.1이다.

```pyrepl
>>> from decimal import Decimal
>>> Decimal('0.1') + Decimal('0.2') == Decimal('0.3')
True
>>> Decimal('1.10') * Decimal('1.10')
Decimal('1.2100')
```

`1.2100` 을 보라. 유효 자릿수를 **버리지 않는다.** `Decimal`은 값만이 아니라 **정밀도(scale)까지** 들고 다닌다. 회계에서 "1.10달러"와 "1.1달러"가 다른 의미인 것과 같다.

::: danger Decimal(0.1) 과 Decimal('0.1') 은 완전히 다르다
```pyrepl
>>> from decimal import Decimal
>>> Decimal('0.1')
Decimal('0.1')
>>> Decimal(0.1)
Decimal('0.1000000000000000055511151231257827021181583404541015625')
```

`Decimal(0.1)` 은 **이미 오염된 float**를 받는다. `Decimal`은 그 float의 정확한 값을 충실히 옮겨 적을 뿐이다. 오차는 리터럴 `0.1`을 쓴 순간 이미 들어갔다. **`Decimal`은 항상 문자열이나 int로 만들어라.**

같은 이유로 이런 비일관성이 생긴다.

```pyrepl
>>> Decimal('0.1') == 0.1
False
>>> Decimal('0.5') == 0.5
True
```

0.5는 float로도 정확하니 `True`, 0.1은 아니니 `False`. 이게 정확히 함정이다. **0.5로 짠 테스트는 통과하고 실제 데이터에서 터진다.**
:::

정밀도는 컨텍스트로 조절한다. `Decimal`은 **임의 정밀도이되 무한 정밀도는 아니다.**

```pyrepl
>>> from decimal import Decimal, getcontext
>>> Decimal(1) / Decimal(3)
Decimal('0.3333333333333333333333333333')
>>> getcontext().prec = 50
>>> Decimal(1) / Decimal(3)
Decimal('0.33333333333333333333333333333333333333333333333333')
```

기본 `prec`는 28이다. `1/3`은 여전히 잘린다. **`Decimal`은 float 문제를 없애는 게 아니라 10진 문제로 옮긴다.** 10진법으로 유한한 수(금액, 세율, 퍼센트)에서만 정확하다.

### Fraction — 진짜 유리수

$1/3$까지 정확히 원하면 `Fraction`이다. 분자와 분모를 int로 들고 있으므로 **유리수 범위에서 오차가 0**이다.

```pyrepl
>>> from fractions import Fraction
>>> Fraction(1, 3) + Fraction(1, 6)
Fraction(1, 2)
>>> Fraction('0.1') * 3 == Fraction(3, 10)
True
```

`Fraction(0.1)` 은 `Decimal(0.1)` 과 똑같은 함정이 있다.

```pyrepl
>>> Fraction(0.1)
Fraction(3602879701896397, 36028797018963968)
```

float의 정확한 값이 그대로 나온다. 오히려 이게 **float의 진짜 정체를 확인하는 최고의 도구**다.

```pyrepl
>>> import math
>>> Fraction(math.pi).limit_denominator(1000)
Fraction(355, 113)
```

`limit_denominator`는 연분수 전개로 최적 유리수 근사를 찾는다. $355/113$ 은 5세기 조충지가 찾아낸 바로 그 원주율 근사다.

::: perf 정확성의 가격표
```text nolines
                        연산 시간          객체 크기
  int    a + b            9.7 ns           28 bytes
  float  a + b            9.3 ns           24 bytes
  Decimal a + b          42.2 ns          120 bytes
  Fraction a + b        267.0 ns           48 bytes
```

`Decimal` 덧셈은 int의 **약 4.3배**, 객체는 **4.3배** 크다. `Fraction`은 **약 27배** — 매 연산마다 `math.gcd`로 약분하기 때문이다. 10만 개 합산은 `Decimal`이 `float`의 **6배**(3.51ms vs 0.60ms).

(Python 3.14.5 / Windows 기준 실측. CPython의 `decimal`은 C로 구현된 libmpdec을 쓴다. 순수 파이썬 폴백 환경에서는 훨씬 더 느리다.)

**6배는 대부분의 시스템에서 낼 만한 값이다.** 금액 계산이 병목인 서비스는 거의 없다. 반면 `Fraction`을 백만 번 도는 루프에 넣으면 그건 병목이 된다.
:::

::: tip 무엇을 언제 쓰나 — 결정 규칙
| 상황 | 선택 | 이유 |
| --- | --- | --- |
| 개수, 인덱스, 모듈러 | `int` | 정확하고 빠르다 |
| 측정값, 물리량, ML | `float` | 오차보다 속도가 중요. 입력 자체가 부정확 |
| 금액, 세율, 청구서 | `Decimal` | 10진 규칙이 법으로 정해져 있다 |
| 유리수 대수, 정확한 비율, 검증 | `Fraction` | 오차 0. 느려도 상관없는 곳 |
| 신호, 주파수 응답 | `complex` | 파이썬 내장 |

**의심스러우면 `float`로 시작하되, "이 오차가 누적되면 누가 화를 내는가"를 물어라.** 사람이 화를 내면(돈, 법규) `Decimal`. 자연이 화를 내면(수치 해석) 알고리즘을 고쳐야지 타입을 바꿔서는 안 된다.
:::

## int와 float의 경계

두 타입 사이를 오갈 때 조용한 사고가 난다.

```pyrepl
>>> 2**53
9007199254740992
>>> 2**53 + 1
9007199254740993
>>> float(2**53 + 1) == float(2**53)
True
```

**float의 가수는 52비트 + 암묵적 1비트 = 53비트.** $2^{53}$을 넘으면 **연속된 정수를 모두 표현할 수 없다.** $2^{53}+1$은 float가 될 수 없어서 $2^{53}$으로 붙어 버린다.

```pyrepl
>>> 1e16 + 1.0 - 1e16
0.0
>>> 1e16 + 2.0 - 1e16
2.0
```

$10^{16}$ 근처에서는 **float가 셀 수 있는 최소 단위가 2**다. 1을 더해도 아무 일이 없다. JavaScript에 `BigInt`가 추가된 이유이고, JSON으로 큰 ID를 주고받을 때 값이 깨지는 이유다.

::: danger int(math.sqrt(n)) 은 틀린다
코딩테스트에서 실제로 오답을 만드는 코드다.

```pyrepl
>>> import math
>>> n = 99999999999999999999999999
>>> math.isqrt(n)
9999999999999
>>> int(math.sqrt(n))
10000000000000
```

`int(math.sqrt(n))` 이 **1 크다.** $10^{13}$의 제곱은 $10^{26}$ 으로 `n`보다 크다. 원인은 방금 본 것이다 — `math.sqrt`는 int를 **먼저 float로 바꾸므로**, 26자리 정수는 그 순간 53비트에 잘린다.

**`math.isqrt`를 써라.** 3.8부터 있고, 순수 정수 연산으로 $\lfloor\sqrt{n}\rfloor$ 을 **정확히** 구한다.

속도는 오히려 반대다 — 같은 `n`으로 실측하면(Python 3.14.5 / Windows, timeit) `math.isqrt(n)` 122ns, `int(math.sqrt(n))` 62ns. `int(math.sqrt(n))`이 **약 2배 더 빠르다.** float 변환·sqrt 연산이 정수 전용 뉴턴법보다 하드웨어에 더 가깝기 때문이다. 하지만 **정답이 틀리면 속도는 의미가 없다.** 여기서 고를 기준은 성능이 아니라 정확성이다.

```python
def is_prime(n: int) -> bool:
    if n < 2:
        return False
    for i in range(2, math.isqrt(n) + 1):   # ✅ int(n**0.5)+1 이 아니다
        if n % i == 0:
            return False
    return True
```

`+1` 을 붙이는 관용구 때문에 어긋난 1이 우연히 메워지는 경우가 많다. 그래서 **이 버그는 아주 늦게 발견된다.**
:::

int → float 변환이 아예 불가능하면 예외가 난다. 그런데 float끼리의 오버플로는 조용하다.

```pyrepl
>>> float(2**2000)
Traceback (most recent call last):
  ...
OverflowError: int too large to convert to float
>>> 1e308 * 10
inf
```

**비대칭이다.** 명시적 변환은 거부하고, 산술은 `inf`를 준다. IEEE 754가 `inf`를 정의하므로 후자는 "정상 동작"이기 때문이다. 그래서 `inf`가 된 계산은 조용히 흘러가다가 한참 뒤에 `nan`으로 나타난다.

```pyrepl
>>> import math
>>> math.inf - math.inf
nan
>>> nan = float('nan')
>>> nan == nan
False
>>> [nan] == [nan]
True
```

마지막 줄이 이상해 보인다. `nan != nan` 인데 왜 리스트 비교는 `True`인가? 컨테이너 비교는 **먼저 `is`로 확인하고 그 다음에 `==`를 본다**(identity implies equality 최적화). 같은 객체이므로 `==`까지 가지 않는다. [1.1 객체, 이름, 참조](#/objects-names)의 정체성 이야기가 여기서 되돌아온다.

## 비트 연산

정수를 **비트 집합**으로 볼 때 쓴다.

```pyrepl
>>> 0b1100 & 0b1010
8
>>> 0b1100 | 0b1010
14
>>> 0b1100 ^ 0b1010
6
>>> 1 << 100
1267650600228229401496703205376
```

`1 << 100` 이 그냥 된다. **비트 연산도 임의 정밀도다.** C에서 `1 << 100` 은 미정의 동작(undefined behavior)이다.

::: deep 음수의 비트는 몇 개인가
파이썬은 정수를 **부호 + 크기**로 저장한다고 했다. 그런데 비트 연산은 **무한 자릿수 2의 보수**인 것처럼 동작한다.

```pyrepl
>>> ~5
-6
>>> -5 & 0xFF
251
>>> -1 >> 100
-1
>>> bin(-5)
'-0b101'
```

`-1 >> 100` 이 `-1`이다. 32비트 정수라면 오른쪽 시프트를 100번 하면 뭐가 됐든 사라져야 한다. 파이썬에서는 `-1`이 `...11111111` 이라는 **무한한 1의 나열**이라 아무리 밀어도 `-1`이다.

`bin(-5)` 가 `'-0b101'` 인 것도 보라. 2의 보수 비트 패턴이 아니라 **부호와 절댓값**을 따로 보여 준다. 저장 방식(부호+크기)이 여기서 드러난다. 반면 연산은 2의 보수인 척한다. **두 모델이 공존한다.**

고정 폭이 필요하면 마스킹한다.

```pyrepl
>>> -5 & 0xFFFFFFFF          # 32비트 부호 없는 정수로 보기
4294967291
>>> (5).to_bytes(2, 'big')
b'\x00\x05'
>>> int.from_bytes(b'\x01\x00', 'big')
256
```

음수 시프트는 예외다. C의 미정의 동작을 파이썬은 명시적 에러로 바꿨다.

```pyrepl
>>> 1 << -1
Traceback (most recent call last):
  ...
ValueError: negative shift count
```

::: cote 비트마스크 관용구
```python
mask |= 1 << i          # i번 원소 추가
mask &= ~(1 << i)       # i번 원소 제거
mask ^= 1 << i          # i번 원소 토글
if mask & (1 << i):     # i번 원소 포함?
mask & -mask            # 최하위 1비트만 남기기 (2의 보수 트릭)
bin(mask).count('1')    # ❌ 문자열을 만든다
mask.bit_count()        # ✅ 3.10+. CPU popcount 명령
n.bit_length()          # 최상위 1비트 위치
```

`bit_count()`가 3.10에 들어오기 전까지 모두가 `bin(n).count('1')` 을 썼다. 문자열 객체를 만들었다 버리는 짓이다.

```pyrepl
>>> (255).bit_count()
8
>>> (1 << 200).bit_count()
1
>>> (0).bit_length(), (255).bit_length()
(0, 8)
```

`(1 << 200).bit_count()` 가 동작한다 — 200비트 정수에도 그냥 쓴다. 비트마스크 DP는 [7.24 비트마스크](#/bitmask)에서 본격적으로 다룬다.
:::

::: warn bool은 int다
```pyrepl
>>> isinstance(True, int)
True
>>> True + True
2
>>> sum([True, True, False])
2
>>> {1: 'a', True: 'b'}
{1: 'b'}
```

`bool`은 `int`의 **서브클래스**다. `hash(True) == hash(1)` 이고 `True == 1` 이므로 딕셔너리에서 **같은 키**로 취급된다. 키가 `1`로 남고 값만 덮인 걸 보라.

`sum(조건들)` 로 개수를 세는 관용구는 이 사실을 이용한 것이다. 편하지만, 딕셔너리 키에 `0/1`과 `False/True`를 섞으면 조용히 뭉개진다. [1.6 dict](#/dict)에서 해시의 관점으로 다시 본다.
:::

## 요약

- **`int`는 30비트 digit 배열**이다. 오버플로가 없는 대신 `0`도 28바이트고, 큰 수 곱셈은 카라추바로 $O(n^{1.585})$ 다.
- **`str(bigint)`는 4300자리 제한**이 있다. DoS 방어다. `hex()`/`bin()`은 제한 없다.
- **`float`는 IEEE 754 binary64**다. 밑이 2이므로 `0.1`은 애초에 저장되지 않는다. `0.1+0.2`와 `0.3`은 1 ULP 차이다.
- **float 비교는 `math.isclose`.** 단 0과 비교할 때는 `abs_tol`을 직접 줘야 한다.
- **`sum()`은 3.12부터 보상 합산**을 한다. `t += x` 루프보다 정확하다. 완전한 정확도는 `math.fsum`.
- **`round()`는 은행가 반올림**(편향 제거). `round(2.675, 2) == 2.67`은 그것과 무관한 별개 문제다.
- **`//`는 floor, `%`의 부호는 나누는 수를 따른다.** C 습관인 `((x%m)+m)%m` 은 지워라.
- **금액은 `Decimal`, 유리수는 `Fraction`.** 반드시 문자열로 생성한다.
- **`int(math.sqrt(n))` 대신 `math.isqrt(n)`.** $2^{53}$ 위에서 틀린다.
