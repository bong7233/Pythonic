# 1.6 dict — 해시 테이블의 내부

::: lead
`dict` 는 파이썬에서 가장 많이 쓰이는 자료구조다. 당신이 직접 쓰지 않아도 이미 쓰고 있다. 모든 객체의 속성, 모든 모듈의 전역 변수, 모든 클래스의 메서드 테이블이 전부 dict다. 파이썬 인터프리터 자체가 dict 위에서 돈다. 그래서 CPython은 dict에 다른 어떤 자료구조보다 많은 최적화를 쏟아부었다. 이 절은 그 안을 연다. 왜 조회가 5만 배 빠른지, 왜 3.7부터 순서가 보장되는지, 그리고 왜 어떤 dict는 한 줄의 대입으로 메모리가 세 배가 되는지.
:::

## 리스트에서 찾는 게 왜 느린가

문제부터 보자. 20만 개짜리 컨테이너에서 원소 하나를 찾는다.

```python title="lookup_bench.py"
import timeit

setup = "L = list(range(200000)); D = {x: 1 for x in L}; k = 199999"

tl = min(timeit.repeat("k in L", setup, number=200, repeat=5)) / 200
td = min(timeit.repeat("k in D", setup, number=200000, repeat=5)) / 200000

print(f"list  1회: {tl * 1e6:.1f} us")
print(f"dict  1회: {td * 1e9:.1f} ns")
print(f"배수: {round(tl / td)}")
```

```text nolines
list  1회: 670.7 us
dict  1회: 12.6 ns
배수: 53254
```

(Python 3.14.5 / Windows 기준 실측. 절대값은 기기마다 다르지만 **자릿수 차이**는 어디서나 같다.)

5만 3천 배다. 리스트는 처음부터 하나씩 비교한다 — $O(n)$. 20만 번의 비교, 20만 번의 포인터 추적, 20만 번의 캐시 미스.

dict는 **비교를 거의 하지 않는다.** 한 번, 많아야 두세 번. 개수가 20만이든 2천만이든 마찬가지다. 이게 어떻게 가능한가?

## 해시 테이블의 아이디어

발상은 단순하다.

> **키를 보고 그 키가 있어야 할 자리를 계산한다.** 찾는 게 아니라 계산한다.

`hash()` 가 그 계산의 첫 단계다. 임의의 객체를 정수 하나로 뭉갠다.

```pyrepl
>>> hash(42)
42
>>> hash(3.5)
1152921504606846979
>>> hash((1, 2, 3))
529344067295497451
>>> hash("python")
309243305923044716
```

::: note 작은 정수의 해시는 자기 자신이다
`hash(42) == 42`. CPython은 작은 정수를 그대로 해시로 쓴다. 계산이 공짜이기 때문이다. 큰 정수는 $2^{61}-1$ 로 나눈 나머지가 된다.

```pyrepl
>>> hash(2**61 - 1)
0
>>> hash(2**61)
1
>>> hash(2**61 - 2)
2305843009213693950
```

`hash(2**61 - 1)` 이 0이라는 건 그 값이 항상 0번 슬롯을 먼저 노린다는 뜻이다. 이게 문제가 될 수 있다는 사실은 이 절 뒤에서 다시 나온다.
:::

해시는 64비트 정수다. 그런데 테이블의 슬롯은 8개, 16개, 32개뿐이다. 큰 수를 작은 범위로 접어 넣어야 한다.

```pyrepl
>>> [hash(n) & 7 for n in [0, 1, 2, 8, 9, 16, 24]]
[0, 1, 2, 0, 1, 0, 0]
```

`hash & 7` — `% 8` 과 같은 결과지만 비트 연산이라 훨씬 싸다. **이게 CPython 해시 테이블 크기가 항상 2의 거듭제곱인 이유다.** 나눗셈은 CPU에서 가장 느린 정수 연산 중 하나다. 크기가 $2^k$ 이면 나머지 연산이 `& (2^k - 1)` 한 줄이 된다.

대가도 있다. `& (size-1)` 은 **해시의 하위 비트만 본다.** 상위 비트에 아무리 좋은 정보가 있어도 버려진다. 위에서 `0, 8, 16, 24` 가 전부 0번 슬롯으로 몰린 게 그 증거다. 이 대가를 어떻게 갚는지가 다음 이야기다.

## 충돌 — 두 키가 같은 자리를 원할 때

슬롯이 8개인데 키를 100개 넣으면? 당연히 겹친다. 심지어 8개만 넣어도 겹친다. **충돌은 예외가 아니라 정상이다.**

푸는 방법은 크게 둘이다.

| 방식 | 하는 일 | 쓰는 곳 |
| --- | --- | --- |
| **체이닝**(chaining) | 슬롯마다 연결 리스트를 달고 거기 매단다 | Java `HashMap`, C++ `unordered_map` |
| **개방 주소법**(open addressing) | 다른 빈 슬롯을 찾아 거기 넣는다 | **CPython dict/set** |

CPython은 개방 주소법을 쓴다. 이유는 **지역성**이다. 체이닝은 슬롯마다 힙에 흩어진 노드를 포인터로 따라가야 해서 방문마다 캐시 미스가 나지만, 개방 주소법은 하나의 연속된 배열 안에서만 움직인다. 현대 CPU에서 이 차이는 크다.

문제는 **"다른 빈 슬롯을 어떻게 찾을 것인가"** 다. 가장 단순한 답인 선형 탐사(`i+1`, `i+2`, ...)는 **뭉침**(clustering)을 만든다 — 한 번 뭉치기 시작하면 덩어리가 커지고, 걸린 모든 키의 탐색이 같이 느려진다.

::: deep CPython의 탐사 수식
`Objects/dictobject.c` 에 이 주석과 함께 들어 있는 재귀식이 CPython 해시 테이블의 심장이다.

```python title="탐사 순서 모델 — CPython의 재귀식 그대로"
def probe_sequence(h, size=8, steps=12):
    mask = size - 1
    perturb = h & ((1 << 64) - 1)     # 부호 없는 64비트로 취급
    i = h & mask
    yield i
    for _ in range(steps - 1):
        perturb >>= 5                 # PERTURB_SHIFT = 5
        i = mask & (i * 5 + perturb + 1)
        yield i


print(list(probe_sequence(42)))
print(list(probe_sequence(12345)))
```

```text nolines
[2, 4, 5, 2, 3, 0, 1, 6, 7, 4, 5, 2]
[1, 7, 0, 1, 6, 7, 4, 5, 2, 3, 0, 1]
```

두 가지가 동시에 일어난다.

1. **`perturb` 가 해시의 상위 비트를 끌어온다.** 처음에 `& mask` 로 버려졌던 상위 비트가 5비트씩 오른쪽으로 밀려 들어오면서 탐사 경로에 섞인다. 그래서 하위 비트가 같은 키들도 **두 번째 탐사부터는 서로 다른 길로 흩어진다.** `& (size-1)` 이 정보를 버린 대가를 여기서 갚는다.

2. **`perturb` 가 0이 되면 `i = (5i + 1) & mask` 만 남는다.** 이건 $2^k$ 를 법으로 하는 **완전 순회 순열**이다(Knuth). 즉 **결국 모든 슬롯을 정확히 한 번씩 방문한다.** 빈 자리가 하나라도 있으면 반드시 찾는다 — 무한 루프가 절대 나지 않는다는 보장이다.

위 출력에서 `42` 는 9스텝 만에, `12345` 는 10스텝 만에 8개 슬롯 전부를 방문한다. 중간에 같은 슬롯을 다시 밟는 건 `perturb` 가 아직 살아 있는 구간이라 그렇다.
:::

::: note 충돌 처리는 조회의 일부다
슬롯을 찾았다고 끝이 아니다. 거기 있는 키가 내가 찾는 키가 맞는지 확인해야 한다. CPython은 순서대로 값싼 검사부터 한다: **슬롯이 비었나**(없는 키, 끝) → **저장된 해시가 다른가**(정수 비교 한 번, 다르면 다음 슬롯) → **`key is stored_key`인가**(같은 객체면 바로 확정, 인터닝된 문자열이 이득을 본다) → 그래도 확실치 않으면 **`key == stored_key`**(여기서 처음 당신의 `__eq__` 가 불린다). `==` 는 파이썬 레벨 호출이라 비싸다. 해시를 엔트리에 같이 저장해 두는 이유가 바로 두 번째 단계다 — 대부분의 충돌은 정수 비교 한 번으로 걸러진다.
:::

## compact dict — 3.6이 바꾼 것

이제 진짜 이야기다. 파이썬 3.5까지의 dict는 이렇게 생겼다.

```text nolines
  Python 3.5 이전 — 슬롯이 곧 엔트리였다 (size = 8)

  ┌──────┬────────┬───────┬───────┐
  │ slot │  hash  │  key  │ value │
  ├──────┼────────┼───────┼───────┤
  │  0   │  NULL  │       │       │
  │  1   │ hash_a │  'a'  │   1   │
  │  2   │  NULL  │       │       │
  │  3   │  NULL  │       │       │
  │  4   │ hash_c │  'c'  │   3   │
  │  5   │  NULL  │       │       │
  │  6   │ hash_b │  'b'  │   2   │
  │  7   │  NULL  │       │       │
  └──────┴────────┴───────┴───────┘

  8 x 24 = 192 bytes           <- 빈 슬롯도 24바이트를 통째로 낭비한다
  순회 순서: 'a', 'c', 'b'     <- 슬롯 번호 순 = 해시 순 = 사실상 무작위
```

두 가지가 나쁘다. 첫째, **해시 테이블은 절대 꽉 채우지 않는다.** 채우면 충돌이 폭발하기 때문에 CPython은 3분의 2에서 멈춘다. 즉 **슬롯의 최소 3분의 1이 항상 24바이트짜리 빈칸**이다. 둘째, 순회 순서가 해시 순서다. 사람이 넣은 순서와 아무 상관이 없다.

3.6에서 Raymond Hettinger가 이걸 뒤집었다. **테이블을 둘로 쪼갠다.**

```text nolines
  dk_indices                        dk_entries
  (size = 8, 1 byte each)           (usable = 5, 24 bytes each)

  ┌──────┬───────┐                  ┌─────┬────────┬───────┬───────┐
  │ slot │ index │                  │ idx │  hash  │  key  │ value │
  ├──────┼───────┤                  ├─────┼────────┼───────┼───────┤
  │  0   │  -1   │                  │  0  │ hash_a │  'a'  │   1   │
  │  1   │   0   │ ───────────────▶ │  1  │ hash_b │  'b'  │   2   │
  │  2   │  -1   │                  │  2  │ hash_c │  'c'  │   3   │
  │  3   │  -1   │                  │  3  │  ----  │ ----- │ ----- │
  │  4   │   2   │                  │  4  │  ----  │ ----- │ ----- │
  │  5   │  -1   │                  └─────┴────────┴───────┴───────┘
  │  6   │   1   │
  │  7   │  -1   │                  8 x 1 + 5 x 24 = 128 bytes
  └──────┴───────┘
```

- **`dk_indices`** — 진짜 해시 테이블. 슬롯마다 **정수 하나**만 들어 있다. 엔트리 배열의 몇 번째인지. `-1`(`DKIX_EMPTY`)은 빈 슬롯.
- **`dk_entries`** — 실제 데이터. **삽입 순서대로 빈틈없이 차곡차곡 쌓인다.**

빈 슬롯의 낭비가 24바이트에서 **1바이트**로 줄었다. 192 → 128바이트다.

그리고 **순회는 `dk_entries` 를 0번부터 그냥 훑는다.** 해시 테이블을 거치지 않는다. 그 배열은 삽입 순서다. 그러니까 **순서가 보장된다.**

::: hist 순서 보장은 의도가 아니라 부산물이었다
이 점이 중요하다. Hettinger의 제안은 **메모리 절약**이었다. 순서 보장은 그 설계에서 공짜로 딸려 나온 **부작용**이었다.

그래서 3.6 릴리스 노트는 이렇게 못을 박았다. *"이 동작은 구현 세부사항이며 의존해서는 안 된다."* PyPy나 다른 구현이 따라야 할 의무가 없다는 뜻이었다.

그런데 현실은 달랐다. 3.6이 나오자마자 모든 코드가 이 순서에 의존하기 시작했다. 되돌릴 수 없게 되자 Guido van Rossum이 python-dev에서 선언했다.

> "Make it so." — 2017년 12월 15일

**3.7부터 삽입 순서 보존은 언어 명세다.** 이제 모든 파이썬 구현이 지켜야 한다. 사실상의 표준이 명세를 밀어낸 드문 사례다.
:::

### 구조를 직접 확인한다

방금 그린 그림이 진짜인지 확인할 방법이 있다. `sys.getsizeof` 로 재 보는 것이다. 그림이 맞다면 크기를 **정확히** 예측할 수 있어야 한다.

```python title="dict_size_model.py — 그림이 맞다면 이 공식이 맞아야 한다"
import sys


def model(n):
    if n == 0:
        return 64                              # 빈 dict는 전역 empty keys를 공유한다
    size = 8
    while size * 2 // 3 < n:                   # 적재율 2/3를 넘으면 두 배
        size *= 2
    width = 1 if size <= 128 else 2 if size <= 32768 else 4
    usable = size * 2 // 3
    return 48 + 48 + size * width + usable * 24
    #      ^    ^    ^               ^
    #      |    |    |               └─ dk_entries
    #      |    |    └───────────────── dk_indices
    #      |    └────────────────────── PyDictKeysObject 헤더
    #      └─────────────────────────── PyDictObject 본체


for n in [1, 5, 6, 11, 22, 43, 86, 171, 1000, 5000, 21845, 21846, 50000]:
    real = sys.getsizeof({i: i for i in range(n)})
    assert real == model(n), (n, real, model(n))
print("전부 일치")
```

```text nolines
전부 일치
```

**한 바이트도 틀리지 않는다.** 그림이 맞다는 뜻이다. 여기서 읽어 낼 게 두 가지 있다.

**1. 인덱스 너비가 자란다.** 슬롯이 128개 이하면 인덱스는 1바이트로 충분하고, 32768개까지는 2바이트, 그 위는 4바이트다. **CPython은 테이블 크기에 맞춰 인덱스 타입을 바꾼다** — 작은 dict가 압도적으로 많으니 이 최적화가 대부분의 경우에 이긴다.

**2. 증가는 계단이다.** 크기가 바뀌는 지점을 뽑아 보면 이렇다.

```pyrepl
>>> import sys
>>> prev = None
>>> for n in range(300):
...     s = sys.getsizeof({i: i for i in range(n)})
...     if s != prev:
...         print(n, s)
...         prev = s
...
0 64
1 224
6 352
11 632
22 1168
43 2264
86 4688
171 9304
```

85개에서 86개로 넘어가면 2264 → 4688바이트로 **두 배 뛴다.** 리사이즈는 새 테이블을 할당하고 모든 키를 다시 삽입하는 $O(n)$ 작업이다. 평소엔 상각되어 안 보이지만, 그 순간엔 진짜로 일어난다. 키 1000개(내부적으로 슬롯 2048개, 엔트리 1365개)일 때 `sys.getsizeof` 는 36952바이트다. 옛날 구조라면 `2048 x 24 = 49152` 바이트 + 헤더 ≈ 49248바이트였을 것이니, **약 25% 절약**한 셈이다. 딕셔너리가 파이썬 어디에나 있다는 걸 생각하면 인터프리터 전체 메모리에 그대로 반영되는 숫자다.

## str 키 dict는 아예 다른 물건이다

여기 이상한 게 있다.

```pyrepl
>>> import sys
>>> sys.getsizeof({0: 0})
224
>>> sys.getsizeof({"a": 1})
184
```

같은 크기 dict인데 40바이트가 다르다. 왜?

::: deep 유니코드 전용 키 테이블
CPython의 keys 객체는 `dk_kind` 라는 필드로 세 종류를 구분한다. 그중 `DICT_KEYS_UNICODE` 는 **모든 키가 `str` 일 때만** 쓰이는 특수 모드다. 이 모드의 엔트리(`PyDictUnicodeEntry`)는 해시 필드 없이 키·값 포인터 둘만 가져 16바이트다(일반 엔트리는 해시까지 셋을 가져 24바이트). 해시가 없어도 되는 이유는 `str` 객체가 **자기 해시를 자기 안에 캐싱**하기 때문이다 — 문자열의 해시는 한 번 계산하면 그 객체가 죽을 때까지 재사용되니, dict가 또 들고 있을 이유가 없다.

엔트리가 24 → 16바이트, 3분의 1이 사라진다. 앞의 공식에서 24를 16으로 바꾸면 str 키 dict의 크기도 한 바이트까지 정확히 맞는다.

```pyrepl
>>> sys.getsizeof({str(i): i for i in range(1000)})
26032
>>> sys.getsizeof({i: i for i in range(1000)})
36952
```

속도도 다르다. 모든 키가 str이면 CPython은 `__eq__` 를 아예 안 부르는 전용 조회 루틴(`lookdict_unicode`)으로 갈아탄다.

```text nolines
str 키만 있는 dict : 0.0436 s
int 키가 섞인 dict : 0.0487 s        (200만 회 조회, 약 1.12배)
```

**이게 파이썬이 객체 속성과 모듈 전역을 dict로 구현하고도 빠른 이유다.** 그 dict들은 키가 전부 식별자 문자열이라 100% 이 모드에 들어간다.
:::

::: danger int 키 하나가 dict 메모리를 2.8배로 만든다
str 전용 모드는 **str이 아닌 키가 하나라도 들어오면 즉시 무너진다.** 그리고 무너지는 방식이 잔인하다.

```pyrepl
>>> import sys
>>> d = {str(i): i for i in range(1000)}
>>> sys.getsizeof(d)
26032
>>> d[0] = 0                    # 정수 키 딱 하나
>>> sys.getsizeof(d)
73808
```

**26KB → 74KB. 2.835배.** 대입 한 줄로. 엔트리 배열이 16바이트에서 24바이트 형식으로 전부 다시 만들어지고(`DICT_KEYS_GENERAL` 로 강등), 그 재구축이 리사이즈를 타면서 슬롯도 2048에서 4096으로 두 배가 된 결과다.

실전에서 이걸 밟는 곳: JSON을 파싱해 만든 dict에 정수 ID를 키로 끼워 넣을 때, 캐시 dict의 키 타입을 통일하지 않았을 때. **대책은 단순하다. 한 dict의 키 타입을 섞지 마라.**
:::

## 삭제는 자리를 비우지 않는다

`dk_entries` 는 삽입 순서대로 쌓인 배열이다. 그럼 중간 원소를 지우면?

**당길 수 없다.** 당기면 `dk_indices` 에 저장된 모든 인덱스가 어긋난다. 그래서 CPython은 엔트리를 **묘비(tombstone)로 표시만 하고 자리는 그대로 둔다.** 인덱스 슬롯에도 "여긴 비었지만 탐사는 계속하라"는 뜻의 `DKIX_DUMMY`(-2)를 넣는다. 진짜 빈 슬롯(-1)으로 만들면 탐사가 거기서 멈춰서 뒤에 있는 키를 못 찾게 되기 때문이다.

결과가 이렇다.

::: danger dict는 삭제해도 절대 줄어들지 않는다
```python title="shrink_test.py"
import sys, timeit

d = {i: i for i in range(1_000_000)}
for i in range(999_999):
    del d[i]                            # 999,999개 삭제. 1개만 남는다.

fresh = {999_999: 999_999}

print("len  :", len(d), len(fresh))
print("size :", sys.getsizeof(d), sys.getsizeof(fresh))

t1 = timeit.timeit(lambda: list(d), number=200) / 200
t2 = timeit.timeit(lambda: list(fresh), number=200) / 200
print("iter :", round(t1 / t2), "배")
```

```text nolines
len  : 1 1
size : 41943128 224
iter : 4510 배 (같은 기기에서 재실행하면 4천~9천배 사이를 오간다)
```

**원소 1개짜리 dict가 42MB를 쥐고 있다.** 크기 비율은 정확히 `41943128 / 224 ≈ 187,246`배로 재실행해도 항상 똑같다 — 이건 구조가 결정하는 값이라 흔들리지 않는다. 순회 시간 배수는 `fresh` 쪽이 원소 1개라 마이크로초 이하의 잡음에 묻히기 때문에 재실행마다 널뛴다(4천 배가 나올 때도, 9천 배가 나올 때도 있다). **자릿수(천 단위)는 항상 일관되지만 정확한 배수는 믿지 마라.** `list(d)` 가 느린 이유는 확실하다 — 210만 개 엔트리 슬롯을 전부 훑으면서 묘비를 건너뛰어야 하기 때문이다.

**새 키를 넣어도 안 줄어든다.** 아직 `dk_usable` 이 남아 있으면 리사이즈가 안 일어나고, 리사이즈가 안 일어나면 압축도 없다.

```pyrepl
>>> d["new"] = 1
>>> sys.getsizeof(d)
41943128
```

되살리는 방법은 하나뿐이다. **다시 만든다.**

```pyrepl
>>> d = dict(d)
>>> sys.getsizeof(d)
224
```

이건 이론적인 이야기가 아니다. **오래 사는 캐시 dict, 세션 저장소, ROS 노드의 메시지 버퍼** — 넣고 지우기를 반복하는 dict는 실제로 이렇게 부푼다. 삭제가 삽입만큼 잦은 dict는 주기적으로 `d = dict(d)` 로 압축하거나, 애초에 `dict` 대신 크기가 정해진 `collections.deque` 나 `functools.lru_cache` 를 써라. ([5.2 메모리 모델](#/memory))
:::

::: warn 순회 중에 크기를 바꾸면 예외다
```pyrepl
>>> d = {"a": 1, "b": 2, "c": 3}
>>> for k in d:
...     if k == "a":
...         del d[k]
...
Traceback (most recent call last):
  ...
RuntimeError: dictionary changed size during iteration
```

**값만 바꾸는 건 괜찮다.** 크기가 안 변하기 때문이다.

```pyrepl
>>> d = {"a": 1, "b": 2}
>>> for k in d:
...     d[k] = 0
...
>>> d
{'a': 0, 'b': 0}
```

지우면서 순회해야 하면 **키 목록의 스냅샷을 먼저 뜬다.**

```python
for k in list(d):        # ✅ list(d)가 키를 복사해 둔다
    if should_drop(k):
        del d[k]

d = {k: v for k, v in d.items() if not should_drop(k)}    # ✅ 더 나은 방법 — 압축 효과까지 덤
```

두 번째 방법은 새 dict를 만들므로 위에서 본 묘비 문제까지 같이 해결한다.
:::

## `__hash__` / `__eq__` 계약

이제 당신이 만든 객체를 키로 쓸 차례다. 지켜야 할 계약이 있다.

> **1. `a == b` 이면 반드시 `hash(a) == hash(b)` 여야 한다.**
> **2. 키로 쓰이는 동안 해시가 변하면 안 된다.**
> **3. 그 역은 요구되지 않는다.** 해시가 같은데 값이 다른 건 정상이다(= 충돌).

1번이 왜 필요한지는 구조를 보면 자명하다. `a == b` 인데 해시가 다르면 **둘은 서로 다른 슬롯으로 간다.** `d[a] = 1` 한 뒤 `d[b]` 를 찾으면 다른 자리를 뒤지고 `KeyError` 가 난다. 조회의 첫 단계가 해시이지 `==` 가 아니기 때문이다.

::: warn __eq__ 만 정의하면 클래스가 해시 불가능해진다
파이썬이 이걸 강제한다. `__eq__` 를 정의하면 **`__hash__` 가 자동으로 `None` 이 된다.**

```pyrepl
>>> class P:
...     def __init__(self, n): self.n = n
...     def __eq__(self, o): return isinstance(o, P) and self.n == o.n
...
>>> P.__hash__ is None
True
>>> {P(1): "a"}
Traceback (most recent call last):
  ...
TypeError: cannot use 'P' as a dict key (unhashable type: 'P')
```

기본 `object.__hash__` 는 `id()` 기반이다. `__eq__` 를 값 비교로 바꿔 놓고 해시는 그대로 두면 **계약 1번이 즉시 깨진다.** 그래서 파이썬은 "네가 `__eq__` 를 바꿨으니 `__hash__` 도 직접 정하라"며 아예 막아 버린다. **이건 친절한 설계다.** 조용히 깨진 dict보다 시끄러운 `TypeError` 가 백 배 낫다. `@dataclass(frozen=True)` 를 쓰면 둘 다 자동으로 만들어 준다. ([2.6 dataclasses](#/dataclasses))
:::

::: danger 가변 객체를 키로 쓰면 값이 사라진다
계약 2번을 어기면 이렇게 된다.

```pyrepl
>>> class K:
...     def __init__(self, v): self.v = v
...     def __hash__(self): return hash(self.v)
...     def __eq__(self, o): return self.v == o.v
...     def __repr__(self): return f"K({self.v})"
...
>>> k = K(1)
>>> d = {k: "hello"}
>>> k.v = 2                # 키 객체를 수정했다
>>> d
{K(2): 'hello'}
>>> k in d                 # 자기 자신조차 못 찾는다
False
>>> d.get(K(1)), d.get(K(2))
(None, None)
```

**값은 dict 안에 멀쩡히 살아 있는데 어떤 키로도 꺼낼 수 없다.** 예외도 안 난다. 조용한 데이터 손실이다.

`k.v = 2` 가 해시를 바꿨는데, 그 값은 이미 `hash(K(1))` 이 정한 슬롯에 들어가 있다. 이제 `hash(K(2))` 는 다른 슬롯을 가리킨다. 아무도 그 자리를 찾아가지 않는다.

**`list`, `dict`, `set` 이 해시 불가능한 이유가 정확히 이것이다.** 파이썬이 당신을 못 믿어서가 아니라, 가변 객체는 계약 2번을 지킬 수 없기 때문이다.

```pyrepl
>>> {[1, 2]: "a"}
Traceback (most recent call last):
  ...
TypeError: cannot use 'list' as a dict key (unhashable type: 'list')
```

`tuple` 은 해시 가능하다 — **단, 안에 든 게 전부 해시 가능할 때만.** 튜플의 해시는 원소들의 해시로 계산되기 때문이다.

```pyrepl
>>> hash((1, 2, 3))
529344067295497451
>>> hash((1, [2]))
Traceback (most recent call last):
  ...
TypeError: unhashable type: 'list'
```
:::

### `1 == 1.0 == True` 가 만드는 일

계약 1번이 표준 타입에도 적용된다. 그리고 파이썬에서 `1`, `1.0`, `True` 는 **같은 값**이다.

```pyrepl
>>> hash(1) == hash(1.0) == hash(True)
True
>>> 1 == 1.0 == True
True
```

그러므로 dict에게 이 셋은 **같은 키**다.

```pyrepl
>>> {1: "int", 1.0: "float", True: "bool"}
{1: 'bool'}
>>> d = {0: "zero"}
>>> d[False]
'zero'
```

::: warn 키는 안 바뀌고 값만 바뀐다
위 결과를 자세히 보라. `{1: 'bool'}` 이다. `{True: 'bool'}` 이 아니다.

**이미 있는 키에 대입하면 CPython은 값만 갈아 끼운다. 키 객체는 처음 것을 그대로 둔다.** 굳이 키를 교체할 이유가 없기 때문이다(어차피 `==` 니까). 그래서 첫 삽입이 `1` 이면 영원히 `1` 이다.

```pyrepl
>>> {True: "a", 1: "b"}
{True: 'b'}
```

이번엔 `True` 가 먼저라 `True` 가 살아남았다. **키의 정체성은 첫 삽입이 결정하고, 값은 마지막 대입이 결정한다.**

이게 실제 버그가 되는 곳: `Counter` 로 데이터를 세는데 소스에 `1` 과 `True` 가 섞여 있으면 둘이 합산된다. JSON 파싱 결과를 dict 키로 쓸 때도 마찬가지다.
:::

## $O(1)$ 의 진짜 의미, 그리고 최악의 $O(n)$

"dict 조회는 $O(1)$" 이라고 외우는 것과 그게 뭘 뜻하는지 아는 건 다르다.

```python title="flat.py — 크기가 100만 배 커져도 조회 시간이 그대로인가"
import timeit

for n in (10, 1000, 100_000, 10_000_000):
    s = f"d = {{i: i for i in range({n})}}; k = {n - 1}"
    t = min(timeit.repeat("d[k]", s, number=1_000_000, repeat=3)) / 1_000_000
    print(f"{n:>10,} : {t * 1e9:5.1f} ns")
```

```text nolines
        10 :  10.8 ns
     1,000 :  13.8 ns
   100,000 :  13.6 ns
10,000,000 :  13.7 ns
```

(Python 3.14.5 / Windows 기준 실측. 절대값은 기기마다 다르지만 **평탄한 모양**은 어디서나 같다.)

**평탄하다.** 원소가 100만 배 늘어도 조회 시간이 그대로다. 이게 $O(1)$ 의 실물이다.

주목할 건 오히려 10개짜리가 **10.8ns로 더 빠르다**는 것이다. 알고리즘 때문이 아니다. 테이블 전체가 CPU L1 캐시에 통째로 들어가기 때문이다. 100만 개짜리는 매 조회가 사실상 랜덤 메모리 접근이라 캐시 미스를 먹는다. **파이썬 dict에서 $n$ 이 성능에 영향을 주는 유일한 경로는 캐시다.** 이건 빅오가 말해 주지 않는다.

### 그런데 최악은 $O(n)$ 이다

해시 함수가 제 일을 안 하면 어떻게 되는지 보자.

```python title="worst_case.py — 모든 키의 해시가 같다면"
import timeit


class Bad:
    def __hash__(self):
        return 42                    # 모든 인스턴스가 42번 슬롯을 노린다

    def __eq__(self, o):
        return self is o


class Good:
    pass                             # 기본 __hash__ = id 기반


for cls, label in ((Bad, "Bad "), (Good, "Good")):
    for n in (1000, 2000, 4000):
        keys = [cls() for _ in range(n)]
        t = timeit.timeit(lambda: {k: 1 for k in keys}, number=1)
        print(label, n, round(t, 6))
```

```text nolines
Bad  1000 0.0155
Bad  2000 0.0589
Bad  4000 0.2342
Good 1000 3.2e-05
Good 2000 7e-05
Good 4000 0.000131
```

`Good` 은 개수가 두 배면 시간도 두 배다 — $O(n)$ 으로 $n$ 개를 넣었다는 뜻이니 삽입 하나당 $O(1)$.

`Bad` 는 개수가 두 배면 **시간이 네 배**다. 0.0155 → 0.0589 → 0.2342. $O(n^2)$. 삽입 하나가 $O(n)$ 이라는 뜻이다. 4000개에서 이미 **수천 배** 느리다 — 정확히 몇 배인지는 재실행마다 크게 흔들린다. 같은 코드를 8번 돌려 보면 842배가 나올 때도, 1,524배가 나올 때도 있다(`timeit(number=1)` 단발 측정이라 시스템 잡음을 그대로 먹는다). 흔들리지 않는 건 **관계** 쪽이다. `Good` 은 $n$ 이 두 배면 시간도 두 배, `Bad` 는 $n$ 이 두 배면 시간이 네 배 — $O(n)$ 대 $O(n^2)$ 이라는 자릿수 차이는 몇 번을 다시 재도 똑같이 나온다. 모든 키가 42번 슬롯에서 출발해 탐사 사슬을 처음부터 끝까지 훑기 때문이다.

**$O(1)$ 은 해시가 고르게 흩어진다는 가정 위에서만 성립한다.** 그 가정은 당신이 `__hash__` 를 잘못 쓰면 즉시 무너진다.

::: deep 해시 랜덤화 — 이건 보안 문제였다
악의적인 사용자가 일부러 충돌하는 키를 대량으로 보내면 서버를 세울 수 있다. 웹 서버가 쿼리 스트링이나 JSON 본문을 dict로 파싱한다면, 공격자는 $O(n)$ 짜리 입력으로 $O(n^2)$ 의 CPU를 태울 수 있다. 2011년에 실제로 터진 취약점이다(CVE-2012-1150).

파이썬의 대응은 **문자열 해시에 프로세스마다 다른 난수 시드를 섞는 것**이었다(PEP 456, 3.3부터 기본 활성). 그래서 문자열 해시는 실행할 때마다 다르고, `PYTHONHASHSEED` 로 고정하지 않는 한 예측할 수 없다. **랜덤화되는 건 `str` 과 `bytes` 뿐이다** — 정수·튜플·frozenset의 해시는 실행 간에 고정이다.

```bash
$ python -c "print(hash('python'))"
309243305923044716
$ python -c "print(hash('python'))"
-6960366976175296462
$ PYTHONHASHSEED=0 python -c "print(hash('python'))"
4381348739211007536
$ PYTHONHASHSEED=0 python -c "print(hash('python'))"
4381348739211007536
```

실전 함의 두 가지. 첫째, **`hash()` 값을 파일이나 DB에 저장하지 마라.** 다음 실행에서 안 맞는다. 둘째, **`set` 의 순회 순서가 실행마다 달라진다.** 문자열 집합을 순회하는 코드가 어제는 통과하고 오늘은 실패한다면 이게 범인이다. dict는 삽입 순서라서 이 영향을 안 받는다.
:::

## 실전 관용구

### `get` 과 `setdefault`, 그리고 `defaultdict`

```python
# ❌ 두 번 조회한다
if k in d:
    v = d[k]
else:
    v = 0

v = d.get(k, 0)              # ✅ 한 번
```

`setdefault` 는 "없으면 넣고, 있든 없든 그 값을 돌려준다"를 **한 번의 조회로** 한다. 그런데 함정이 있다.

::: warn setdefault 의 기본값은 항상 만들어진다
```pyrepl
>>> calls = 0
>>> class C:
...     def __init__(self):
...         global calls; calls += 1
...
>>> d = {"a": C()}
>>> d.setdefault("a", C())     # 'a'는 이미 있는데도...
<__main__.C object at ...>
>>> calls
2
```

**`C()` 가 두 번 불렸다.** 인자는 호출 **전에** 평가되므로, 키가 있든 없든 `C()` 는 무조건 만들어지고 필요 없으면 버려진다. 기본값을 만드는 게 비싸면(DB 연결, 큰 리스트, 파일 열기) 순수한 낭비다. `defaultdict` 는 **팩토리를 넘기므로** 필요할 때만 부른다.

```python title="세 방법 비교 — 10만 단어 x 10회"
from collections import defaultdict

def m_setdefault(words):
    d = {}
    for w in words:
        d.setdefault(w, []).append(w)
    return d

def m_check(words):
    d = {}
    for w in words:
        if w not in d:
            d[w] = []
        d[w].append(w)
    return d

def m_defaultdict(words):
    d = defaultdict(list)
    for w in words:
        d[w].append(w)
    return d
```

```text nolines
m_setdefault  0.0340
m_check       0.0278
m_defaultdict 0.0189
```

**`setdefault` 가 가장 느리다.** 키가 이미 있는 경우가 많을수록(위 예시는 4종류 단어가 10만 번) 버려지는 `[]` 가 많아지기 때문이다. **그룹핑에는 `defaultdict` 를 써라.** `setdefault` 는 기본값이 리터럴 상수(`0`, `None`)일 때만 쓸 만하다. 자세한 건 [7.6 해시](#/hashing)에서.
:::

### dict 컴프리헨션과 `dict()`

```pyrepl
>>> {k: v for k, v in [("a", 1), ("b", 2)]}
{'a': 1, 'b': 2}
>>> {v: k for k, v in {"a": 1, "b": 2}.items()}     # 키-값 뒤집기
{1: 'a', 2: 'b'}
```

키가 겹치면 **나중 것이 이긴다.** 뒤집기를 할 때 값이 중복이면 조용히 사라지니 주의하라.

::: perf 변환만 할 거면 dict() 가 더 빠르다
```text nolines
dict(pairs)                   0.387          <- 가장 빠르다
for k, v in pairs: d[k] = v   0.460          <- dict()보다 약 1.19배 느리다
{k: v for k, v in pairs}      0.472          <- 셋 중 가장 느리다
```

(쌍 10만 개 x 100회, repeat=4의 최솟값을 3번 더 측정해 평균 — Python 3.14.5 / Windows 기준 실측)

`dict(pairs)` 는 루프 전체가 C 안에서 돈다. **for 루프와 dict 컴프리헨션은 둘 다 파이썬 바이트코드를 매 반복 실행하므로 거의 붙는다** — 순서를 뒤집어 재실행해도 for 루프가 컴프리헨션보다 근소하게(2~3%) 빠르게 나온다. 컴프리헨션이 아주 살짝 더 느린 건, `{...}` 리터럴이 매 반복 `STORE_SUBSCR` 대신 결과를 쌓는 내부 헬퍼 프레임을 거치기 때문이다. 어느 쪽이든 `dict(pairs)` 와의 차이(약 20%)에 비하면 미미하다. **필터링이나 변환 없이 그냥 쌍을 dict로 만들 뿐이라면 `dict()` 를 써라.** 컴프리헨션은 `if` 나 표현식이 필요할 때, for 루프는 매 반복 부수효과(로깅, 카운팅)가 필요할 때 쓴다. ([1.9 컴프리헨션](#/comprehensions))
:::

::: danger fromkeys 의 기본값은 하나의 객체다
```pyrepl
>>> d = dict.fromkeys(["a", "b", "c"], [])
>>> d["a"].append(1)
>>> d
{'a': [1], 'b': [1], 'c': [1]}
```

[1.1 객체, 이름, 참조](#/objects-names)에서 본 `[[0]*3]*3` 과 **정확히 같은 함정**이다. `fromkeys` 는 두 번째 인자를 한 번만 평가하고 그 **참조**를 모든 키에 붙인다.

가변 기본값이 필요하면 `defaultdict(list)` 나 컴프리헨션을 써라.

```python
d = {k: [] for k in ["a", "b", "c"]}      # ✅ 키마다 새 리스트
```

`dict.fromkeys` 가 유용한 곳은 **값이 필요 없을 때**다. 순서를 지키며 중복을 제거하는 관용구가 대표적이다.

```pyrepl
>>> list(dict.fromkeys([3, 1, 3, 2, 1]))
[3, 1, 2]
```

`set` 으로 하면 순서가 깨진다. dict는 삽입 순서를 지키므로 이게 성립한다. **3.7 순서 보장이 언어 명세가 된 덕에 안전하게 쓸 수 있는 관용구다.**
:::

### 3.9의 `|` 연산자

3.9 이전에 두 dict를 합치려면 `{**a, **b}` 나 `a.copy()` 후 `update` 였다. PEP 584가 연산자를 줬다.

```pyrepl
>>> a = {"x": 1, "y": 2}
>>> b = {"y": 99, "z": 3}
>>> a | b
{'x': 1, 'y': 99, 'z': 3}
>>> b | a
{'y': 2, 'z': 3, 'x': 1}
```

**순서는 왼쪽이 정하고, 값은 오른쪽이 이긴다.** `b | a` 를 보라. `'y'` 가 맨 앞인 건 왼쪽 `b` 의 삽입 순서 때문이고, 값이 `2` 인 건 오른쪽 `a` 가 덮어썼기 때문이다. 위에서 본 "키는 첫 삽입, 값은 마지막 대입" 규칙 그대로다.

`|=` 는 제자리 수정이다.

```pyrepl
>>> c = {"x": 1}
>>> c |= {"y": 2}
>>> c
{'x': 1, 'y': 2}
```

`|=` 는 `|` 보다 관대하다. `|` 는 dict끼리만 되지만 `|=` 는 `update` 와 같아서 키-값 쌍의 이터러블도 받는다.

```pyrepl
>>> {"x": 1} | [("k", 1)]
Traceback (most recent call last):
  ...
TypeError: unsupported operand type(s) for |: 'dict' and 'list'
>>> c = {"x": 1}
>>> c |= [("k", 1)]          # 이건 된다
>>> c
{'x': 1, 'k': 1}
```

일부러 이렇게 만들었다 — `|` 는 반환 타입이 명확해야 하는데 `dict | list` 가 무슨 타입인지 모호하다. `|=` 는 왼쪽이 dict임이 확정이라 애매함이 없다.

### dict 뷰는 집합이다

`.keys()` 는 리스트가 아니라 **뷰**다. 집합 연산을 그대로 지원해서 두 dict의 공통 키·차집합을 바로 구할 수 있다.

```pyrepl
>>> a = {"x": 1, "y": 2}
>>> a.keys() | {"z"}
{'y', 'x', 'z'}
>>> a.keys() & {"x"}
{'x'}
```

(결과가 `set` 이므로 원소 순서는 해시 랜덤화의 영향을 받는다 — 재실행하면 다른 순서로 나올 수 있다. 내용물은 항상 같다.) `.items()` 도 값이 해시 가능하면 집합 연산이 된다. `.values()` 는 안 된다 — 값은 중복될 수 있고 해시 가능하다는 보장도 없다.

::: cote 코딩테스트 포인트
**1. `in` 을 리스트에 쓰면 그 자리에서 시간 초과다.**

```python
# ❌ O(n) x m회 = O(nm)
if x in visited_list:

# ✅ O(1) x m회 = O(m)
if x in visited_set:
```

절 첫머리의 5만 3천 배가 이거다. **`in` 을 쓸 컨테이너는 무조건 `set` 이나 `dict`.** 순회만 할 거면 리스트가 낫다.

**2. 그룹핑은 `defaultdict`.**

```python
from collections import defaultdict

graph = defaultdict(list)
for u, v in edges:
    graph[u].append(v)
    graph[v].append(u)
```

인접 리스트를 이렇게 만든다. `setdefault` 보다 빠르고 짧다. ([7.13 그래프](#/graph))

**3. 튜플을 키로 써라.** 2차원 좌표, 상태 조합을 dict 키로 쓸 때 튜플이 정석이다.

```python
dist = {}
dist[(r, c)] = 0                 # 좌표를 키로
memo[(i, mask)] = result         # 비트마스크 DP 상태를 키로
```

단, **`dist[(r, c)]` 보다 2차원 리스트 `dist[r][c]` 가 몇 배 빠르다** — 튜플을 매번 새로 만들고 해싱해야 하기 때문이다. 좌표 범위가 정해져 있고 조밀하면 리스트를, 희소하거나 범위를 모르면 dict를 써라. ([7.20 DP 기초](#/dp-basics))

**4. `Counter` 는 dict의 서브클래스다.** `most_common()` 은 정렬이라 $O(n \log n)$ 이다. 최댓값 하나만 필요하면 `max(cnt, key=cnt.get)` 이 $O(n)$ 으로 더 빠르다. 그리고 일부 저지 사이트는 파이썬 풀이를 저격하려고 `[0, 2**20, 2**21, ...]` 처럼 `hash(int)` 가 전부 같은 슬롯으로 몰리는 입력을 넣는다 — 당했다 싶으면 키를 `x ^ 0x9e3779b9` 같은 값으로 섞어 넣어 우회한다.
:::

## dict의 나머지 얼굴들

`dict` 는 순서 있는 매핑이 됐다. 그럼 `OrderedDict` 는 이제 쓸모없나? 아니다. **`dict` 의 `==` 는 순서를 무시하지만 `OrderedDict` 의 `==` 는 순서까지 본다.**

```pyrepl
>>> from collections import OrderedDict
>>> {"a": 1, "b": 2} == {"b": 2, "a": 1}
True
>>> OrderedDict(a=1, b=2) == OrderedDict(b=2, a=1)
False
>>> import sys
>>> sys.getsizeof({"a": 1}), sys.getsizeof(OrderedDict(a=1))
(184, 344)
```

`OrderedDict` 는 `move_to_end()` 도 준다 — LRU 캐시를 직접 구현할 때 필수다. 대신 메모리는 거의 두 배다. **순서 비교나 `move_to_end` 가 필요 없으면 `dict` 를 써라.**

::: note dict는 파이썬의 골격이다
이 절에서 배운 게 dict라는 타입 하나에만 해당하는 게 아니다.

- **`obj.__dict__`** — 객체 속성. `obj.x` 는 dict 조회다.
- **모듈 전역** — `globals()` 가 dict다. 전역 변수 읽기가 dict 조회인 이유이자, [5.3 파이썬 레벨 최적화](#/py-optimize)에서 "지역 변수가 전역보다 빠르다"고 말하는 이유다(지역 변수는 배열 인덱싱이고 전역은 dict 조회다).
- **클래스 네임스페이스** — 메서드 탐색이 MRO를 따라가며 각 클래스의 dict를 뒤진다. ([1.13 상속, MRO](#/inheritance))
- **`**kwargs`** — dict다.
- **`set`** — 값 없는 dict나 다름없다. 같은 탐사 수식, 같은 계약. ([1.7 set과 frozenset](#/sets))

앞에서 본 **str 전용 키 테이블**이 왜 그렇게까지 최적화됐는지 이제 보일 것이다. 저 dict들은 전부 키가 식별자 문자열이다. dict를 1% 빠르게 만들면 파이썬 전체가 1% 빨라진다.
:::

## 요약

- **dict는 해시 테이블이다.** 키를 찾는 게 아니라 키가 있을 자리를 **계산한다.** 20만 원소에서 리스트 대비 5만 배 빠르다.
- **테이블 크기는 항상 2의 거듭제곱**이라 `hash & (size-1)` 로 슬롯을 구한다. 나눗셈이 없다. 대신 상위 비트가 버려지는데, 탐사 수식의 `perturb` 가 그걸 되살린다.
- **충돌은 개방 주소법**으로 푼다. `i = (5i + perturb + 1) & mask` — 상위 비트를 섞으면서도 결국 모든 슬롯을 방문한다.
- **3.6의 compact dict는 인덱스 배열과 엔트리 배열로 쪼갠 구조**다. 메모리를 25% 아꼈고, **순서 보장은 그 부산물**이었다. 3.7부터는 언어 명세다.
- **키가 전부 `str` 이면 dict는 다른 구조를 쓴다**(엔트리 16바이트, 전용 조회 루틴). **`str` 아닌 키 하나가 섞이면 메모리가 2.8배 뛴다.**
- **삭제해도 dict는 절대 줄지 않는다.** 묘비만 남는다. 되살리려면 `dict(d)` 로 다시 만들어라.
- **계약: `a == b` 면 `hash(a) == hash(b)`, 그리고 키인 동안 해시는 불변.** `__eq__` 만 정의하면 파이썬이 해시를 막아 준다 — 고마운 일이다.
- **$O(1)$ 은 해시가 고르다는 가정 위에 있다.** 무너지면 삽입 하나가 $O(n)$, 전체가 $O(n^2)$ 이 된다.
