# 7.26 기하 알고리즘

::: lead
좌표평면 위의 문제는 코딩테스트에서 유독 기피 대상이다. 아이디어는 어렵지 않은데 **부동소수점 오차 하나로 통째로 틀리기** 때문이다. `atan2`, `sqrt`, 기울기 나눗셈에 의존하면 맞는 로직도 예제에서만 통과하고 채점 서버에서는 WA(Wrong Answer)를 받는다. 이 절의 목표는 하나다 — **나눗셈과 삼각함수를 쓰지 않고, 정수 곱셈과 뺄셈만으로 방향·교차·볼록 껍질을 판정하는 법**을 익힌다. 그 핵심 도구가 CCW(counter-clockwise) 판정이다.
:::

## 방향을 묻는 질문에 나눗셈으로 답하면 안 되는 이유

세 점 $A$, $B$, $C$가 있을 때 "$A \to B \to C$로 이으면 왼쪽(반시계)으로 도는가, 오른쪽(시계)으로 도는가?"는 기하 알고리즘 전체의 기초 질문이다. 볼록 껍질도, 선분 교차도, 다각형 넓이도 결국 이 질문의 반복이다.

직관적으로 떠오르는 방법은 두 벡터의 **기울기를 비교**하는 것이다.

```python title="slope_naive.py — 기울기로 방향을 판단하는 나쁜 코드"
def turn_naive(a, b, c):
    # 기울기 AB와 AC를 비교한다 -- 나눗셈이 들어간다
    slope_ab = (b[1] - a[1]) / (b[0] - a[0])  # b[0] == a[0]이면 ZeroDivisionError
    slope_ac = (c[1] - a[1]) / (c[0] - a[0])  # c[0] == a[0]이면 ZeroDivisionError
    return slope_ac - slope_ab
```

이 코드는 두 가지로 무너진다. 첫째, `b[0] == a[0]`(수직선)이면 그 자리에서 `ZeroDivisionError`다. 매번 예외 처리를 넣어야 한다. 둘째, 나눗셈은 부동소수점을 만들고, 부동소수점은 오차를 만든다.

```pyrepl
>>> a, b, c = (0.1, 0.1), (0.2, 0.2), (0.3, 0.30000000000000004)
>>> # a, b, c는 사실 한 직선 위에 있어야 하는 점들이다 (기울기 1)
>>> cross = (b[0]-a[0])*(c[1]-a[1]) - (b[1]-a[1])*(c[0]-a[0])
>>> cross
3.469446951953614e-18
```

세 점은 원래 완벽하게 일직선(collinear)이어야 하는데, 부동소수점 계산 결과는 정확히 0이 아니라 `3.469446951953614e-18`이 나온다. 이 값의 부호로 "왼쪽인지 오른쪽인지"를 판단하면, **수학적으로는 일직선인 경우조차 미세한 오차 때문에 왼쪽 또는 오른쪽으로 잘못 분류**된다. 볼록 껍질 알고리즘은 바로 이 부호 하나로 점을 넣을지 뺄지 결정하기 때문에, 이 오차가 최종 답을 뒤집는다.

::: warn 부동소수점으로 방향을 판정하지 마라
`atan2`로 각도를 구해 비교하는 방법도 같은 문제를 안고 있다. 각도 계산에는 삼각함수와 나눗셈이 들어가고, 서로 다른 두 각이 부동소수점 오차 때문에 같은 값으로 뭉개지거나, 반대로 같은 각이 미세하게 갈라진다. **기하 알고리즘에서 나눗셈과 삼각함수는 최후의 수단이다.** 다음 절에서 볼 외적(cross product) 기반 판정은 곱셈과 뺄셈만 쓴다.
:::

## CCW: 외적 하나로 방향을 정확히 판정한다

벡터 $\vec{AB}$와 $\vec{AC}$의 **외적**(cross product)의 부호가 곧 회전 방향이다.

$$\text{cross} = (B_x - A_x)(C_y - A_y) - (B_y - A_y)(C_x - A_x)$$

```python title="ccw.py — 정수 연산만으로 방향 판정"
def ccw(a, b, c):
    val = (b[0] - a[0]) * (c[1] - a[1]) - (b[1] - a[1]) * (c[0] - a[0])
    if val > 0:
        return 1    # 반시계 (왼쪽으로 꺾임)
    if val < 0:
        return -1   # 시계 (오른쪽으로 꺾임)
    return 0        # 일직선 (collinear)
```

```pyrepl
>>> ccw((0, 0), (1, 0), (1, 1))
1
>>> ccw((0, 0), (1, 1), (1, 0))
-1
>>> ccw((0, 0), (1, 0), (2, 0))
0
```

좌표가 정수이기만 하면 이 값도 **항상 정확한 정수**다. 나눗셈이 없으니 오차가 낄 틈이 없다.

::: deep 외적의 값 자체도 의미가 있다 — 넓이의 두 배
`cross`의 절댓값은 삼각형 $ABC$ 넓이의 정확히 두 배다. 즉 `abs(val) / 2`가 삼각형 넓이(신발끈 공식, shoelace formula)다. 부호는 방향, 절댓값은 넓이 — 외적 하나가 두 가지 정보를 동시에 준다. 이 성질은 다각형 넓이를 구하는 문제(백준 2166 등)에서 그대로 쓰인다. 좌표가 정수면 넓이가 정수 또는 반정수로 정확히 나온다는 것도 이 절 뒷부분의 "정수 좌표를 써라"는 조언과 같은 이유다.
:::

## 선분 교차 판정

두 선분 $P_1P_2$와 $P_3P_4$가 서로 교차하는지 판정하는 문제(백준 17370 등)는 CCW 네 번으로 끝난다. 핵심 아이디어: **$P_1$과 $P_2$가 직선 $P_3P_4$를 기준으로 서로 반대쪽에 있고, 동시에 $P_3$과 $P_4$가 직선 $P_1P_2$를 기준으로 서로 반대쪽에 있으면 두 선분은 교차한다.**

```text nolines
     P1                    P3
       \                  /
        \      X         /       두 선분이 서로를 "가로지른다"
         \    /\        /        -> P1, P2가 선분 P3P4의 양쪽에 있고
          \  /  \      /            P3, P4가 선분 P1P2의 양쪽에 있다
           \/    \    /
           /\     \  /
          /  \     \/
        P4    -    P2
```

```python title="segments_intersect.py — 일반 교차 + 겹치는 선분(collinear) 처리"
def on_segment(a, b, p):
    """p가 이미 a, b와 일직선임을 전제로, p가 선분 a-b의 bounding box 안에 있는지만 확인한다."""
    return min(a[0], b[0]) <= p[0] <= max(a[0], b[0]) and \
           min(a[1], b[1]) <= p[1] <= max(a[1], b[1])


def segments_intersect(p1, p2, p3, p4):
    d1 = ccw(p3, p4, p1)
    d2 = ccw(p3, p4, p2)
    d3 = ccw(p1, p2, p3)
    d4 = ccw(p1, p2, p4)

    if d1 != d2 and d3 != d4:      # 일반적인 경우: 서로 다른 부호면 교차
        return True

    # 부호가 0인 경우(일직선) -- 끝점이 상대 선분 위에 얹혀 있는지 따로 확인
    if d1 == 0 and on_segment(p3, p4, p1):
        return True
    if d2 == 0 and on_segment(p3, p4, p2):
        return True
    if d3 == 0 and on_segment(p1, p2, p3):
        return True
    if d4 == 0 and on_segment(p1, p2, p4):
        return True

    return False
```

`d1 != d2 and d3 != d4`만으로는 부족하다. **한 선분의 끝점이 다른 선분 위에 정확히 얹히거나, 두 선분이 일직선 위에서 겹치는 경우**는 `ccw`가 0을 반환하므로 별도의 `on_segment` 검사가 필요하다. 이 네 가지 예외 케이스를 빠뜨리는 것이 이 문제 유형의 가장 흔한 오답 원인이다.

### 정답 검증: 방향 판정을 전혀 쓰지 않는 방법과 대조

이 구현이 맞는지 확인하려면, `ccw`를 아예 쓰지 않는 **독립적인 방법**과 대조해야 의미가 있다. 두 직선의 교점을 유리수(`Fraction`)로 정확히 풀고, 그 교점이 두 선분 구간 안에 들어오는지 확인하는 방식을 썼다.

```python title="brute_intersect_check.py — Fraction으로 두 직선의 교점을 직접 계산"
from fractions import Fraction as F

def brute_intersect_check(p1, p2, p3, p4):
    # 퇴화 선분(길이 0, 즉 시작점==끝점) 먼저 걸러낸다 -- 아래 denom==0 분기는
    # "직선 P1P2 자체가 정의되지 않는" 이 경우를 처리하지 못한다.
    if p1 == p2 and p3 == p4:
        return p1 == p3
    if p1 == p2:
        return ccw(p3, p4, p1) == 0 and on_segment(p3, p4, p1)
    if p3 == p4:
        return ccw(p1, p2, p3) == 0 and on_segment(p1, p2, p3)

    x1, y1 = p1; x2, y2 = p2
    x3, y3 = p3; x4, y4 = p4
    denom = (x2 - x1) * (y4 - y3) - (y2 - y1) * (x4 - x3)

    if denom == 0:
        cross = (x2 - x1) * (y3 - y1) - (y2 - y1) * (x3 - x1)
        if cross != 0:
            return False              # 평행하지만 다른 직선

        def proj(p):                  # 방향벡터로 1차원 좌표화
            return (p[0]-x1)*(x2-x1) + (p[1]-y1)*(y2-y1)
        lo1, hi1 = 0, proj(p2)
        lo2, hi2 = sorted((proj(p3), proj(p4)))
        return hi1 >= lo2 and hi2 >= min(lo1, hi1)

    t = F((x3-x1)*(y4-y3) - (y3-y1)*(x4-x3), denom)
    u = F((x3-x1)*(y2-y1) - (y3-y1)*(x2-x1), denom)
    return F(0) <= t <= F(1) and F(0) <= u <= F(1)
```

무작위 정수 좌표로 3,000쌍의 선분을 만들어 두 구현을 대조했다.

```pyrepl
>>> mismatches = 0
>>> for _ in range(3000):
...     # 무작위 좌표 4쌍으로 두 선분을 만들고 양쪽 결과를 비교
...     if segments_intersect(*pts) != brute_intersect_check(*pts):
...         mismatches += 1
>>> mismatches
0
```

**3,000쌍 전부 일치했다.** (Python 3.14.5 기준 실행, 좌표 범위 -1000~1000.) 이 대조 패턴 — "빠른 구현을 전혀 다른 원리의 느린 구현과 맞춰 본다" — 은 [7.12 유니온 파인드](#/union-find)에서 쓴 것과 같은 검증 방식이다.

::: warn '0/3000'은 좌표 범위가 넓을 때만 성립하는 결과다
처음 버전의 `brute_intersect_check`는 `p1 == p2`(길이 0인 퇴화 선분)를 걸러내지 않았다. 이 경우 `denom`과 `cross`가 항상 0이 되어 `denom == 0` 분기로 들어가고, `proj` 계산도 방향벡터가 `(0, 0)`이라 전부 0으로 뭉개지면서 **실제 기하와 상관없이 항상 `True`를 반환**하는 버그가 있었다. 좌표 범위를 -1000~1000이나 -100~100으로 잡으면 무작위 4점 중 두 점이 우연히 일치할 확률이 낮아 3,000쌍에서 이 버그가 드러나지 않았다(0/3000). 하지만 범위를 -5~5로 좁히면 `p1 == p2`인 표본이 실제로 나오면서 3,000쌍 중 20건이 불일치했다(예: `p1=p2=(-8,-8)`, `p3=(3,-7)`, `p4=(1,-6)` — `ccw(p3,p4,p1)=13≠0`이라 실제로는 만나지 않는데 옛 `brute_intersect_check`는 `True`를 반환했다). 위 코드는 `p1==p2` / `p3==p4` 퇴화 케이스를 앞에서 따로 처리해 이 버그를 없앴고, 좌표 범위를 -1000, -100, -20, -5로 바꿔 가며 다시 돌려도 전부 0/3000이 나오는 것을 확인했다.
:::

::: cote 선분 교차 문제의 함정 세 가지
1. **끝점끼리 닿기만 해도 교차로 친다.** $P_2 = P_3$처럼 한 점에서 맞닿는 경우도 대부분의 문제에서 "교차"로 인정한다. `d1==0`류 분기를 빠뜨리면 이런 경우를 놓친다.
2. **두 선분이 같은 직선 위에서 겹친다(collinear overlap).** 이때는 `ccw` 네 개가 전부 0이 될 수 있다. `on_segment`로 구간이 겹치는지 반드시 따로 확인해야 한다.
3. **입력이 실수 좌표라도 문제 제약을 보면 정수로 스케일할 수 있는 경우가 많다.** 소수점 둘째 자리까지라면 100을 곱해 정수로 바꿔라. 이 절 뒤에서 다시 다룬다.
:::

## 볼록 껍질이 필요한 이유 — 브루트포스가 막히는 지점

**볼록 껍질**(convex hull)은 평면 위 점들을 모두 감싸는 가장 작은 볼록다각형이다. "울타리로 모든 점을 최소 길이로 둘러싸라", "가장 멀리 떨어진 두 점을 찾아라"(회전하는 캘리퍼스의 전 단계) 같은 문제의 뼈대다.

가장 단순하게 떠오르는 방법은 이렇다. **모든 점의 쌍 $(a, b)$에 대해, 나머지 점이 전부 직선 $ab$의 한쪽에만 있으면 $ab$는 볼록 껍질의 변이다.**

```python title="convex_hull_bruteforce.py — 모든 쌍을 검사하는 O(n^3) 버전"
def convex_hull_bruteforce(points):
    pts = list(set(points))
    n = len(pts)
    hull_edges = []
    for i in range(n):
        for j in range(n):
            if i == j:
                continue
            a, b = pts[i], pts[j]
            side = None
            ok = True
            for k in range(n):               # 나머지 점 전부 검사
                if k == i or k == j:
                    continue
                c = ccw(a, b, pts[k])
                if c == 0:
                    continue
                if side is None:
                    side = c
                elif c != side:               # 양쪽에 점이 있다 -> ab는 변이 아니다
                    ok = False
                    break
            if ok:
                hull_edges.append((a, b))
    return hull_edges
```

점 쌍이 $O(n^2)$개고, 쌍마다 나머지 $O(n)$개 점을 검사하니 최악 $O(n^3)$이다. 실제로 재 보면 확실히 느려진다.

| n | 브루트포스 O(n³) | 모노톤 체인 |
| --- | --- | --- |
| 50 | 0.0021초 | 0.000044초 |
| 100 | 0.0084초 | 0.000084초 |
| 200 | 0.0300초 | 0.000165초 |
| 400 | 0.1417초 | 0.000316초 |
| 800 | 0.5885초 | 0.000651초 |

(Python 3.14.5 / Windows 기준 실측. 같은 무작위 점 집합, 두 알고리즘 결과는 동일한 볼록 껍질임을 확인했다.)

$n=800$에서 브루트포스는 0.59초, 모노톤 체인은 0.00065초다 — **약 900배 차이.** 코딩테스트에서 흔히 등장하는 $n=10^5$대라면 브루트포스는 아예 시도조차 못 한다(이 표의 증가 추세만 봐도 몇 시간 단위로 뛴다). 이 절에서 다룰 정렬 기반 알고리즘은 $O(n \log n)$으로 끝낸다.

::: cote 신호: 언제 볼록 껍질을 떠올려야 하는가
- "모든 점을 감싸는 다각형/울타리/철조망"
- "가장 먼 두 점 사이의 거리"(회전하는 캘리퍼스와 결합)
- $n$이 $10^4 \sim 10^6$ 수준으로 커서 $O(n^2)$조차 위험한 기하 문제
- 백준 1708(볼록 껍질), 10254(고속도로) 같은 문제가 전형이다

이런 제약 조건에서 알고리즘을 역추론하는 훈련은 [8.4 문제 유형 분류와 신호 읽기](#/problem-signals)에서 더 다룬다.
:::

## 앤드류 모노톤 체인: $O(n \log n)$ 볼록 껍질

교과서에 흔히 나오는 **그라함 스캔**(Graham scan)은 기준점 하나를 잡고 나머지를 각도순으로 정렬한 뒤 스택으로 훑는다. 각도 정렬에는 `atan2`가 끼어들기 쉬워 오차에 취약하다.

**앤드류 모노톤 체인**(Andrew's monotone chain)은 아이디어가 더 단순하다. 점을 **$x$좌표(같으면 $y$좌표) 기준으로 정렬**한 뒤, 아래쪽 껍질과 위쪽 껍질을 각각 한 번씩 훑어서 만든다. 정렬만 순수하게 좌표 비교로 하니 각도 계산이 아예 없다.

```text nolines
정렬된 점을 왼쪽에서 오른쪽으로 훑으며 아래쪽 껍질을 만든다.
스택 맨 위 두 점과 새 점이 오른쪽(시계)으로 꺾이면 맨 위를 버린다.

   추가 전:  ... A -- B          새 점 C가 오른쪽으로 꺾이게 만들면
   추가 후:  ... A ------ C      B를 버리고 C를 붙인다 (오목해지는 걸 막는다)
                 \      /
                  \    /
                   B(버려짐)
```

```python title="convex_hull_monotone.py — 앤드류 모노톤 체인"
def convex_hull_monotone(points):
    pts = sorted(set(points))          # x, y 순으로 정렬 -- O(n log n)의 지배적 비용
    if len(pts) <= 2:
        return pts

    def half(pts_):
        hull = []
        for p in pts_:
            # 오른쪽으로 꺾이거나(<=0) 일직선이면 직전 점을 버린다
            while len(hull) >= 2 and ccw(hull[-2], hull[-1], p) <= 0:
                hull.pop()
            hull.append(p)
        return hull

    lower = half(pts)             # 왼쪽 -> 오른쪽으로 아래쪽 껍질
    upper = half(pts[::-1])       # 오른쪽 -> 왼쪽으로 위쪽 껍질
    return lower[:-1] + upper[:-1]     # 양 끝은 겹치니 하나씩 제거
```

`while` 조건의 `<= 0`이 핵심이다. **0(일직선)도 버린다** — 볼록 껍질의 변 위에 다른 점과 일직선으로 놓인 점은 굳이 꼭짓점으로 남길 필요가 없기 때문이다(이렇게 만든 껍질을 "엄격한(strict) 볼록 껍질"이라 부른다). 아래쪽 껍질과 위쪽 껍질을 각각 만들고 이어 붙이면 전체 껍질이 완성된다.

### 정답 검증: 볼록 껍질의 정의로 직접 확인

이번에는 다른 알고리즘과 비교하는 대신, **볼록 껍질의 정의 자체**로 검증한다. (1) 껍질을 이루는 변들이 전부 같은 방향(반시계)으로 꺾여야 하고, (2) 원본 점 전부가 그 다각형의 안쪽 또는 경계 위에 있어야 한다. 둘 다 `ccw` 하나로 확인할 수 있다.

```python title="hull_is_valid.py — 정의 기반 검증"
def hull_is_valid(hull, all_pts):
    n = len(hull)
    for i in range(n):
        a, b, c = hull[i], hull[(i + 1) % n], hull[(i + 2) % n]
        if ccw(a, b, c) < 0:          # 반시계로 도는 도중 오른쪽으로 꺾이면 안 된다
            return False
    for i in range(n):
        a, b = hull[i], hull[(i + 1) % n]
        for p in all_pts:
            if ccw(a, b, p) < 0:      # 어느 변의 바깥에 점이 있으면 껍질이 틀렸다
                return False
    return True
```

무작위 점(3 ~ 15개) 200세트에 대해 돌려 봤다.

```pyrepl
>>> ok_count = sum(hull_is_valid(convex_hull_monotone(pts), pts) for pts in random_pointsets)
>>> ok_count
200
```

**200세트 전부 통과했다.**

### 실측: 큰 입력에서 $O(n \log n)$이 유지되는가

```python title="bench_hull_scaling.py"
for n in [1000, 10000, 100000, 1000000]:
    pts = [(random.randint(-10**6, 10**6), random.randint(-10**6, 10**6)) for _ in range(n)]
    t0 = time.perf_counter()
    convex_hull_monotone(pts)
    t1 = time.perf_counter()
```

| n | 실행 시간 |
| --- | --- |
| 1,000 | 0.0008초 |
| 10,000 | 0.0083초 |
| 100,000 | 0.0892초 |
| 1,000,000 | 1.5662초 |

(Python 3.14.5 / Windows 기준 실측.)

$n$이 10배씩 늘 때 시간도 대략 10 ~ 17배 는다 — 로그 인자가 얹힌 **거의 선형**에 가까운 증가다. `sorted()`가 전체 비용을 지배하고(파이썬의 Timsort는 [7.4 정렬](#/sorting)에서 다뤘다), 나머지 껍질 구축 자체는 각 점을 스택에 넣고 빼는 것이 전부라 선형이다.

::: perf 정렬이 곧 병목이다
모노톤 체인의 시간 복잡도가 $O(n \log n)$인 이유는 알고리즘 자체가 아니라 **`sorted()` 한 줄** 때문이다. 아래쪽/위쪽 껍질을 만드는 `half()` 함수는 각 점이 스택에 최대 한 번 들어가고 한 번 빠지므로 $O(n)$이다. 이미 정렬된 입력이 주어지는 문제라면(드물지만) 정렬을 생략해 진짜 $O(n)$까지 내릴 수 있다.
:::

## 부동소수점 대신 정수 좌표를 써라

이 절 도입부에서 본 오차(`3.469446951953614e-18`)를 다시 떠올려 보자. 좌표가 실수라면 `ccw`의 반환값이 이론상 0이어야 할 때조차 미세한 양수나 음수로 나올 수 있다. 볼록 껍질의 `while ccw(...) <= 0` 같은 판정문은 이 미세한 오차 하나로 **점을 넣거나 빼는 결정이 완전히 뒤집힌다.**

```pyrepl
>>> a, b = (1, 1), (2, 2)
>>> c_int = (3, 3)
>>> (b[0]-a[0])*(c_int[1]-a[1]) - (b[1]-a[1])*(c_int[0]-a[0])
0
>>> c_float = (0.3, 0.30000000000000004)
>>> a_f, b_f = (0.1, 0.1), (0.2, 0.2)
>>> (b_f[0]-a_f[0])*(c_float[1]-a_f[1]) - (b_f[1]-a_f[1])*(c_float[0]-a_f[0])
3.469446951953614e-18
```

같은 기하학적 관계(세 점이 일직선)인데 정수 좌표로 계산하면 정확히 `0`이 나오고, 부동소수점으로 계산하면 `0`이 아닌 아주 작은 수가 나온다. **파이썬 `int`는 임의 정밀도라 오버플로도 없다** ([1.2 숫자와 수치 연산](#/numbers)). 좌표를 정수로 유지할 수 있다면 오차 걱정 자체가 사라진다.

::: danger 좌표가 실수로 주어진다고 그대로 float으로 계산하지 마라
문제가 좌표를 소수점 이하 몇 자리까지의 실수로 준다면, **그 자릿수만큼 10의 거듭제곱을 곱해 정수로 바꿔라.** 예를 들어 소수점 둘째 자리까지라면 100을 곱한다.

```python
# ❌ 실수 좌표를 그대로 쓰면 ccw의 부호가 오차로 흔들릴 수 있다
p = (0.12, 3.45)

# ✅ 스케일링해서 정수로 바꾼다 (문제의 좌표 정밀도를 미리 확인해야 한다)
p_scaled = (12, 345)   # 100을 곱하고 int로
```

**정말로 정수화가 불가능한 경우**(무리수가 섞이는 기하 변환 등)에만 부동소수점을 쓰고, 그때도 `==` 대신 `abs(a - b) < eps` 형태의 허용 오차 비교를 쓴다. 하지만 이건 최후의 수단이다. 코딩테스트에 나오는 좌표 문제의 대다수는 애초에 정수로 설계되어 있다.
:::

::: tip 표준 라이브러리에는 기하 함수가 없다 — 템플릿을 외워라
[7.12 유니온 파인드](#/union-find)의 `UnionFind`처럼, `ccw`와 `segments_intersect`, `convex_hull_monotone`도 파이썬 표준 라이브러리에 없다. 이 절의 세 함수를 그대로 외워서 들고 들어갈 만한 템플릿으로 삼아라. [8.10 실전 템플릿 모음](#/templates)에 정리해 둔다.
:::

## 요약

- **CCW**(외적의 부호)로 세 점의 회전 방향을 판정한다. 정수 좌표라면 나눗셈·삼각함수 없이 곱셈과 뺄셈만으로 정확하다.
- **선분 교차**는 CCW 네 번으로 일반적인 경우를 잡고, 부호가 0인 경우(끝점 접촉, collinear 겹침)는 `on_segment`로 따로 확인해야 한다.
- **볼록 껍질**을 모든 점 쌍으로 브루트포스하면 $O(n^3)$이라 $n$이 몇백만 되어도 느려진다(실측: $n=800$에서 브루트포스 0.59초 vs 모노톤 체인 0.00065초, 약 900배 차이).
- **앤드류 모노톤 체인**은 정렬 후 아래/위쪽 껍질을 각각 스캔해 $O(n \log n)$에 끝낸다. 각도 계산이 없어 그라함 스캔보다 오차에 강하다.
- 정답 검증은 다른 알고리즘(Fraction 기반 교점 계산)이나 **정의 자체**(볼록성 + 전체 점 포함)로 대조하면 된다. 실측으로 전부 일치를 확인했다.
- **좌표는 가능하면 정수로 유지하라.** 실수 좌표는 방향 판정의 부호를 미세하게 흔들어 알고리즘 전체를 틀리게 만들 수 있다.
