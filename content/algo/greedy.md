# 7.19 그리디

::: lead
[7.18 재귀와 백트래킹](#/backtracking)에서는 답이 될 수 있는 모든 후보를 가지치기하며 뒤져 봤다. 그리디는 정반대다. **매 순간 가장 좋아 보이는 선택 하나만 하고, 절대 되돌아보지 않는다.** 후보를 남겨 두지 않으므로 빠르다. 문제는 "지금 당장 좋아 보이는 것"이 "전체로 봤을 때도 좋은 것"이라는 보장이 전혀 없다는 점이다. 이 절의 목표는 그 보장이 **언제 성립하고 언제 깨지는지**를 증명 수준으로 이해하는 것이다. 코딩테스트에서 그리디는 맞으면 가장 짧고 빠른 풀이지만, 틀리면 부분점수조차 안 나오는 함정이기도 하다.
:::

## 왜 그리디인가 — 브루트포스로 먼저 당해 보기

회의실 하나를 여러 팀이 쓰겠다고 신청했다. 시작·종료 시각이 정해진 신청서가 여러 장 있고, **겹치지 않게 최대한 많은 신청을 수락**하고 싶다. 이것이 활동 선택 문제(activity selection)다.

가장 순진한 접근은 가능한 모든 부분집합을 검사해 그중 겹치지 않는 것들 중 가장 큰 집합을 찾는 것이다.

```python title="brute_force_activity.py"
from itertools import combinations

def brute_force(intervals):
    n = len(intervals)
    for r in range(n, 0, -1):          # 큰 크기부터 내려가며 검사
        for combo in combinations(range(n), r):
            chosen = sorted(combo, key=lambda i: intervals[i][0])
            if all(intervals[a][1] <= intervals[b][0]
                   for a, b in zip(chosen, chosen[1:])):
                return r               # 겹치지 않는 조합을 찾으면 그게 최댓값
    return 0
```

크기 $r$짜리 부분집합의 개수는 $\binom{n}{r}$이고, 전체를 다 보면 $2^n$에 가깝다. 실제로 $n$을 늘려 가며 재 보자.

```python title="bench_bf.py — 실행 결과는 아래 표"
import random, time
random.seed(1)
for n in (10, 15, 18, 20):
    intervals = [(s := random.randint(0, n), s + random.randint(1, 5)) for _ in range(n)]
    t0 = time.perf_counter(); brute_force(intervals); t1 = time.perf_counter()
```

| n | 브루트포스 시간 |
| --- | --- |
| 10 | 0.0004초 |
| 15 | 0.0191초 |
| 18 | 0.1870초 |
| 20 | 0.7793초 |

(Python 3.14.5 / Windows 기준 실측.) $n$이 2씩만 늘어도 시간이 4배 가까이 뛴다 — $2^n$의 전형적인 곡선이다. $n$이 30만 돼도 이 방식은 몇 분을 넘긴다. 백준·프로그래머스의 활동 선택류 문제는 보통 $n$이 수만~수십만이다. 지수 시간으로는 애초에 승부가 안 된다.

이 문제에는 놀랍도록 단순한 해법이 있다. **종료 시각이 가장 빠른 것부터 순서대로 보면서, 지금까지 고른 것과 안 겹치면 무조건 고른다.** 그게 전부다.

```python title="greedy_activity.py"
def greedy(intervals):
    order = sorted(intervals, key=lambda x: x[1])   # 종료 시각 오름차순
    count = 0
    last_end = float("-inf")
    for s, e in order:
        if s >= last_end:        # 직전에 고른 것과 안 겹친다
            count += 1
            last_end = e
    return count
```

정렬이 지배 비용이므로 $O(n \log n)$이다. 실제로 $n$을 100만까지 키워도 문제없다. 재현 가능하도록 시드를 고정하고 직접 재 봤다.

```python title="bench_greedy.py — 실행 결과는 아래 표"
import random, time
random.seed(1)
for n in (1000, 10000, 100000, 1_000_000):
    intervals = [(s := random.randint(0, n), s + random.randint(1, 5)) for _ in range(n)]
    t0 = time.perf_counter(); count = greedy(intervals); t1 = time.perf_counter()
    print(n, t1 - t0, count)
```

| n | 그리디 실행 시간 | 선택된 개수 |
| --- | --- | --- |
| 1,000 | 0.000148초 | 352 |
| 10,000 | 0.001345초 | 3,433 |
| 100,000 | 0.018855초 | 34,806 |
| 1,000,000 | 0.371860초 | 347,217 |

(Python 3.14.5 / Windows 기준, `random.seed(1)` 고정 실측.) 선택된 개수는 인스턴스 생성 방식(구간 길이를 1~5 사이 무작위로 준 것)에 좌우되므로, 시드나 생성 로직이 다르면 이 숫자 자체는 달라진다 — **자릿수와 비율**(대략 전체의 35% 안팎)이 재현되는 게 핵심이지 정확한 값이 고정된 상수는 아니다. 시간 쪽은 $n$이 10배가 될 때마다 약 9~20배로 늘어난다 — $\log n$이 조금씩 커지는 것까지 반영된 $O(n \log n)$ 곡선이다. $n=20$짜리를 브루트포스로 0.78초 걸려 풀던 문제를, 그리디는 $n=1{,}000{,}000$에서 0.37초에 끝낸다.

**두 함수가 실제로 같은 답을 내는지도 확인해야 한다.** 무작위 소규모 인스턴스 2,000개에서 두 결과를 대조했다.

```python title="검증"
def make_random():
    n = random.randint(1, 8)
    return [(s := random.randint(0, 10), s + random.randint(1, 5)) for _ in range(n)]

random.seed(42)
mismatches = sum(
    1 for _ in range(2000)
    if brute_force(inst := make_random()) != greedy(inst)   # 같은 인스턴스를 두 함수에 넣는다
)
print(mismatches)   # 0
```

```text nolines
trials done, mismatches = 0
```

2,000번 모두 일치했다.

::: warn `make_random()`을 두 번 부르면 다른 문제를 비교하게 된다
`brute_force(make_random()) != greedy(make_random())`처럼 쓰면 `make_random()`이 비교 연산자 양쪽에서 **각각 한 번씩, 총 두 번** 호출된다. 그러면 `brute_force`와 `greedy`가 서로 다른 무작위 인스턴스를 받아 계산하게 되고, 둘의 답이 달라도 그건 알고리즘이 틀려서가 아니라 애초에 다른 문제를 풀어서다. 실제로 이 버그가 있는 코드를 그대로 돌리면(`random.seed(42)` 고정) 2,000번 중 1,403번이나 "불일치"로 나온다 — 그리디가 1,403번 틀린 게 아니라 비교 자체가 무의미했던 것이다. 위 코드처럼 `inst := make_random()`으로 **인스턴스를 변수에 먼저 고정**하고 그 하나를 양쪽에 넣어야 진짜 비교가 된다. 이 실수는 겉보기에 멀쩡해서 코드 리뷰로도 잘 안 걸린다 — 실행 결과 숫자가 이상하면 항상 "같은 입력을 넣고 있는가"부터 의심하라.
:::

**그런데 "우연히 맞았다"와 "항상 맞는다"는 다르다.** 반례 하나가 남아 있으면 시험장에서 그 반례가 걸린다. 이제 왜 이 그리디가 **항상** 옳은지 증명할 차례다.

## 그리디가 통하는 조건

모든 문제에 그리디가 통하지 않는다. 통하려면 두 성질이 **동시에** 성립해야 한다.

**탐욕적 선택 속성**(greedy choice property): 지금 이 순간 가장 좋아 보이는 선택을 해도, 그 선택을 포함하는 전역 최적해가 항상 존재한다. 다시 말해 "당장의 최선"이 "전역 최적으로 가는 길 위"에 있다는 뜻이다.

**최적 부분 구조**(optimal substructure): 첫 선택을 하고 난 뒤 남은 부분 문제의 최적해가, 원래 문제 전체의 최적해 안에 그대로 들어 있다. 이 부분은 [7.20 동적 계획법 기초](#/dp-basics)에서 다루는 DP의 전제 조건과 똑같다.

::: note DP와 그리디는 형제다
둘 다 최적 부분 구조를 요구한다. 차이는 **탐욕적 선택 속성**의 유무다. DP는 "지금 선택이 나중에 어떤 영향을 줄지 몰라서" 모든 선택지를 남겨 두고 부분 문제의 답을 전부 계산한다. 그리디는 "지금 선택이 항상 최선으로 이어짐이 증명됐으므로" 하나만 남기고 나머지는 버린다. **그리디가 통한다는 것은 DP를 풀 필요가 없다는 강한 주장이다.** 그 주장은 반드시 증명이 있어야 믿을 수 있다.
:::

::: cote 그리디처럼 보이는데 DP인 문제
"정렬해서 순서대로 처리하면 되지 않을까"라는 직감이 드는 순간이 가장 위험하다. **직감이 증명을 대신할 수 없다.** 그리디로 몇 개 예제를 통과했다고 안심하지 마라. 예제는 보통 반례를 피해서 만들어지지 않는다 — 그냥 우연히 없을 뿐이다. 이 절 뒷부분에서 다룰 배낭 문제, 코인 교환이 바로 "그럴듯한 그리디가 조용히 틀리는" 대표 사례다. [8.4 문제 유형 분류](#/problem-signals)에서 이 신호를 구분하는 법을 더 다룬다.
:::

## 교환 논증으로 활동 선택 문제 증명하기

**교환 논증**(exchange argument)은 그리디의 정당성을 증명하는 표준 기법이다. 아이디어는 하나다.

> 그리디가 아닌 어떤 최적해가 있다고 가정하자. 그 최적해를 그리디의 선택과 **바꿔치기해도 나빠지지 않는다**는 것을 보이면, 그리디도 최적이라는 결론이 나온다.

활동 선택 문제에 적용해 보자. 증명할 명제는 이렇다.

> **종료 시각이 가장 빠른 활동을 첫 선택에 포함하는 최적해가 항상 존재한다.**

**증명.** $A$를 임의의 최적해라 하자. $A$ 안에서 종료 시각이 가장 빠른 활동을 $a^*$, 전체 활동 중 종료 시각이 가장 빠른 것을 $g$라 하자 ($g$가 그리디의 첫 선택이다).

- $a^* = g$면 끝이다. 이미 그리디의 선택이 최적해에 들어 있다.
- $a^* \ne g$라면, $g$의 종료 시각이 $a^*$보다 빠르거나 같다는 정의에 의해 $g$는 $A$에 있는 $a^*$를 포함한 다른 모든 활동과 **절대 겹치지 않는다.** $A$에서 $a^*$를 빼고 그 자리에 $g$를 넣은 새 집합 $A' = (A \setminus \{a^*\}) \cup \{g\}$를 만들자.
  - $A'$의 크기는 $A$와 같다 (하나 빼고 하나 넣었으니).
  - $A'$의 원소들은 서로 겹치지 않는다 — $g$는 $a^*$ 대신 들어왔을 뿐, $A$의 나머지와의 관계는 그대로고, $g$ 자체가 어느 것과도 안 겹친다는 것도 방금 보였다.
  - 따라서 $A'$도 $A$와 크기가 같은 유효한 해다. **$A$가 최적이었으므로 $A'$도 최적이다.**

$A'$는 $g$를 포함하는 최적해다. $\blacksquare$

이 증명이 하는 일은 딱 하나다. **"그리디의 선택을 포함하지 않는 최적해가 있다면, 그걸 그리디의 선택을 포함하는 다른 최적해로 바꿀 수 있다"**는 것을 보이는 것. 한 번 이게 성립하면, 나머지는 귀납이다 — 첫 선택 이후 남은 문제(고른 활동과 안 겹치는 나머지 활동들 중 최대 선택)는 **크기만 작아진 똑같은 문제**이므로 같은 논증을 반복할 수 있다. 이게 최적 부분 구조다.

::: deep 왜 "종료 시각"이고 "시작 시각"이나 "길이"가 아닌가
직관적으로 다른 정렬 기준도 그럴듯해 보인다. 각 기준이 실제로 어디서 깨지는지 반례로 걸러 보자. **"시작이 빠른 것부터"와 "길이가 짧은 것부터"는 실패하는 지점이 서로 다르므로, 각각에 맞는 반례가 따로 필요하다.**

```python title="기준 2: 시작이 빠른 것부터 — 반례"
intervals_start = [(1, 2), (0, 10), (3, 4), (5, 6)]
# 시작 순 정렬: (0,10), (1,2), (3,4), (5,6)
# (0,10)을 먼저 고르면 나머지 세 개와 모두 겹쳐 총 1개만 선택된다.
# 종료 기준이었다면 (1,2),(3,4),(5,6) 세 개를 골랐을 최적해다.
```

`(0, 10)`은 시작이 가장 빠르지만 고르는 순간 나머지 전부와 겹친다. **"먼저 시작한다"는 사실은 "나중에 다른 것들을 방해하지 않는다"를 보장하지 않는다.**

```python title="기준 1: 길이가 짧은 것부터 — 반례"
intervals_length = [(0, 4), (4, 8), (3, 5)]
# 길이 순 정렬: (3,5)[길이2] -> (0,4)[길이4] -> (4,8)[길이4]
# (3,5)를 먼저 고르면 last_end=5가 되어 (0,4)도 (4,8)도 다 걸려 넘어간다.
# 실제 최적은 (0,4),(4,8) 두 개인데, 길이 기준 그리디는 (3,5) 하나만 고른다.
```

짧은 구간 `(3, 5)`가 가운데 끼어들어, 서로 겹치지 않는 두 구간 `(0, 4)`와 `(4, 8)`을 동시에 막아 버린다. 길이가 짧다는 것도 "다른 선택의 여지를 남긴다"는 보장이 되지 않는다. 앞서 본 `intervals = [(1, 2), (0, 10), (3, 4), (5, 6)]`에 길이 기준을 적용하면 마침 `(0, 10)`이 네 구간 중 가장 길어서 정렬 순서상 맨 뒤로 밀리고, 그 결과 `(1,2),(3,4),(5,6)` 세 개가 뽑혀 최적과 우연히 일치한다 — 이건 길이 기준이 옳다는 뜻이 아니라, **이 특정 데이터가 길이 기준의 반례가 되지 못할 뿐**이다. 반례는 기준마다 따로 골라야 한다는 걸 보여 주는 좋은 예다.

반면 "종료가 빠르다"는 것은 정확히 "이 뒤로 남는 시간이 가장 많다"는 뜻이고, 그게 바로 다음 선택의 여지를 최대로 남긴다. 이 차이가 왜 종료 시각 기준만 증명이 성립하는지의 핵심이다.
:::

## 활동 선택 문제 풀어 보기

작은 예제로 그리디가 실제로 어떻게 움직이는지 보자.

| 활동 | A | B | C | D | E | F | G |
| --- | --- | --- | --- | --- | --- | --- | --- |
| 시작 | 1 | 2 | 3 | 0 | 5 | 8 | 6 |
| 종료 | 3 | 4 | 5 | 7 | 9 | 10 | 8 |

종료 시각 순으로 정렬하면 `A(1,3) → B(2,4) → C(3,5) → G(6,8) → E(5,9) → F(8,10) → D(0,7)`이다. 순서대로 훑는다.

```text nolines
A(1,3)  시작 1 >= last_end(-inf)   선택.  last_end = 3
B(2,4)  시작 2 <  last_end(3)      건너뜀 (A와 겹침)
C(3,5)  시작 3 >= last_end(3)      선택.  last_end = 5
D(0,7)  시작 0 <  last_end(5)      건너뜀
G(6,8)  시작 6 >= last_end(5)      선택.  last_end = 8
E(5,9)  시작 5 <  last_end(8)      건너뜀
F(8,10) 시작 8 >= last_end(8)      선택.  last_end = 10
```

실제로 실행한 결과도 이와 같다.

```pyrepl
>>> intervals = [("A",1,3),("B",2,4),("C",3,5),("D",0,7),("E",5,9),("F",8,10),("G",6,8)]
>>> order = sorted(intervals, key=lambda x: x[2])
>>> [name for name, s, e in order]
['A', 'B', 'C', 'D', 'G', 'E', 'F']
>>> last_end = float("-inf"); chosen = []
>>> for name, s, e in order:
...     if s >= last_end:
...         chosen.append(name); last_end = e
...
>>> chosen
['A', 'C', 'G', 'F']
```

`A, C, G, F` 네 개를 골랐다. 7개 중 겹치지 않는 조합으로 4개보다 많이 고를 수 없다는 것은 앞서의 증명이 보장한다 — 브루트포스로 확인해도 같은 값이 나온다.

## 그리디가 실패하는 경우 — 배낭 문제

교환 논증이 통하지 않는 문제에 그리디를 억지로 적용하면 **조용히 틀린 답**이 나온다. 가장 유명한 사례가 배낭 문제(knapsack)다.

무게 용량 50인 배낭과 세 물건이 있다.

| 물건 | 무게 | 가치 | 가치/무게 |
| --- | --- | --- | --- |
| 노트북 | 10 | 60 | 6.0 |
| 카메라 | 20 | 100 | 5.0 |
| 그림 | 30 | 120 | 4.0 |

**분할 가능 배낭**(fractional knapsack) — 물건을 쪼개서 일부만 담아도 되는 버전 — 에서는 "가치/무게 비율이 높은 것부터 담을 수 있는 만큼 담는다"가 정확히 최적이다. 교환 논증이 그대로 통한다: 비율이 더 높은 물건의 아주 작은 조각과 더 낮은 물건의 같은 무게만큼을 바꿔치기하면 가치가 늘거나 같으므로, 최적해는 항상 비율 순서대로 채운 형태다.

```python title="fractional_knapsack.py"
def fractional_knapsack(items, capacity):
    order = sorted(items, key=lambda x: x[1] / x[0], reverse=True)
    remaining = capacity
    value = 0.0
    for weight, worth in order:
        if weight <= remaining:
            remaining -= weight
            value += worth
        else:
            value += worth * (remaining / weight)   # 남은 만큼만 쪼개 담는다
            break
    return value
```

같은 순서(비율 내림차순: 노트북 → 카메라 → 그림)로 **0/1 배낭**(쪼갤 수 없는 버전)에 그대로 적용하면 어떻게 될까.

```python title="01_knapsack_greedy_wrong.py"
def knapsack_greedy(items, capacity):
    order = sorted(items, key=lambda x: x[1] / x[0], reverse=True)
    remaining = capacity
    value = 0
    taken = []
    for weight, worth in order:
        if weight <= remaining:      # 통째로 담거나 아예 포기
            remaining -= weight
            value += worth
            taken.append(worth)
    return value, taken
```

세 가지 방법을 직접 실행해서 대조했다.

```python title="검증"
from itertools import combinations

items = [(10, 60), (20, 100), (30, 120)]
capacity = 50

# 0/1 배낭 브루트포스(정답)
best_val, best_combo = 0, None
for r in range(len(items) + 1):
    for combo in combinations(range(len(items)), r):
        w = sum(items[i][0] for i in combo)
        v = sum(items[i][1] for i in combo)
        if w <= capacity and v > best_val:
            best_val, best_combo = v, combo
```

```text nolines
0/1 배낭 브루트포스 최적해: (카메라, 그림)  가치: 220
0/1 배낭 그리디(위 코드) 결과: (노트북, 카메라)  가치: 160
분할 가능 배낭 그리디 결과 가치: 240.0
```

**그리디는 160을 답으로 내지만 실제 최적은 220이다.** 27% 낮은, 틀린 답을 아무 에러도 없이 "정답인 척" 내놓는다. 이게 그리디의 가장 위험한 실패 방식이다 — 크래시하지 않고 그럴듯한 값을 준다.

::: danger 왜 여기서 교환 논증이 깨지는가
분할 가능 배낭에서는 "비율 낮은 물건의 아주 작은 조각"과 "비율 높은 물건의 같은 무게만큼"을 자유롭게 바꿔치기할 수 있다. 이 미세 조정이 증명의 핵심이었다. **0/1 배낭에서는 물건을 통째로만 넣고 뺄 수 있어서 이 미세 조정 자체가 불가능하다.** 노트북(10)+카메라(20)=30을 담고 남은 용량 20에는, 남은 유일한 후보인 그림(30)이 통째로 안 들어간다 — 쪼갤 수 없으니 포기해야 한다. 반면 카메라+그림(50)을 담으면 용량을 정확히 다 쓰면서 더 큰 가치를 얻는다. **탐욕적 선택 속성이 깨지는 순간, 국소 최선이 전역 최선을 배반한다.**

0/1 배낭의 올바른 풀이는 그리디가 아니라 DP다. [7.21 동적 계획법 심화](#/dp-advanced)에서 다룬다.
:::

::: cote 배낭 문제 신호 구분하기
- "물건을 쪼갤 수 있다", "비율만큼 부분적으로 담아도 된다" → 그리디(비율 정렬)로 충분하다.
- "각 물건은 담거나 안 담거나 둘 중 하나다(0/1)" → 그리디로 되는 것처럼 보여도 반례가 있다. DP로 가라.
- 문제 지문에 "부분적으로", "비율대로" 같은 말이 없고 물건이 개별 단위(사람 수, 상자 개수)라면 십중팔구 0/1이다.
:::

## 그리디가 실패하는 경우 — 코인 교환

거스름돈 문제도 교환 논증이 화폐 단위에 따라 성립하기도, 깨지기도 하는 대표 사례다. "가장 큰 단위 동전부터 쓸 수 있는 만큼 쓴다"는 그리디를 두 화폐 체계에 적용해 봤다.

```python title="coin_greedy.py"
def greedy_coin(coins, target):
    coins = sorted(coins, reverse=True)
    count, used, remaining = 0, [], target
    for c in coins:
        while remaining >= c:
            remaining -= c
            count += 1
            used.append(c)
    return count, used

def dp_min_coins(coins, target):
    dp = [0] + [float("inf")] * target
    for i in range(1, target + 1):
        for c in coins:
            if c <= i:
                dp[i] = min(dp[i], dp[i - c] + 1)
    return dp[target]
```

```pyrepl
>>> greedy_coin([1, 3, 4], 6)
(3, [4, 1, 1])
>>> dp_min_coins([1, 3, 4], 6)
2
```

동전 `{1, 3, 4}`로 6원을 만들 때, 그리디는 `4+1+1`로 3개를 쓰지만 **`3+3`이면 2개로 끝난다.** 그리디가 틀렸다.

반면 한국 동전 체계(`1, 5, 10, 50, 100, 500`)에서는 무작위 500개 금액으로 시험했을 때 그리디와 DP가 **단 한 건도 다르지 않았다.**

```pyrepl
>>> import random
>>> random.seed(0)
>>> krw = [1, 5, 10, 50, 100, 500]
>>> mismatch = sum(
...     1 for _ in range(500)
...     if greedy_coin(krw, (t := random.randint(1, 2000)))[0] != dp_min_coins(krw, t)
... )
>>> mismatch
0
```

::: deep 정준 화폐 체계(canonical coin system)
그리디가 항상 옳은 화폐 체계를 **정준(canonical)** 이라 부른다. 한국·미국 동전처럼 **각 단위가 그 아래 단위들의 조합보다 "효율적으로" 큰 체계**는 대부분 정준이다 (엄밀한 판정에는 별도의 조건 검사가 필요하지만, 코딩테스트에 나오는 대부분의 "일상적인" 화폐 단위는 정준이라고 봐도 된다). `{1, 3, 4}`처럼 인위적으로 만든 단위 집합이 문제에 등장한다면 **그리디를 의심하고 DP로 가라.** 문제 지문이 "화폐 단위가 특이하다" 또는 "임의의 단위가 주어진다"고 하면 이건 그리디가 아니라 DP 신호다.
:::

::: cote 백준/프로그래머스에서 코인 그리디를 믿어도 되는 경우
- 화폐 단위가 실제 지폐·동전(1, 5, 10, 50, 100, 500, 1000, 5000, ...)이면 그리디로 충분한 경우가 대부분이다.
- 문제가 "임의의 동전 단위 배열이 주어진다"고 명시하면 **그리디를 쓰지 말고 DP로 풀어라.** 이건 함정 문제의 전형이다.
- 확신이 안 서면 작은 반례를 손으로 만들어 그리디와 DP를 비교해 보고 판단하라. 실전에서 몇 초 걸리지 않는다.
:::

## 회의실 배정 — 최소 회의실 개수

활동 선택 문제의 변형으로, "최대한 많이 수락하기"가 아니라 **"모든 신청을 다 받아주려면 회의실이 몇 개 필요한가"**를 묻는 버전도 자주 나온다. 여기서는 힙이 등장한다.

```python title="min_meeting_rooms.py"
import heapq

def min_rooms(meetings):
    if not meetings:
        return 0
    meetings = sorted(meetings, key=lambda x: x[0])   # 시작 시각 순
    heap = []   # 현재 쓰이는 회의실들의 "종료 예정 시각"
    for s, e in meetings:
        if heap and heap[0] <= s:
            heapq.heapreplace(heap, e)   # 가장 먼저 끝나는 방을 재사용
        else:
            heapq.heappush(heap, e)      # 새 방 하나 더 필요
    return len(heap)
```

여기서 그리디의 선택은 "시작 시각 순으로 훑으면서, 지금 가장 먼저 비는 회의실이 있으면 그걸 재사용하고, 없으면 새로 연다"이다. 힙의 루트(`heap[0]`)가 항상 "가장 먼저 끝날 회의실"이라는 사실을 [7.8 힙과 우선순위 큐](#/heap)에서 이미 다뤘다.

```pyrepl
>>> min_rooms([(0, 30), (5, 10), (15, 20)])
2
>>> min_rooms([(1, 4), (2, 5), (7, 9), (3, 6)])
3
```

두 결과 모두 "특정 시각에 동시에 진행 중인 회의 수의 최댓값"을 전수 조사해 얻은 값과 정확히 일치한다 (무작위 3,000개 인스턴스로 검증. 불일치 0건). 이 문제의 그리디가 정당한 이유도 같은 종류의 교환 논증으로 증명된다 — "임의의 시각에 동시에 열려야 하는 회의 수"는 절대 줄일 수 없는 하한이고, 그리디는 정확히 그 하한만큼만 회의실을 쓴다는 것을 보이면 된다.

## 요약

- 그리디는 **탐욕적 선택 속성**(당장의 최선이 전역 최적으로 이어짐)과 **최적 부분 구조**가 둘 다 성립할 때만 옳다. 하나라도 깨지면 반례가 존재한다.
- 정당성은 **교환 논증**으로 증명한다 — 그리디가 아닌 최적해가 있다면, 그것을 그리디의 선택으로 바꿔도 나빠지지 않음을 보인다.
- 활동 선택 문제는 **종료 시각이 빠른 순**으로 고르는 것이 정답이다. 시작 시각이나 길이 기준은 반례가 있다.
- 분할 가능 배낭은 비율 그리디가 정확하지만, **0/1 배낭은 같은 그리디가 조용히 틀린 답을 낸다** — 크래시 없이 최적보다 낮은 값을 준다. 이럴 땐 DP로 가야 한다.
- 코인 교환도 화폐 단위가 정준(canonical)이 아니면 그리디가 실패한다. 문제에 "임의의 동전 단위"가 나오면 그리디를 의심하라.
- 회의실 최소 개수 문제는 힙으로 "가장 먼저 끝나는 자원을 재사용"하는 그리디로 풀린다.
- 브루트포스는 지수 시간, 그리디는 정렬 비용인 $O(n \log n)$이다 — 실측으로도 $n=20$에서 0.78초 걸리던 것이 $n=1{,}000{,}000$에서 0.3초로 끝난다.
