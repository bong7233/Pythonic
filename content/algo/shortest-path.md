# 7.15 최단 경로

::: lead
[7.14 BFS/DFS 응용](#/bfs-dfs)에서 본 최단 거리는 전부 **모든 간선의 비용이 1**이라는 전제 위에 있었다. 도로에 비용(거리, 시간, 톨게이트 요금)이 붙는 순간 BFS는 답을 낼 수 없다. 이 절은 가중치가 있는 그래프에서 최단 경로를 구하는 네 가지 도구 — 다익스트라, 벨만-포드, 플로이드-워셜, A* — 를 다룬다. 넷 다 "간선을 따라가며 거리를 갱신한다"는 같은 아이디어의 변주지만, 저마다 다른 제약과 다른 비용 위에서 동작한다. 어느 것도 만능이 아니고, 코딩테스트에서는 **문제의 제약 조건을 보고 넷 중 하나를 즉시 골라야 한다.**
:::

## 가중치가 있으면 BFS로 안 된다

BFS가 최단 거리를 보장하는 이유는 단순하다. **레벨 순서로 노드를 방문하기 때문에, 처음 도달한 순간이 곧 최소 간선 수 경로다.** 간선마다 비용이 다르면 이 전제가 무너진다.

```python title="weighted_bfs_fail.py"
from collections import deque


def bfs_hop_count(graph, start):
    dist = {start: 0}
    q = deque([start])
    while q:
        u = q.popleft()
        for v, _ in graph[u]:
            if v not in dist:
                dist[v] = dist[u] + 1     # 간선 수만 센다 — 가중치 무시
                q.append(v)
    return dist


graph = {
    "A": [("B", 1), ("C", 10)],
    "B": [("C", 1)],
    "C": [],
}
print(bfs_hop_count(graph, "A"))
```

```text
{'A': 0, 'B': 1, 'C': 1}
```

BFS는 `C`까지의 거리를 `1`이라 답한다. `A → C` 간선 하나를 탔으니 "홉 수"로는 맞다. 하지만 실제 비용은 `10`이다. 진짜 최단 비용 경로는 `A → B → C` (`1 + 1 = 2`)인데, BFS는 이 경로를 볼 수 있는 눈이 없다. **BFS는 간선 수를 세지, 비용을 더하지 않는다.** 이걸 억지로 고치려면 결국 다익스트라를 다시 발명하게 된다.

::: cote 문제 신호
"거리", "비용", "시간", "요금"처럼 간선마다 다른 값이 붙어 있고 최소 합을 구하라는 문제면 가중치 그래프다. 백준 "최단경로"(1753), "타임머신"(11657), "플로이드"(11404)가 이 절에서 다루는 세 알고리즘에 각각 대응한다. [8.4 문제 유형 분류](#/problem-signals)에서 제약 조건으로 역산하는 법을 더 다룬다.
:::

## 다익스트라 — 확정된 것부터 넓혀간다

[7.8 힙과 우선순위 큐](#/heap)에서 미리 본 지연 삭제(lazy deletion) 패턴을 그대로 쓴다. 아이디어는 하나다. **아직 확정 안 된 노드 중 지금까지 알려진 거리가 가장 짧은 것부터 확정한다.** 모든 가중치가 음이 아니라면, 힙에서 막 뽑힌 노드의 거리는 그 시점에 나올 수 있는 가장 작은 값이므로 더는 줄어들 수 없다 — 이게 다익스트라가 성립하는 유일한 근거다.

```python title="dijkstra.py"
import heapq


def dijkstra(graph, start):
    dist = {node: float("inf") for node in graph}
    dist[start] = 0
    pq = [(0, start)]
    while pq:
        d, u = heapq.heappop(pq)
        if d > dist[u]:
            continue          # 낡은 항목 — 이미 더 짧은 경로로 갱신됐다
        for v, w in graph[u]:
            nd = d + w
            if nd < dist[v]:
                dist[v] = nd
                heapq.heappush(pq, (nd, v))
    return dist
```

같은 노드가 힙에 여러 번 들어갈 수 있다는 게 핵심이다. 노드를 지우는 연산이 없는 `heapq`의 한계를, **낡은 항목은 팝될 때 걸러낸다**는 전략으로 우회한다.

### 브루트포스와 대조해서 정답부터 확인한다

무식하지만 확실한 오라클을 만든다. **모든 단순 경로를 다 열거해서 최솟값을 취한다.**

```python title="oracle_all_paths.py"
def brute_all_paths(graph, start):
    best = {n: float("inf") for n in graph}
    best[start] = 0

    def dfs(u, d, visited):
        if d < best[u]:
            best[u] = d
        for v, w in graph[u]:
            if v not in visited:
                visited.add(v)
                dfs(v, d + w, visited)
                visited.remove(v)

    dfs(start, 0, {start})
    return best
```

무작위로 만든 소규모 방향 그래프 30개에서 두 결과를 대조하면 전부 일치한다.

```text
검증 통과: 30개 무작위 그래프에서 dijkstra == brute_all_paths
```

## heapq 없이 짤 수도 있다 — 그런데 왜 안 쓰는가

다익스트라의 원래 정의는 "아직 확정 안 된 노드 중 거리가 최소인 것을 매번 선형 탐색으로 찾는다"이다. 힙이 없던 시절의 구현이자, 지금도 그래프가 조밀하면($E \approx V^2$) 여전히 유효한 선택이다.

```python title="dijkstra_array.py — 힙 없는 버전"
def dijkstra_array(n, adj, start):
    INF = float("inf")
    dist = [INF] * n
    visited = [False] * n
    dist[start] = 0
    for _ in range(n):
        u = -1
        best = INF
        for i in range(n):
            if not visited[i] and dist[i] < best:
                best = dist[i]
                u = i
        if u == -1:
            break
        visited[u] = True
        for v, w in adj[u]:
            if dist[u] + w < dist[v]:
                dist[v] = dist[u] + w
    return dist
```

이 버전은 매 확정마다 $O(V)$짜리 선형 탐색을 하므로 전체가 $O(V^2)$다. 힙 버전은 $O((V+E)\log V)$다. **그래프가 성겨서 $E \ll V^2$일 때** 이 차이가 극적으로 벌어진다.

::: perf 실측: 성긴 그래프에서 heap vs 배열
평균 차수 4인 성긴 무작위 그래프에서 두 구현의 결과가 같음을 확인한 뒤 시간을 쟀다.

| $n$ | heap 버전 | 배열 버전 $O(V^2)$ | 배율 |
| --- | --- | --- | --- |
| 1,000 | 0.0006초 | 0.0224초 | 38.1배 |
| 5,000 | 0.0032초 | 0.5710초 | 178.9배 |
| 10,000 | 0.0076초 | 2.3449초 | 308.2배 |
| 30,000 | 0.0305초 | 20.5398초 | 674.3배 |

(Python 3.14.5 / Windows 기준 실측. 두 구현의 결과가 완전히 같음을 `assert`로 확인.) $n$이 세 배 늘 때마다 배율이 계속 커진다 — $O(V^2)$과 $O(V\log V)$의 격차가 그래프가 커질수록 벌어지는 걸 그대로 보여준다. **백준·프로그래머스에서 정점 수가 몇천을 넘고 그래프가 성기면 배열 버전은 시간 초과다.**
:::

## 다익스트라가 음수 가중치에서 실패하는 이유

다익스트라의 유일한 전제는 "확정한 노드의 거리는 다시는 줄어들지 않는다"였다. 음수 간선이 있으면 이 전제가 깨진다. **나중에 확정될, 지금은 멀어 보이는 경로가 음수 간선을 만나 갑자기 더 짧아질 수 있기 때문이다.**

```python title="dijkstra_visited.py — 교과서적인 '확정하면 끝' 버전"
import heapq


def dijkstra_visited(graph, start):
    dist = {n: float("inf") for n in graph}
    dist[start] = 0
    visited = set()
    pq = [(0, start)]
    while pq:
        d, u = heapq.heappop(pq)
        if u in visited:
            continue
        visited.add(u)
        for v, w in graph[u]:
            if v in visited:
                continue          # 이미 확정된 노드는 다시 갱신하지 않는다
            nd = d + w
            if nd < dist[v]:
                dist[v] = nd
                heapq.heappush(pq, (nd, v))
    return dist
```

```python title="반례"
graph = {
    "A": [("B", 1), ("C", 2)],
    "B": [],
    "C": [("D", 1)],
    "D": [("B", -10)],
}
# 진짜 최단 경로: A -> C -> D -> B = 2 + 1 - 10 = -7
```

```text
실제 정답      : {'A': 0, 'B': -7, 'C': 2, 'D': 3}
visited 버전    : {'A': 0, 'B': 1, 'C': 2, 'D': 3}
```

`A`에서 출발하면 `B`가 거리 `1`로 가장 먼저 확정된다. `C → D → B` 라는, 나중에 발견되는 경로가 `-7`이라는 더 짧은 값을 만들어 내지만 **`B`는 이미 확정되어 다시는 갱신되지 않는다.** 결과는 조용히 틀린다 — 예외도, 경고도 없다.

::: danger 음수 가중치 문제에 다익스트라를 쓰면
컴파일 에러도, 런타임 에러도 나지 않는다. **그냥 답이 틀린다.** 채점 서버가 준 반례 하나에 걸리기 전까지는 로컬 테스트를 통과할 수도 있다. 문제에 "가중치는 음수일 수 있다"거나 "요금 할인/캐시백" 같은 표현이 있으면 다익스트라부터 의심해라.
:::

::: deep 그럼 이 절 맨 위의 `dijkstra` 코드는 왜 안전해 보이는가
이 절 맨 위 `dijkstra` 함수는 `visited` 집합이 없고 `if d > dist[u]: continue`로 낡은 항목만 거른다. 이 버전으로 방금 그 반례를 돌리면 우연히 정답(`-7`)이 나온다 — `B`가 확정 처리 자체를 하지 않고, 더 짧은 값이 나올 때마다 계속 다시 큐에 넣기 때문이다. 실제로 무작위로 만든 음수 간선 그래프(사이클 없음) 148개에서 이 "느슨한" 버전과 벨만-포드(뒤에서 다룬다)의 결과를 전부 대조했을 때 **불일치는 0건**이었다.

그렇다고 이 버전을 음수 가중치에 써도 된다는 뜻은 아니다. 이건 우연이지, 보장이 아니다.

- **시간 복잡도 보장이 사라진다.** 힙이 담을 수 있는 항목 수에 제한이 없어서, 최악의 경우 지수적으로 많은 재삽입이 일어날 수 있다. $O((V+E)\log V)$라고 주장할 근거가 없다.
- **음수 사이클이 있으면 무한 루프다.** 사이클을 돌 때마다 거리가 계속 줄어드니 큐가 절대 비지 않는다.

이 "느슨한" 다익스트라는 사실 **SPFA**(Shortest Path Faster Algorithm)라는, 벨만-포드를 큐 기반으로 최적화한 알고리즘과 같은 계열이다. 이론적 보장이 없다는 같은 이유로 실전에서는 잘 안 쓴다. 음수 가중치가 나오면 그냥 다음 절의 벨만-포드로 가라.
:::

## 벨만-포드 — 전부 $V-1$번 봐서 이긴다

벨만-포드는 다익스트라처럼 "다음에 볼 노드를 영리하게 고르지" 않는다. 대신 **모든 간선을 $V-1$번 반복해서 그냥 다 갱신한다.** 영리함을 포기하는 대신 음수 가중치에서도 정확하다는 보장을 얻는다.

```python title="bellman_ford.py"
def bellman_ford(n, edges, start):
    dist = [float("inf")] * n
    dist[start] = 0
    for _ in range(n - 1):
        updated = False
        for u, v, w in edges:
            if dist[u] != float("inf") and dist[u] + w < dist[v]:
                dist[v] = dist[u] + w
                updated = True
        if not updated:
            break                  # 더 갱신될 게 없으면 일찍 끝낸다
    has_negative_cycle = False
    for u, v, w in edges:
        if dist[u] != float("inf") and dist[u] + w < dist[v]:
            has_negative_cycle = True
    return dist, has_negative_cycle
```

::: deep 왜 $V-1$번이면 충분한가
최단 경로는 (음수 사이클이 없다는 전제 아래) **정점을 최대 $V-1$개 거친다.** 정점이 $V$개인 그래프에서 사이클 없이 만들 수 있는 가장 긴 경로가 $V-1$개의 간선이기 때문이다. 한 번의 전체 간선 순회는 "경로 길이를 최소 1칸 늘려서 확정하는" 효과가 있으므로, $V-1$번 반복하면 가능한 모든 길이의 경로가 다 반영된다. $V$번째 반복에서도 여전히 갱신이 일어난다면, 그건 $V-1$칸으로 표현 못 할 만큼 유리한 경로가 있다는 뜻이고 — 유한한 그래프에서 그런 일은 **음수 사이클**이 있을 때만 가능하다.
:::

`brute_all_paths` 오라클과 무작위로 생성한 음수 간선(사이클 없음) 그래프 34개를 대조하면 전부 일치한다.

```text
검증 통과: 음수 간선 포함 34개 그래프에서 bellman_ford == brute oracle
```

### 음수 사이클 검출

벨만-포드의 진짜 쓸모는 최단 거리 자체보다 **"음수 사이클이 있는가"를 판정하는 것**일 때가 많다. $V-1$번 반복 후에도 $V$번째에서 갱신이 일어나면 음수 사이클이 존재한다.

```python title="음수 사이클 검증"
# 1 -> 2 -> 3 -> 1 사이클의 합 = -1 -1 -1 = -3 < 0
neg_cycle_edges = [(0, 1, 1), (1, 2, -1), (2, 3, -1), (3, 1, -1)]
_, has_cycle = bellman_ford(4, neg_cycle_edges, 0)
assert has_cycle is True

no_cycle_edges = [(0, 1, 1), (1, 2, -1), (2, 3, 5)]
_, has_cycle = bellman_ford(4, no_cycle_edges, 0)
assert has_cycle is False
```

```text
음수 사이클 검출 성공: True
음수 사이클 없음 판정 성공: [0, 1, 0, 5]
```

::: cote 백준 "타임머신"(11657) 유형
"시간을 되돌릴 수 있다"거나 "무한히 이득을 볼 수 있으면 -1을 출력하라"는 문제는 거의 항상 음수 사이클 검출이다. $V-1$번째 반복까지는 정상적으로 최단 거리를 구하고, $V$번째 반복에서 갱신이 발생하는 정점이 **사이클에 물려 있는지, 아니면 사이클에서 도달 가능한 정점인지**까지 구분해야 하는 문제도 있다 — 단순히 "사이클 있음/없음"만 볼지, "그 사이클로부터 도달 가능한 모든 정점의 거리를 -∞로 만들지"까지 볼지는 문제 조건을 반드시 확인해라.
:::

::: perf 벨만-포드는 다익스트라보다 느리다
벨만-포드는 매 반복마다 **모든 간선**을 보므로 전체 $O(VE)$다. 다익스트라의 $O((V+E)\log V)$보다 그래프가 조밀할수록 훨씬 느려진다. **음수 가중치가 없다면 벨만-포드를 쓸 이유가 없다.** 이건 "일반적으로 더 안전한 알고리즘"이 아니라 "음수 가중치라는 특정 상황에서만 필요한 알고리즘"이다.
:::

## 플로이드-워셜 — 모든 쌍의 거리를 한 번에

지금까지는 "출발점 하나에서 모든 정점까지"였다. **모든 정점 쌍의 최단 거리**가 다 필요하면 매번 다익스트라를 $V$번 돌리는 것도 방법이지만($O(V(V+E)\log V)$), 정점 수가 작을 때는 더 간단한 방법이 있다.

```python title="floyd_warshall.py"
def floyd_warshall(n, edges):
    INF = float("inf")
    dist = [[INF] * n for _ in range(n)]
    for i in range(n):
        dist[i][i] = 0
    for u, v, w in edges:
        if w < dist[u][v]:
            dist[u][v] = w
    for k in range(n):              # 경유지 k
        for i in range(n):
            if dist[i][k] == INF:
                continue
            for j in range(n):
                if dist[i][k] + dist[k][j] < dist[i][j]:
                    dist[i][j] = dist[i][k] + dist[k][j]
    return dist
```

핵심은 **점화식 하나**다. `k`를 0부터 하나씩 늘려가며 "지금까지 고려한 경유지($0 \dots k$)만 거쳐서 갈 수 있는 `i`-`j` 최단 거리"를 갱신한다.

$$dist[i][j] = \min(dist[i][j],\ dist[i][k] + dist[k][j])$$

바깥 루프가 `k`라는 게 중요하다. `k`를 하나 확정할 때마다 "경유지 후보에 `k`를 추가해도 되는 모든 쌍"이 한 번에 갱신된다. 삼중 루프이므로 $O(n^3)$이다.

음이 아닌 가중치의 무작위 그래프 20개를 만들어, 모든 `(src, dst)` 쌍에 대해 `floyd_warshall`의 결과와 각 `src`에서 돌린 `dijkstra`의 결과를 대조했다.

```text
검증 완료: 20개 그래프 x 모든 (src,dst) 쌍에서 불일치 0건
```

::: warn `INF + INF` 함정
파이썬의 `float("inf")`는 `inf + inf == inf`라 안전하지만, **정수 오버플로가 있는 언어(C++, Java)에서 `INF`를 너무 크게 잡으면 `INF + INF`가 오버플로로 음수가 되어 조용히 틀린 최단 거리를 만든다.** 파이썬은 정수 크기 제한이 없어([1.2 숫자와 수치 연산](#/numbers)) 이 문제가 없지만, `dist[i][k] == INF`처럼 도달 불가능 여부를 먼저 걸러야 한다는 습관은 파이썬에서도 유효하다 — 안 걸러도 결과는 맞지만 쓸모없는 덧셈을 $n^3$번 반복하는 건 낭비다.
:::

::: perf 실측: $O(n^3)$ 곡선
평균 차수 4인 무작위 그래프에서 정점 수를 두 배씩 늘리며 측정했다.

| $n$ | 걸린 시간 | 직전 대비 배율 |
| --- | --- | --- |
| 50 | 0.0021초 | — |
| 100 | 0.0176초 | 8.18배 |
| 200 | 0.1221초 | 6.95배 |
| 400 | 0.9230초 | 7.56배 |

(Python 3.14.5 / Windows 기준 실측.) $n$이 두 배가 될 때마다 시간이 대략 **8배**($=2^3$)씩 늘어난다 — $O(n^3)$의 정의 그대로다. $n = 400$만 돼도 1초에 육박한다. **플로이드-워셜은 정점 수가 수백 이하일 때만 현실적이다.** 백준 문제에서 $n \le 500$ 근처로 제약이 걸려 있으면 플로이드-워셜을 노린 문제라고 봐도 된다.
:::

## A* — 목적지를 아는 다익스트라

지금까지의 세 알고리즘은 **목적지가 어디인지 모른 채** 모든 방향으로 고르게 탐색한다. 하지만 코딩테스트에서든 실제 내비게이션에서든 "출발점에서 도착점까지"라는 구체적인 질문이 훨씬 많다. 목적지를 안다면, **목적지 쪽을 향하는 탐색을 우선시**해서 쓸데없는 방향을 덜 볼 수 있다. 이게 A*다.

다익스트라는 "여기까지 오는 데 확실히 든 비용" $g(n)$만으로 우선순위를 정한다. A*는 여기에 **"여기서 목적지까지 대충 얼마나 남았는가"를 추정하는 휴리스틱** $h(n)$을 더한다.

$$f(n) = g(n) + h(n)$$

```python title="astar_grid.py — 격자 위 A*"
import heapq


def neighbors(grid, pos):
    size = len(grid)
    i, j = pos
    for di, dj in ((1, 0), (-1, 0), (0, 1), (0, -1)):
        ni, nj = i + di, j + dj
        if 0 <= ni < size and 0 <= nj < size and grid[ni][nj] == 0:
            yield (ni, nj), 1   # grid[i][j] == 1 이면 벽


def manhattan(a, b):
    return abs(a[0] - b[0]) + abs(a[1] - b[1])


def astar(grid, start, goal):
    dist = {start: 0}
    pq = [(manhattan(start, goal), 0, start)]
    while pq:
        f, d, u = heapq.heappop(pq)
        if d > dist.get(u, float("inf")):
            continue
        if u == goal:
            return d
        for v, w in neighbors(grid, u):
            nd = d + w
            if nd < dist.get(v, float("inf")):
                dist[v] = nd
                heapq.heappush(pq, (nd + manhattan(v, goal), nd, v))
    return float("inf")
```

다익스트라와 코드가 거의 같다. 딱 하나, 힙에 넣는 우선순위가 `d`(실제 비용) 대신 `d + manhattan(v, goal)`(실제 비용 + 추정 잔여 비용)이라는 점만 다르다.

::: warn 휴리스틱은 아무렇게나 고르면 안 된다
A*가 항상 최단 경로를 찾으려면 휴리스틱이 **허용 가능(admissible)** 해야 한다 — 실제 남은 비용을 **절대 과대평가하지 않아야** 한다. 격자에서 상하좌우로만 움직이고 간선 비용이 전부 1이라면, 맨해튼 거리는 항상 실제 거리 이하이므로 허용 가능하다. 대각선 이동이 가능한데 맨해튼 거리를 쓰면 과대평가가 생겨 최단 경로를 놓칠 수 있다 — 그럴 땐 체비쇼프 거리나 유클리드 거리를 써야 한다. 휴리스틱을 잘못 고르면 **더 빨라지는 대신 틀린 답을 낸다.**
:::

::: cote $h(n) = 0$이면 A*는 다익스트라다
휴리스틱을 그냥 `0`으로 두면 $f(n) = g(n)$이 되어 다익스트라와 완전히 같아진다. 즉 **A*는 다익스트라의 일반화**다. 코딩테스트에서 격자 위 최단 경로 문제가 나오면 "목적지가 명확히 정해져 있는가"를 확인해라. 여러 목적지 후보 중 아무 데나 가도 되는 문제라면 좋은 휴리스틱을 찾기 어려워 다익스트라나 BFS가 더 간단할 때가 많다.
:::

::: perf 실측: A*가 실제로 덜 본다
벽이 20%인 무작위 격자에서 좌상단에서 우하단까지 최단 경로를 구하며, 힙에서 뽑힌(방문한) 노드 수를 셌다.

| 격자 크기 | 최단 거리 | 다익스트라 방문 수 | A* 방문 수 | A*가 덜 본 비율 |
| --- | --- | --- | --- | --- |
| 20×20 | 38 | 308 | 206 | 1.50배 |
| 40×40 | 78 | 1,251 | 612 | 2.04배 |
| 60×60 | 118 | 2,879 | 1,665 | 1.73배 |
| 80×80 | 158 | 5,121 | 3,201 | 1.60배 |

(Python 3.14.5 / Windows 기준 실측. 두 방법이 찾은 최단 거리는 모두 동일함을 `assert`로 확인.) 두 방법이 찾는 **최단 거리 값은 완전히 같다.** 다만 A*는 목적지 방향으로 탐색을 편향시켜서 **방문 노드 수를 1.5~2배 줄인다.** 격자가 클수록, 그리고 장애물이 목적지 방향을 크게 우회시키지 않을수록 이 이득은 더 커진다.
:::

## 언제 무엇을 쓰는가

| 상황 | 알고리즘 | 복잡도 |
| --- | --- | --- |
| 가중치 ≥ 0, 출발점 하나, 도착점 전부 필요 | 다익스트라 (heapq) | $O((V+E)\log V)$ |
| 가중치 ≥ 0, 그래프가 조밀함($E \approx V^2$) | 다익스트라 (배열) | $O(V^2)$ |
| 가중치에 음수 포함 | 벨만-포드 | $O(VE)$ |
| "이득이 무한히 커질 수 있는가?"를 물음 (음수 사이클 판정) | 벨만-포드 | $O(VE)$ |
| 모든 정점 쌍의 최단 거리, $V$가 작음(수백 이하) | 플로이드-워셜 | $O(V^3)$ |
| 출발점·도착점이 명확하고 좋은 휴리스틱이 있음 | A* | 다익스트라와 같거나 더 빠름(실전) |
| 모든 간선 비용이 1 | BFS ([7.14](#/bfs-dfs)) | $O(V+E)$ |

::: cote 실전에서 순서대로 물어라
1. **가중치가 전부 1인가?** → BFS로 끝난다. 다익스트라를 쓸 필요가 없다.
2. **음수 가중치가 있는가?** → 벨만-포드. 여기서 다익스트라를 쓰면 조용히 틀린다.
3. **모든 쌍이 필요하고 $V$가 작은가($\le$ 수백)?** → 플로이드-워셜. 코드가 압도적으로 짧다.
4. **출발-도착이 정해져 있고 격자·좌표 기반이라 좋은 휴리스틱이 보이는가?** → A*.
5. **그 외 대부분** → 다익스트라(heapq). 코딩테스트에서 가장 자주 쓰는 기본값이다.

이 판단을 시험장에서 몇 초 안에 끝내는 습관은 [8.1 코딩테스트의 구조와 전략](#/cote-strategy)이 강조하는 "제출 전 체크리스트"의 일부다. 실제 문제로 반복 훈련하는 건 [8.4 문제 유형 분류와 신호 읽기](#/problem-signals), [8.7 실전 풀이 II — 탐색/그래프](#/drill-search)에서 이어진다.
:::

경로 자체(거리뿐 아니라 실제 지나온 노드 목록)가 필요하면, 갱신될 때마다 `prev[v] = u`를 기록해 두고 도착점부터 거꾸로 따라가며 복원한다. 다익스트라·벨만-포드·A* 모두 같은 방식이 통한다.

```python title="경로 복원"
def reconstruct(prev, goal):
    path = [goal]
    while path[-1] in prev:
        path.append(prev[path[-1]])
    return path[::-1]
```

## 요약

- BFS는 간선 수만 센다. 가중치가 있으면 BFS로는 안 되고, 다익스트라부터 시작한다.
- 다익스트라는 "확정된 노드의 거리는 다시 줄지 않는다"는 가정 위에서 동작한다. `heapq` + 지연 삭제로 $O((V+E)\log V)$에 구현한다. 그래프가 조밀하면 배열 기반 $O(V^2)$ 구현이 오히려 나을 수 있다.
- 음수 가중치가 있으면 다익스트라는 **조용히 틀린다.** 예외도 경고도 없다. 이럴 땐 벨만-포드를 써라.
- 벨만-포드는 모든 간선을 $V-1$번 반복해서 갱신한다. $V$번째 반복에서도 갱신이 일어나면 음수 사이클이 있다는 뜻이다.
- 플로이드-워셜은 경유지를 바깥 루프로 둔 삼중 루프로 모든 쌍의 최단 거리를 $O(V^3)$에 구한다. $V$가 수백을 넘으면 비현실적이다.
- A*는 목적지까지의 추정 비용(휴리스틱)을 더해 다익스트라의 탐색 방향을 목적지 쪽으로 편향시킨다. 휴리스틱이 과대평가하지 않아야 정확성이 보장된다. $h=0$이면 다익스트라와 같다.
- 넷 다 "확정된 거리를 바탕으로 이웃을 갱신한다"는 같은 골격이다. 차이는 **어떤 순서로, 몇 번 갱신하는가**에 있다.
