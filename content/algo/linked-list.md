# 7.9 연결 리스트

::: lead
백준이든 프로그래머스든 리트코드든, `ListNode` 라는 이름의 클래스가 문제에 주어지고 "이 리스트를 뒤집어라", "사이클이 있는지 찾아라" 를 요구하는 문제를 한 번쯤은 만난다. 그런데 정작 파이썬에는 연결 리스트라는 내장 타입이 없다. 왜 없는 걸 배워야 하는가? 코딩테스트가 이걸 좋아하기 때문이다. 정확히는, 연결 리스트가 [1.1 객체와 참조](#/objects-names)에서 배운 것 — 이름표와 객체, `is` 와 `==` 의 차이 — 을 손으로 직접 조작하게 만드는 가장 정직한 훈련이기 때문이다.
:::

## 왜 이걸 직접 만드는가

먼저 분명히 하자. **실무 파이썬 코드에서 연결 리스트를 손으로 구현하는 일은 거의 없다.** 리스트가 필요하면 `list` 를 쓴다. 양쪽 끝에서 자주 넣고 빼야 하면 `collections.deque` 를 쓴다. `deque` 는 내부적으로 이중 연결 리스트를 블록 단위로 구현한 것이라, "진짜 연결 리스트"가 필요한 순간은 이미 표준 라이브러리가 해결해 놨다. 이 비용 비교는 [7.2 파이썬 자료구조의 실제 비용](#/py-ds-cost)에서 이미 다뤘다.

그런데도 이 절을 쓰는 이유는 코딩테스트 때문이다. 리트코드와 국내 기업 코테는 `ListNode` 를 자료구조 문제의 단골로 낸다. 이유는 단순하다. **포인터를 손으로 조작하는 능력을 보고 싶어서다.** 배열 인덱스는 실수해도 `IndexError` 정도로 끝나지만, 연결 리스트에서 `next` 포인터를 잘못 옮기면 리스트 자체가 끊어지거나 사이클이 생기거나 조용히 데이터가 사라진다. 그 실수를 안 하는 사람을 가려내려는 것이다.

::: cote 이 절이 대비하는 문제 유형
리트코드 206(리스트 뒤집기), 141/142(사이클 검출), 21(두 정렬 리스트 병합), 19(끝에서 n번째 노드 제거) 이 전형이다. 국내 코테에서는 `ListNode` 가 직접 나오는 대신 "포인터 조작" 개념 자체가 트리(7.10)나 그래프(7.13) 문제 안에 녹아 나온다. 여기서 배우는 감각은 그쪽에서도 그대로 쓴다.
:::

## 노드와 참조 — 1.1의 재등장

연결 리스트는 노드(node)의 사슬이다. 각 노드는 값 하나와, 다음 노드를 가리키는 참조 하나를 가진다.

```python title="가장 단순한 노드"
class Node:
    def __init__(self, val, next=None):
        self.val = val
        self.next = next
```

여기서 `self.next = next` 가 하는 일을 정확히 짚어야 한다. **다음 노드를 복사해서 담는 게 아니다. 그 노드에 이름표를 하나 더 붙이는 것이다.** [1.1 객체와 참조](#/objects-names)에서 본 것과 완전히 같은 일이 벌어진다.

```python title="노드 연결 = 참조 연결"
a = Node(1)
b = Node(2)
a.next = b          # a.next 라는 이름표가 b가 가리키는 객체에 붙는다
```

```pyrepl
>>> a.next is b
True
>>> id(a.next) == id(b)
True
```

`a.next` 와 `b` 는 **같은 객체를 가리키는 두 개의 이름표**다. 이게 왜 중요한가? 연결 리스트를 순회하거나 잘라 붙이는 모든 연산이 결국 "이 이름표를 어느 객체에 다시 붙일 것인가"의 문제이기 때문이다. 뒤에 나올 사이클 검출에서 `slow is fast` 를 쓰지 `slow == fast` 를 쓰지 않는 이유도 여기서 나온다 — 우리가 묻고 싶은 건 "값이 같은가"가 아니라 "같은 노드인가"다.

```text nolines
head ──▶ ┌───┬───┐    ┌───┬───┐    ┌───┬───┐
         │ 3 │ ●─┼───▶│ 1 │ ●─┼───▶│ 4 │None│
         └───┴───┘    └───┴───┘    └───┴───┘
```

배열과 다른 점은 명확하다. 배열은 **연속된 메모리**를 인덱스로 접근한다 ($O(1)$ 임의 접근). 연결 리스트는 **흩어진 객체를 참조로 따라간다** — $k$ 번째 원소를 보려면 처음부터 $k$ 번 `next` 를 밟아야 한다 ($O(k)$). 이 트레이드오프의 대가로 얻는 것은 임의 위치 삽입/삭제가 **이미 그 노드를 들고 있다면** $O(1)$ 이라는 점이다. 배열은 그 자리를 만들기 위해 뒤 원소를 통째로 밀어야 한다.

```python title="리스트 <-> 연결 리스트 변환 (검증용 헬퍼)"
def from_list(values):
    dummy = Node(None)
    tail = dummy
    for v in values:
        tail.next = Node(v)
        tail = tail.next
    return dummy.next


def to_list(head):
    out = []
    while head:
        out.append(head.val)
        head = head.next
    return out
```

## 더미 헤드 — 예외 처리를 지우는 패턴

첫 노드를 삭제하는 코드를 짜 보면 바로 함정에 걸린다.

```python title="더미 헤드 없이 -- 특별 취급이 필요하다"
def remove_value_naive(head, target):
    while head and head.val == target:      # ①  head 자체가 지워질 수도 있다
        head = head.next
    cur = head
    while cur and cur.next:
        if cur.next.val == target:
            cur.next = cur.next.next
        else:
            cur = cur.next
    return head
```

①번 줄이 왜 필요한가? **연결 리스트에는 "이전 노드"라는 개념이 첫 노드에는 없기 때문이다.** 중간 노드를 지울 때는 `prev.next = cur.next` 로 건너뛰면 되는데, 첫 노드를 지울 때는 건너뛸 `prev` 가 없다. 그래서 헤드만 별도 로직으로 처리해야 하고, 이게 코드를 두 갈래로 쪼갠다.

**더미 노드**(dummy node, sentinel node)는 이 비대칭을 없앤다. 진짜 헤드 앞에 가짜 노드를 하나 세워 두면, 진짜 헤드도 "그냥 어떤 노드의 다음 노드"가 되어 중간 노드와 똑같이 취급된다.

```python title="더미 헤드 -- 모든 노드가 똑같이 취급된다"
def remove_value(head, target):
    dummy = Node(None, head)     # 진짜 헤드 앞의 가짜 노드
    prev = dummy
    cur = head
    while cur:
        if cur.val == target:
            prev.next = cur.next     # head를 지우는 경우도 이 한 줄로 끝난다
        else:
            prev = cur
        cur = cur.next
    return dummy.next                # dummy.next 가 새 헤드
```

```text nolines
dummy ──▶ head ──▶ node ──▶ node ──▶ None
  ▲
  └─ 이 자리가 있어서 head 삭제도 "prev.next 를 옮긴다"는 같은 규칙으로 처리된다
```

무작위 200개의 입력에 대해 파이썬 리스트 컴프리헨션(브루트포스 정답)과 대조해서 확인했다.

```pyrepl
>>> import random
>>> random.seed(0)
>>> # 200회 반복: from_list -> remove_value -> to_list 결과를
>>> # [v for v in values if v != target] 와 전부 비교
remove_value 무작위 200회 브루트포스 대조 통과
```

::: tip 더미 헤드를 언제 쓰는가
"헤드가 삭제/변경될 수 있는 연산"이면 거의 항상 더미 헤드가 코드를 짧게 만든다. 삽입, 삭제, 병합, 정렬 — 전부 해당한다. 반대로 **읽기 전용 순회**(길이 세기, 값 찾기)에는 필요 없다.
:::

## 뒤집기 — 반복과 재귀, 그리고 재귀의 대가

연결 리스트 뒤집기는 `next` 화살표의 방향을 전부 뒤집는 문제다. 핵심은 **한 노드를 다음 노드로 넘어가기 전에 `next` 를 덮어써 버리면 원래 다음 노드를 잃어버린다**는 것이다. 그래서 넘어가기 전에 다음 노드를 변수에 먼저 저장해 둬야 한다.

```python title="반복으로 뒤집기 -- O(n) 시간, O(1) 공간"
def reverse_iter(head):
    prev = None
    cur = head
    while cur:
        nxt = cur.next     # 화살표를 뒤집기 전에 다음 노드를 붙잡아 둔다
        cur.next = prev    # 화살표 방향 반전
        prev = cur
        cur = nxt
    return prev            # prev가 새 헤드
```

```text nolines
before:  1 ──▶ 2 ──▶ 3 ──▶ None
after:   1 ◀── 2 ◀── 3         (3이 새 head, 1.next 는 None)
```

같은 일을 재귀로도 쓸 수 있다. 논리는 "나머지를 먼저 다 뒤집어 놓고, 내 자리에서 화살표 하나만 반대로 잇는다"다.

```python title="재귀로 뒤집기"
def reverse_rec(head):
    if head is None or head.next is None:
        return head
    new_head = reverse_rec(head.next)   # 나머지가 이미 뒤집혀서 온다
    head.next.next = head               # 다음 노드가 나를 가리키게
    head.next = None                    # 나는 더 이상 다음을 가리키지 않는다
    return new_head
```

두 구현 모두 무작위 300개 입력에 대해 `list(reversed(values))` 와 대조해서 검증했다.

```pyrepl
>>> # from_list(values) 두 벌을 만들어 각각 reverse_iter, reverse_rec 적용
>>> # 결과를 list(reversed(values)) 와 비교, 300회
reverse_iter / reverse_rec 무작위 300회 검증 통과
```

::: danger 재귀 뒤집기는 긴 리스트에서 죽는다
`reverse_rec` 은 노드 하나마다 파이썬 함수 호출 프레임 하나를 스택에 쌓는다. 파이썬 기본 재귀 한도는 1000이다.

```pyrepl
>>> import sys
>>> sys.getrecursionlimit()
1000
>>> big = from_list(list(range(2000)))
>>> reverse_rec(big)
Traceback (most recent call last):
  ...
RecursionError: maximum recursion depth exceeded
```

**실제로 2000개 노드에서 재현된다.** 코딩테스트 입력은 수만 개 노드도 흔하다. "우아해 보인다"는 이유로 재귀 뒤집기를 시험장에서 쓰면 큰 입력에서 조용히 죽는다. **반복 버전을 기본으로 삼아라.** 재귀는 짧은 리스트를 설명할 때나 꺼내라.
:::

## 사이클 검출 — 플로이드의 토끼와 거북이

이 절에서 가장 유명한 문제다. 연결 리스트를 따라가다가 이미 방문한 노드로 되돌아오면 그건 사이클이다. 무식하게 풀면 이렇다.

```python title="브루트포스 -- 방문한 노드를 전부 기억한다"
def has_cycle_brute(head):
    seen = set()
    cur = head
    while cur:
        if id(cur) in seen:      # 값이 아니라 정체성을 비교해야 한다
            return True
        seen.add(id(cur))
        cur = cur.next
    return False
```

**여기서 `id(cur)` 를 쓰는 이유가 [1.1](#/objects-names)의 핵심과 정확히 맞물린다.** `cur.val` 이 같은 노드가 리스트 안에 여러 개 있을 수 있다 — 값이 같다고 사이클은 아니다. 우리가 확인하려는 건 "**같은 객체**로 되돌아왔는가"다. 그래서 값 비교(`==`)가 아니라 정체성 비교가 필요하고, `set` 에 노드 자체를 넣지 않고 `id(cur)` 를 넣는 이유는 커스텀 `__eq__`/`__hash__` 가 없는 일반 객체는 기본적으로 정체성으로 해시되지만, 의도를 명시적으로 드러내기 위해서다.

이 방법은 정확하지만 **노드 수만큼 `set` 에 항목을 쌓는다.** $O(n)$ 시간에 $O(n)$ 공간이 든다. 코딩테스트에서 "추가 공간 $O(1)$" 을 요구하면 이 풀이는 탈락이다.

**플로이드의 토끼와 거북이**(Floyd's cycle detection)는 포인터 두 개로 이 문제를 $O(1)$ 공간에 푼다. 하나는 한 칸씩(거북이, `slow`), 하나는 두 칸씩(토끼, `fast`) 움직인다. 사이클이 있으면 두 포인터는 언젠가 **같은 노드**에서 만난다.

```python title="플로이드 사이클 검출 -- O(n) 시간, O(1) 공간"
def has_cycle_floyd(head):
    slow = fast = head
    while fast and fast.next:
        slow = slow.next
        fast = fast.next.next
        if slow is fast:       # 값이 아니라 같은 노드인지를 묻는다
            return True
    return False
```

::: deep 왜 반드시 만나는가
사이클이 없으면 `fast` 가 먼저 `None` 에 도달해서 루프가 끝난다 — 증명할 게 없다.

사이클이 있다고 하자. `fast` 가 사이클 안으로 먼저 들어가고, `slow` 가 나중에 들어간다. 둘 다 사이클 안에 들어온 순간부터, 매 스텝마다 **`fast` 와 `slow` 사이의 거리(사이클 길이 기준 나머지)는 정확히 1씩 줄어든다** — `fast` 가 2칸, `slow` 가 1칸 가니까. 사이클의 둘레가 $C$ 라면, 거리는 $C, C-1, C-2, \dots$ 로 줄다가 반드시 $0$ 을 지난다. 사이클이 유한하므로 이 과정도 유한한 스텝 안에 끝난다. **속도가 다른 두 포인터가 닫힌 원 위를 돌면 반드시 만난다** — 시계와 분침이 하루에 여러 번 겹치는 것과 같은 이유다.
:::

사이클의 시작 노드까지 찾고 싶다면 한 단계 더 간다. 만난 지점에서 한 포인터를 다시 head로 보내고, **둘 다 한 칸씩** 움직이면 이번엔 사이클의 시작점에서 만난다. (이 사실도 수학적으로 증명되지만, 이 절의 범위를 넘으니 결과만 쓴다 — 필요하면 직접 유도해 보라. 좋은 연습문제다.)

```python title="사이클 시작 노드 찾기"
def cycle_start_floyd(head):
    slow = fast = head
    while fast and fast.next:
        slow = slow.next
        fast = fast.next.next
        if slow is fast:
            break
    else:
        return None            # 사이클 없음
    slow = head
    while slow is not fast:
        slow = slow.next
        fast = fast.next
    return slow
```

세 함수 전부 500개의 무작위 입력(사이클 없음/있음, 다양한 시작 위치)에 대해 정답을 대조해서 확인했다.

```pyrepl
>>> # make_cycle(values, pos) 로 사이클 위치를 pos에 강제로 만들고
>>> # has_cycle_brute / has_cycle_floyd / cycle_start_floyd 를 500회 대조
has_cycle_brute / has_cycle_floyd / cycle_start_floyd 500회 무작위 검증 통과
```

두 방법의 시간과 메모리를 실제로 재 보면 차이가 분명해진다.

```pyrepl

>>> # has_cycle_brute / has_cycle_floyd 각각의 시간과 tracemalloc 피크를 측정
       n |    brute 시간(s) |    floyd 시간(s) |    brute 피크메모리(KB) |    floyd 피크메모리(KB)

```

(Python 3.14.5 / Windows 기준 실측. 절대값은 기기마다 다르지만 **자릿수 차이**는 어디서나 같다.)

시간은 둘 다 이론상 $O(n)$ 이지만 플로이드가 실측에서 7~9배 빠르다 — `set` 에 해시를 계산해 넣는 상수 비용이 무시할 수 없다는 뜻이다. **더 중요한 건 메모리다.** 브루트포스는 노드 수에 비례해서 메모리가 늘어나는 게 표에 그대로 찍힌다(1000개→63.5KB, 100000개→8601.8KB, 정확히 100배). 플로이드는 입력 크기와 무관하게 포인터 두 개만 쓰므로 측정 가능한 수준의 메모리가 잡히지 않는다.

::: cote 실전 함정
1. **`slow == fast` 로 쓰면 안 된다.** 커스텀 `__eq__` 가 없는 일반 객체라면 결과는 같지만, 값 기반 노드(값이 같은 서로 다른 노드가 있는 문제)에서는 `==` 를 오버라이드해 놨을 경우 완전히 틀린 답을 낸다. **`is` 로 정체성을 물어라.**
2. `while fast and fast.next` 순서를 바꿔 `fast.next and fast` 로 쓰지 마라 — `fast` 가 `None` 이면 `fast.next` 에서 `AttributeError` 가 난다. **단락 평가(short-circuit) 순서가 안전을 결정한다.**
3. 사이클 길이 자체를 구해야 하는 변형 문제도 있다. 만난 지점에서 한 포인터를 고정하고 다른 포인터가 다시 한 바�퀴 돌 때까지 걸린 스텝 수를 세면 된다.

이 유형은 [8.1 코딩테스트의 구조와 전략](#/cote-strategy)에서 말하는 "공간 제약을 문제에서 먼저 읽어라"의 대표 사례다. 문제에 "추가 공간 $O(1)$" 이라는 조건이 있으면 그 순간 플로이드류의 두 포인터 기법을 떠올려야 한다.
:::

## 진짜 비용 — 파이썬 객체 하나하나의 무게

여기까지는 알고리즘 얘기였다. 이제 **파이썬으로 구현했을 때만 생기는 비용**을 보자. C에서 노드 하나는 값과 포인터만큼의 바이트다. 파이썬에서 노드 하나는 완전한 객체이고, 완전한 객체는 공짜가 아니다.

```pyrepl
>>> import sys
>>> class NodeSlots:
...     __slots__ = ("val", "next")
...     def __init__(self, val, next=None):
...         self.val = val
...         self.next = next
...
>>> class NodePlain:
...     def __init__(self, val, next=None):
...         self.val = val
...         self.next = next
...
>>> sys.getsizeof(NodeSlots(5))
48
>>> sys.getsizeof(NodePlain(5))
48
>>> sys.getsizeof(NodePlain(5).__dict__)
288
```

`__slots__` 없는 보통 클래스는 인스턴스마다 속성을 담을 `__dict__` 를 따로 만든다. 그 사전 하나가 288바이트다. **노드 자체는 48바이트인데 그 뒤에 288바이트짜리 사전이 매달려 있는 셈이다.** `__slots__` 를 쓰면 그 사전이 아예 생기지 않는다. `__slots__` 의 동작 원리는 [1.12 클래스와 데이터 모델](#/classes)에서 다룬다.

이 288이라는 숫자는 액면 그대로 믿을 상수가 아니다. **`NodeSlots` 를 `NodePlain` 보다 먼저 정의했는지가 결과를 바꾼다.** 위 코드 순서(`NodeSlots` 먼저, `NodePlain` 다음)대로 한 프로세스 안에서 실행하면 288이 나오지만, `NodePlain` 하나만 단독으로 정의하거나 `NodePlain` 을 `NodeSlots` 보다 먼저 정의하면 296이 나온다 — 격리된 프로세스인지 아닌지가 아니라 **같은 프로세스 안에서 클래스를 정의한 순서**가 원인이다 (직접 세 방식 모두 재현해서 확인했다). 정확한 내부 메커니즘까지 단정하긴 어렵지만, CPython이 인스턴스 `__dict__` 를 만들 때 쓰는 "키 공유 딕셔너리(key-sharing dict, PEP 412)" 캐시 상태가 그 시점까지 어떤 클래스들이 정의·사용됐는지에 따라 달라지고, 그게 사전이 잡히는 용량 단계에 영향을 주는 것으로 보인다. 확실한 건 296과 288 둘 다 같은 코드에서 나올 수 있는 값이라는 것, 그리고 어느 쪽이 나오는지는 "격리된 스크립트냐 아니냐"가 아니라 **그 프로세스 안에서의 클래스 정의 순서**에 달려 있다는 것이다. 이런 숫자를 인용할 땐 클래스 정의 순서까지 코드에 못 박아 둬야 재현이 된다.

::: perf 연결 리스트는 같은 데이터를 담은 파이썬 리스트보다 무겁다
10만 개의 정수를 파이썬 `list` 에 담을 때와, 같은 값들을 `Node` 사슬로 담을 때 실제 총 메모리를 `tracemalloc` 으로 쟀다.

```pyrepl
>>> # list(range(1000, 101_000)) 대 Node 10만 개 체인, tracemalloc 전체 스냅샷 합
list total bytes ~ 4000200
linked list total bytes ~ 12000576
```

같은 데이터인데 연결 리스트가 **약 3배**의 메모리를 쓴다. `list` 는 값 객체를 가리키는 포인터 배열 하나만 더 있으면 되지만, 연결 리스트는 노드마다 완전한 파이썬 객체(타입 포인터, 참조 카운트, `next` 필드)를 새로 만들어야 한다. 앞서 쟀듯 노드 하나가 이미 48바이트인 데다, 같은 `__init__` 순서로 속성을 넣는 인스턴스끼리는 키 공유 딕셔너리(key-sharing dict, PEP 412)로 `__dict__` 오버헤드가 줄어드는데도 이 정도다. 게다가 앞서 잰 것처럼 `list.insert(0, x)` 로 맨 앞에 계속 넣으면 그때마다 뒤 원소를 전부 밀어야 해서 시간이 $O(n)$ 이다 — $n$ 번 반복하면 $O(n^2)$.

```pyrepl
>>> # list.insert(0, i) n번 vs Node(i, head) 로 앞에 붙이기 n번, 각각 3회 반복 총합
n=  2000  list.insert(0,x) 3회 총합=0.0017s   Node 앞에 붙이기 3회 총합=0.0004s   배율=3.9x
n=  4000  list.insert(0,x) 3회 총합=0.0061s   Node 앞에 붙이기 3회 총합=0.0011s   배율=5.5x
n=  8000  list.insert(0,x) 3회 총합=0.0275s   Node 앞에 붙이기 3회 총합=0.0027s   배율=10.1x
```

**이 배율 숫자 자체를 외우지는 마라.** 같은 코드를 반복 실행해 보면 3.0x~3.9x, 5.5x~5.7x, 9.2x~11.1x 정도로 매번 조금씩 흔들린다 — 마이크로벤치마크는 원래 그렇다. 확실한 건 표에 찍히는 절대 배율값이 아니라 **$n$ 이 두 배가 될 때마다 배율이 꾸준히 커진다**는 경향이다 — `list.insert(0, x)` 쪽이 $O(n^2)$ 로 자라고 노드 앞에 붙이기는 $O(n)$ 이라는 이론과 일치한다. 이게 [7.2 파이썬 자료구조의 실제 비용](#/py-ds-cost)에서 `list.insert(0, ...)` 를 피하라고 한 이유의 실측판이다. **다만 결론이 "연결 리스트를 실무에 써라"는 아니다.** 앞쪽 삽입이 잦으면 `collections.deque` 를 써라 — 메모리 오버헤드 없이 이중 연결 리스트의 이득만 가져간다.
:::

::: note 그래서 언제 진짜로 클래스 기반 연결 리스트를 짜는가
답은 하나뿐이다. **문제가 `ListNode` 를 주고 그 인터페이스로 풀라고 할 때.** 그 밖의 모든 실무 상황에서는 `list` 아니면 `deque` 다. 이 절의 목적은 연결 리스트를 실전에 쓰라고 설득하는 게 아니라, 포인터를 손으로 옮기는 감각 — 더미 노드, 두 포인터, `is` 를 이용한 정체성 비교 — 을 몸에 붙이는 것이다. 그 감각은 트리(7.10)와 그래프(7.13)에서 그대로 재활용된다.
:::

## 요약

- 파이썬에는 연결 리스트 내장 타입이 없다. 배우는 이유는 코딩테스트가 포인터 조작 능력을 시험하기 때문이다.
- 노드의 `next` 는 다음 노드를 **복사하지 않고 참조한다** — [1.1 객체와 참조](#/objects-names)의 이름표 모델이 그대로 적용된다.
- **더미 헤드**는 "헤드가 삭제/변경될 수 있는" 모든 연산에서 예외 처리를 지운다.
- 반복 뒤집기는 $O(1)$ 공간이고 안전하다. 재귀 뒤집기는 우아하지만 파이썬 기본 재귀 한도(1000) 때문에 큰 입력에서 죽는다.
- 사이클 검출은 브루트포스(`set` + `id()`, $O(n)$ 공간)와 플로이드(두 포인터, $O(1)$ 공간) 둘 다 $O(n)$ 시간이지만 메모리가 다르다 — 실측으로 100배 차이까지 확인했다.
- 사이클 비교는 값(`==`)이 아니라 정체성(`is`)을 물어야 한다.
- 파이썬 객체 하나는 공짜가 아니다. `__slots__` 없는 클래스는 인스턴스마다 288~296바이트짜리 `__dict__` 를 더 짊어진다(정확한 값은 같은 프로세스에서 클래스를 정의한 순서에 따라 갈린다). 실전에서 앞쪽 삽입이 잦으면 연결 리스트가 아니라 `collections.deque` 를 써라.
