# 7.24 비트마스크

::: lead
정수 하나로 집합 전체를 표현할 수 있다면 어떨까. 원소가 있는지 확인하는 것도, 추가하는 것도, 두 집합의 교집합을 구하는 것도 전부 CPU 명령 하나로 끝난다면? 비트마스크는 정확히 이 아이디어다 — 정수의 각 비트를 원소 하나에 대응시켜, 파이썬의 `set`이 하는 일을 정수 연산자(`| & ^ ~`)로 대체한다. 단순한 트릭처럼 보이지만, 이 절을 넘기면 외판원 순회 같은 "브루트포스로는 절대 못 푸는" 문제가 갑자기 실행 가능한 문제로 바뀐다. [7.20 동적 계획법 기초](#/dp-basics)에서 상태를 표현하는 법을 배웠다면, 비트마스크는 그 상태 중 하나인 "지금까지 무엇을 골랐는가"를 정수 하나로 압축하는 방법이다.
:::

## 브루트포스가 막히는 지점 — 외판원 순회

다음 문제를 생각해 보자. 도시가 $n$개 있고, 도시 사이의 이동 비용이 주어진다. 도시 0에서 출발해 나머지 도시를 전부 한 번씩 방문하고 다시 0으로 돌아오는 최소 비용 경로를 구하라. 이른바 외판원 순회(TSP) 문제다.

가장 직관적인 방법은 방문 순서를 전부 나열해 보는 것이다.

```python title="tsp_brute.py — 순열을 전부 시도한다"
import itertools

def tsp_brute(n, dist):
    best = float("inf")
    for perm in itertools.permutations(range(1, n)):
        route = (0,) + perm
        cost = sum(dist[route[k]][route[k + 1]] for k in range(n - 1))
        cost += dist[route[-1]][route[0]]
        best = min(best, cost)
    return best
```

$n$개 도시를 방문하는 순서는 $(n-1)!$가지다. 실제로 재 보면 이렇다.

| n | 브루트포스 시간 | $(n-1)!$ |
| --- | --- | --- |
| 8 | 0.0028초 | 5,040 |
| 9 | 0.0244초 | 40,320 |
| 10 | 0.2540초 | 362,880 |

(Python 3.14.5 / Windows 기준 실측.)

$n$이 하나 늘 때마다 시간이 대략 $n$배로 는다. $n=19$면 $(n-1)! = 18! = 6{,}402{,}373{,}705{,}728{,}000$ — $10^{15}$ 자릿수다. 여기서 $n$을 하나만 더 늘려도 $19! = 121{,}645{,}100{,}408{,}832{,}000$로 $10^{17}$ 자릿수까지 뛴다. 코딩테스트에서 $n \le 20$ 정도의 TSP류 문제는 이 방식으로 절대 못 뚫는다.

::: cote 신호: 언제 비트마스크를 의심해야 하는가
문제의 제약 조건을 보고 바로 판단할 수 있어야 한다.

- **$n \le 20$ 정도로 작다** — $2^{20} \approx 10^6$, $2^n \times n$ 규모 상태 공간이 시간 안에 들어온다는 뜻이다.
- "모든 원소를 방문했는지", "지금까지 어떤 부분집합을 골랐는지"가 상태에 들어간다.
- "순서가 중요한 게 아니라 무엇을 골랐는지가 중요하다" — 순열이 아니라 부분집합이 본질이다.

$n \le 20$이라는 제약은 그 자체로 "이 문제는 $2^n$을 상태로 쓰라"는 출제자의 신호다. 제약 조건만 보고 알고리즘을 역추론하는 훈련은 [8.4 문제 유형 분류와 신호 읽기](#/problem-signals)에서 더 다룬다.
:::

핵심 관찰은 이렇다. 브루트포스가 낭비하는 것은 **"어떤 순서로 방문했는가"라는 정보다.** 하지만 다음에 어디로 갈 수 있는지 결정하는 데 필요한 것은 순서 전체가 아니라 딱 두 가지뿐이다 — **지금까지 방문한 도시의 집합**과 **현재 위치**. 방문 순서가 달라도 방문한 집합과 현재 위치가 같으면 이후의 최적 선택은 똑같다. 이 관찰이 상태 공간을 $(n-1)!$에서 $2^n \times n$으로 줄인다. 그리고 "방문한 집합"을 표현하는 가장 효율적인 방법이 바로 비트마스크다.

## 정수를 집합처럼 쓰기: 네 가지 연산

원소 $i$가 집합에 있는지를 정수의 $i$번째 비트로 표현한다. 비트가 1이면 포함, 0이면 미포함이다. 필요한 연산은 네 가지뿐이다.

| 연산 | 의미 | 코드 |
| --- | --- | --- |
| 추가 | 원소 $i$를 넣는다 | `mask \| (1 << i)` |
| 삭제 | 원소 $i$를 뺀다 | `mask & ~(1 << i)` |
| 포함 확인 | 원소 $i$가 있는가 | `(mask >> i) & 1` |
| 토글 | 원소 $i$를 뒤집는다 | `mask ^ (1 << i)` |

```pyrepl
>>> mask = 0
>>> mask |= (1 << 0)      # 0 추가
>>> mask |= (1 << 2)      # 2 추가
>>> mask |= (1 << 3)      # 3 추가
>>> bin(mask)
'0b1101'
>>> (mask >> 2) & 1       # 2가 있는가?
1
>>> (mask >> 1) & 1       # 1이 있는가?
0
>>> mask &= ~(1 << 2)     # 2 삭제
>>> bin(mask)
'0b1001'
>>> mask ^= (1 << 1)      # 1 토글 (없었으니 추가됨)
>>> bin(mask)
'0b1011'
```

**삭제 연산이 왜 되는지가 핵심이다.** `~(1 << i)`는 $i$번째 비트만 0이고 나머지는 전부 1인 수다. C였다면 `int`가 32비트나 64비트로 고정되어 있어서 "나머지는 전부 1"을 정확히 표현하려면 비트 폭을 알아야 한다. 파이썬의 `int`는 [1.2 숫자와 수치 연산](#/numbers)에서 봤듯 **임의 정밀도**라, 음수를 무한히 이어지는 2의 보수로 취급한다. `~(1 << i)`는 정확히 "$i$번째 비트만 0이고 그 위로도 아래로도 전부 1"인 값이 되고, `mask`와 AND하면 다른 비트에 전혀 영향을 주지 않는다. 비트 폭을 신경 쓸 필요가 없다는 뜻이다.

::: danger 연산자 우선순위 함정
비트마스크 코드에서 실제로 자주 나오는 버그다.

```python
mask = 0b0100   # 원소 2만 있다
i = 2
print(mask & 1 << i == 1)   # False — 원하는 결과가 아니다!
```

파이썬에서 `<<`가 `&`보다, `&`가 `==`보다 먼저 묶인다. 그래서 이 식은 `(mask & (1 << i)) == 1`로 해석된다. `mask & (1 << i)`는 원소가 있으면 $2^i$가 나오지 **1이 나오는 게 아니다.** $i=2$면 결과가 4이므로 `== 1`은 항상 거짓이다.

```python
print(bool(mask & 1 << i))     # ✅ 참/거짓만 필요하면 이렇게
print((mask >> i) & 1 == 1)    # ✅ 정확히 0 또는 1이 필요하면 이렇게
```

**포함 여부를 확인할 때는 `& (1<<i)`의 결과를 `1`과 비교하지 마라.** `bool()`로 감싸거나, 오른쪽으로 시프트해서 맨 끝 비트만 남긴 뒤 비교하라.
:::

## 부분집합 전체 순회

원소가 $n$개면 부분집합은 $2^n$개다. $0$부터 $2^n - 1$까지의 정수 각각이 정확히 하나의 부분집합에 대응한다.

```python title="all_subsets.py — 원소 4개의 모든 부분집합"
n = 4
for mask in range(1 << n):
    elems = [i for i in range(n) if (mask >> i) & 1]
    print(f"{mask:04b} -> {elems}")
```

이렇게 만든 16개 부분집합을 `itertools.chain(*combinations(...))`로 만든 파워셋과 대조하면 **완전히 일치한다** — 원소 4개에서 두 방식 모두 정확히 16개, 내용도 같다. 비트마스크 순회가 `itertools`의 조합 생성기를 대체할 수 있다는 뜻이다. 다만 비트마스크 쪽이 원소 개수와 무관하게 `mask`라는 정수 하나로 상태를 표현하므로, DP의 배열 인덱스나 딕셔너리 키로 바로 쓸 수 있다는 게 결정적인 차이다.

::: warn 전체 순회는 $n$이 커지면 바로 못 쓴다
`1 << n`가지를 전부 도는 비용은 $O(2^n)$이다. $n=20$이면 약 백만, $n=25$면 약 3천만, $n=30$이면 10억을 넘는다. 코딩테스트의 시간 제한(보통 1~2초)을 감안하면 **$n \le 20$ 근처**가 전체 부분집합 순회의 현실적 한계다. [7.1 시간·공간 복잡도](#/complexity)에서 다룬 "$10^8$번 연산 = 대략 1초"라는 감각을 여기에도 그대로 적용하면 된다.
:::

## 최하위 비트 추출: `n & -n`

마스크에 들어 있는 원소를 하나씩 꺼내야 할 때 `n & -n`이라는 표현이 자주 쓰인다. 이 값은 **최하위(가장 낮은 자리) 1비트만 남기고 나머지를 전부 0으로 만든다.**

```text nolines
  x       = 0010 1100
  -x      = 1101 0100     <- 파이썬은 무한 정밀도라 2의 보수를 그대로 적용한 것과 같다
  x & -x  = 0000 0100     <- 최하위 1비트만 남는다
```

왜 되는가: $-x$는 $x$의 2의 보수, 즉 $\sim x + 1$이다. $x$의 최하위 1비트보다 아래는 전부 0이므로 그 부분은 보수를 취해도(전부 0 → 전부 1) 그대로고, `+1`을 하면 그 0들이 다시 캐리(carry)를 일으키며 원래의 최하위 1비트 자리까지 뒤집는다. 결과적으로 최하위 1비트 위쪽은 원래 값과 정확히 반대가 되고, `x & -x`를 하면 그 반대인 부분이 전부 0으로 걸러지고 최하위 1비트만 살아남는다.

```python title="lowbit.py — 마스크의 원소를 하나씩 꺼낸다"
def bits_of(mask):
    result = []
    while mask:
        lb = mask & -mask              # 최하위 1비트
        idx = lb.bit_length() - 1      # 그 비트의 위치
        result.append(idx)
        mask ^= lb                     # 그 비트를 지운다
    return result
```

이 함수가 "각 자리를 `(mask >> i) & 1`로 일일이 검사하는" 브루트포스와 같은 결과를 내는지 무작위 20비트 마스크 2,000개로 검증하면 **전부 일치한다.** `n & -n`은 이 절 바깥에서도 계속 나온다 — [7.25 세그먼트 트리와 펜윅 트리](#/segment-tree)의 펜윅 트리(BIT)가 인덱스를 이동할 때 정확히 이 트릭을 쓴다.

## 부분집합의 부분집합 순회

비트마스크 DP를 짜다 보면 "이 집합의 모든 부분집합(submask)"을 순회해야 하는 경우가 나온다 — 배낭 문제를 그룹으로 나누거나, SOS(Sum over Subsets) DP를 짤 때다. 순진하게 짜면 $0$부터 `mask`까지 전부 돌면서 `sub & mask == sub`인 것만 거르면 되지만, 이건 부분집합이 아닌 것까지 다 검사하느라 낭비가 크다.

더 빠른 방법이 있다. **`sub - 1`을 `mask`와 AND하면 다음으로 작은 submask가 바로 나온다.**

```python title="submasks.py — mask의 모든 submask 순회"
def submasks(mask):
    sub = mask
    while True:
        yield sub
        if sub == 0:
            break
        sub = (sub - 1) & mask
```

`mask = 0b1011`(원소 {0, 1, 3})로 실제로 돌려 보면 이런 순서가 나온다.

```text nolines
1011   <- 시작은 mask 자신
1010
1001
1000
0011
0010
0001
0000   <- 여기서 멈춘다 (공집합)
```

`mask`의 원소가 3개이므로 submask는 정확히 $2^3 = 8$개다. 이 트릭이 "$0$부터 `mask`까지 걸러 가며 도는" 브루트포스와 같은 집합을 만드는지 무작위 12비트 마스크 500개로 검증하면 **전부 일치한다.**

::: perf 전체 submask 순회의 총 비용은 $O(3^n)$이다
"$n$비트 마스크 각각에 대해 그 submask를 전부 도는" 이중 반복문의 총 반복 횟수가 얼마인지 헷갈리기 쉽다. $O(4^n)$처럼 보이지만 실제로는 $O(3^n)$이다.

```python
n = 12
total = 0
for mask in range(1 << n):
    sub = mask
    while True:
        total += 1
        if sub == 0:
            break
        sub = (sub - 1) & mask
print(total)        # 531441
print(3 ** 12)      # 531441
```

$n=12$에서 총 반복 횟수를 실제로 세어 보면 정확히 $531{,}441$번이고, $3^{12}$도 정확히 같은 값이다. 이유는 이렇다 — 원소 $i$ 하나를 놓고 보면 (전체 마스크에 없음), (전체 마스크에 있지만 submask에는 없음), (둘 다 있음) 세 가지 경우가 나오고, 이게 $n$개 원소에 대해 독립적으로 곱해지므로 $3^n$이다. $n=20$이면 $3^{20} \approx 3.5 \times 10^9$로 이미 시간 초과권이니 주의하라.
:::

## 비트마스크 DP: 외판원 문제로 되돌아가기

이제 앞서 낸 문제로 돌아간다. 상태를 `dp[mask][last]` = "`mask`에 해당하는 도시들을 이미 방문했고 마지막으로 `last`에 있을 때의 최소 비용"으로 정의한다.

```python title="tsp_bitmask.py — 방문 집합 + 현재 위치를 상태로"
def tsp_bitmask(n, dist):
    FULL = (1 << n) - 1
    INF = float("inf")
    dp = [[INF] * n for _ in range(1 << n)]
    dp[1][0] = 0                          # {0}만 방문, 현재 위치 0
    for mask in range(1 << n):
        for last in range(n):
            cur = dp[mask][last]
            if cur == INF or not (mask & (1 << last)):
                continue
            for nxt in range(n):
                if mask & (1 << nxt):      # 이미 방문했으면 건너뛴다
                    continue
                nmask = mask | (1 << nxt)
                cand = cur + dist[last][nxt]
                if cand < dp[nmask][nxt]:
                    dp[nmask][nxt] = cand
    return min(dp[FULL][last] + dist[last][0] for last in range(n))
```

`itertools.permutations`를 쓴 브루트포스와 $n=4$부터 $8$까지 무작위 거리 행렬로 대조하면 **모든 경우에서 답이 정확히 일치한다.** 그리고 이제 브루트포스가 손도 못 대는 크기에서 시간을 재 보자.

| n | bitmask DP 시간 | 상태 공간 $2^n \times n$ |
| --- | --- | --- |
| 10 | 0.0025초 | 10,240 |
| 14 | 0.0523초 | 229,376 |
| 16 | 0.2746초 | 1,048,576 |
| 18 | 1.3550초 | 4,718,592 |

(Python 3.14.5 / Windows 기준 실측.)

$n=18$까지도 1.4초 안에 끝난다. 같은 $n=18$을 순열로 돌면 $17! \approx 3.6 \times 10^{14}$가지라 우주가 끝날 때까지도 못 끝낸다. $n=20$으로 넓혀 봐도 상태 공간은 $2^{20} \times 20 \approx 2.1 \times 10^7$로 여전히 다룰 만한 크기지만, 순열 개수 $19! \approx 1.2 \times 10^{17}$는 비교조차 무의미하다. 이것이 "순서 전체가 아니라 방문한 집합만 상태로 삼는다"는 관찰이 만들어 낸 차이다.

::: cote 비트마스크 DP, 어디서 또 나오는가
외판원 순회는 비트마스크 DP의 가장 유명한 예일 뿐이다. 실전에서 자주 만나는 다른 패턴들:

- **집합 분할**: $n$개 원소를 몇 개의 그룹으로 나누는 최소/최대 비용 (배낭의 일반화)
- **작업 배정**: $n$명의 일꾼에게 $n$개의 일을 배정하는 최소 비용 (이분 매칭의 DP 버전)
- **상태에 "지금까지 어떤 조건을 만족했는가"를 압축**해야 하는 모든 문제

공통점은 하나다. **원소 개수 $n$이 작고(대개 $\le 20$), 순서가 아니라 "무엇을 골랐는가"가 중요하다.** 비트마스크 DP의 다른 유형들(배낭, LIS, LCS, 구간 DP와의 조합)은 [7.21 동적 계획법 심화](#/dp-advanced)에서 이어서 다룬다. 실전 문제로 손을 풀고 싶다면 [8.8 실전 풀이 III — DP](#/drill-dp)로 가라.
:::

## 파이썬 int는 왜 비트마스크 크기 제한이 사실상 없는가

C나 Java에서 비트마스크를 정수로 표현하면 `int`(32비트)나 `long`(64비트)에 갇힌다. 원소가 64개를 넘으면 `bitset`이나 `BigInteger` 같은 별도 자료구조가 필요하다. 파이썬은 다르다. [1.2 숫자와 수치 연산](#/numbers)에서 봤듯 `int`는 **임의 정밀도**라, 비트 수가 몇 개든 그냥 정수 하나로 표현된다.

```pyrepl
>>> mask = 0
>>> for i in [3, 50, 100, 150, 199]:
...     mask |= (1 << i)
>>> mask.bit_length()
200
>>> import sys
>>> sys.getsizeof(mask)
52
```

200개의 도시가 있어도, 심지어 20,000개의 원소가 있어도 마스크는 그냥 정수 하나다. `1 << i`도, `mask | (1 << i)`도 코드가 전혀 달라지지 않는다.

::: perf "제한 없음"이 "공짜"라는 뜻은 아니다
CPython의 `int`는 내부적으로 30비트씩 묶은 "digit" 배열이다([1.2 숫자와 수치 연산](#/numbers)). 비트 수가 늘면 이 배열도 늘어나고, 비트 연산은 배열을 훑어야 하니 비트 수에 비례해서 느려진다.

```python
def bench(bits, iters=200_000):
    m = (1 << bits) - 1
    for _ in range(iters):
        m | 1
```

| 비트 수 | `getsizeof` | OR 연산 20만 회 시간 |
| --- | --- | --- |
| 30 | 28 bytes | 0.0027초 |
| 300 | 64 bytes | 0.0050초 |
| 3,000 | 424 bytes | 0.0069초 |
| 30,000 | 4,024 bytes | 0.0128초 |

(Python 3.14.5 / Windows 기준 실측.)

메모리는 비트 수에 거의 선형으로 늘고, 연산 시간도 비트 수가 1,000배 늘 때 5배 정도 늘어난다 — 완전한 상수는 아니지만 배열 길이에 비례하는 선형 비용이다. **코딩테스트에서 흔한 $n \le 20{\sim}30$ 정도의 비트마스크는 CPU 워드 하나(64비트)에 통째로 들어가므로 사실상 상수 시간이라고 봐도 된다.** 원소가 수만 개를 넘어가는 비트셋을 다뤄야 한다면 그때는 이 선형 비용이 실제로 체감된다.
:::

::: tip 언제 set 대신 비트마스크를 쓰는가
`set`도 원소 추가·확인·삭제가 평균 $O(1)$이니 얼핏 비슷해 보인다. 하지만 DP의 상태로 쓸 때는 차이가 크다.

- **정수는 리스트의 인덱스로 바로 쓸 수 있다.** `dp[mask][last]`처럼. `set`은 해시할 수 있어도 인덱스로는 못 쓴다.
- **정수 비교·복사가 훨씬 싸다.** `set` 두 개를 비교하거나 복사하는 비용은 원소 수에 비례하지만, 정수는 그냥 정수다.
- **원소 개수가 작을 때(대개 $\le 60$)는 비트 연산이 파이썬 객체 오버헤드가 있는 `set` 연산보다 빠르다.**

반대로 원소 종류가 정수가 아니거나(문자열, 튜플 등), 원소 개수가 아주 많거나 동적으로 늘어난다면 `set`이 낫다. [7.7 set과 frozenset](#/sets)에서 `set`의 내부 구조를 다시 보라.
:::

## 요약

- 정수의 각 비트를 원소 하나에 대응시키면 추가(`|`)·삭제(`& ~`)·포함 확인(`>> & 1`)·토글(`^`)이 전부 비트 연산 하나로 끝난다.
- `& (1<<i)`의 결과는 0 아니면 $2^i$이지 1이 아니다 — `== 1`과 비교하지 말고 `bool()`로 감싸거나 오른쪽 시프트 뒤 비교하라.
- `range(1 << n)`으로 전체 부분집합을 순회할 수 있다. 비용은 $O(2^n)$이라 $n \le 20$ 언저리가 현실적 한계다.
- `n & -n`은 최하위 1비트만 남긴다 — 마스크에서 원소를 하나씩 꺼낼 때 쓰고, 펜윅 트리에서도 같은 트릭이 쓰인다([7.25 세그먼트 트리와 펜윅 트리](#/segment-tree)).
- `sub = (sub-1) & mask` 로 한 마스크의 모든 submask를 돌 수 있다. 전체 마스크에 대해 이걸 다 하면 총 비용은 $O(3^n)$이다.
- 상태를 "방문 순서 전체"가 아니라 "방문한 집합 + 현재 위치"로 압축하면 $(n-1)!$짜리 문제가 $O(2^n n^2)$짜리 문제가 된다 — 외판원 순회의 비트마스크 DP로 실측 확인했다.
- 파이썬 `int`는 임의 정밀도라 비트마스크 크기 제한이 사실상 없다. 다만 비트 수가 커지면 연산 비용도 선형으로 커진다 — 공짜 확장은 아니다.
