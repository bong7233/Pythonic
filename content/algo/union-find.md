# 7.12 유니온 파인드

::: lead
"이 두 노드는 같은 그룹에 속하는가?"라는 질문에 매번 그래프를 다시 훑어서 답하면 코딩테스트에서는 반드시 시간 초과가 난다. 유니온 파인드(union-find, disjoint set)는 이 질문을 **거의 상수 시간**에 답하게 해 주는 자료구조다. 아이디어는 단순하다 — 각 원소가 자기 그룹의 대표(루트)를 가리키게 하고, 대표가 같으면 같은 그룹이라고 판단한다. 문제는 "거의 상수 시간"이 공짜로 나오지 않는다는 것이다. 이 절에서는 그 상수를 실제로 만들어 내는 두 가지 최적화 — 경로 압축과 union by rank — 를 직접 구현하고, 왜 그것들이 필요한지 실측으로 증명한다.
:::

## 브루트포스가 막히는 지점

다음 문제를 생각해 보자. 정점 $n$개짜리 그래프에 간선이 하나씩 추가된다. 매번 "지금 두 정점 $a$, $b$가 연결되어 있는가?"를 answer해야 한다. 백준 1976(여행가자), 20040(사이클 게임) 같은 문제의 뼈대다.

가장 직관적인 방법은 간선이 추가될 때마다 BFS/DFS로 두 정점이 같은 컴포넌트에 있는지 확인하는 것이다. 간선이 $m$개, 질의가 $q$번이면 매 질의마다 $O(n)$이 드니 전체 $O(qn)$이다. $n, q$가 각각 $10^5$면 $10^{10}$ — 확실한 시간 초과다.

핵심 관찰: 이 문제는 그래프 전체 구조가 아니라 **"누가 누구와 같은 그룹인가"만** 알면 된다. 간선 방향도, 실제 경로도 필요 없다. 이렇게 문제를 좁히면 훨씬 가벼운 자료구조로 충분하다 — 그게 분리집합이다.

::: cote 신호: 언제 유니온 파인드를 떠올려야 하는가
문제에 이런 조건이 보이면 유니온 파인드를 의심하라.

- "그룹으로 묶어라", "같은 팀/네트워크/집합인지 확인하라"
- 간선이 **하나씩 추가되기만 하고 삭제는 없다** (동적 연결성, 오프라인이 아니라면)
- 최종적으로 "몇 개의 그룹이 있는가", "이 둘을 이으면 사이클이 생기는가"를 묻는다
- 최소 신장 트리(크루스칼) 문제 — [7.16 최소 신장 트리](#/mst)에서 그대로 재사용한다

반대로 **간선이 삭제되기도 한다면** 유니온 파인드 기본형은 못 쓴다. 오프라인으로 시간을 거꾸로 처리하거나 다른 자료구조가 필요하다.

이런 신호를 문제 본문에서 빠르게 읽어 내는 훈련은 [8.1 코딩테스트의 구조와 전략](#/cote-strategy)에서 다룬다 — 제한 시간과 $n$의 크기만 봐도 "이 문제는 유니온 파인드다"를 확신할 수 있어야 시험장에서 시간을 아낀다.
:::

## 분리집합의 표현: parent 배열

분리집합은 각 그룹을 **트리**로 표현한다. 그룹의 대표(루트)는 자기 자신을 부모로 가리키는 노드다. 배열 하나로 전체 숲(forest)을 표현할 수 있다.

```text nolines
초기 상태 — 모든 원소가 독립된 그룹, 자기 자신이 루트다.

index : 0  1  2  3  4  5
parent: 0  1  2  3  4  5
```

연산은 두 개뿐이다.

- `find(x)`: `x`가 속한 그룹의 루트를 반환한다.
- `union(a, b)`: `a`가 속한 그룹과 `b`가 속한 그룹을 하나로 합친다.

가장 단순한 구현은 이렇다.

```python title="uf_naive.py — 최적화 없는 버전"
class NaiveUF:
    def __init__(self, n):
        self.parent = list(range(n))

    def find(self, x):
        while self.parent[x] != x:
            x = self.parent[x]
        return x

    def union(self, a, b):
        ra, rb = self.find(a), self.find(b)
        if ra == rb:
            return False          # 이미 같은 그룹 -> 합칠 필요 없음
        self.parent[ra] = rb      # a의 루트를 b의 루트 밑에 건다
        return True
```

`find`는 부모를 따라 루트까지 올라가고, `union`은 한쪽 루트를 다른 쪽 루트에 매단다. 코드는 짧지만 함정이 하나 있다. **매다는 방향을 아무렇게나 정하면 트리가 한쪽으로 길게 늘어질 수 있다.**

```text nolines
union(0,1), union(1,2), union(2,3), union(3,4) 를 순서대로 처리하면:

0 ──▶ 1 ──▶ 2 ──▶ 3 ──▶ 4      <- 사슬(chain). 깊이가 n에 비례한다.
```

이 상태에서 `find(0)`을 부르면 `parent`를 4번 타고 올라가야 한다. $n$개를 이런 순서로 이으면 트리의 깊이가 $n$에 비례하고, `find` 한 번이 $O(n)$이 된다. `union`도 내부에서 `find`를 두 번 부르니 마찬가지다.

## 실측: 최적화 없이는 얼마나 느린가

말로만 하지 말고 재 보자. 사슬을 만든 뒤 무작위 위치에서 `find`를 2,000번 호출해서 걸리는 시간을 잰다.

```python title="uf_bench.py — 사슬을 만든 뒤 find 2,000회"
def worst_case_chain_ops(n):
    return [(i, i + 1) for i in range(n - 1)]

def bench(cls, n, n_finds):
    uf = cls(n)
    for a, b in worst_case_chain_ops(n):
        uf.union(a, b)
    for q in random_queries(n, n_finds):
        uf.find(q)
```

| n | Naive (최적화 없음) | 배수 |
| --- | --- | --- |
| 1,000 | 0.0189초 | - |
| 5,000 | 0.0921초 | 4.9x |
| 10,000 | 0.1975초 | 2.1x |
| 50,000 | 0.9699초 | 4.9x |
| 100,000 | 1.9229초 | 2.0x |

(Python 3.14.5 / Windows 기준 실측, `find` 호출 횟수는 2,000회로 고정. 절대값은 기기마다 다르지만 **`n`이 10배 늘 때 시간도 거의 10배 는다는 추세**는 어디서나 같다.)

`find` 호출 횟수를 고정했는데도 시간이 `n`에 비례해서 늘어난다. 사슬의 평균 깊이가 `n`에 비례하기 때문이다. 코딩테스트에서 $n=10^5$, 질의가 $10^5$번이면 이 구현은 그대로 시간 초과다.

::: warn 매다는 순서가 성능을 결정한다
방금 본 사슬은 **최악의 union 순서**를 일부러 골라서 만든 것이다. 운이 좋으면 트리가 안 늘어질 수도 있다. 문제는 코딩테스트에서 입력이 항상 운 좋게 들어온다는 보장이 없다는 것이다. **입력에 의존하지 않는 성능**을 만들어야 하고, 그게 다음에 나오는 union by rank다.
:::

## 최적화 1: union by rank (또는 size)

아이디어는 단순하다. **항상 낮은 트리를 높은 트리 밑에 붙인다.** 그러면 트리 높이가 함부로 늘지 않는다.

높이를 직접 추적하는 대신 **랭크**(rank, 대략적인 높이의 상한)를 쓴다. 랭크가 같은 두 트리를 합칠 때만 랭크가 1 늘고, 그 외에는 안 늘어난다.

```python title="uf_rank.py — union by rank 추가"
class RankOnlyUF:
    def __init__(self, n):
        self.parent = list(range(n))
        self.rank = [0] * n

    def find(self, x):
        while self.parent[x] != x:
            x = self.parent[x]
        return x

    def union(self, a, b):
        ra, rb = self.find(a), self.find(b)
        if ra == rb:
            return False
        if self.rank[ra] < self.rank[rb]:
            ra, rb = rb, ra        # ra가 항상 랭크가 크거나 같은 쪽
        self.parent[rb] = ra
        if self.rank[ra] == self.rank[rb]:
            self.rank[ra] += 1
        return True
```

이렇게만 해도 트리 높이가 $O(\log n)$으로 묶인다. 증명의 핵심은 이렇다 — 랭크가 $k$인 트리가 만들어지려면 랭크 $k-1$짜리 두 개가 합쳐져야 하고, 그러려면 최소 $2^k$개의 원소가 필요하다. 그러니 $n$개 원소로는 랭크가 $\log_2 n$을 넘을 수 없다.

**size 기준으로 union하는 것도 결과는 같다.** 랭크 대신 각 그룹의 원소 개수를 세고, 작은 쪽을 큰 쪽에 붙인다. 논리는 동일하고, 덤으로 "이 그룹에 몇 개가 들어 있는가"를 바로 알 수 있다는 장점이 있다 — 백준 4195(친구 네트워크)처럼 매 union 직후 그룹 크기를 answer해야 하는 문제에 딱 맞는다.

## 최적화 2: 경로 압축

union by rank만으로는 높이가 $O(\log n)$이라 이미 실용적으로 충분히 빠르다. 하지만 **경로 압축**(path compression)을 더하면 사실상 상수 시간까지 내려간다.

아이디어: `find(x)`로 루트를 찾고 나면, 그 경로 위에 있던 모든 노드를 **루트에 바로 붙여 버린다.** 다음번에 그 노드들을 다시 찾을 때는 한 걸음이면 된다.

```text nolines
압축 전 (사슬):                압축 후 find(0) 호출 결과:

0 ──▶ 1 ──▶ 2 ──▶ 3 ──▶ 4        0 ──┐
                                  1 ──┤
                                  2 ──┼──▶ 4
                                  3 ──┘
```

재귀로 짜면 한 줄이지만, **사슬 깊이가 $10^5$면 재귀 깊이도 $10^5$라 `RecursionError`가 난다.** 그래서 반복문 버전을 쓴다.

```python title="uf_full.py — 경로 압축 + union by rank"
class UnionFind:
    def __init__(self, n):
        self.parent = list(range(n))
        self.rank = [0] * n
        self.count = n              # 현재 그룹(연결 요소) 개수

    def find(self, x):
        root = x
        while self.parent[root] != root:   # 1단계: 루트를 찾는다
            root = self.parent[root]
        while self.parent[x] != root:      # 2단계: 경로 위 모든 노드를 루트에 직결
            self.parent[x], x = root, self.parent[x]
        return root

    def union(self, a, b):
        ra, rb = self.find(a), self.find(b)
        if ra == rb:
            return False
        if self.rank[ra] < self.rank[rb]:
            ra, rb = rb, ra
        self.parent[rb] = ra
        if self.rank[ra] == self.rank[rb]:
            self.rank[ra] += 1
        self.count -= 1
        return True

    def connected(self, a, b):
        return self.find(a) == self.find(b)
```

::: warn 재귀 경로 압축은 깊은 사슬에서 터진다
흔히 보는 교과서 코드는 이렇다.

```python
def find(self, x):
    if self.parent[x] != x:
        self.parent[x] = self.find(self.parent[x])  # ❌ n이 크면 RecursionError
    return self.parent[x]
```

한 줄로 예쁘지만, union by rank 없이 이 코드만 쓰면 사슬이 만들어질 수 있고 재귀 깊이가 그대로 노출된다. `sys.setrecursionlimit`을 늘려서 땜질할 수도 있지만, 반복문 버전(위 `UnionFind.find`)을 쓰는 게 근본적인 해결책이다. 이 책에서는 항상 반복문 버전을 쓴다.
:::

## 실측: 경로 압축 + union by rank의 효과

같은 사슬-생성 시나리오에 rank만 적용한 버전과, rank+경로 압축 둘 다 적용한 버전을 추가로 재 보자.

| n | Naive | RankOnly | Full (rank+압축) |
| --- | --- | --- | --- |
| 1,000 | 0.0220초 | 0.0005초 | 0.0005초 |
| 5,000 | 0.1111초 | 0.0003초 | 0.0004초 |
| 10,000 | 0.2152초 | 0.0003초 | 0.0006초 |
| 50,000 | 1.1777초 | 0.0003초 | 0.0003초 |
| 100,000 | 2.4466초 | 0.0004초 | 0.0004초 |

(Python 3.14.5 / Windows 기준 실측. 같은 사슬 생성 순서, `find` 2,000회 기준.)

이 표를 보고 "RankOnly와 Full이 거의 똑같으니 경로 압축은 이 시나리오에서 별 효과가 없구나, union by rank가 이미 높이를 $O(\log n)$으로 눌러놨으니까"라고 결론 내리고 싶어진다. **실제로 높이를 재보면 이 결론은 틀렸다.**

```python title="height_check.py — 사슬 순서로 union한 뒤 실제 트리 높이 측정"
def tree_height(uf, n):
    h = 0
    for i in range(n):
        d, x = 0, i
        while uf.parent[x] != x:
            x = uf.parent[x]
            d += 1
        h = max(h, d)
    return h
```

`union(0,1), union(1,2), union(2,3), ...` 순서로 사슬을 만든 뒤 `RankOnlyUF`의 실제 높이를 재면, $n=1{,}000$부터 $100{,}000$까지 **전부 높이가 1**이다. $O(\log n)$이 아니라 완전히 평평한 별(star) 모양이 된다. 이유는 union by rank의 타이브레이크 규칙에 있다 — `union(a, b)`를 부를 때 두 루트의 랭크가 같으면 그중 하나(코드에서는 `rb`)를 다른 하나 밑에 붙이고 랭크를 1 올린다. 사슬 순서에서는 항상 "이미 랭크가 더 크거나 같은 기존 루트" 밑에 새 원소 하나가 매번 그대로 붙기 때문에, 루트의 랭크만 계속 올라갈 뿐 트리 높이는 처음부터 끝까지 1에 머문다. 즉 **이 union 순서는 union-by-rank 입장에서 전혀 적대적이지 않은 입력**이라, RankOnly 혼자만으로도 사실상 $O(1)$이 되어 버려서 경로 압축이 끼어들 여지가 없다. `n`이 늘어도 rank/full 두 열이 모두 0.0003~0.0006초 근방(측정 잡음 수준)에 머무는 이유가 이것이다 — 반면 Naive는 사슬 그대로 깊이가 `n`에 비례해서 늘어나므로 시간도 그만큼 는다. 그 결과 Naive 대비 배수는 고정된 "약 117배"가 아니라 **`n`이 커질수록 함께 커진다** — $n=1{,}000$에서 약 40배였던 것이 $n=100{,}000$에서는 약 6,000배 넘게 벌어진다. RankOnly와 Full의 시간이 서로 거의 구별 안 되는 것도 우연이 아니라, 둘 다 같은 별 모양 트리에서 출발하기 때문이다.

그렇다면 경로 압축이 진짜로 효과를 내는 상황은 어떻게 만드나? **union-by-rank 타이브레이크에 유리하지 않은, 진짜로 균형 잡힌 병합 순서**를 쓰면 된다 — 인접한 원소끼리 먼저 짝지어 합치고, 그 결과를 다시 짝지어 합치는 식으로 이진 트리처럼 병합하면 각 병합마다 양쪽 트리의 랭크가 똑같이 커지면서 실제로 $O(\log n)$ 높이가 나온다.

```python title="balanced_merge.py — 실제로 O(log n) 높이가 나오는 병합 순서"
def balanced_merge_ops(n):
    ops, groups = [], list(range(n))
    while len(groups) > 1:
        nxt = []
        for i in range(0, len(groups) - 1, 2):
            ops.append((groups[i], groups[i + 1]))
            nxt.append(groups[i])
        if len(groups) % 2 == 1:
            nxt.append(groups[-1])
        groups = nxt
    return ops
```

| n | RankOnly 높이 | $\log_2 n$ |
| --- | --- | --- |
| 1,000 | 9 | 10.0 |
| 5,000 | 12 | 12.3 |
| 10,000 | 13 | 13.3 |
| 50,000 | 15 | 15.6 |
| 100,000 | 16 | 16.6 |

이번엔 실측대로 $O(\log n)$이 나온다. 이 균형 트리에서 같은 2,000개 질의를 30라운드 반복해서 조회하면(즉 같은 노드를 계속 다시 찾으면) 경로 압축의 효과가 비로소 드러난다.

| n | RankOnly (30라운드) | Full (30라운드) | 배수 |
| --- | --- | --- | --- |
| 10,000 | 0.0099초 | 0.0040초 | 약 2.5배 |
| 100,000 | 0.0178초 | 0.0040초 | 약 4.5배 |

(Python 3.14.5 / Windows 기준 실측, best-of-5.)

RankOnly는 매 조회마다 높이 $O(\log n)$짜리 경로를 다시 타고 올라가야 하므로 `n`이 커질수록 조회 시간도 함께 늘어난다. Full은 첫 라운드에서 한 번 경로를 압축해 놓으면 그다음부터는 거의 한 걸음으로 끝나서 `n`이 커져도 시간이 거의 그대로다(약 0.0040초 근방 고정). **경로 압축의 진짜 위력은 사슬을 딱 한 번씩만 조회하는 상황이 아니라, 같은 원소를 반복해서 조회하는 상황에서 나온다** — 그리고 그 효과를 보려면 union by rank에 유리하지 않은(즉 진짜로 $O(\log n)$ 높이가 나오는) union 순서가 먼저 필요하다.

::: perf 총 연산 횟수를 늘려도 연산당 비용이 늘지 않는다
이번엔 다른 실험이다. `union` $n-1$번과 `find` $n$번을 무작위 순서로 섞어서 실행하고, 연산 하나당 평균 시간을 잰다.

| n | 총 시간 | 연산당 시간 | 이전 대비 배수 |
| --- | --- | --- | --- |
| 1,000 | 0.0004초 | 196.2ns | - |
| 10,000 | 0.0053초 | 266.3ns | 13.6x |
| 100,000 | 0.0531초 | 265.5ns | 10.0x |
| 1,000,000 | 1.1231초 | 561.5ns | 21.1x |

(Python 3.14.5 / Windows 기준 실측.)

$n$이 10배씩 늘 때 총 시간도 대략 10~21배 느는 정도다 — **초선형이 아니라 거의 선형이다.** 연산 하나당 걸리는 시간(약 170~560ns)은 $n$이 1,000배 늘어도 몇 배 수준 안에서만 움직이고, `n`에 비례해서 커지지는 않는다(측정 환경에 따라 상수 항의 흔들림 폭은 달라질 수 있다). 이게 "경로 압축 + union by rank를 같이 쓰면 연산 하나가 사실상 상수 시간"이라는 주장의 실측 근거다.
:::

## 왜 "사실상 상수"인가 — 역 아커만 함수

경로 압축과 union by rank를 함께 쓰면, $n$개의 원소에 대해 $m$번의 연산을 수행하는 총 비용은

$$O((n + m)\,\alpha(n))$$

이다. 여기서 $\alpha(n)$은 **역 아커만 함수**(inverse Ackermann function)로, 아커만 함수의 역함수다.

::: deep 역 아커만 함수는 얼마나 느리게 자라는가
아커만 함수 $A(n)$은 상상하기 힘들 만큼 빨리 자란다 — $A(4)$만 해도 우주에 있는 원자 수보다 큰 자릿수를 가진다. 역함수 $\alpha(n)$은 그 반대로, **상상하기 힘들 만큼 느리게 자란다.**

구체적으로, $\alpha(n) \le 4$가 성립하는 $n$의 범위는 $n$이 $2^{2^{2^{65536}}}$ (2를 65536번 거듭제곱한 탑)보다 작을 때까지다. 우주에 존재하는 원자 수가 대략 $10^{80}$인데, 이 값은 그것과 비교도 안 되게 크다.

**결론: 현실에서 등장하는 모든 $n$에 대해 $\alpha(n) \le 4$다.** 그래서 이론적으로는 "$O(\alpha(n))$"이라고 쓰지만, 실무에서는 "**사실상 상수 시간**"이라고 불러도 무방하다. Tarjan(1975)이 이 상계를 증명했다. 증명 자체는 이 책의 범위를 넘으니, 여기서는 **경로 압축 하나만으로도 이미 충분히 빠르고, union by rank까지 더하면 실측으로 확인한 것처럼 상수에 가까운 성능이 나온다**는 사실만 기억하면 된다.
:::

## 정답 검증: 브루트포스와 대조

구현이 맞는지 손으로 하나하나 확인하는 대신, **집합의 리스트**로 만든 브루트포스와 무작위 연산을 대조한다.

```python title="uf_check.py — 브루트포스 대조 검증"
def brute_force_check(n, ops):
    groups = [{i} for i in range(n)]         # 그룹을 집합의 리스트로 표현

    def bf_union(a, b):
        ga = next(g for g in groups if a in g)
        gb = next(g for g in groups if b in g)
        if ga is gb:
            return False
        ga |= gb
        groups.remove(gb)
        return True

    uf = UnionFind(n)
    for kind, a, b in ops:
        if kind == "union":
            assert bf_union(a, b) == uf.union(a, b)
        else:
            ga = next(g for g in groups if a in g)
            assert (b in ga) == uf.connected(a, b)
    assert len(groups) == uf.count
```

5회 무작위 시행($n$은 5~40, 연산은 10~80개)을 돌려 본 결과 **전부 일치**했다(`trial 0: n=30, ops=73 -> OK` 형태로 5건 모두 통과). 이 패턴 — "느리지만 명백히 맞는 구현과 대조한다" — 은 정렬, 그래프 탐색 등 다른 알고리즘을 검증할 때도 그대로 쓸 수 있다. [7.1 시간·공간 복잡도](#/complexity)에서 다룬 "일단 맞게, 그다음 빠르게"의 실천이다.

## 응용 1: 사이클 검출과 크루스칼 예고

무방향 그래프에서 간선 $(a, b)$를 추가하기 직전에 `find(a) == find(b)`이면, $a$와 $b$는 **이미 같은 그룹에 속해 있다는 뜻**이고, 이 간선을 더하면 사이클이 생긴다.

```python title="cycle.py — union의 반환값이 곧 사이클 여부다"
def has_cycle(n, edges):
    uf = UnionFind(n)
    for a, b in edges:
        if not uf.union(a, b):     # union이 False면 이미 같은 그룹이었다는 뜻
            return True
    return False
```

DFS로 부모 노드를 제외하고 이미 방문한 정점을 다시 만나는지 확인하는 방식과 결과를 대조하면 무작위 그래프 20개에서 **모두 일치**한다. 유니온 파인드 쪽이 코드가 훨씬 짧고, 간선이 스트리밍으로 들어오는 상황(전체 그래프를 한 번에 안 주는 경우)에 특히 유리하다.

이 패턴이 그대로 **크루스칼 최소 신장 트리 알고리즘**의 핵심이다. 간선을 가중치 순으로 정렬한 뒤, 사이클을 만들지 않는 간선만 순서대로 채택하면 된다 — "사이클을 만들지 않는다"를 확인하는 부분이 정확히 지금 만든 `union`이다. 자세한 것은 [7.16 최소 신장 트리](#/mst)에서 이어 간다.

## 응용 2: 그룹 개수 세기

`count` 필드 하나로 "지금까지 몇 개의 서로 다른 그룹이 있는가"를 $O(1)$에 답할 수 있다. `union`이 성공(두 그룹이 실제로 합쳐짐)할 때마다 1씩 줄이면 된다.

```python title="groups.py"
def count_groups(n, edges):
    uf = UnionFind(n)
    for a, b in edges:
        uf.union(a, b)
    return uf.count
```

BFS로 컴포넌트 개수를 세는 것과 무작위 그래프 20개에서 대조한 결과 **모두 일치**한다. 백준 1717(집합의 표현), 1976(여행가자) 같은 문제가 정확히 이 패턴이다.

::: cote 자주 나오는 함정 세 가지
1. **인덱스 기준을 맞춰라.** 문제가 정점을 1번부터 준다면 `parent` 배열을 `n+1` 크기로 만들고 0번은 안 쓰거나, 아예 입력을 받을 때 1을 빼서 0-based로 통일하라. 절반의 실수가 여기서 나온다.
2. **`union`의 반환값을 버리지 마라.** "사이클 검출"과 "그룹 개수 세기" 둘 다 `union`이 **실제로 합쳤는지 아닌지**에 의존한다. 이미 같은 그룹인데 다시 합치려는 시도를 성공으로 잘못 세면 답이 틀린다.
3. **초기화를 매 테스트 케이스마다 새로 하라.** 여러 테스트 케이스가 한 번에 들어오는 문제(예: 여러 개의 그래프)에서 `UnionFind` 인스턴스를 재사용하면 이전 케이스의 그룹 정보가 남아 틀린 답이 나온다.
:::

::: tip 표준 라이브러리에는 없다 — 직접 짜서 외워 둬라
파이썬 표준 라이브러리에는 유니온 파인드가 없다. [7.6 해시](#/hashing)의 `Counter`, `defaultdict`처럼 가져다 쓸 수 있는 게 아니라 **매번 직접 구현**해야 한다. 이 절의 `UnionFind` 클래스는 그대로 외워서 코딩테스트에 들고 들어갈 만한 20줄짜리 템플릿이다. [8.10 실전 템플릿 모음](#/templates)에 다시 정리된다.
:::

## 요약

- 분리집합은 각 그룹을 트리로 표현하고, `parent` 배열 하나로 전체 숲을 나타낸다.
- 최적화 없이는 union 순서에 따라 트리가 사슬처럼 늘어져 `find`가 $O(n)$까지 느려진다 — 실측으로 확인했다.
- **union by rank(또는 size)**: 낮은 트리를 높은 트리 밑에 붙여 높이를 $O(\log n)$으로 묶는다.
- **경로 압축**: `find`가 지나간 경로의 모든 노드를 루트에 직결한다. 반복문으로 구현해야 깊은 사슬에서 재귀 깊이 문제가 안 생긴다.
- 둘을 함께 쓰면 $m$번의 연산이 $O((n+m)\alpha(n))$ — 역 아커만 함수는 현실적인 모든 $n$에서 4 이하이므로 **사실상 상수 시간**이다.
- `union`의 반환값(성공/실패)이 곧 **사이클 검출**의 근거이고, 이는 [7.16 최소 신장 트리](#/mst)의 크루스칼 알고리즘 그대로 이어진다.
- `count` 필드로 **그룹 개수 세기**를 $O(1)$에 처리한다.

2. 백준 스타일 입력 — 정점이 1번부터 $n$번까지 주어지고, 간선 $m$개가 주어진 뒤 "1번과 $n$번이 연결되어 있는가?"를 묻는 문제를 이 절의 `UnionFind`로 풀어라. 인덱스 처리에 특히 주의하라.
