# 7.8 힙과 우선순위 큐

::: lead
"가장 작은(또는 가장 큰) 원소를 계속 뽑아야 하는데, 그사이에 새 원소도 계속 들어온다." 이 문장이 나오면 힙이다. 정렬된 리스트를 유지하는 것도, 매번 `min()`을 부르는 것도 틀린 답은 아니지만 느리다. 이 절에서는 왜 느린지 직접 측정하고, `heapq`가 그 문제를 어떻게 $O(\log n)$으로 눌러 담는지, 그리고 힙을 코딩테스트에 쓸 때 반드시 걸려 넘어지는 함정 두 가지를 다룬다. [7.7 스택과 큐](#/stack-queue)가 "양 끝"만 신경 썼다면, 힙은 "가장 작은 것"에 집중한다.
:::

## 왜 힙이 필요한가 — 브루트포스로 당해 보기

작업 스케줄러를 만든다고 하자. 우선순위가 낮은 작업이 계속 들어오고(`push`), 그때그때 우선순위가 가장 높은(값이 가장 작은) 작업을 꺼내 처리한다(`pop`). 가장 순진한 구현은 리스트에 넣어 두고 꺼낼 때마다 `min()`으로 찾는 것이다.

```python title="brute_force_scheduler.py"
def brute_force(ops):
    buf = []
    for kind, val in ops:
        if kind == "push":
            buf.append(val)
        else:
            m = min(buf)      # 매번 전체를 훑는다 — O(n)
            buf.remove(m)     # 찾은 자리를 지우는 것도 O(n)
```

`push`는 $O(1)$이지만 `pop`은 매번 리스트 전체를 훑어야 한다. `push`와 `pop`이 번갈아 $n$번 일어나면 총 비용은 $O(n^2)$이다. 실제로 재 보자.

```python title="heap_motivate.py"
import random, time, heapq

def with_heap(ops):
    h = []
    for kind, val in ops:
        if kind == "push":
            heapq.heappush(h, val)
        else:
            heapq.heappop(h)
```

| n | brute_force | heap | 배율 |
| --- | --- | --- | --- |
| 2,000 | 0.0013초 | 0.0001초 | 8.9배 |
| 4,000 | 0.0048초 | 0.0003초 | 15.9배 |
| 8,000 | 0.0187초 | 0.0006초 | 29.6배 |
| 16,000 | 0.0755초 | 0.0013초 | 57.1배 |

(Python 3.14.5 / Windows 기준 실측. `push` 60%, `pop` 40% 확률로 뒤섞은 연산열.) $n$이 두 배가 될 때마다 브루트포스 시간은 네 배 가까이 뛴다 — $O(n^2)$의 전형적인 곡선이다. 반면 힙은 거의 그대로다. 이 차이가 이 절 전체의 존재 이유다.

::: cote 문제 신호
"실시간으로 최솟값/최댓값을 유지하며 삽입·삭제가 계속 일어난다"는 문장이 보이면 `heapq`부터 떠올려라. 백준 "카드 정렬하기", "최소 힙", 프로그래머스 "더 맵게" 같은 문제가 전형적이다. [8.4 문제 유형 분류](#/problem-signals)에서 이런 신호를 더 다룬다.
:::

## heapq: 배열 하나로 구현된 완전 이진 트리

`heapq`는 별도의 클래스를 주지 않는다. **평범한 파이썬 리스트를 힙 규칙에 맞게 관리해 주는 함수들의 모음**이다. 그 규칙은 하나뿐이다.

> **모든 노드의 값은 자식 노드의 값보다 작거나 같다.** (최소 힙 조건)

이 트리는 배열 인덱스만으로 표현된다. 인덱스 `i`의 자식은 `2*i+1`, `2*i+2`이고 부모는 `(i-1)//2`다. 포인터가 하나도 없다.

```text nolines
index:   0   1   2   3   4   5   6
value:   1   3   2   7   9   4   8

              1                <- index 0
           /     \
          3        2           <- index 1, 2
        /   \    /   \
       7     9  4     8        <- index 3, 4, 5, 6
```

부모(1)가 항상 자식(3, 2)보다 작다. 하지만 **형제 사이의 순서는 보장하지 않는다** — 3과 2 중 어느 게 더 작은지는 힙 조건과 무관하다. 그래서 힙은 "완전히 정렬된 목록"이 아니라 "최솟값을 빠르게 꺼내기 위한 최소한의 질서"만 유지한다. 정렬보다 훨씬 싸게 먹히는 이유가 여기 있다.

```pyrepl
>>> import heapq
>>> h = [5, 1, 8, 3, 9, 2]
>>> heapq.heapify(h)          # 리스트를 제자리에서 힙으로 바꾼다
>>> h
[1, 3, 2, 5, 9, 8]
>>> heapq.heappop(h)          # 항상 인덱스 0, 즉 최솟값
1
>>> h
[2, 3, 8, 5, 9]
>>> heapq.heappush(h, 0)
>>> h
[0, 3, 2, 5, 9, 8]
```

`heappop`이 꺼내는 값은 항상 `h[0]`이다. 원소를 빼고 나면 마지막 원소를 루트로 올린 뒤 자식과 비교하며 아래로 내려보낸다(sift-down). `heappush`는 끝에 넣고 부모와 비교하며 위로 올려보낸다(sift-up). 두 연산 모두 트리의 높이, 즉 $\log_2 n$ 번을 넘지 않는 비교로 끝난다.

::: note 힙은 정렬된 자료구조가 아니다
`h[1]`이 `h[2]`보다 작다는 보장은 없다. **오직 `h[0]`만 "전체 중 최솟값"임이 보장된다.** 힙을 순서대로 순회하고 싶다면 `heappop`을 반복해야 하고, 그건 사실상 힙 정렬(heap sort)이다.
:::

## heapify가 $O(n)$인 이유

리스트를 힙으로 바꾸는 방법은 두 가지다.

1. `heapq.heapify(lst)` — 한 번에 변환.
2. 빈 힙에 `heappush`를 $n$번 반복.

둘 다 결과는 같은 힙 조건을 만족하는 배열이 되지만, **비용은 다르다.** `heapify`는 $O(n)$, `heappush` $n$번은 최악의 경우 $O(n \log n)$이다. 직관과 어긋난다 — $n$개를 하나씩 넣는 게 왜 한 번에 넣는 것보다 이론상 더 비쌀 수 있는가?

답은 **sift-down과 sift-up이 비대칭**이라는 데 있다. `heapify`는 리스트의 뒷부분(리프에 가까운 쪽)부터 시작해 앞으로 오면서 각 노드를 sift-down한다. **리프 노드는 애초에 내려갈 곳이 없으므로 즉시 끝난다.** 그런데 전체 노드의 절반은 리프다. 4분의 1은 리프 바로 위(최대 1칸만 내려감), 8분의 1은 그 위(최대 2칸)... 이런 식으로 **깊이가 깊어질수록(=많이 움직일 수 있는 노드일수록) 그런 노드의 개수는 기하급수적으로 줄어든다.** 이 급수를 합하면 $O(n)$으로 수렴한다.

반면 `heappush`는 매번 **트리의 맨 아래에서 시작해 위로** 올라간다. 최악의 경우(계속 최솟값이 들어오는 경우) 매번 루트까지, 즉 $\log n$번 비교한다.

::: deep 실제로 비교 횟수를 세어 검증한다
`__lt__`를 오버로드해 비교가 몇 번 일어나는지 직접 세 볼 수 있다.

```python title="비교 횟수 계측"
import heapq

class Counted:
    count = 0
    __slots__ = ("v",)
    def __init__(self, v):
        self.v = v
    def __lt__(self, other):
        Counted.count += 1
        return self.v < other.v
```

**입력 순서와 무관하게 `heapify`는 선형이다.**

| n | 내림차순 heapify | 오름차순 heapify |
| --- | --- | --- |
| 1,000 | 1,490회 (1.49/n) | 1,985회 (1.99/n) |
| 10,000 | 14,990회 (1.50/n) | 19,986회 (2.00/n) |
| 100,000 | 149,988회 (1.50/n) | 199,983회 (2.00/n) |
| 1,000,000 | 1,499,986회 (1.50/n) | 1,999,980회 (2.00/n) |

$n$이 커져도 원소당 비교 횟수(`/n`)가 **일정한 상수에 수렴한다** — $O(n)$의 정의 그대로다. 입력 순서가 바뀌어도(내림차순 vs 오름차순) 상수만 다를 뿐 선형인 건 똑같다.

이번엔 `heappush`를 $n$번 반복한 경우다. **의도적으로 최악의 입력**(내림차순 — 새 원소가 항상 지금까지의 최솟값이라 매번 루트까지 올라감)을 넣으면 성장 곡선이 드러난다.

| n | heappush 합 | 원소당(c/n) | $\log_2 n$ |
| --- | --- | --- | --- |
| 1,000 | 7,987회 | 7.99 | 9.97 |
| 10,000 | 113,631회 | 11.36 | 13.29 |
| 100,000 | 1,468,946회 | 14.69 | 16.61 |
| 1,000,000 | 17,951,445회 | 17.95 | 19.93 |

원소당 비교 횟수가 $n$이 10배가 될 때마다 일정하게 늘어난다 — $\log n$과 나란히 움직인다. `heapify`의 "원소당 상수"와 다르게, `heappush` 반복은 **원소당 비용 자체가 $n$에 따라 계속 커진다.** 이게 $O(n)$과 $O(n \log n)$의 실측 차이다.

(단, 무작위 데이터에서는 `heappush` 한 번이 평균적으로 상수 시간에 가깝다 — 새로 들어온 값이 우연히 최솟값 근처일 확률이 낮기 때문이다. **최악의 경우에만 $O(\log n)$이 실제로 드러난다.** 코딩테스트에서 입력이 적대적으로 정렬돼 있을 수 있다는 걸 기억해라.)
:::

::: perf 그래서 실전에서는
$n$개짜리 리스트가 이미 있고 그걸 통째로 힙으로 만들 거라면 **반드시 `heapq.heapify(lst)`를 써라.** 빈 리스트에 `for x in lst: heappush(h, x)`를 도는 건 이론상으로도, 실측으로도 더 느리다.
:::

## heapq는 최소 힙만 지원한다 — 최대 힙 트릭

`heapq`에는 `heappop_max` 같은 함수가 없다. **오직 최소 힙**이다. 최댓값을 빠르게 뽑고 싶다면 값을 **부호 반전**해서 넣는다.

```pyrepl
>>> import heapq
>>> nums = [5, 1, 8, 3, 9, 2]
>>> min_heap = nums[:]
>>> heapq.heapify(min_heap)
>>> max_heap = [-x for x in nums]
>>> heapq.heapify(max_heap)
>>> [heapq.heappop(min_heap) for _ in range(3)]
[1, 2, 3]
>>> [-heapq.heappop(max_heap) for _ in range(3)]
[9, 8, 5]
```

넣을 때 부호를 뒤집고, 꺼낼 때 다시 뒤집는다. 그뿐이다. 튜플의 일부만 최대 힙으로 쓰고 싶을 때도 같은 트릭을 그 자리에만 적용한다 — 예를 들어 "값은 최대로, 인덱스는 최소로" 뽑고 싶으면 `(-value, index)`를 넣는다.

::: warn 왜 `heapq`는 애초에 최대 힙을 안 주는가
설계자는 최소 힙 하나만 제공하고 최대 힙이 필요하면 부호를 반전하라고 명시했다(공식 문서에 나온다). 최소/최대 힙을 별도 API로 나누면 코드가 두 배가 되고, 실수로 섞어 쓰는 버그(최소 힙 함수에 최대 힙을 넣는 등)가 더 잦아진다는 판단이다. 부호 반전은 한 줄이면 되니 API를 늘릴 이유가 없다는 것이다. `heapq.nlargest`/`heapq.nsmallest`는 내부에서 이 트릭과 동등한 처리를 이미 해 주므로, 상위 $k$개만 필요하면 직접 부호를 반전할 필요도 없다.
:::

## k번째 원소를 $O(n \log k)$에 구하기

"배열에서 $k$번째로 작은/큰 원소를 구하라"는 코딩테스트 단골 문제다. 브루트포스는 정렬 후 인덱싱, $O(n \log n)$이다. $k$가 작다면 이보다 빠르게 할 수 있다 — **크기 $k$로 고정된 힙만 유지하면 된다.**

$k$번째로 작은 값을 구하고 싶다면, **크기 $k$짜리 최대 힙**을 유지하면서 새 원소가 힙의 최댓값보다 작을 때만 교체한다. 힙에는 항상 "지금까지 본 것 중 가장 작은 $k$개"만 남고, 그 힙의 루트(최댓값)가 곧 $k$번째로 작은 값이다.

```python title="kth_smallest.py"
import heapq

def kth_smallest(nums, k):
    heap = [-x for x in nums[:k]]   # 앞 k개로 최대 힙 시작 (부호 반전)
    heapq.heapify(heap)
    for x in nums[k:]:
        if -x > heap[0]:            # x가 현재 힙의 최댓값보다 작다는 뜻
            heapq.heapreplace(heap, -x)   # pop + push를 한 번에
    return -heap[0]
```

`heapq.heapreplace`는 `heappop` 다음 `heappush`를 하는 것과 결과는 같지만 **힙 크기가 절대 $k+1$로 늘어나지 않아 한 번의 sift만 일어난다.** 힙 크기가 항상 $k$로 고정되므로 매 원소당 비용은 $O(\log k)$, 전체는 $O(n \log k)$다. $k \ll n$이면 $O(n \log n)$인 전체 정렬보다 훨씬 싸다.

```python title="검증: 힙과 전체 정렬이 같은 결과를 내는가"
import random
import heapq
import time

def k_smallest_via_heap(nums, k):
    heap = [-x for x in nums[:k]]      # 크기 k 최대 힙으로 "가장 작은 k개"를 유지
    heapq.heapify(heap)
    for x in nums[k:]:
        if -x > heap[0]:
            heapq.heapreplace(heap, -x)
    return sorted(-v for v in heap)

def k_smallest_via_sort(nums, k):
    return sorted(nums)[:k]

random.seed(0)
data = [random.randint(0, 10**9) for _ in range(2_000_000)]
k = 10

t0 = time.perf_counter()
heap_result = k_smallest_via_heap(data, k)
t1 = time.perf_counter()
sort_result = k_smallest_via_sort(data, k)
t2 = time.perf_counter()

assert heap_result == sort_result   # 두 방법이 같은 답을 내는지 직접 확인
print(f"heap: {t1 - t0:.4f}s  sort: {t2 - t1:.4f}s")
```

::: deep 실측: $O(n \log k)$ vs $O(n \log n)$
$n=2{,}000{,}000$, $k=10$짜리 배열에서 "가장 작은 10개"를 구하는 두 방법을 비교했다.

```text nolines
heap  (크기 k=10 유지) : 0.0399초
sort  (전체 정렬)       : 0.2914초
비율                    : 7.30배
```

(Python 3.14.5 / Windows 기준 실측. 결과는 두 방법 모두 동일함을 `assert`로 확인.) $k$가 10처럼 작고 $n$이 클수록 차이는 더 벌어진다. $k$가 $n$에 근접하면 이 이점은 사라진다 — 그럴 땐 그냥 정렬하는 게 코드도 간단하고 상수도 작다.
:::

::: cote 손으로 만들지 말고 표준 라이브러리부터
지금 만든 로직은 사실 `heapq.nsmallest(k, iterable)`과 `heapq.nlargest(k, iterable)`이 이미 내부에서 거의 동일하게 처리해 준다. `key` 인자도 받는다.

```pyrepl
>>> import heapq
>>> data = [30, 5, 88, 2, 47, 19]
>>> heapq.nsmallest(3, data)
[2, 5, 19]
>>> heapq.nlargest(3, data, key=lambda x: -x)   # key로 뒤집기도 가능
[2, 5, 19]
```

**손으로 부호 반전 힙을 짜는 건 $k$가 고정 상수가 아니거나(예: 스트리밍 중 매번 다른 $k$), 커스텀 비교 로직·타이브레이크가 필요할 때만 하라.** 단순히 "가장 작은/큰 $k$개"가 필요하면 `nsmallest`/`nlargest`로 끝난다.
:::

## 튜플을 힙에 넣을 때 생기는 함정

우선순위 큐에는 보통 `(우선순위, 데이터)` 튜플을 넣는다. 문제는 **튜플 비교가 첫 원소가 같을 때 두 번째 원소로 넘어간다**는 것이다.

```pyrepl
>>> import heapq
>>> h = []
>>> heapq.heappush(h, (2, {"name": "a"}))
>>> heapq.heappush(h, (2, {"name": "b"}))
Traceback (most recent call last):
  ...
TypeError: '<' not supported between instances of 'dict' and 'dict'
```

우선순위(첫 원소)가 `2`로 같으니 힙은 **두 번째 원소끼리 비교**해서 순서를 정하려 한다. 그런데 딕셔너리는 `<`를 지원하지 않는다. 즉시 `TypeError`다. 운 나쁘게 두 번째 원소가 비교 가능한 타입(문자열, 숫자)이면 **에러는 안 나지만, 의도치 않은 기준으로 순서가 정해진다** — 우선순위가 같을 때 어느 게 먼저 나올지가 알파벳 순서 같은 엉뚱한 규칙에 좌우되는 것이다.

::: danger 우선순위가 같을 때 세 번째 요소로 타이브레이크를 넣어라
표준적인 해법은 **단조 증가하는 카운터**를 튜플 가운데에 끼워 넣는 것이다. 카운터는 항상 유일하므로 세 번째 원소(원본 데이터)까지 비교가 내려갈 일이 없다.

```python title="counter_tiebreak.py"
import heapq
import itertools

counter = itertools.count()
h = []
heapq.heappush(h, (2, next(counter), {"name": "a"}))
heapq.heappush(h, (2, next(counter), {"name": "b"}))
heapq.heappush(h, (1, next(counter), {"name": "c"}))

while h:
    print(heapq.heappop(h))
```

```text nolines
(1, 2, {'name': 'c'})
(2, 0, {'name': 'a'})
(2, 1, {'name': 'b'})
```

우선순위가 같은 `(2, ...)` 둘은 **먼저 넣은 순서(카운터 값)대로** 나온다 — 안정 정렬처럼 동작한다. 카운터가 항상 유일한 값을 주므로 딕셔너리까지 비교가 내려갈 일이 아예 없다.
:::

::: cote 실전 체크리스트
- 힙에 넣는 원소가 `(우선순위, 데이터)`이고 데이터가 비교 불가능한 객체(dict, 커스텀 클래스, 리스트)라면 **무조건 카운터를 끼워 넣어라.** 데이터가 문자열·숫자라서 지금 당장 에러가 안 난다고 안전한 게 아니다.
- 커스텀 객체를 우선순위 큐에 직접 넣고 싶다면 [1.14 특수 메서드](#/dunder)에서 다루는 `__lt__`를 정의해 비교 기준을 명시하는 방법도 있다. `dataclass(order=True)`를 쓰면 필드 순서대로 자동 비교가 생긴다.
- `heapq`는 힙 조건이 깨진 리스트를 넣어도 에러 없이 조용히 잘못된 결과를 낸다. 직접 리스트를 조작한 뒤에는 `heapify`를 다시 부르지 않으면 안 된다.
- 이런 함정들은 실전에서 "분명 로직은 맞는데 왜 틀렸지"라는 상태로 시간을 잡아먹는다. 제출 전 체크리스트로 미리 걸러내는 습관은 [8.1 코딩테스트의 구조와 전략](#/cote-strategy)에서 더 다룬다.
:::

## 다익스트라 예고 — 힙이 진짜 힘을 발휘하는 곳

지금까지는 힙을 "우선순위대로 꺼내는 큐"로만 썼다. 그래프 알고리즘에서는 한 단계 더 나아간다 — **"거리가 갱신될 때마다 갱신된 값으로 다시 넣고, 꺼낼 때 낡은 값이면 버린다"**(지연 삭제, lazy deletion)는 패턴이다.

```python title="dijkstra_preview.py"
import heapq

def dijkstra(graph, start):
    dist = {node: float("inf") for node in graph}
    dist[start] = 0
    pq = [(0, start)]
    while pq:
        d, u = heapq.heappop(pq)
        if d > dist[u]:
            continue          # 더 짧은 경로가 이미 반영됐다 — 낡은 항목 무시
        for v, w in graph[u]:
            nd = d + w
            if nd < dist[v]:
                dist[v] = nd
                heapq.heappush(pq, (nd, v))
    return dist
```

같은 노드가 여러 번 힙에 들어갈 수 있다는 게 핵심이다. 힙에서 그 노드를 찾아 값을 갱신하는 대신 **그냥 새 값을 또 넣고, 나중에 꺼낼 때 낡았으면 버린다.** 힙에 원소를 지우는 연산이 없다는 한계를 이렇게 우회한다. 왜 이 방식이 정확한지, 왜 이게 최선인지는 [7.15 최단 경로](#/shortest-path)에서 제대로 다룬다. 지금은 "힙 하나로 그래프 탐색의 우선순위를 관리한다"는 그림만 기억하면 된다.

## 요약

- `heapq`는 리스트를 힙 규칙(부모 ≤ 자식)에 맞게 관리하는 함수 모음이지 별도 자료구조가 아니다. `h[0]`이 항상 최솟값이다.
- 이미 있는 리스트를 힙으로 만들 땐 `heapify`(O(n))를 써라. `heappush`를 $n$번 반복하면 최악의 경우 $O(n \log n)$이다 — 실측으로도 차이가 드러난다.
- `heapq`는 최소 힙만 지원한다. 최대 힙은 부호를 반전해서 넣고 뺄 때 다시 반전한다.
- $k$번째 최소/최대 원소는 크기 $k$로 고정한 힙으로 $O(n \log k)$에 구한다. 표준 라이브러리에는 이미 `heapq.nsmallest`/`nlargest`가 있다.
- 튜플을 넣을 때 첫 원소가 같으면 두 번째 원소로 비교가 넘어간다. 데이터가 비교 불가능하면 `TypeError`, 비교 가능하면 의도치 않은 정렬이 일어난다. `itertools.count()`로 유일한 타이브레이커를 끼워라.
- 힙은 그래프 알고리즘에서 "매번 다시 넣고 낡은 항목은 버리는" 지연 삭제 패턴으로 확장된다. 다익스트라가 대표 사례다.
