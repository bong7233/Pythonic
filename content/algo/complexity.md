# 7.1 시간·공간 복잡도

::: lead
이제부터는 코딩테스트다. 정답을 맞히는 것과 **시간 안에** 맞히는 것은 다른 문제고, 파이썬으로 그 둘을 다 하려면 빅오를 제대로 알아야 한다. 그런데 "빅오만 알면 된다"는 것도 틀린 말이다. 같은 $O(n)$끼리도 실전에서 몇 배씩 차이 나고, 코딩테스트 커뮤니티에 떠도는 "1초에 1억 번" 공식은 파이썬에서 그대로 믿으면 시간 초과를 받는다. 이 절은 빅오의 정의부터 시작해서, 그 정의가 숨기는 것들 — 상수, 인터프리터 오버헤드, 재귀 깊이 — 까지 실측으로 채운다.
:::

## 브루트포스가 왜 막히는가

문제 하나로 시작하자. 정수 배열에서 합이 `target` 이 되는 두 원소의 인덱스를 찾는다 (Two Sum). 가장 먼저 떠오르는 풀이는 모든 쌍을 다 본다.

```python title="brute force — O(n^2)"
def two_sum_brute(nums, target):
    n = len(nums)
    for i in range(n):
        for j in range(i + 1, n):
            if nums[i] + nums[j] == target:
                return (i, j)
    return None
```

정답이라는 확신을 실제로 검증부터 하자. 무식하게 다 보는 이 버전을 기준(oracle)으로 놓고, 뒤에 나올 최적화 버전과 무작위 입력에서 결과를 대조한다.

```python title="정답 대조"
import random

def two_sum_hash(nums, target):
    seen = {}
    for i, x in enumerate(nums):
        need = target - x
        if need in seen:
            return (seen[need], i)
        seen[x] = i
    return None

random.seed(0)
for _ in range(200):
    n = random.randint(2, 30)
    nums = random.sample(range(-50, 50), n)
    i, j = random.sample(range(n), 2)
    target = nums[i] + nums[j]
    r1 = two_sum_brute(nums, target)
    r2 = two_sum_hash(nums, target)
    assert nums[r1[0]] + nums[r1[1]] == target
    assert nums[r2[0]] + nums[r2[1]] == target
print("통과: 두 구현 모두 유효한 쌍을 찾는다")
```

```text
통과: 두 구현 모두 유효한 쌍을 찾는다
```

두 구현 다 맞다. 문제는 **속도**다. 실제로 입력 크기를 키우면서 재 보자. 매번 정답이 없는 최악의 경우(끝까지 다 훑어야 하는 경우)로 맞춘다.

| $n$ | brute force $O(n^2)$ | hash $O(n)$ |
| --- | --- | --- |
| 1,000 | 10.4 ms | — |
| 2,000 | 41.4 ms | — |
| 4,000 | 171.3 ms | — |
| 8,000 | 676.1 ms | — |
| 10,000 | — | 0.66 ms |
| 100,000 | — | 6.36 ms |
| 1,000,000 | — | 63.7 ms |
| 2,000,000 | — | 131.0 ms |

(Python 3.14.5 / Windows 기준 실측. 절대값은 기기마다 다르지만 자릿수 차이는 어디서나 같다.)

패턴이 보인다. brute force는 $n$ 이 두 배가 될 때마다 시간이 **네 배**가 된다(10.4 → 41.4 → 171.3 → 676.1, 매번 약 4배). hash 방식은 $n$ 이 열 배가 될 때 시간도 **열 배** 근처로 늘어난다(0.66 → 6.36 → 63.7, 매번 약 10배). 이게 바로 $O(n^2)$ 과 $O(n)$ 의 차이가 눈에 보이는 방식이다. $n$ 이 8,000만 돼도 brute force는 이미 0.7초에 육박한다. 백준 기준 시간 제한 1~2초짜리 문제에 $n = 10^5$ 가 나온다면, $O(n^2)$ 은 애초에 승산이 없다.

::: cote 제약 조건에서 시간복잡도를 역산하라
문제의 $n$ 범위를 보면 요구되는 복잡도가 거의 정해진다. 이건 [8.4 문제 유형 분류와 신호 읽기](#/problem-signals)에서 체계적으로 다루지만, 지금 감을 잡아 두면 좋다.

| $n$ 범위 | 통과 가능한 복잡도 |
| --- | --- |
| $n \le 10$ | $O(n!)$, $O(2^n \cdot n)$ |
| $n \le 20$ | $O(2^n)$ |
| $n \le 5{,}000$ | $O(n^2)$ |
| $n \le 100{,}000$ | $O(n \log n)$ |
| $n \le 10{,}000{,}000$ | $O(n)$ |
| $n$ 이 훨씬 크거나 시간 1초 이하 | $O(\log n)$, $O(1)$ |

$n \le 10^5$ 인데 이중 반복문이 떠오른다면, 그건 알고리즘을 잘못 골랐다는 신호다.
:::

## 빅오란 무엇인가

빅오는 **상한**(upper bound)을 말하는 표기법이다. 정의는 이렇다.

$$f(n) = O(g(n)) \iff \exists\, c > 0,\ n_0 \text{ 인 상수가 있어서 } n \ge n_0 \text{인 모든 } n \text{에 대해 } f(n) \le c \cdot g(n)$$

말로 풀면: **입력이 충분히 커진 뒤부터, $f(n)$ 은 $g(n)$ 에 어떤 상수를 곱한 값보다 절대 커지지 않는다.** 여기서 세 가지가 바로 따라 나온다.

1. **"충분히 커진 뒤"** — 작은 $n$ 에서는 빅오가 아무것도 보장하지 않는다. $n_0$ 이전 구간은 정의 밖이다.
2. **"상수를 곱한 값"** — 상수 $c$ 는 얼마든 커도 된다. 빅오는 그 상수가 얼마인지 말하지 않는다.
3. **상한**이라는 것 — $f(n) = O(n^2)$ 이라고 해도 실제로는 훨씬 빠를 수 있다(느슨한 상한). 정확한 증가율을 말하고 싶으면 $\Theta$ (빅세타)를 쓴다. 코딩테스트에서는 관용적으로 빅오를 "정확한 차수"처럼 쓰지만, 엄밀히는 상한일 뿐이다.

## 흔한 오해: 빅오는 최악을 말하지, 상수를 말하지 않는다

두 가지 오해가 항상 같이 온다.

**오해 1 — 빅오는 평균을 말한다.** 아니다. 별다른 언급이 없으면 빅오는 보통 **최악의 경우**(worst case)를 말한다. 퀵 정렬은 평균 $O(n \log n)$ 이지만 최악은 $O(n^2)$ 이다 ([7.4 정렬](#/sorting)에서 이유를 다룬다). 코딩테스트에서 "평균은 빠른데 최악에 걸리는" 입력을 일부러 주는 문제도 있다.

**오해 2 — $O(n)$ 이면 다 똑같이 빠르다.** 아니다. 빅오는 상수 $c$ 를 지워 버리기 때문에, 같은 $O(n)$ 이라도 그 $c$ 가 7배, 10배 차이 나면 실행 시간도 그만큼 차이 난다. 다음 절에서 실측한다.

## 실측: 같은 O(n)인데 왜 이렇게 다른가

리스트의 모든 원소를 더하는 두 가지 방법을 비교해 보자. **알고리즘 자체는 완전히 같다** — 앞에서부터 한 번씩 순회하며 누적한다. 둘 다 명백히 $O(n)$ 이다.

```python title="같은 O(n), 다른 구현"
def slow_sum(arr):
    total = 0
    for x in arr:          # 매 반복마다 바이트코드 명령을 하나씩 실행
        total += x
    return total

def fast_sum(arr):
    return sum(arr)         # C로 짜인 루프 하나가 통째로 돈다
```

```pyrepl
>>> import timeit
>>> arr = list(range(1_000_000))
>>> min(timeit.repeat(lambda: slow_sum(arr), number=10, repeat=5)) / 10 * 1000
15.671
>>> min(timeit.repeat(lambda: fast_sum(arr), number=10, repeat=5)) / 10 * 1000
2.123
```

같은 $O(n)$, 같은 알고리즘인데 **7.4배** 차이가 난다. 빅오 표기로는 이 둘을 구분할 방법이 없다 — 둘 다 그냥 $O(n)$ 이다. 차이는 상수 $c$ 에서 온다.

::: perf 상수가 어디서 오는가
`for x in arr: total += x` 는 반복마다 파이썬 바이트코드 여러 개(`FOR_ITER`, `STORE_FAST`, `LOAD_FAST`, `BINARY_OP`, ...)를 인터프리터 루프가 하나씩 해석해서 실행한다. `sum(arr)` 은 C로 작성된 `builtin_sum` 함수 하나가 호출되고, 그 안의 반복은 파이썬 바이트코드를 전혀 거치지 않는다. 매 원소마다 "인터프리터 디스패치 비용"을 내느냐 안 내느냐의 차이가 7배로 나타난다.

이 원칙은 이 파트 전체에 반복해서 나온다. **가능하면 내장 함수·표준 라이브러리에 반복을 맡겨라.** `sum`, `max`, `min`, `sorted`, `any`, `all`, `map`, 컴프리헨션 모두 C 레벨 루프를 쓴다. [7.2 파이썬 자료구조의 실제 비용](#/py-ds-cost)에서 이 표를 자료구조별로 정리한다.
:::

## 왜 시간복잡도만으로는 부족한가

빅오는 **점근적**(asymptotic) 이야기다. $n$ 이 무한대로 갈 때 어떤 함수가 이기는지를 말한다. 그런데 코딩테스트의 $n$ 은 무한대가 아니라 구체적인 숫자다 — $10^5$, $10^6$. 이 구간에서는 상수가 승부를 가른다.

극단적인 예로, "$n$ 이 작을 때는 $O(n^2)$ 이 $O(n \log n)$ 보다 빠를 수 있다"는 것도 실제로 벌어진다. 삽입 정렬은 $O(n^2)$ 이지만 상수가 아주 작아서, 작은 배열($n \lesssim 20$)에서는 병합 정렬의 $O(n \log n)$ 보다 빠르다. 그래서 실전 정렬 알고리즘(Timsort 포함)은 작은 구간을 삽입 정렬로 처리한다. [7.4 정렬](#/sorting)에서 Timsort 내부를 본다.

::: deep 점근 표기의 세 가지 이름
- $O(g)$ — 상한. "이보다 나쁘지 않다."
- $\Omega(g)$ — 하한. "이보다 좋지 않다."
- $\Theta(g)$ — 상한이면서 하한. "정확히 이 차수다."

코딩테스트에서 "이 알고리즘은 $O(n \log n)$ 이다"라고 할 때 실제로 의미하는 건 대부분 $\Theta(n \log n)$ 이다. 엄밀한 빅오만 따지면 $O(n^2)$ 이라고 해도 틀린 말은 아니다(더 느슨한 상한이니까). 관용적 표현과 엄밀한 정의가 다르다는 것만 알아 두면 된다.
:::

빅오가 못 잡아내는 또 하나는 **인터프리터 오버헤드** 그 자체다. C나 Java와 비교하면 파이썬은 같은 알고리즘, 같은 빅오라도 태생적으로 느리다. 다음 절에서 그 크기를 실측한다.

## "1초에 1억 번"은 파이썬에서 틀렸다

코딩테스트 커뮤니티에는 "시간 제한 1초 = 대략 1억(10^8) 번의 연산"이라는 어림값이 퍼져 있다. 이건 **C/C++ 기준**이다. 파이썬으로 실측해 보자.

```python title="빈 루프 vs 실제 연산"
import time

N = 10_000_000
start = time.perf_counter()
for i in range(N):
    pass
elapsed = time.perf_counter() - start
print(f"빈 루프: {elapsed:.4f}s -> 초당 {N/elapsed:,.0f}회")

N = 100_000_000
start = time.perf_counter()
total = 0
for i in range(N):
    if i % 2 == 0:
        total += i
elapsed = time.perf_counter() - start
print(f"비교+누적: {elapsed:.4f}s -> 초당 {N/elapsed:,.0f}회")
```

```text
빈 루프: 0.1110s -> 초당 90,056,501회
비교+누적: 4.2731s -> 초당 23,402,255회
```

(Python 3.14.5 / Windows 기준 실측.)

숫자를 곱씹어 볼 가치가 있다. **아무 일도 안 하는** 빈 루프는 초당 9천만 번 가까이 돈다 — "1억"이라는 숫자에 근접한다. 그런데 여기에 `if` 비교 하나와 누적 대입 하나만 더해도 초당 **2,340만 번**으로 떨어진다. 실제 코딩테스트 코드는 최소한 이 정도의 연산은 몸통에 들어간다. 1억 번을 채우는 데 4.27초가 걸렸다 — 즉 시간 제한 1초를 기준으로 삼는다면 실제로 감당할 수 있는 연산 횟수는 **1000만~2000만** 정도로 봐야 한다.

::: cote "1억 번"을 그대로 믿으면 생기는 일
백준 같은 저지에서 시간 제한 1초, $n = 10^7$ 짜리 문제를 보고 "1억 번의 10분의 1이니 여유 있다"고 판단하면 위험하다. 순수 파이썬 반복문으로 $10^7$ 번을 도는 코드는 위 실측 기준으로 이미 0.3~0.5초를 잡아먹는다. 안에 함수 호출, 딕셔너리 접근, 문자열 연산이 하나라도 더 들어가면 순식간에 1초를 넘긴다.

그래서 파이썬 코딩테스트 전략은 둘 중 하나다.

1. **반복 횟수 자체를 줄인다** — 더 나은 알고리즘(빅오를 낮춘다).
2. **반복을 C 레벨로 밀어낸다** — 내장 함수, `sys.stdin` 일괄 읽기, `numpy` 벡터화.

2번은 [8.2 파이썬 입출력 최적화](#/io-optimize), [8.3 시간 초과를 피하는 관용구](#/tle)에서 본격적으로 다룬다.
:::

## 공간복잡도와 재귀 깊이의 한계

시간만큼 중요한 게 공간복잡도다. 정의는 같은 방식이다 — 입력 크기 $n$ 에 대해 **추가로** 쓰는 메모리가 어떻게 늘어나는가. `입력 자체`는 보통 세지 않는다.

가장 흔히 잊는 공간이 **호출 스택**이다. 재귀 함수는 호출할 때마다 스택 프레임을 하나씩 쌓는다. 깊이 $d$ 만큼 재귀하면 $O(d)$ 만큼의 스택 공간을 쓴다.

```text nolines
depth(3)
  |-- depth(2)
        |-- depth(1)
              |-- depth(0) -> 0   프레임 4개가 동시에 스택에 쌓여 있다
```

파이썬은 이 스택이 무한정 자라는 걸 막기 위해 재귀 깊이에 상한을 둔다.

```python title="재귀 깊이 한계"
import sys

print(sys.getrecursionlimit())

def depth(n):
    if n == 0:
        return 0
    return 1 + depth(n - 1)

try:
    depth(10_000)
except RecursionError as e:
    print("RecursionError:", e)
```

```text
1000
RecursionError: maximum recursion depth exceeded
```

기본값은 **1000**이다. 그래프 문제에서 정점이 몇만 개만 돼도 DFS를 순수 재귀로 짜면 이 한계에 그대로 걸린다. 한도를 늘릴 수는 있다.

```python title="한도 늘리기"
sys.setrecursionlimit(20_000)
print(depth(15_000))   # 15000
```

```text
15000
```

::: danger setrecursionlimit는 만병통치약이 아니다
`sys.setrecursionlimit()` 이 정하는 건 **파이썬이 감지하는 논리적 한도**일 뿐이다. 실제로 재귀가 쓰는 건 운영체제가 스레드에 할당한 **C 스택**이고, 그 크기는 보통 몇 MB로 고정돼 있다. 한도를 너무 크게 올리면 파이썬의 `RecursionError` 가 뜨기 전에 **C 스택이 먼저 터져서 인터프리터 자체가 세그멘테이션 폴트로 죽는다.** `RecursionError`는 최소한 예외라서 잡아서 처리라도 할 수 있지만, 스택 오버플로는 프로그램을 그냥 죽인다.

코딩테스트에서 재귀 깊이가 문제될 걸 알면(그래프 정점 수가 많다, 트리 높이가 크다) 정공법은 한도를 올리는 게 아니라 **재귀를 반복문 + 명시적 스택으로 바꾸는 것**이다. 이건 [7.13 그래프 표현과 순회](#/graph)와 [7.18 재귀와 백트래킹](#/backtracking)에서 실제 코드로 다룬다.
:::

::: cote 재귀 깊이는 시험장에서 실제로 자주 나오는 함정이다
백준 문제 중 트리·그래프의 깊이가 $10^5$ 를 넘는 경우가 흔하다. 재귀로 DFS를 짜면 로컬에서는 잘 돌다가 채점 서버에서 `RecursionError` 또는 런타임 에러를 받는다. 흔한 대처:

```python
import sys
sys.setrecursionlimit(10**6)   # 임시방편. 큰 입력에서는 여전히 위험하다
```

이걸로 통과하는 문제도 있지만, **정석은 반복문 기반 DFS/BFS로 바꾸는 것**이다. 재귀 깊이 자체가 골칫거리인 문제는 애초에 "재귀로 풀지 말라"는 신호일 때가 많다.
:::

시간복잡도와 마찬가지로, 공간복잡도도 **상수**가 있다. 리스트 하나를 새로 만드는 것과 제자리에서 고치는 것은 둘 다 특정 빅오로 표현되지만 실제 메모리 사용량은 다르다. 이 얘기는 [7.2 파이썬 자료구조의 실제 비용](#/py-ds-cost)과 [5.2 메모리 모델과 측정](#/memory)에서 `sys.getsizeof`, `tracemalloc`으로 직접 잰다.

## 요약

- 빅오 $O(g(n))$ 은 **상한**이다 — 상수 $c$ 와 문턱값 $n_0$ 이 존재해서 $n \ge n_0$ 부터 $f(n) \le c \cdot g(n)$ 이라는 뜻이다.
- 별다른 말이 없으면 빅오는 **최악의 경우**를 말한다. 평균이 아니다.
- **같은 $O(n)$ 이라도 상수가 다르면 실전 속도는 크게 다르다.** 실측으로 7배 차이를 봤다 — 파이썬 루프 vs `sum()`.
- 빅오는 점근적이라 인터프리터 오버헤드 같은 "고정 비용"을 설명하지 못한다. 파이썬은 C보다 태생적으로 느리다.
- "1초에 1억 번"은 C 기준이다. 파이썬은 실질 연산이 섞이면 **초당 1000만~2000만 번** 정도로 잡아야 안전하다.
- 재귀는 $O(d)$ 만큼 호출 스택을 쓴다. 기본 재귀 한도는 1000이고, `setrecursionlimit`을 함부로 올리면 C 스택 오버플로로 프로그램이 죽을 수 있다.
- 제약 조건 $n$ 을 보면 요구되는 복잡도가 대략 정해진다. 이 감각이 코딩테스트 전략의 출발점이다.

       return count_down(n - 1, acc + [n])
