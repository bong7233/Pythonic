# 7.4 정렬

::: lead
정렬은 코딩테스트에서 가장 많이 호출하는 함수이면서, 가장 안 배우고 넘어가는 주제다. `sorted(x)` 한 줄이면 끝나기 때문이다. 하지만 그 한 줄 뒤에서 벌어지는 일을 모르면 두 가지를 놓친다. 하나는 **왜 파이썬의 정렬이 남들이 손으로 짠 어떤 정렬보다도 실전 데이터에서 빠른가**이고, 다른 하나는 **다중 조건 정렬에서 왜 순서를 반대로 짜야 맞는가**다. 이 절에서 둘 다 실측과 코드로 뜯어본다.
:::

## 정렬을 직접 짜면 왜 시간초과가 나는가

코딩테스트 문제에 "정렬"이 필요하다는 신호가 보이면 대부분은 그냥 `sorted()` 를 쓰면 된다. 그런데 가끔 "정렬 정도는 나도 짤 수 있다"며 버블 정렬이나 선택 정렬을 손으로 짜는 사람이 있다. 그 대가를 실측으로 보자.

```python title="버블 정렬 vs 내장 정렬"
import random
import time

random.seed(2)


def bubble_sort(a):
    a = a[:]
    n = len(a)
    for i in range(n):
        for j in range(n - 1 - i):
            if a[j] > a[j + 1]:
                a[j], a[j + 1] = a[j + 1], a[j]
    return a


for n in (500, 1000, 2000, 4000):
    data = [random.randint(0, 10**6) for _ in range(n)]
    t0 = time.perf_counter(); bubble_sort(data); t1 = time.perf_counter()
    t2 = time.perf_counter(); sorted(data); t3 = time.perf_counter()
    print(f"n={n:>6}  bubble={t1-t0:.4f}s  sorted={t3-t2:.5f}s  배수={(t1-t0)/(t3-t2):.0f}x")
```

```text nolines
n=   500  bubble=0.0045s  sorted=0.00003s  배수=  156x
n=  1000  bubble=0.0187s  sorted=0.00006s  배수=  317x
n=  2000  bubble=0.0801s  sorted=0.00013s  배수=  640x
n=  4000  bubble=0.3443s  sorted=0.00027s  배수= 1287x
```

(Python 3.14.5 / Windows 기준 실측.) $n$ 이 두 배가 될 때마다 버블 정렬 시간은 **약 4배**로 뛴다 ($O(n^2)$). 내장 `sorted()` 는 거의 그대로다.

::: note 배수 자체는 실행할 때마다 크게 흔들린다
이 코드를 그대로 세 번 더 돌리면 배수가 $n=500$ 에서는 140~177x, $n=4000$ 에서는 1097~1556x 사이를 오간다. 이유는 단순하다 — `sorted()` 쪽 시간이 수십 마이크로초 단위라 `time.perf_counter()` 한 번 호출로는 측정 자체의 잡음(캐시 상태, OS 스케줄링)이 결과의 상당 부분을 차지한다. 반면 버블 정렬은 밀리초~수백 밀리초 단위라 잡음의 비중이 작다. **그래서 정확한 배수 숫자보다 방향이 중요하다.** $n$ 이 두 배 될 때 버블 정렬 시간이 4배로 뛰는 $O(n^2)$ 곡선과, `sorted()` 가 그 $n$ 범위에서 거의 안 움직이는 곡선 — 이 두 방향성은 몇 번을 다시 재도 흔들리지 않는다.
:::

백준 기준 $N=100{,}000$ 이면 버블 정렬은 $10^{10}$ 번 비교, 1초 제한을 수백 배 넘긴다. **정렬은 절대 직접 짜지 마라.** 파이썬 내장 정렬이 당신이 5분 안에 짤 수 있는 어떤 정렬보다도 빠르고 정확하다. 이건 [8.3 시간 초과를 피하는 관용구](#/tle)의 첫 번째 규칙이기도 하다.

그럼 내장 정렬은 정확히 뭘 하길래 이렇게 빠른가.

## `list.sort()` 의 정체 — Timsort

파이썬(그리고 Java, Android, V8, Rust의 일부)이 쓰는 정렬 알고리즘은 **Timsort**다. 팀 피터스(Tim Peters)가 2002년 CPython을 위해 설계했다. 병합 정렬과 삽입 정렬을 섞은 **하이브리드**다.

### 핵심 관찰: 현실의 데이터는 완전히 무작위가 아니다

교과서의 병합 정렬은 데이터를 무작위로 취급하고 무조건 반씩 쪼갠다. Timsort의 출발점은 다른 질문이다. **실전 데이터에는 이미 정렬된 구간(run)이 섞여 있는 경우가 많다.** 로그 파일은 시간순으로 대부분 정렬돼 있고, 이미 한 번 정렬한 리스트에 값 몇 개만 추가된 경우도 흔하다. 이런 구조를 무시하고 매번 반으로 쪼개는 건 낭비다.

Timsort는 이렇게 동작한다.

1. **자연 발생 run을 찾는다.** 배열을 앞에서부터 훑으며 이미 오름차순(또는 내림차순)인 구간을 찾는다. 내림차순 run은 찾은 즉시 뒤집어서 오름차순으로 만든다 — 이러면 두 방향 모두 "이미 정렬된 덩어리"로 재활용된다.
2. **run이 너무 짧으면 삽입 정렬로 최소 길이까지 늘린다.** 최소 run 길이(minrun)는 배열 크기에 따라 32~64 사이로 계산된다. 작은 조각에서는 삽입 정렬이 병합 정렬보다 실제로 빠르다 — 비교 횟수는 $O(n^2)$ 이지만 $n$ 이 작을 때 상수가 작고 분기 예측이 잘 되기 때문이다.
3. **run들을 병합 정렬로 합친다.** 이때도 무작정 합치지 않고, 스택에 쌓인 run의 크기 균형을 보며 언제 어떤 두 run을 합칠지 규칙(불변식)을 지킨다. 균형이 깨지면 성능이 $O(n \log n)$ 을 벗어날 수 있어서다.
4. **갤로핑(galloping) 모드.** 병합 중 한쪽 run에서 연속으로 여러 번 이겼다면, 그 run에 앞으로도 유리한 원소가 몰려 있을 가능성이 높다고 보고 이분 탐색으로 건너뛴다.

```text nolines
data:   3  4  8  9  2  1  6  7  0  5
run A: [3  4  8  9]                    <- 이미 오름차순, 그대로 재사용
run B:             [2  1] -> [1  2]    <- 내림차순 run, 뒤집어서 재사용
run C:                     [6  7]      <- 오름차순
run D:                           [0]   <- 길이 1, 삽입 정렬로 확장
run E:                              [5]
```

::: deep 왜 "적응형(adaptive)" 정렬이라 부르는가
Timsort의 최선 성능은 $O(n)$ 이다. 배열이 이미 완전히 정렬돼 있으면 run 하나로 끝나고 병합이 아예 필요 없다. 최악은 $O(n \log n)$ 으로, 일반 병합 정렬과 같다. **입력의 기존 순서에 따라 실제 비용이 달라진다**는 뜻에서 적응형이라 부른다. 퀵정렬이나 힙정렬은 입력이 정렬돼 있어도 최선의 이점을 못 본다 — 이게 Timsort가 실전에서 선택되는 이유다.
:::

말로만 하지 말고 실측하자. 완전 무작위, 이미 정렬됨, 완전 역순, 그리고 **1%만 무작위로 섞인 "거의 정렬된" 데이터**를 비교한다.

```python title="Timsort의 적응성 실측"
import random
import timeit

random.seed(0)

def make_almost_sorted(n):
    lst = list(range(n))
    k = max(1, n // 100)                  # 1%만 무작위로 흩뜨린다
    idxs = random.sample(range(n), k)
    vals = [lst[i] for i in idxs]
    random.shuffle(vals)
    for i, v in zip(idxs, vals):
        lst[i] = v
    return lst

for n in (100_000, 1_000_000):
    r = [random.randint(0, 10**6) for _ in range(n)]
    s = list(range(n))
    rv = list(range(n, 0, -1))
    a = make_almost_sorted(n)
    t_r  = timeit.timeit(lambda: sorted(r), number=5) / 5
    t_s  = timeit.timeit(lambda: sorted(s), number=5) / 5
    t_rv = timeit.timeit(lambda: sorted(rv), number=5) / 5
    t_a  = timeit.timeit(lambda: sorted(a), number=5) / 5
    print(f"n={n:>9}  무작위={t_r*1000:7.3f}ms  정렬됨={t_s*1000:6.3f}ms  "
          f"역순={t_rv*1000:6.3f}ms  거의정렬={t_a*1000:7.3f}ms  무작위/정렬됨={t_r/t_s:.1f}x")
```

```text nolines
n=   100000  무작위=  9.998ms  정렬됨= 0.396ms  역순= 0.364ms  거의정렬=  0.839ms  무작위/정렬됨=25.2x
n=  1000000  무작위=149.067ms  정렬됨= 4.385ms  역순= 4.530ms  거의정렬= 15.806ms  무작위/정렬됨=34.0x
```

(Python 3.14.5 / Windows 기준 실측.) **이미 정렬된 100만 개 배열은 무작위 배열보다 34배 빠르게 끝난다.** 역순 배열도 거의 같은 속도다 — 뒤집기 한 번으로 정렬된 run이 되기 때문이다. 1%만 흐트러진 "거의 정렬된" 배열도 완전 무작위보다 10배 가까이 빠르다. 퀵정렬 계열이었다면 역순 입력에서 오히려 최악의 경우($O(n^2)$)에 걸릴 수 있다는 걸 생각하면, 이 적응성은 우연이 아니라 설계다.

::: note $n=100{,}000$ 배수는 재실행마다 20~36x 사이를 오간다
`정렬됨` 케이스는 절대 시간이 0.3~0.5ms 밖에 안 돼서 `timeit` 의 `number=5` 로는 반복 횟수가 부족하다. 같은 코드를 세 번 더 돌리면 $n=100{,}000$ 의 무작위/정렬됨 배수는 20.6x, 25.2x, 32.3x, 36.4x 로 흔들린다. $n=1{,}000{,}000$ 은 절대 시간이 커서 훨씬 안정적이다 — 26~36x 사이에서 30x 안팎으로 수렴한다. **"이미 정렬된 입력이 수십 배 빠르다"는 방향과 자릿수는 확실하지만, "정확히 19.0배"처럼 소수점 하나까지 단정하면 다음 실행에서 바로 틀린다.**
:::

::: cote 코딩테스트 포인트 — "이미 정렬돼 있다"를 활용하라
문제에서 입력이 시간순, 좌표순 등으로 **부분적으로라도 정렬돼 들어오는 경우**, `sorted()` 를 호출하는 비용을 두려워하지 마라. Timsort는 그런 입력에서 사실상 $O(n)$ 에 가깝게 끝난다. 반대로 "정렬된 배열에 원소 하나만 추가로 넣고 다시 정렬해야 한다"는 상황이면 매번 `sorted()` 를 다시 부르지 말고 [7.5 이분 탐색](#/binary-search)의 `bisect.insort` 로 삽입 위치를 $O(\log n)$ 에 찾아 넣는 걸 고려하라.
:::

## 왜 안정 정렬인가 — 다중 키 정렬의 기반

**안정 정렬(stable sort)** 이란 *"정렬 키가 같은 원소들 사이의 원래 상대 순서를 정렬 후에도 유지한다"* 는 뜻이다. Timsort는 안정 정렬이다. 이건 우연이 아니라 명세다 — CPython 문서가 이걸 보장한다.

왜 중요한가? 안정성이 없으면 **여러 조건으로 순서를 매기는 것 자체가 불가능**해지기 때문이다.

```python title="안정성이 다중 키 정렬을 가능하게 한다"
students = [
    ("김철수", 3, 88),
    ("이영희", 1, 92),
    ("박민수", 3, 95),
    ("최지우", 1, 92),
    ("정하나", 2, 79),
]  # (이름, 학년, 점수)

# 2차 키(점수, 내림차순)로 먼저 정렬한 뒤 1차 키(학년, 오름차순)로 다시 정렬한다
by_score = sorted(students, key=lambda s: -s[2])
final = sorted(by_score, key=lambda s: s[1])
print(final)
```

```pyrepl
>>> final
[('이영희', 1, 92), ('최지우', 1, 92), ('정하나', 2, 79), ('박민수', 3, 95), ('김철수', 3, 88)]
```

**주의: 순서가 거꾸로다.** 낮은 우선순위 키(점수)로 먼저 정렬하고, 높은 우선순위 키(학년)로 **나중에** 정렬해야 맞는다. 안정 정렬이 아니었다면 학년으로 다시 정렬하는 순간 같은 학년 안의 점수 순서가 뒤죽박죽됐을 것이다. 안정 정렬이기 때문에 **1학년 안에서는 점수 내림차순이 그대로 보존된 채** 학년 순으로만 재배치된다.

실제로 튜플 키로 한 번에 정렬한 것과 정확히 같은 결과인지 확인해 보자.

```python title="검증: 두 단계 정렬 == 튜플 키 한 번"
one_pass = sorted(students, key=lambda s: (s[1], -s[2]))
print(final == one_pass)
```

```pyrepl
>>> final == one_pass
True
```

같다. 실전에서는 굳이 두 단계로 나누지 않고 처음부터 튜플 키를 쓰는 게 훨씬 흔하다 — 아래 절에서 다룬다. 하지만 **"낮은 우선순위부터, 높은 우선순위를 나중에"** 라는 순서 자체는 GUI 표의 다단 정렬(엑셀의 "정렬 기준 추가"), SQL의 `ORDER BY a, b` 가 내부적으로 기대는 것과 같은 원리다. 이 원리를 모르면 다단 정렬을 짤 때마다 결과가 왜 이런지 헤맨다.

::: hist 안정 정렬이 아닌 알고리즘도 있다
퀵정렬, 힙정렬은 일반적으로 **안정적이지 않다.** 원소를 교환하는 방식 자체가 같은 키를 가진 원소들의 상대 순서를 보장하지 않는다. C의 표준 라이브러리 `qsort` 가 안정성을 보장하지 않는 이유가 이거다. 파이썬이 처음부터 **명세로 안정성을 보장**한 건 의도적인 설계 결정이다 — 다단 정렬이라는 실전에서 압도적으로 자주 쓰이는 패턴을 언어 차원에서 지원하기 위해서다.
:::

## `key=` 와 `cmp_to_key` — 비교 방법을 알려주는 두 가지 방식

파이썬 2에는 `cmp(a, b)` 형태의 비교 함수를 직접 넘기는 방식이 있었다. 파이썬 3부터는 **`key=` 방식이 기본이자 사실상 유일하게 권장되는 방식**이다.

- **`key=f`**: 각 원소를 `f(원소)` 로 변환한 **값**을 기준으로 비교한다. 정렬 전에 **각 원소마다 딱 한 번만** `f` 를 호출한다.
- **`cmp_to_key(g)`**: 두 원소를 직접 비교하는 함수 `g(a, b)` (음수/0/양수 반환)를 `key` 로 쓸 수 있게 감싸주는 `functools` 의 어댑터다. 비교가 일어날 때마다 `g` 가 호출된다.

```python title="key vs cmp_to_key — 같은 결과, 다른 호출 횟수"
import functools

def cmp(a, b):
    if a[1] != b[1]:
        return a[1] - b[1]      # 학년 오름차순
    return b[2] - a[2]          # 점수 내림차순

cmp_result = sorted(students, key=functools.cmp_to_key(cmp))
print(cmp_result == one_pass)
```

```pyrepl
>>> cmp_result == one_pass
True
```

결과는 같지만 비용은 다르다. `key=` 는 $n$ 개 원소에 대해 $n$ 번만 변환 함수를 부르고, 그 뒤로는 캐시된 변환값끼리 비교한다. `cmp_to_key` 는 병합 과정에서 일어나는 **모든 비교**($O(n \log n)$ 번)마다 파이썬 함수 호출이 발생한다.

::: perf key= 가 cmp_to_key 보다 빠른 이유를 실측
```python
import functools, random, timeit

random.seed(3)
data = [(random.randint(0, 100), random.randint(0, 100)) for _ in range(50_000)]

t_key = timeit.timeit(lambda: sorted(data, key=lambda x: (x[0], -x[1])), number=20)
t_cmp = timeit.timeit(
    lambda: sorted(data, key=functools.cmp_to_key(
        lambda a, b: (a[0] - b[0]) or (b[1] - a[1]))),
    number=20,
)
print(f"key= : {t_key:.3f}s   cmp_to_key: {t_cmp:.3f}s   배수: {t_cmp/t_key:.1f}x")
```

```text nolines
key= : 0.311s   cmp_to_key: 1.048s   배수: 3.4x
```

(Python 3.14.5 / Windows 기준 실측.) `key=` 가 약 3.4배 빠르다. 재실행 두 번을 더 해 봐도 3.4~3.7x 사이로 안정적이다 — $5$ 만 개면 비교 횟수가 $O(n \log n) \approx 78$ 만 번인데, `cmp_to_key` 는 그 비교 하나하나에서 파이썬 람다 호출과 튜플 언패킹, `or` 단락 평가까지 거치는 반면 `key=` 는 원소당 딱 한 번 튜플을 만들고 그 뒤론 C 레벨 튜플 비교만 하기 때문이다. **비교 로직을 튜플 키로 표현할 수 있으면 항상 `key=` 를 써라.** `cmp_to_key` 는 "두 원소의 순서가 여러 필드에 걸친 복잡한 조건문으로만 정의되고 튜플로 못 풀어낼 때"의 최후 수단이다.
:::

이 최적화는 사실 이름이 있다 — **데코레이트-정렬-언데코레이트(Decorate-Sort-Undecorate, DSU)** 패턴이라 부른다. `key=` 방식 자체가 이 아이디어를 내장 API로 만든 것이다: 원소를 (변환값, 원본) 쌍으로 감쌌다가, 변환값으로만 정렬하고, 원본을 꺼낸다. 파이썬 내부에서 `sorted(data, key=f)` 는 대략 다음과 동일한 일을 하되, C 레벨에서 훨씬 빠르게 처리한다.

```python title="key= 가 내부적으로 하는 일 (개념 시연)"
decorated = [(f(x), i, x) for i, x in enumerate(data)]   # i는 안정성 보장용 tie-breaker
decorated.sort()
result = [x for _, _, x in decorated]
```

## `sorted()` 와 `list.sort()` — 새 객체냐 제자리 수정이냐

이름의 형태에서 답이 보인다. [1.1 객체, 이름, 참조](#/objects-names)에서 본 경험칙 그대로다 — **동사형은 제자리 수정, 형용사·명사형은 새 객체 반환.**

```python title="반환값 확인"
lst = [3, 1, 2]
ret = lst.sort()
print(ret, lst)          # None [1, 2, 3]  — 제자리 수정, 반환값은 None

lst2 = [3, 1, 2]
new_lst = sorted(lst2)
print(new_lst, lst2)     # [1, 2, 3] [3, 1, 2]  — 새 리스트, 원본 그대로
```

```pyrepl
>>> ret, lst
(None, [1, 2, 3])
>>> new_lst, lst2
([1, 2, 3], [3, 1, 2])
```

| | `list.sort()` | `sorted()` |
| --- | --- | --- |
| 대상 | `list` 메서드 (list 전용) | 내장 함수 (모든 이터러블) |
| 반환값 | `None` | 새 `list` |
| 원본 | 제자리 수정 | 그대로 |
| 메모리 | 추가 할당 없음(정확히는 병합용 임시 버퍼만) | 새 리스트 전체 할당 |

::: danger `lst = lst.sort()` — None을 대입하는 사고
```python
# ❌ ret는 항상 None이다
scores = [3, 1, 2]
scores = scores.sort()
print(scores)          # None !!
```

`sort()` 는 **제자리에서 바꾸고 `None` 을 반환한다.** 반환값을 다시 대입하면 리스트 자체를 날려버린다. 반환값이 필요하면 `sorted()`, 원본을 바꿔도 되면 `.sort()` — 둘 중 하나지, 섞어 쓰면 안 된다.
:::

::: cote 함수 인자로 받은 리스트를 정렬해도 되는가
[1.1](#/objects-names)에서 봤듯 `.sort()` 는 호출자의 리스트를 조용히 바꾼다. 코딩테스트 함수를 짤 때 **입력으로 받은 배열을 그대로 정렬해도 괜찮은지** 문제 요구사항을 확인하라. 원본을 보존해야 한다면 `sorted(arr)` 로 새 리스트를 만들거나, 함수 맨 앞에서 `arr = arr[:]` 로 복사한 뒤 `.sort()` 를 써라. 채점 시스템이 같은 배열을 여러 함수에 재사용하는 유형(멀티 테스트케이스)에서 이걸 놓치면 두 번째 테스트케이스부터 틀린다.
:::

또 하나 실전에서 자주 놓치는 인자: `reverse=True` 는 정렬 후 뒤집는 게 아니라 **비교 방향 자체를 반대로 하는 것**이라, 안정성이 그대로 유지된다. `sorted(x)[::-1]` 과 `sorted(x, reverse=True)` 는 키가 유일하면 결과가 같지만, 같은 키가 있으면 **동점자의 상대 순서가 반대로** 나온다.

```pyrepl
>>> data = [(1, "a"), (1, "b"), (2, "c")]
>>> sorted(data, key=lambda x: x[0], reverse=True)
[(2, 'c'), (1, 'a'), (1, 'b')]
>>> sorted(data, key=lambda x: x[0])[::-1]
[(2, 'c'), (1, 'b'), (1, 'a')]
```

동점자 `(1, "a")` 와 `(1, "b")` 의 순서가 정반대다. **동점자 순서가 문제 조건에 걸려 있다면 `[::-1]` 대신 반드시 `reverse=True` 를 써라.**

## 여러 조건으로 정렬하기

### 튜플 키가 기본 무기다

파이썬 튜플은 **사전식(lexicographic) 순서**로 비교된다 — 첫 원소가 다르면 거기서 결판나고, 같으면 다음 원소로 넘어간다. 이게 정확히 "1차 키, 2차 키, 3차 키..." 다중 정렬 조건과 맞아떨어진다.

```pyrepl
>>> (1, 5) < (1, 7)
True
>>> (1, 5) < (2, 0)
True
>>> (3, "b") < (3, "a")
False
```

```python title="세 조건으로 정렬: 학년 오름 -> 점수 내림 -> 이름 오름"
sorted(students, key=lambda s: (s[1], -s[2], s[0]))
```

### 음수 트릭: 방향을 섞는다

`reverse=True` 는 **모든 키에 일괄 적용**된다. "학년은 오름차순, 점수는 내림차순"처럼 **키마다 방향이 다르면** `reverse=True` 하나로는 안 된다. 이때 숫자 키는 **부호를 뒤집으면** 오름차순 비교가 그대로 내림차순처럼 동작한다.

```pyrepl
>>> items = [("a", 3), ("b", 1), ("c", 3), ("d", 2)]
>>> sorted(items, key=lambda x: (-x[1], x[0]))
[('a', 3), ('c', 3), ('d', 2), ('b', 1)]
```

숫자(`x[1]`)는 내림차순, 문자열(`x[0]`)은 오름차순으로 섞였다. `-x[1]` 을 오름차순으로 정렬하는 것은 `x[1]` 을 내림차순으로 정렬하는 것과 정확히 같다 — 부호가 대소 관계를 뒤집기 때문이다.

::: warn 음수 트릭은 숫자에만 통한다
```python
words = ["banana", "apple", "cherry"]
# ❌ 문자열에는 단항 마이너스가 없다
sorted(words, key=lambda w: -w)
# TypeError: bad operand type for unary -: 'str'
```

문자열, 튜플처럼 **부호가 없는 타입**은 이 트릭을 못 쓴다. 세 가지 우회로가 있다.

1. **그 키만 따로 `reverse=True` 로 두 단계 정렬한다** (안정성을 이용, 이 절 앞부분 참고).
2. **정수로 매핑한다.** 예를 들어 등급이 `"A", "B", "C"` 라면 `{"A": 0, "B": 1, "C": 2}` 로 바꿔 부호를 뒤집는다.
3. **`cmp_to_key` 로 비교 함수를 직접 쓴다** — 가장 유연하지만 가장 느리다.

```python
# ✅ 우회 2: 문자를 정수로 매핑해 음수 트릭을 되살린다
rank = {"A": 0, "B": 1, "C": 2}
sorted(students_by_grade, key=lambda s: (-s.score, rank[s.grade]))
```
:::

::: cote 코딩테스트 자주 나오는 조합
- **좌표 정렬**: `sorted(points, key=lambda p: (p[0], p[1]))` — x 오름차순, 같으면 y 오름차순. 스위핑 알고리즘의 전처리로 압도적으로 자주 나온다.
- **회의실 배정류**: `sorted(intervals, key=lambda x: (x[1], x[0]))` — 끝나는 시간 오름차순. 그리디 정당성의 전제가 된다. [7.19 그리디](#/greedy)에서 다시 다룬다.
- **문자열 조합해서 가장 큰/작은 수 만들기**: 단순 사전식 정렬이 아니라 `cmp_to_key(lambda a, b: (b+a > a+b) - (a+b > b+a))` 처럼 "이어 붙인 결과"를 비교해야 하는 대표적인 `cmp_to_key` 필수 케이스다. 두 문자열 `a`, `b` 를 원소 자체로 비교할 수 없고 `a+b` vs `b+a` 라는 **관계**로만 정의되기 때문이다.
:::

## 정렬 기반 알고리즘 패턴: 좌표 압축

정렬은 그 자체로 답이 되는 경우도 있지만, **다른 알고리즘의 전처리 단계**로 훨씬 자주 쓰인다. 대표적인 예가 **좌표 압축(coordinate compression)**이다.

세그먼트 트리, 펜윅 트리 같은 자료구조는 배열 인덱스가 $0$ 부터 $n-1$ 처럼 촘촘해야 효율적으로 동작한다. 그런데 실제 값은 좌표(최대 $10^9$)처럼 듬성듬성 클 수 있다. **값들의 순위(rank)만 남기고 실제 크기를 버리는 것**이 좌표 압축이다.

```python title="좌표 압축"
raw = [50, 10, 30, 10, 90, 30]

uniq_sorted = sorted(set(raw))                       # 1) 중복 제거 후 정렬
compress = {v: i for i, v in enumerate(uniq_sorted)}  # 2) 값 -> 순위 매핑
compressed = [compress[v] for v in raw]               # 3) 원본을 순위로 치환

print(uniq_sorted)   # [10, 30, 50, 90]
print(compressed)    # [2, 0, 1, 0, 3, 1]
```

```pyrepl
>>> compressed
[2, 0, 1, 0, 3, 1]
```

큰 값끼리의 **상대적인 대소 관계**만 보존하고 절댓값은 버렸다. 이제 `compressed` 의 값은 전부 $0$ 부터 $3$ 사이라 배열 인덱스로 바로 쓸 수 있다. `bisect.bisect_left` 로도 같은 결과를 얻을 수 있다 — 이미 정렬된 `uniq_sorted` 안에서 각 값의 위치를 이분 탐색으로 찾는 것이다.

```python title="bisect로도 같은 압축"
import bisect
compressed_bisect = [bisect.bisect_left(uniq_sorted, v) for v in raw]
assert compressed == compressed_bisect
```

::: cote 좌표 압축이 실제로 쓰이는 곳
- 좌표값이 $10^9$ 규모인데 세그먼트 트리·펜윅 트리로 구간 질의를 해야 할 때 ([7.25 세그먼트 트리](#/segment-tree))
- "몇 번째로 큰 수인가"를 반복해서 물어야 할 때 (순위만 필요, 실제 값은 안 필요)
- 인버전 카운트(교환 횟수 세기) 문제에서 값 범위를 인덱스 범위로 줄일 때

**정렬은 $O(n \log n)$, 압축 자체는 $O(n)$.** 전체 비용은 정렬이 지배한다. 좌표 압축을 빼먹고 원래 좌표값으로 배열을 잡으면 `MemoryError` 나 시간 초과로 직행한다 — 좌표값이 배열 크기가 되기 때문이다.
:::

## 실측: 정말 $O(n \log n)$ 인가

지금까지 "이론상 $O(n \log n)$" 이라고만 말했다. 입력 크기를 10배씩 늘려가며 실제로 걸리는 시간을 재서 확인하자. $O(n)$ 이라면 시간도 10배, $O(n^2)$ 이라면 100배가 돼야 한다. $O(n \log n)$ 은 그 사이, $\log_2 10 \approx 3.32$ 배만큼 이론 배수에 더 얹힌 값이 나와야 한다.

```python title="정렬 시간의 스케일링 실측"
import random, timeit, math

random.seed(1)
sizes = [10_000, 100_000, 1_000_000]
times = []
for n in sizes:
    data = [random.randint(0, 10**9) for _ in range(n)]
    t = timeit.timeit(lambda: sorted(data), number=7) / 7
    times.append(t)

for i in range(1, len(sizes)):
    ratio_n = sizes[i] / sizes[i - 1]
    ratio_t = times[i] / times[i - 1]
    theory = (sizes[i] * math.log2(sizes[i])) / (sizes[i - 1] * math.log2(sizes[i - 1]))
    print(f"n x{ratio_n:.0f} -> 실측 시간 x{ratio_t:.2f}  (n log n 이론 배수 x{theory:.2f})")
```

```text nolines
n x10 -> 실측 시간 x13.98  (n log n 이론 배수 x12.50)
n x10 -> 실측 시간 x13.32  (n log n 이론 배수 x12.00)
```

(Python 3.14.5 / Windows 기준 실측.) $O(n)$ 이었다면 배수가 10 근처, $O(n^2)$ 이었다면 100 근처여야 한다. 실측값 13.3~14.0은 $O(n \log n)$ 이론값(12.0~12.5)에 가깝다 — 파이썬 함수 호출 오버헤드와 캐시 효과가 약간 더해진 정도다. **정렬은 배열 크기가 10배 늘어도 시간은 100배가 아니라 15배 안팎으로 늘어난다**는 걸 몸으로 확인했다.

::: note 왜 정확히 12.5배가 아니라 14배인가
$O$ 표기는 **점근적** 비율이다. 상수항과 저차항($cn \log n + dn + e$ 의 $d$, $e$)이 실측값을 이론값에서 살짝 밀어낸다. [7.1 시간·공간 복잡도](#/complexity)에서 다룬 "상수의 무게"가 여기서도 그대로 적용된다. 자릿수(10배 vs 100배)가 맞는지가 중요하지, 소수점까지 일치할 필요는 없다.
:::

## 요약

- **Timsort**는 병합 정렬과 삽입 정렬의 하이브리드이며, 실전 데이터에 흔한 부분 정렬 구간(run)을 찾아 재사용하는 **적응형** 알고리즘이다. 이미 정렬됐거나 역순인 입력에서 무작위 입력보다 수십 배 빠르다.
- Timsort는 **안정 정렬**이다. 같은 키를 가진 원소의 상대 순서를 보존하며, 이 덕분에 "낮은 우선순위 키로 먼저, 높은 우선순위 키로 나중에" 정렬하면 다단 정렬이 성립한다.
- `key=` 는 원소당 한 번만 변환 함수를 호출하고, `cmp_to_key` 는 비교마다 호출한다. 튜플로 표현되는 조건이면 항상 `key=`.
- `list.sort()` 는 제자리 수정 + `None` 반환, `sorted()` 는 새 리스트 반환. 반환값을 헷갈려 대입하면 데이터를 날린다.
- **튜플 키**는 다중 조건 정렬의 기본이고, **숫자 키의 부호를 뒤집는 트릭**으로 오름차순·내림차순을 자유롭게 섞을 수 있다. 문자열 등 부호가 없는 타입에는 이 트릭이 안 통한다.
- **좌표 압축**은 정렬을 이용해 큰 값의 범위를 촘촘한 인덱스로 줄이는 전처리다. 세그먼트 트리 등 인덱스 기반 자료구조와 짝을 이룬다.
- 정렬의 $O(n \log n)$ 은 실측으로도 확인된다 — 입력이 10배 늘면 시간은 100배가 아니라 15배 안팎이다.
