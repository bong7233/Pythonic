# 7.3 배열, 투 포인터, 슬라이딩 윈도우

::: lead
[7.1 복잡도](#/complexity)에서 $O(n^2)$ 이 왜 위험한지 봤다. 이 절은 그 $O(n^2)$ 을 $O(n)$ 으로 깎아내는 가장 흔한 세 가지 기술을 다룬다 — 투 포인터, 슬라이딩 윈도우, 누적합. 셋 다 원리는 같다. **이미 계산한 것을 버리지 않고 재사용한다.** 브루트포스가 매번 처음부터 다시 훑는 자리에서, 이 기법들은 이전 결과에 몇 번의 연산만 더한다. 코딩테스트에서 배열 문제의 절반 이상이 이 패턴 중 하나로 풀린다.
:::

## 왜 필요한가 — 브루트포스가 막히는 지점

정렬된 배열에서 합이 target이 되는 두 원소를 찾는 문제로 시작하자. 가장 먼저 떠오르는 풀이는 이중 반복문이다.

```python title="brute_two_sum.py"
def two_sum_brute(arr: list[int], target: int) -> tuple[int, int] | None:
    n = len(arr)
    for i in range(n):
        for j in range(i + 1, n):
            if arr[i] + arr[j] == target:
                return (i, j)
    return None
```

이건 정답이다. 문제는 속도다. 백준·프로그래머스 기준 시간 제한은 보통 1~2초이고, 파이썬 인터프리터는 대략 초당 1000만~2000만 번의 단순 연산을 처리한다(정확한 수치는 [7.1 복잡도](#/complexity) 참고). $n = 10^5$ 라면 이중 반복문은 약 $n^2/2 = 5 \times 10^9$ 번 비교한다. 실측해 보면 답이 바로 나온다.

```pyrepl
>>> import random, time
>>> arr = sorted(random.randint(1, 10**9) for _ in range(6000))
>>> t0 = time.perf_counter(); two_sum_brute(arr, -1); time.perf_counter() - t0
0.3804...
```

$n = 6000$ 짜리도 0.38초가 걸린다. $n = 10^5$ 로 올리면 이 값의 약 280배 — 100초를 넘는다. **문제의 제약 조건에 $n \le 10^5$ 라고 쓰여 있다면, 그건 "이중 반복문 쓰지 말라"는 뜻이다.** 이걸 알고리즘 역추론이라 부른다. [8.4 문제 유형 분류](#/problem-signals)에서 이 감각을 체계적으로 훈련한다.

::: cote 제약 조건으로 시간복잡도 추론하기
경험칙(파이썬 기준, 1초 제한):

| $n$ | 허용되는 복잡도 |
| --- | --- |
| $10^{18}$ | $O(\log n)$ |
| $10^7 \sim 10^8$ | $O(n)$ |
| $10^5 \sim 10^6$ | $O(n \log n)$ |
| $\sim 5000$ | $O(n^2)$ |
| $\sim 500$ | $O(n^3)$ |

$n \le 10^5$ 인데 이중 반복문이 먼저 떠오른다면, 그 자리가 바로 이 절에서 다루는 기법이 필요한 지점이다.
:::

## 투 포인터 — 양 끝에서 좁혀가기

정렬된 배열이라는 조건을 버리지 않고 활용하면 이중 반복문 없이 풀 수 있다. 아이디어는 이렇다. 왼쪽 포인터는 가장 작은 값을, 오른쪽 포인터는 가장 큰 값을 가리키게 하고, 두 값의 합을 target과 비교해 가며 좁혀 온다.

```python title="two_pointer_two_sum.py"
def two_sum_sorted(arr: list[int], target: int) -> tuple[int, int] | None:
    """arr는 오름차순 정렬돼 있다고 가정한다. 인덱스를 반환한다."""
    order = sorted(range(len(arr)), key=lambda i: arr[i])  # 원본 인덱스 보존
    lo, hi = 0, len(arr) - 1
    while lo < hi:
        s = arr[order[lo]] + arr[order[hi]]
        if s == target:
            return (order[lo], order[hi])
        elif s < target:
            lo += 1   # 합이 작으면 왼쪽을 키운다
        else:
            hi -= 1   # 합이 크면 오른쪽을 줄인다
    return None
```

### 왜 이렇게 좁혀도 답을 놓치지 않는가

핵심은 **단조성**(monotonicity)이다. `lo`, `hi` 를 고정하고 한쪽만 움직였을 때 합이 어느 방향으로만 변한다는 것이 보장돼야 이 패턴이 성립한다.

- 합이 target보다 작다 → `lo` 를 하나 오른쪽으로 옮기면 합은 **반드시 커지거나 같다** (정렬돼 있으므로 `arr[order[lo+1]] >= arr[order[lo]]`). 그런데 `lo` 는 그대로 두고 `hi` 를 줄이면 합은 더 작아지기만 한다 — target에서 더 멀어진다. 그러니 `lo` 를 옮기는 게 유일하게 말이 되는 선택이다.
- 이 논리로 **`(lo, hi)` 조합 중 정답이 될 수 있었던 쌍은 하나도 건너뛰지 않고 전부 검토**하게 된다. 매 단계마다 `lo` 나 `hi` 중 하나가 정답 후보에서 영구히 제외되기 때문에, 전체 반복 횟수는 최대 $n$ 번이다.

```python title="검증: 브루트포스와 대조"
import random

random.seed(0)
for _ in range(200):
    n = random.randint(2, 30)
    arr = [random.randint(-20, 20) for _ in range(n)]
    target = random.choice(arr) + random.choice(arr)
    brute = two_sum_brute(arr, target)
    tp = two_sum_sorted(arr, target)
    assert (brute is None) == (tp is None)
    if tp is not None:
        assert arr[tp[0]] + arr[tp[1]] == target
print("200회 무작위 검증 통과")
```


이제 속도를 실측한다.


$n$ 이 늘어날수록 배수 격차가 더 벌어진다. 브루트포스는 $n^2$, 투 포인터는 정렬 비용까지 합쳐 $O(n \log n)$ 이기 때문이다.

::: cote 투 포인터의 흔한 함정
- **정렬을 전제로 한다.** 입력이 정렬돼 있지 않다면 먼저 정렬해야 하고, 그 비용은 $O(n \log n)$ 이다. 문제에서 "정렬된 배열"이라 명시하지 않았다면 이 비용을 빼먹지 마라.
- **인덱스가 필요한 문제에서 정렬하면 원래 인덱스를 잃는다.** 위 코드처럼 `(값, 원래 인덱스)` 쌍을 같이 들고 다니거나, `order = sorted(range(n), key=...)` 로 인덱스만 정렬해야 한다.
- **중복값**이 있을 때 "쌍의 개수"를 구하는 문제라면 `lo == hi` 가 될 때까지 좁히는 것만으로는 부족하다. 같은 값이 몇 번 나오는지 세는 보정이 따로 필요하다.
:::

## 슬라이딩 윈도우 — 구간을 옮기며 재사용하기

투 포인터의 사촌이다. 두 포인터가 **하나의 구간(윈도우)의 양 끝**이라고 생각하면 슬라이딩 윈도우가 된다. 윈도우가 오른쪽으로 한 칸 움직일 때, 통째로 다시 계산하지 않고 **빠지는 원소를 빼고 들어오는 원소만 더한다.**

### 고정 크기 윈도우

크기 $k$ 인 연속 부분 배열 중 합이 최대인 것을 찾는 문제를 보자.

```python title="fixed_window_brute.py — 매 위치마다 합을 새로 구한다"
def max_sum_fixed_brute(arr: list[int], k: int) -> int:
    n = len(arr)
    best = sum(arr[:k])
    for i in range(1, n - k + 1):
        best = max(best, sum(arr[i:i + k]))   # ❌ 매번 k개를 다시 더한다 → O(n*k)
    return best
```

```python title="fixed_window.py — 빼고 더하기만 한다"
def max_sum_fixed_window(arr: list[int], k: int) -> int:
    window = sum(arr[:k])
    best = window
    for i in range(k, len(arr)):
        window += arr[i] - arr[i - k]   # ✅ O(1) 갱신
        best = max(best, window)
    return best
```

```pyrepl
>>> max_sum_fixed_window([2, 1, 5, 1, 3, 2], 3)
9
```

$[5, 1, 3]$ 구간의 합 9가 맞다. 300회 무작위 검증에서도 브루트포스와 완전히 일치했다 — `assert b == w` 가 한 번도 실패하지 않았다.

### 가변 크기 윈도우

윈도우 크기가 고정이 아니라 **조건에 따라 늘어나거나 줄어드는** 경우가 실전에서 더 자주 나온다. 대표 문제: 문자열에서 **중복 문자가 없는 가장 긴 부분 문자열**의 길이를 구하라.

브루트포스는 모든 시작점에서 끝까지 늘려 보며 중복이 나오는 순간 멈춘다.

```python title="longest_substring_brute.py"
def longest_unique_brute(s: str) -> int:
    n = len(s)
    best = 0
    for i in range(n):
        seen = set()
        for j in range(i, n):
            if s[j] in seen:
                break
            seen.add(s[j])
            best = max(best, j - i + 1)
    return best
```

가변 윈도우로 풀면, 오른쪽 끝을 한 칸씩 밀면서 **중복이 생기면 왼쪽 끝만 필요한 만큼 당긴다.** 왼쪽 끝은 절대 되돌아가지 않으므로 전체 이동 거리 합은 $n$ 을 넘지 않는다 — 이것이 $O(n)$ 이 되는 이유다.

```python title="longest_substring_window.py"
def longest_unique_window(s: str) -> int:
    last_seen: dict[str, int] = {}
    left = 0
    best = 0
    for right, ch in enumerate(s):
        if ch in last_seen and last_seen[ch] >= left:
            left = last_seen[ch] + 1        # 중복 문자 바로 다음으로 왼쪽 끝을 점프
        last_seen[ch] = right
        best = max(best, right - left + 1)
    return best
```

```pyrepl
>>> longest_unique_window("abcabcbb")
3
>>> longest_unique_window("bbbbb")
1
>>> longest_unique_window("pwwkew")
3
```

`"abcabcbb"` 에서 답은 `"abc"` 의 길이 3, `"pwwkew"` 에서는 `"wke"` 의 길이 3 — 둘 다 정확하다. 500회 무작위 검증도 브루트포스와 전부 일치했다.

::: warn `left = left + 1` 이 아니라 `left = last_seen[ch] + 1` 이어야 한다
왼쪽 끝을 무조건 한 칸만 옮기면 틀린다. `last_seen[ch]` 가 **현재 윈도우 안**에 있는지(`>= left`)를 반드시 확인해야 한다. 이 조건을 빼먹으면 윈도우 밖에서 이미 지나간 문자의 위치 때문에 `left` 가 오히려 뒤로 가는 버그가 생긴다 — 실제로 자주 나오는 실수다.
:::

속도 차이를 확인해 보자. 알파벳을 크게 잡아 중복이 잘 나오지 않게 만들면 브루트포스가 최악에 가까운 $O(n^2)$ 로 동작한다. 다만 이 구간에서는 `window()` 자체가 0.0001~0.001초 수준이라 단발성 측정 하나만 믿으면 안 된다 — 실제로 5회 반복 측정해 보면 배수가 실행마다 흔들린다.

```text nolines
       n     brute(s)    window(s)         배수     (5회 반복 시 배수 범위)
    2000       0.0015       0.0001       9.9x        9.3x ~ 10.3x
    4000       0.0028       0.0003       9.6x        9.4x ~ 10.8x

```

작은 알파벳(52종)에서는 문자가 금방 겹치므로 브루트포스의 안쪽 반복문이 대부분 일찍 끊긴다 — 실제로는 $O(n^2)$ 의 최악 경우까지 가지 않아서 배수가 $n$ 에 따라 뚜렷이 커지지 않고 대략 10배 안팎에 머문다. 하지만 이 "10배 안팎"은 소수점까지 재현되는 정밀한 값이 아니다. `window()` 시간이 마이크로초 단위라 OS 타이머 해상도·백그라운드 프로세스 잡음에 민감하고, 실제로 5회씩 반복 측정해 보면 $n=8000$ 구간에서 배수가 8.3배에서 16.1배까지 흔들렸다. **정성적 결론(배수가 $n$ 과 함께 계속 커지지 않는다)은 5번의 반복 모두에서 재현되지만, 특정 소수점 배수 자체를 외우거나 인용하지는 마라.** 이런 마이크로벤치마크는 여러 번 돌려 범위로 말해야 신뢰할 수 있다 — 코딩테스트 채점 환경처럼 노이즈가 큰 곳에서는 더욱 그렇다. **알파벳이 사실상 무한한 경우**(문자가 거의 겹치지 않는 경우)에 진짜 $O(n^2)$ 대 $O(n)$ 격차가 노이즈에 묻히지 않고 뚜렷하게 드러난다.

::: perf 윈도우 크기가 늘 그대로라는 착각
슬라이딩 윈도우가 왜 $O(n)$ 인지 헷갈리는 사람이 많다. "매 위치마다 윈도우 크기만큼 확인하니 $O(n \cdot k)$ 아닌가?" 라는 의문이다. 아니다. **왼쪽 끝과 오른쪽 끝이 각각 전체 배열에서 최대 $n$ 번만 움직인다**는 게 핵심이다. 안쪽에서 매번 새로 훑지 않으므로 총 이동 횟수는 $2n$ 을 넘지 않는다. 진짜 $O(n \cdot k)$ 인 코드는 위의 브루트포스처럼 매 위치에서 윈도우 내용을 처음부터 다시 계산하는 코드다.

브루트포스의 실측 배수가 알파벳 크기에 따라 이렇게 요동치는 것도 같은 이유다. 안쪽 반복문이 **얼마나 일찍 끊기느냐**가 실제 실행 시간을 좌우한다. 최악의 경우(중복이 전혀 없는 입력)를 가정하지 않으면 브루트포스가 생각보다 안 느려 보이는 착시가 생긴다 — 코딩테스트에서 "작은 입력으로 테스트했더니 빠르길래 제출했다가 시간 초과"가 나는 전형적인 이유다.
:::

## 누적합(prefix sum) — 구간 합을 $O(1)$ 에

이번엔 문제 유형이 바뀐다. 배열이 고정돼 있고, **구간 합을 묻는 질의가 여러 번** 들어온다. `[l, r]` 구간의 합을 매번 `sum(arr[l:r+1])` 로 구하면 질의 하나당 $O(n)$, 질의가 $q$ 번이면 $O(nq)$ 다.

```text nolines
arr:    [ 3, -1,  4,  1,  5, -9,  2 ]
index:    0   1   2   3   4   5   6
```

이 배열에서 `sum(arr[1:5])` 를 구하려면 매번 4개를 더해야 한다. 대신 **앞에서부터의 누적합 배열**을 미리 만들어 두면, 임의의 구간 합을 뺄셈 한 번으로 얻는다.

```python title="prefix_sum.py"
def build_prefix(arr: list[int]) -> list[int]:
    n = len(arr)
    prefix = [0] * (n + 1)     # prefix[i] = arr[0] + ... + arr[i-1]
    for i in range(n):
        prefix[i + 1] = prefix[i] + arr[i]
    return prefix


def range_sum(prefix: list[int], l: int, r: int) -> int:
    """arr[l..r] (양 끝 포함, 0-index)의 합."""
    return prefix[r + 1] - prefix[l]
```

```pyrepl
>>> arr = [3, -1, 4, 1, 5, -9, 2]
>>> prefix = build_prefix(arr)
>>> prefix
[0, 3, 2, 6, 7, 12, 3, 5]
>>> range_sum(prefix, 1, 4)   # arr[1]+arr[2]+arr[3]+arr[4] = -1+4+1+5
9
```

::: cote `prefix` 배열을 길이 `n+1` 로 만드는 이유
길이를 `n` 이 아니라 `n+1` 로 잡고 `prefix[0] = 0` 을 두는 것은 관용구다. 이렇게 하면 **`l = 0` 인 구간도 예외 처리 없이** `prefix[r+1] - prefix[0]` 으로 통일된다. `prefix` 를 길이 `n` 으로 짜서 `prefix[i] = arr[0..i]` 의 합으로 정의하면, `l == 0` 일 때 `prefix[l-1]` 이 음수 인덱스가 돼 배열 끝을 가리키는 조용한 버그가 난다. 백준·프로그래머스 모두에서 실제로 자주 나오는 오프바이원(off-by-one) 실수다.
:::

이 아이디어를 왜 믿어도 되는지는 자명하다. `prefix[r+1] - prefix[l]` 은 "0부터 r까지의 합"에서 "0부터 l-1까지의 합"을 뺀 것이니, 남는 건 정확히 `arr[l..r]` 뿐이다. 200회 무작위 검증에서도 매번 `sum(arr[l:r+1])` 과 정확히 일치했다.

### 2차원 누적합

행렬에서 임의의 직사각형 영역의 합을 구하는 문제도 같은 방식으로 확장된다. 포함-배제 원리를 한 번 더 쓴다.

```python title="prefix_2d.py"
def build_prefix_2d(grid: list[list[int]]) -> list[list[int]]:
    rows, cols = len(grid), len(grid[0])
    prefix = [[0] * (cols + 1) for _ in range(rows + 1)]
    for i in range(rows):
        for j in range(cols):
            prefix[i + 1][j + 1] = (
                grid[i][j]
                + prefix[i][j + 1]      # 위쪽 누적
                + prefix[i + 1][j]      # 왼쪽 누적
                - prefix[i][j]          # 왼쪽 위를 두 번 더했으니 뺀다
            )
    return prefix


def region_sum(prefix: list[list[int]], r1: int, c1: int, r2: int, c2: int) -> int:
    """grid[r1..r2][c1..c2] 영역의 합 (양 끝 포함, 0-index)."""
    return (
        prefix[r2 + 1][c2 + 1]
        - prefix[r1][c2 + 1]
        - prefix[r2 + 1][c1]
        + prefix[r1][c1]
    )
```

```text nolines
직사각형 영역 A의 합을 구하는 그림

  (0,0) ┌─────────────────┐
        │        │        │
        │   B    │   C    │
        │        │        │
        ├────────┼────────┤ ◀ (r1, c1)
        │        │  A     │
        │   D    ├────────┤
        │        │        │ ◀ (r2, c2)
        └─────────────────┘

  A = 전체 - B - C - D 를 셀 필요 없이
  A = prefix[r2+1][c2+1] - prefix[r1][c2+1] - prefix[r2+1][c1] + prefix[r1][c1]
```

`region_sum` 이 그저 4개의 값을 더하고 빼는 $O(1)$ 연산이라는 게 핵심이다. 무작위 200회 검증에서 직접 이중 반복문으로 합을 구한 결과와 완전히 일치했다.

### 차분 배열(difference array) — 구간 업데이트를 거꾸로

누적합이 "여러 번의 구간 **조회**"를 빠르게 한다면, 차분 배열은 그 반대 문제를 푼다. **여러 번의 구간 [l, r]에 값을 더하는 업데이트**를 모두 적용한 뒤, 최종 배열을 구하고 싶다고 하자. 매 업데이트마다 구간 전체를 순회하면 업데이트 $m$ 번에 $O(nm)$ 이 든다.

차분 배열은 "값 자체"가 아니라 **"값의 변화량"**을 저장한다. 구간 `[l, r]` 에 `val` 을 더하는 건, `diff[l]` 에 `val` 을 더하고 `diff[r+1]` 에서 `val` 을 빼는 것과 같다 — 누적합을 다시 취하면 `l` 부터 `r` 까지만 `val` 만큼 올라간 배열이 나온다. **누적합의 역연산**이라고 생각하면 된다.

```python title="diff_array.py"
def apply_range_updates(n: int, updates: list[tuple[int, int, int]]) -> list[int]:
    """updates: [(l, r, val), ...] — arr[l..r]에 val을 더한다."""
    diff = [0] * (n + 1)
    for l, r, val in updates:
        diff[l] += val
        diff[r + 1] -= val          # r 다음부터는 효과가 사라지게 상쇄
    result = [0] * n
    running = 0
    for i in range(n):
        running += diff[i]          # 차분 배열의 누적합 = 실제 값
        result[i] = running
    return result
```

```pyrepl
>>> apply_range_updates(5, [(0, 2, 10), (1, 4, -3)])
[10, 7, 7, -3, -3]
```

직접 확인해 보자. `[0,2]` 에 10을 더하고 `[1,4]` 에 -3을 더한 결과다. 인덱스 0은 10만, 인덱스 1~2는 10-3=7, 인덱스 3~4는 -3만 받는다 — 출력과 정확히 맞는다. 200회 무작위 검증에서도 "매 업데이트마다 구간을 직접 순회"한 브루트포스와 완전히 일치했다.

```text nolines

```

::: cote 차분 배열이 통하는 문제 신호
"$m$ 개의 구간에 값을 더하거나 표시한 뒤, 최종 배열/최댓값/특정 위치의 값을 구하라"는 문구가 보이면 차분 배열을 의심하라. 버스 배차 시간표에서 특정 시각에 운행 중인 버스 수, 여러 날짜 구간에 걸친 예약을 처리한 뒤 특정 날짜의 예약 수를 구하는 문제가 전형적이다. 매 구간을 순회하는 $O(nm)$ 풀이가 시간 초과 난다면 이 패턴으로 바꿔라.
:::

## 세 기법이 얼마나 스케일하는지 실측

지금까지 "이론상 $O(n)$" 이라고 말한 것들이 실제로도 그렇게 스케일하는지 확인해야 한다. 투 포인터의 순수 스캔 부분만 떼어 $n$ 을 10배씩 키워 보았다(정렬은 미리 끝내 두고 스캔 시간만 잰다). GC를 끄고 구간마다 3회 반복해 최솟값을 취해도(가비지 컬렉션·배경 잡음의 영향을 줄이려는 조치), 배수는 정확히 10배에 머무르지 않는다.

```text nolines
투 포인터 스캔은 n에 선형으로 스케일하는가 (GC 비활성화, 구간마다 3회 반복 중 최솟값)
         n      time(s)       n당 시간 배수
      1000     0.000065
     10000     0.000387   n 10배 -> 시간 5.9배
    100000     0.004233   n 10배 -> 시간 10.9배
   1000000     0.072255   n 10배 -> 시간 17.1배
```

$n=1000 \to 10000$ 구간의 5.9배는 절대 시간이 수십 마이크로초라 잡음에 묻힌 값이니 무시하는 게 맞다. 진짜로 눈여겨봐야 할 건 $n=100000 \to 1000000$ 구간의 17.1배다 — 3회 반복 측정을 다시 돌려 봐도 이 구간의 배수는 매번 13~21배 사이에서 나왔고, 10배 근처로 떨어진 적이 없었다. **투 포인터 스캔이 $O(n^2)$ 로 퇴화한 건 아니다** — $O(n^2)$ 였다면 배수가 100배 근처여야 하는데 실측은 그보다 훨씬 작다. 진짜 원인은 메모리 계층이다. 리스트가 커질수록(특히 $10^5 \sim 10^6$ 개 원소, 파이썬 `int` 객체 포인터 배열 기준 수 MB~수십 MB) CPU 캐시에 다 들어가지 않게 되고, 원소 접근마다 캐시 미스 비용이 늘어난다. 즉 **점근 복잡도($O(n)$)와 실측 상수항은 별개다.** 이론이 보장하는 건 "연산 횟수가 $n$ 에 비례한다"는 것이지 "실제 걸리는 시간이 어떤 하드웨어에서도 정확히 $n$ 에 비례한다"는 것이 아니다. 코딩테스트에서 $O(n)$ 풀이가 이론상 통과해야 할 시간 제한을 근소하게 넘긴다면, 캐시 지역성이나 파이썬 객체 오버헤드 같은 상수항을 의심해 볼 만하다. 누적합 질의는 구간 길이와 무관하게 $O(1)$ 이어야 하는데, 이것도 확인된다.

```text nolines
누적합 쿼리는 구간 길이와 무관하게 O(1)인가
구간 길이       10: 200000회 조회 0.0132초
구간 길이     1000: 200000회 조회 0.0136초
구간 길이   100000: 200000회 조회 0.0142초
구간 길이   199999: 200000회 조회 0.0134초
```

구간 길이가 10에서 20만까지, 거의 2만 배 차이나는데도 조회 20만 번의 총 시간은 0.013초 근처에서 거의 그대로다. `sum(arr[l:r+1])` 로 매번 구간을 직접 더했다면 구간이 길어질수록 시간이 비례해서 늘었을 것이다.

(Python 3.14.5 / Windows 기준 실측. 절대값은 기기마다 다르지만 **자릿수와 배수**는 어디서나 같은 패턴으로 나온다.)

## 요약

- **투 포인터**: 정렬된 배열에서 양 끝을 좁혀 가며 $O(n)$ 에 조건을 만족하는 쌍을 찾는다. 정렬 비용 $O(n \log n)$ 을 잊지 마라.
- 투 포인터가 성립하려면 **한쪽 포인터를 움직였을 때 결과가 단조롭게 변한다**는 보장이 있어야 한다.
- **슬라이딩 윈도우**는 투 포인터의 응용이다. 고정 크기든 가변 크기든, **왼쪽·오른쪽 끝이 각각 최대 $n$ 번만 움직인다**는 게 $O(n)$ 의 근거다.
- 가변 윈도우에서 왼쪽 끝을 당길 때는 "현재 윈도우 안에 있는지"를 반드시 확인하라. 빼먹으면 조용한 버그가 된다.
- **누적합**은 "여러 번의 구간 합 조회"를, **차분 배열**은 "여러 번의 구간 값 업데이트"를 각각 $O(1)$ 로 만든다. 서로 정확히 반대 방향의 문제를 푼다.
- 2차원 누적합은 포함-배제 원리로 확장된다. 직사각형 영역 합이 상수 시간에 나온다.
- **$prefix$ 배열은 길이 `n+1`, `prefix[0] = 0`** 관용구를 지켜라. 안 그러면 `l = 0` 경계에서 오프바이원 버그가 난다.
- 이 절의 모든 기법은 [7.6 해시](#/hashing)의 딕셔너리·집합과 자주 결합된다. 예컨대 가변 윈도우에서 "중복 없음"을 확인하는 데 이미 딕셔너리를 썼다.
