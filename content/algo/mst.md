# 7.16 최소 신장 트리

::: lead
[7.12 유니온 파인드](#/union-find)에서 "사이클을 만드는가"를 거의 상수 시간에 답하는 법을 만들었다. [7.8 힙과 우선순위 큐](#/heap)에서는 "지금까지 본 것 중 가장 작은 값"을 계속 뽑아내는 법을 만들었다. 이 절은 그 둘을 각각 재료로 써서 같은 문제를 두 가지 다른 방식으로 푼다 — **모든 노드를 연결하되 간선의 총 가중치를 최소로 하라.** 도로를 놓는 비용, 네트워크 케이블을 까는 비용, 전력망을 잇는 비용이 전부 이 문제다. 브루트포스가 왜 조합 폭발로 죽는지부터 보고, 크루스칼과 프림이 각각 어떤 자료구조로 그 폭발을 피하는지, 그리고 둘 중 무엇을 언제 골라야 하는지를 실측으로 답한다.
:::

## 문제 정의: 신장 트리와 최소 신장 트리

가중치가 있는 무방향 연결 그래프 $G = (V, E)$가 주어졌다고 하자. **신장 트리**(spanning tree)는 $G$의 모든 정점을 포함하면서 간선은 정확히 $|V|-1$개만 쓰는 부분 그래프다 — 트리이므로 사이클이 없고, 정점 개수보다 간선이 하나 적다. 같은 그래프에서 신장 트리는 보통 여러 개 만들 수 있고, 각각 간선 가중치의 합이 다르다. **최소 신장 트리**(MST, minimum spanning tree)는 그 합이 가장 작은 신장 트리다.

```text nolines
       1
   A ─────B          A, B, C, D 를 모두 연결하는 신장 트리 후보:
   │ \     │
  4│  \2   │3          A-B, A-C, A-D  (합 1+4+5=10)
   │   \   │           A-B, B-C, A-D  (합 1+2+5=8)   <- 더 싸다
   C ─────D            A-B, B-C, C-D  (합 1+2+6=9)
       6
```

간선이 몇 개 안 되는 이 그림에서도 신장 트리 후보가 여러 개고, 합이 제각각이다. 정점이 많아지면 후보 수는 순식간에 커진다.

::: cote 문제 신호: 언제 MST를 떠올려야 하는가
- "모든 도시/컴퓨터/노드를 연결하되 비용을 최소화하라"는 문장.
- 그래프가 **무방향**이고 **가중치**가 있다.
- 최종 답이 "연결에 필요한 최소 비용" 하나의 숫자다 (특정 두 점 사이의 최단 거리가 아니다 — 그건 [7.15 최단 경로](#/shortest-path)).
- 백준 1197(최소 스패닝 트리), 4386(별자리 만들기), 6497(전력난) 같은 문제가 전형적이다.
:::

## 브루트포스: 조합 폭발과 정면으로 만난다

가장 순진한 접근은 "모든 간선 부분집합 중 크기 $|V|-1$이고 사이클이 없는 것을 다 만들어 보고, 가중치 합이 최소인 걸 고른다"이다. 코드로 짜면 이렇다.

```python title="brute_force_mst.py — 모든 (|V|-1)-부분집합을 검사"
import itertools


class NaiveUF:  # 사이클 검사에만 쓰는 용도, 최적화는 생략
    def __init__(self, n):
        self.parent = list(range(n))

    def find(self, x):
        while self.parent[x] != x:
            x = self.parent[x]
        return x

    def union(self, a, b):
        ra, rb = self.find(a), self.find(b)
        if ra == rb:
            return False
        self.parent[ra] = rb
        return True


def brute_force_mst(n, edges):
    """edges: (weight, a, b) 리스트. 모든 (n-1)-조합을 검사한다."""
    best = None
    for combo in itertools.combinations(range(len(edges)), n - 1):
        uf = NaiveUF(n)
        total = 0
        ok = True
        for idx in combo:
            w, a, b = edges[idx]
            if not uf.union(a, b):     # 사이클이 생기면 신장 트리가 아니다
                ok = False
                break
            total += w
        if ok and (best is None or total < best):
            best = total
    return best
```

문제는 검사해야 할 조합의 개수다. 간선이 $m$개인 완전 그래프에서 크기 $n-1$짜리 부분집합은 $\binom{m}{n-1}$개다. 정점 개수를 늘려 가며 실제로 몇 개인지, 그리고 실제로 얼마나 걸리는지 재 보자.

```text nolines
n=4  |E|=6   조합 수=20              시간 0.0000초  결과 43
n=6  |E|=15  조합 수=3,003           시간 0.0023초  결과 83
n=7  |E|=21  조합 수=54,264          시간 0.0448초  결과 87
n=8  |E|=28  조합 수=1,184,040       시간 1.1468초  결과 88
```

(Python 3.14.5 / Windows 기준 실측, 정점 8개짜리 완전 그래프.) $n$이 4에서 8로 딱 두 배 됐을 뿐인데 조합 수는 20에서 118만으로, 시간은 눈에 안 보이던 수준에서 1초 이상으로 뛰었다. 실행을 멈추고 조합 수만 계산해 보면 더 명확하다.

```text nolines
n=9   |E|=36   조합 수=30,260,340
n=10  |E|=45   조합 수=886,163,135
n=12  |E|=66   조합 수=1,074,082,795,968
n=15  |E|=105  조합 수=91,748,617,512,913,200
```

정점이 15개인 완전 그래프에서 조합 수는 $9.17 \times 10^{16}$이다. 코딩테스트에서 정점 수는 보통 $10^3 \sim 10^5$다. 이 브루트포스는 **정점 10개도 못 넘기고 죽는다.** $\binom{m}{n-1}$이 조합 수라는 것 자체가 이 접근이 지수적(사실은 초지수적)으로 폭발한다는 뜻이다.

핵심 관찰: 모든 조합을 만들어 볼 필요가 없다. **간선을 가중치 순으로 하나씩 검토하면서, 사이클을 안 만드는 것만 그리디하게 채택해도 항상 최적이 된다.** 이걸 보장하는 게 다음의 컷 속성이다.

## 그리디가 통하는 이유: 컷 속성

그래프의 정점을 두 그룹 $S$와 $V \setminus S$로 나누는 것을 **컷**(cut)이라고 한다. 두 그룹을 잇는 간선 중 가중치가 가장 작은 것을 **최소 컷 간선**이라 하자.

> **컷 속성(cut property)**: 어떤 컷을 잡아도, 그 컷을 가로지르는 간선 중 가중치가 가장 작은 것은 반드시 어떤 MST에 포함될 수 있다.

```text nolines
       S 그룹          V\S 그룹
   ┌─────────┐      ┌─────────┐
   │  A   B  │      │  C   D  │
   │   \ /   │  3   │   \ /   │
   │    X────┼──────┼────X    │   <- 컷을 가로지르는 간선 중
   │   / \   │  7   │   / \   │      가중치 3인 것이 최소
   └─────────┘      └─────────┘
```

증명은 짧다. 최소 컷 간선을 $e = (u, v)$라 하고, 이걸 포함하지 않는 어떤 MST $T$가 있다고 가정하자. $T$는 트리이므로 $u$에서 $v$로 가는 경로가 유일하게 있고, 그 경로는 반드시 컷을 가로지른다 — 즉 그 경로 위에 컷을 가로지르는 간선 $e'$이 최소 하나 있다. $e$는 컷을 가로지르는 간선 중 최소이므로 $w(e) \le w(e')$이다. $T$에서 $e'$을 빼고 $e$를 넣어도 여전히 신장 트리이고, 총 가중치는 늘지 않는다. 그러니 $e$를 포함하는 MST가 항상 존재한다.

**크루스칼과 프림은 이 성질을 서로 다른 방식으로 활용하는 두 알고리즘일 뿐이다.**

- 크루스칼은 전역적으로 가장 싼 간선부터 훑으면서, 아직 서로 다른 그룹에 속한 두 정점을 잇는 순간마다 컷 속성을 적용한다.
- 프림은 "지금까지 트리에 포함된 정점들"과 "아직 아닌 정점들"이라는 컷 하나를 계속 유지하면서, 그 컷을 가로지르는 최소 간선을 매번 골라 트리를 키운다.

## 크루스칼: 정렬 + 유니온 파인드

크루스칼의 전략은 단순하다. **모든 간선을 가중치 순으로 정렬한 뒤, 앞에서부터 보면서 사이클을 안 만드는 간선만 채택한다.** "사이클을 안 만드는가"를 확인하는 부분이 정확히 [7.12 유니온 파인드](#/union-find)에서 만든 `union`의 반환값이다.

```python title="kruskal.py"
class UnionFind:
    def __init__(self, n):
        self.parent = list(range(n))
        self.rank = [0] * n

    def find(self, x):
        root = x
        while self.parent[root] != root:
            root = self.parent[root]
        while self.parent[x] != root:
            self.parent[x], x = root, self.parent[x]
        return root

    def union(self, a, b):
        ra, rb = self.find(a), self.find(b)
        if ra == rb:
            return False              # 이미 같은 그룹 -> 사이클이 생긴다
        if self.rank[ra] < self.rank[rb]:
            ra, rb = rb, ra
        self.parent[rb] = ra
        if self.rank[ra] == self.rank[rb]:
            self.rank[ra] += 1
        return True


def kruskal(n, edges):
    """edges: (weight, a, b) 리스트. 반환값: (MST 총 가중치, 채택된 간선 리스트)."""
    uf = UnionFind(n)
    total = 0
    used = []
    for w, a, b in sorted(edges):     # 가중치 오름차순
        if uf.union(a, b):
            total += w
            used.append((a, b, w))
    return total, used
```

`sorted(edges)`가 튜플을 첫 원소(가중치) 기준으로 정렬해 준다. 여기서 `union`이 `False`를 반환하는 간선은 그냥 버린다 — 이미 같은 그룹인 두 정점을 잇는 것이므로 사이클만 만들 뿐이다.

::: warn 간선이 `(weight, a, b)` 순서인 이유
튜플 정렬은 첫 원소부터 비교한다. 가중치를 맨 앞에 둬야 `sorted(edges)` 한 줄로 가중치 순 정렬이 끝난다. `(a, b, weight)` 순으로 저장해 놓고 `sorted(edges, key=lambda e: e[2])`를 매번 쓰는 것보다 이쪽이 더 짧고 빠르다. 단, 가중치가 같은 간선이 많으면 그다음 비교 기준(정점 번호)으로 넘어간다 — 정점이 정수라 에러는 안 나지만, [7.8 힙](#/heap)에서 본 "튜플 비교가 다음 원소로 번진다"는 함정과 같은 종류다.
:::

## 정답 검증: 브루트포스와 대조

구현이 실제로 맞는지 손으로 확인하는 대신, 앞서 만든 브루트포스와 무작위 그래프로 대조한다.

```python title="mst_verify.py — 브루트포스 대조"
import random


def random_connected_graph(n, extra_edges, seed):
    rnd = random.Random(seed)
    nodes = list(range(n))
    rnd.shuffle(nodes)
    edges = set()
    for i in range(1, n):              # 무작위 스패닝 트리로 연결성부터 보장
        a, b = nodes[i], nodes[rnd.randint(0, i - 1)]
        edges.add((min(a, b), max(a, b), rnd.randint(1, 50)))
    while len(edges) < (n - 1) + extra_edges:
        a, b = rnd.randint(0, n - 1), rnd.randint(0, n - 1)
        if a != b:
            edges.add((min(a, b), max(a, b), rnd.randint(1, 50)))
    return [(w, a, b) for a, b, w in edges]


random.seed(2024)          # 전역 random을 쓰는 n, extra_edges도 재현되려면 이 줄이 필수다
for trial in range(8):
    n = random.randint(4, 7)
    extra = random.randint(0, 4)
    edges = random_connected_graph(n, extra, seed=1000 + trial)
    bf = brute_force_mst(n, edges)
    kr, _ = kruskal(n, edges)
    assert bf == kr, (trial, bf, kr)
    print(f"trial {trial}: n={n} edges={len(edges)} brute={bf} kruskal={kr} -> OK")
```

trial 2: n=7 edges=8 brute=168 kruskal=168 -> OK

(Python 3.14.5 실측, `random.seed(2024)`로 재현 가능, 8회 시행 전부 일치.)

::: danger `random.randint`을 루프 안에서 쓸 때 흔한 실수
`random_connected_graph` 내부는 `random.Random(seed)`로 자기만의 난수열을 갖지만, 루프의 `n = random.randint(4, 7)`과 `extra_edges`는 **전역 `random` 모듈**을 쓴다. 루프 시작 전에 `random.seed(...)`를 걸지 않으면 이 둘은 매 실행마다 달라지고, `n`이 달라지면 `random_connected_graph`에 넘기는 시드가 같아도 그래프 자체가 달라진다 — "seed를 고정했는데 결과가 매번 다르다"는 버그의 전형이다. 전역 상태를 쓰는 난수 호출과 로컬 `Random` 인스턴스를 섞을 때는 **전역 쪽도 반드시 시드를 고정**해야 재현된다.
:::

이 패턴 — 느리지만 명백히 맞는 구현과 대조하기 — 은 [7.12 유니온 파인드](#/union-find)에서도 쓴 것과 똑같다. 알고리즘을 새로 짤 때마다 반복할 가치가 있다.

## 프림: 힙으로 컷을 관리한다

프림은 다른 그림에서 출발한다. **트리를 한 정점에서 시작해서 매번 "지금 트리"와 "바깥"을 잇는 가장 싼 간선 하나씩을 붙여 나간다.** "지금 트리에 아직 없는 정점 중, 트리와 가장 가깝게 이어지는 정점이 누구인가"를 매번 빠르게 찾아야 하는데, 이게 정확히 [7.8 힙](#/heap)이 잘하는 일이다.

```python title="prim_heap.py"
import heapq


def prim(n, adj, start=0):
    """adj[u] = [(v, weight), ...] 형태의 인접 리스트."""
    visited = [False] * n
    total = 0
    used = []
    pq = [(0, start, -1)]        # (간선 가중치, 정점, 어디서 왔는가)
    count = 0
    while pq and count < n:
        w, u, parent = heapq.heappop(pq)
        if visited[u]:
            continue              # 낡은 항목 -- 7.8의 다익스트라 예고와 같은 지연 삭제 패턴
        visited[u] = True
        count += 1
        total += w
        if parent != -1:
            used.append((parent, u, w))
        for v, wt in adj[u]:
            if not visited[v]:
                heapq.heappush(pq, (wt, v, u))
    if count < n:
        return None, None          # 그래프가 끊겨 있어 신장 트리가 없다
    return total, used
```

이 코드는 [7.8 힙](#/heap)에서 미리 본 다익스트라와 뼈대가 똑같다 — **정점을 확정할 때마다 이웃을 힙에 밀어 넣고, 꺼낼 때 이미 확정된(낡은) 항목이면 버린다.** 차이는 딱 하나다. 다익스트라는 힙에 "시작점부터의 누적 거리"를 넣고, 프림은 "트리에 붙는 간선 하나의 가중치"를 넣는다. 전역 누적값이냐 개별 간선값이냐의 차이가 최단 경로와 최소 신장 트리를 가른다.

::: warn 다익스트라와 헷갈리기 쉬운 지점
프림의 힙에는 **`(그 간선의 가중치, 정점, 부모)`**를 넣는다. 다익스트라처럼 `(시작점부터의 누적 거리, 정점)`을 넣으면 안 된다. 이 둘을 헷갈리면 코드는 돌아가지만 답이 틀린다 — 프림은 "전체 경로 비용"이 아니라 "트리에 다음 한 걸음을 붙이는 비용"만 본다.
:::

## 크루스칼 vs 프림: 같은 답인가

같은 그래프에서 시작 정점만 다르게 잡아도 프림은 항상 같은 총 가중치를 낸다는 것, 그리고 크루스칼과도 항상 일치한다는 것을 확인해 보자 (개별 간선 구성은 가중치가 같은 간선이 있으면 달라질 수 있지만, **총 가중치는 항상 같다** — MST의 총 가중치는 유일하다).

```text nolines

trial 1: n=53 edges=80  kruskal=1027 prim=1027 -> OK
trial 2: n=60 edges=139 kruskal=683  prim=683  -> OK
trial 3: n=43 edges=95  kruskal=649  prim=649  -> OK

```

(Python 3.14.5 실측, $n$ 21~60 사이 무작위 연결 그래프 10개, 전부 일치.)

## 밀집도에 따른 프림의 두 번째 버전: 배열

교과서에서 "프림은 $O(E \log V)$, 크루스칼은 $O(E \log E)$"라고 가르치는데, 이 둘을 곧이곧대로 비교하면 오해가 생긴다. 무방향 그래프에서 $E \le V^2$이므로 $\log E \le 2\log V$다. 즉 **$O(E \log E)$와 $O(E \log V)$는 사실 같은 오더다.** 힙 기반 프림이 크루스칼보다 점근적으로 유리하다는 통념은 정확하지 않다.

진짜 차이는 다른 곳에서 나온다. 프림은 힙 대신 **길이 $V$짜리 배열**로도 구현할 수 있다 — 매번 "아직 트리에 없는 정점 중 트리에 가장 가깝게 붙는 정점"을 배열을 선형 탐색해서 찾는 방식이다.

```python title="prim_array.py — O(V^2), 힙을 안 쓴다"
def prim_array(n, mat, start=0):
    """mat[u][v] = 가중치 (없으면 inf)."""
    INF = float("inf")
    dist = [INF] * n
    visited = [False] * n
    dist[start] = 0
    total = 0
    for _ in range(n):
        u = -1
        best = INF
        for i in range(n):                    # 선형 탐색 -- O(V)
            if not visited[i] and dist[i] < best:
                best = dist[i]
                u = i
        if u == -1:
            break                              # 그래프가 끊겨 있다
        visited[u] = True
        total += dist[u]
        for v in range(n):                     # 인접 정점 갱신도 O(V)
            if not visited[v] and mat[u][v] < dist[v]:
                dist[v] = mat[u][v]
    return total
```

바깥 루프가 $V$번, 안의 두 선형 탐색이 각각 $O(V)$이니 전체는 $O(V^2)$다. **간선 개수 $E$가 아니라 정점 개수 $V$만으로 비용이 정해진다.** 그래프가 밀집해서 $E$가 $V^2$에 가까워지면, $O(V^2)$은 $O(E \log V) = O(V^2 \log V)$보다 **로그 배만큼 작다.** 이게 "밀집 그래프에는 배열 기반 프림"이라는 조언의 진짜 근거다.

## 실측: 희소 그래프에서는 오히려 크루스칼이 앞선다

정점 대비 간선이 적은(평균 차수 4) 그래프에서 크루스칼과 힙 기반 프림을 비교했다.

| $V$ | $E$ | 크루스칼 | 힙 프림 | 배율(크루스칼/프림) |
| --- | --- | --- | --- | --- |
| 1,000 | 2,000 | 0.0006초 | 0.0006초 | 0.99 |
| 10,000 | 20,000 | 0.0079초 | 0.0089초 | 0.89 |
| 100,000 | 200,000 | 0.1066초 | 0.1486초 | 0.72 |

(Python 3.14.5 / Windows 기준 실측, 3회 중 최솟값.) 배율이 1보다 작다는 것은 **크루스칼이 오히려 더 빠르다**는 뜻이다. 이론상 두 알고리즘의 오더가 같다는 걸 이미 확인했으니 놀랍지 않다 — 승부는 상수항에서 갈린다. `sorted()`는 C로 구현된 Timsort를 **한 번** 호출한다. 힙 기반 프림은 매 정점을 확정할 때마다 이웃 전부를 `heappush`로 **개별 호출**하고, 낡은 항목을 `heappop`으로 걸러낸다 — 파이썬 함수 호출 오버헤드가 간선 수만큼 쌓인다. 희소 그래프에서는 이 오버헤드 차이가 실측에 그대로 드러난다.

::: perf "이론상 같은 오더"가 실전에서 뒤집히는 예
이 결과는 [7.1 시간·공간 복잡도](#/complexity)에서 강조한 "빅오가 같아도 상수가 다르면 승부가 뒤집힌다"의 실제 사례다. 코딩테스트에서 그래프가 희소하다면(간선이 정점 수의 몇 배 수준), **크루스칼을 기본으로 삼아도 손해 볼 일이 거의 없다.**
:::

밀집도를 올려서(정점 쌍의 50%, 90%가 간선으로 연결된 경우) 세 구현을 다시 비교했다.

| $V$ | $E$ (밀도 0.5) | 크루스칼 | 힙 프림 | 배열 프림 |
| --- | --- | --- | --- | --- |
| 200 | 10,059 | 0.0029초 | 0.0014초 | 0.0015초 |
| 400 | 40,119 | 0.0127초 | 0.0059초 | 0.0062초 |
| 600 | 90,122 | 0.0292초 | 0.0129초 | 0.0140초 |

| $V$ | $E$ (밀도 0.9) | 크루스칼 | 힙 프림 | 배열 프림 |
| --- | --- | --- | --- | --- |
| 300 | 40,458 | 0.0126초 | 0.0054초 | 0.0032초 |
| 500 | 112,348 | 0.0366초 | 0.0169초 | 0.0092초 |
| 800 | 287,730 | 0.0987초 | 0.0658초 | 0.0241초 |

(Python 3.14.5 / Windows 기준 실측, 3회 중 최솟값. 별도로 $V=500$, 밀도 0.3인 그래프에서 세 구현의 결과가 정확히 4129로 일치함을 확인했다.)

밀도 0.5에서는 프림 두 버전이 크루스칼보다 2배 안팎 빠르고, 힙 프림과 배열 프림은 거의 차이가 없다. **밀도 0.9로 올리면 배열 프림이 확실히 앞서기 시작한다** — $V=800$에서 크루스칼 0.0987초, 힙 프림 0.0658초, 배열 프림 0.0241초로 크루스칼 대비 약 4.1배 빠르다. 간선이 많아질수록 $E \log V$의 로그 배 손해가 실측에 쌓여 나타나는 것이다.

## 선택 기준 정리

::: cote 실전에서 무엇을 골라야 하는가
1. **평소엔 크루스칼을 기본값으로 삼아라.** 코드가 짧고([7.12 유니온 파인드](#/union-find)를 이미 외웠다면 10줄만 더 쓰면 된다), 희소 그래프에서 실측상 더 빠르거나 최소한 밀리지 않는다.
2. **간선이 정점 수의 제곱에 가깝게 많다면(밀집 그래프) 배열 기반 프림을 써라.** $O(V^2)$이 $O(E \log V)$보다 확실히 앞선다.
3. **간선 리스트가 아니라 인접 행렬로 입력이 주어지는 문제**(정점 수가 작고 $V \times V$ 행렬로 가중치를 주는 유형)라면 애초에 배열 프림이 구현도 더 자연스럽다.
4. **"간선이 스트리밍으로 들어온다" 또는 "전체 간선을 한 번에 못 받는다"**면 프림보다 크루스칼이 낫다 — 정렬만 되면 순서대로 처리할 수 있다.
:::

## 함정: MST가 아예 존재하지 않는 경우

그래프가 연결되어 있지 않으면 신장 트리 자체가 없다. 프림 구현에서 `count < n`으로 끝나면 그런 경우다.

```python
result, edges = prim(n, adj)
if result is None:
    print("그래프가 연결되어 있지 않음 -- 신장 트리 없음")
```

크루스칼도 마찬가지다 — 모든 간선을 다 봤는데 `union`이 성공한 횟수가 $n-1$번이 안 됐다면 컴포넌트가 여러 개다.

```python
def kruskal_checked(n, edges):
    uf = UnionFind(n)
    total, unioned = 0, 0
    for w, a, b in sorted(edges):
        if uf.union(a, b):
            total += w
            unioned += 1
    return total if unioned == n - 1 else None
```

::: danger 자주 하는 실수 세 가지
1. **연결성 확인을 빼먹는다.** 문제가 "그래프가 항상 연결되어 있다"고 보장하지 않는 이상, `unioned == n - 1` 체크를 반드시 넣어라. 안 그러면 컴포넌트가 여러 개인데도 "부분적으로 이어진 것들의 합"을 정답인 척 낸다.
2. **방향 그래프에 그대로 적용한다.** MST는 무방향 그래프의 개념이다. 방향 그래프에서 비슷한 문제는 **최소 스패닝 아보레센스**(minimum spanning arborescence, Chu-Liu/Edmonds 알고리즘)로 별개다. 문제가 방향 그래프인데 크루스칼·프림을 그대로 쓰면 조용히 틀린다.
3. **음수 가중치를 걱정한다.** 다익스트라와 달리 MST 알고리즘은 **음수 가중치가 있어도 정상 동작한다** — 컷 속성의 증명 어디에도 가중치의 부호에 대한 가정이 없다. 이 부분에서 다익스트라의 제약과 헷갈리지 마라. [7.15 최단 경로](#/shortest-path)에서 음수 가중치가 왜 다익스트라를 깨는지 다룬다.
:::

## 요약

- MST는 무방향 가중 그래프의 모든 정점을 최소 비용으로 잇는 신장 트리를 찾는 문제다. 답(총 가중치)은 유일하지만 간선 구성은 여러 개일 수 있다.
- 브루트포스(모든 간선 조합 검사)는 $\binom{|E|}{|V|-1}$개를 검사해야 해서 정점 10개 언저리에서 이미 실행 불가능하다.
- **컷 속성**이 그리디 접근을 정당화한다 — 어떤 컷이든 그것을 가로지르는 최소 가중치 간선은 항상 어떤 MST에 속한다.
- **크루스칼**: 간선을 정렬하고, [7.12 유니온 파인드](#/union-find)로 사이클을 걸러내며 채택한다. $O(E \log E)$.
- **프림**: [7.8 힙](#/heap)으로 "트리와 바깥을 잇는 최소 간선"을 관리한다. 다익스트라와 뼈대가 같다. $O(E \log V)$ — 단, $E \log E$와 실제로는 같은 오더다.
- 배열 기반 프림은 $O(V^2)$이고, 밀집 그래프($E$가 $V^2$에 가까움)에서 로그 배만큼 실측으로 앞선다. 희소 그래프에서는 오히려 크루스칼이 상수항에서 앞선다는 것을 실측으로 확인했다.
- 그래프가 끊겨 있으면 신장 트리가 없다 — 반드시 연결성을 확인하라. 방향 그래프에는 적용할 수 없다. 음수 가중치는 문제없다.
- 이 절의 그리디 정당화 패턴(컷 속성)은 [7.19 그리디](#/greedy)에서 다루는 교환 논증과 같은 계열의 증명 기법이다. 실전 문제 풀이는 [8.7 실전 풀이 II — 탐색/그래프](#/drill-search)에서 이어진다.
