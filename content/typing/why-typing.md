# 2.1 왜 타입 힌트인가

::: lead
파이썬은 변수를 선언할 때 타입을 요구하지 않는다. `x = 5` 라고 쓰면 그만이다. 그런데 이 책은 이제부터 아홉 개 절에 걸쳐 타입을 붙이는 법을 가르친다. 모순처럼 보인다. 이 절의 목표는 그 모순을 푸는 것이다. 타입 힌트가 정확히 무엇을 하고, 무엇을 하지 않는지 실측으로 확인한다. 이걸 제대로 이해하지 못하면 타입 힌트를 "귀찮은 장식"이나 "컴파일러 흉내"로 오해하게 되고, 둘 다 틀렸다.
:::

## 자유의 청구서

작은 스크립트에서는 동적 타이핑이 순수한 이득이다. 타입을 선언하지 않아도 되니 빨리 쓸 수 있다.

```python title="처음엔 문제없다"
def total_price(order):
    return order["price"] * order["quantity"]


order = {"price": 15000, "quantity": 3}
print(total_price(order))
```

```pyrepl
>>> total_price({"price": 15000, "quantity": 3})
45000
```

문제는 이 코드가 파일 하나짜리 스크립트로 끝나지 않을 때 시작된다. 다른 개발자가 `order` 딕셔너리를 만드는 코드를 고치면서 키 이름을 `"price"` 에서 `"unit_price"` 로 바꿨다고 하자. `total_price` 함수는 프로젝트 어딘가 다른 파일에 있어서 그 개발자 눈에 띄지 않았다.

```python title="키가 바뀐 걸 아무도 모른다"
order = {"unit_price": 15000, "quantity": 3}   # 다른 파일에서 만든 값
print(total_price(order))
```

```pyrepl
>>> total_price({"unit_price": 15000, "quantity": 3})
Traceback (most recent call last):
  ...
KeyError: 'price'
```

이 에러는 **그 코드 경로가 실행될 때만** 발생한다. `total_price` 가 자주 호출되지 않는 함수라면, 이 버그는 테스트도, 코드 리뷰도, 심지어 몇 주간의 실사용도 통과한 뒤 프로덕션에서 터진다. 딕셔너리는 어떤 키가 들어 있는지 아무 데도 적혀 있지 않다. `order` 라는 이름만 봐서는 이 함수가 무엇을 기대하는지 알 길이 없다. [1.1 객체·이름·참조](#/objects-names)에서 봤듯, 파이썬의 이름은 객체에 붙은 이름표일 뿐 그 객체가 무엇인지 보장하지 않는다.

코드가 100줄일 때는 이게 문제가 안 된다. 함수가 몇 개 안 되니 머릿속으로 다 기억한다. 코드가 10만 줄, 작성자가 20명이 되면 이야기가 다르다. **아무도 전체를 기억하지 못한다.** 함수 하나를 호출할 때마다 그 정의로 가서 무엇을 넘겨야 하는지 확인해야 한다면, 그것만으로 생산성이 무너진다. 동적 타이핑이 주는 자유는 공짜가 아니다. 청구서는 나중에, 코드베이스가 커진 뒤에 날아온다.

## 점진적 타이핑이라는 타협

이 문제에 대한 한 가지 답은 "그럼 정적 타입 언어를 쓰라"는 것이다. Java나 Rust는 처음부터 타입을 강제한다. 하지만 파이썬 커뮤니티는 다른 길을 골랐다. **점진적 타이핑**(gradual typing)이다.

PEP 484(2014, 파이썬 3.5)가 이 길을 열었다. 핵심 아이디어는 이렇다.

> 타입을 **선택적으로** 붙인다. 붙이면 검사받고, 안 붙이면 예전처럼 자유롭다. 그리고 — 이게 제일 중요한데 — **타입을 붙이든 안 붙이든 프로그램이 실행되는 방식은 똑같다.**

이건 "일단 타입 없이 빠르게 프로토타입을 만들고, 코드가 자라면 타입을 얹어서 굳힌다"는 현실적인 작업 흐름을 정확히 반영한다. 함수 하나에만 타입을 붙이고 나머지는 그대로 둬도 된다. 파일 하나만 검사하고 나머지는 나중에 해도 된다. 자바스크립트에서 타입스크립트로 넘어가는 것과 철학이 같다.

그런데 이 타협에는 대가가 있다. **타입 힌트는 힌트일 뿐, 강제가 아니다.** 파이썬 인터프리터는 타입 힌트를 신경 쓰지 않는다. 이건 주장이 아니라 지금 바로 확인할 수 있는 사실이다.

## 실측: 타입 힌트는 실행에 아무 영향이 없다

일부러 틀린 타입 힌트를 단 함수를 만들어 보자.

```python title="proof1.py"
def add(a: int, b: int) -> str:      # 반환 타입이 틀렸다: 실제로는 int를 반환한다
    return a + b


def greet(name: int) -> None:        # name은 str이어야 자연스러운데 int라고 우겼다
    print(f"안녕, {name}!")


result = add(3, 4)
print(result, type(result))

greet("파이썬")                      # int라고 선언한 자리에 str을 넘긴다

print(add.__annotations__)
```

`add` 는 `int` 두 개를 받아 `str` 을 반환한다고 선언했다. 실제로는 정수 덧셈을 하고 정수를 반환한다. `greet` 는 `name: int` 라고 선언해 놓고 문자열을 받는다. 파이썬은 둘 다 아무 문제없이 실행한다.

```bash
python proof1.py
```

```text nolines
7 <class 'int'>
안녕, 파이썬!
{'a': <class 'int'>, 'b': <class 'int'>, 'return': <class 'str'>}
```

에러 없이 끝까지 실행됐다. `7 <class 'int'>` — `str` 을 반환한다고 약속한 함수가 `int` 를 반환했는데도 아무 일도 없다. `greet("파이썬")` 도 `TypeError` 하나 없이 인사말을 출력한다.

마지막 줄 `add.__annotations__` 가 이 절의 핵심을 보여준다. 타입 힌트는 **함수 객체에 매달린 메타데이터 딕셔너리**일 뿐이다. `{'a': <class 'int'>, ...}` 라는 평범한 딕셔너리로 저장되고, 인터프리터는 이 딕셔너리를 실행 경로 어디에서도 들여다보지 않는다. 함수를 호출할 때 파이썬 바이트코드는 `CALL` 명령을 실행할 뿐, "이 인자가 선언된 타입과 맞는지" 확인하는 단계는 애초에 존재하지 않는다.

::: deep 어노테이션은 언제, 어떻게 평가되는가
`def add(a: int, b: int) -> str:` 을 파이썬이 실행하는 순간, `int` 와 `str` 이라는 표현식이 **평가되고** 그 결과가 함수 객체의 `__annotations__` 딕셔너리에 저장된다. 이건 단순한 대입문과 다르지 않다.

```pyrepl
>>> def f(a: int) -> str: ...
>>> f.__annotations__
{'a': <class 'int'>, 'return': <class 'str'>}
>>> type(f.__annotations__)
<class 'dict'>
```

즉 어노테이션은 함수가 **정의될 때 한 번** 평가되어 저장되는 값이다. `if` 문 조건처럼 매 호출마다 검사되는 게 아니다. 이 저장 시점과 평가 방식은 [2.9 런타임 타입 정보](#/runtime-typing)에서 `Annotated`, 지연 평가(`from __future__ import annotations`)와 함께 더 깊이 다룬다.
:::

이 사실 하나가 타입 힌트를 둘러싼 오해 대부분의 뿌리다. 타입 힌트는 **문서이자 약속**이지, **경비원**이 아니다. 약속을 어겨도 인터프리터는 아무 말 하지 않는다. 그럼 이 약속을 지키는지 누가 확인하는가? 사람이 아니라 **별도의 프로그램**이다.

## 그런데도 타입을 다는 이유

런타임에 아무 효과가 없다면, 왜 타입을 다는가? 답은 간단하다. **타입 힌트를 읽는 건 인터프리터가 아니라 다른 도구들이다.**

### 정적 검사기가 버그를 실행 전에 잡는다

방금 만든 `proof1.py` 는 실행하면 조용히 통과했다. 이번엔 실행하지 말고 **정적 타입 검사기**로 같은 파일을 검사해 보자. `pyright` 를 `uv`로 즉석 실행한다.

```bash
uvx pyright proof1.py
```

```text nolines
c:\...\proof1.py:2:12 - error: Type "int" is not assignable to return type "str"
    "int" is not assignable to "str" (reportReturnType)
c:\...\proof1.py:12:7 - error: Argument of type "Literal['파이썬']" cannot be assigned to parameter "name" of type "int" in function "greet"
    "Literal['파이썬']" is not assignable to "int" (reportArgumentType)
2 errors, 0 warnings, 0 informations
```

프로그램을 단 한 줄도 실행하지 않고, `pyright` 는 소스 코드만 읽어서 두 개의 오류를 정확히 찾아낸다. 첫 번째는 `add` 의 반환 타입 선언과 실제 반환값의 불일치, 두 번째는 `greet` 에 `int` 대신 문자열 리터럴 `'파이썬'` 을 넘긴 것. 흥미로운 건 두 번째 메시지다 — `pyright` 는 단순히 "타입이 str이다"가 아니라 **`Literal['파이썬']`** 이라고, 그 문자열 리터럴 자체까지 추론해서 알려준다.

같은 파일을 `mypy` 로도 검사해 보면 다른 도구라도 같은 결함을 잡는다는 걸 확인할 수 있다.

```bash
uvx mypy proof1.py
```

```text nolines
proof1.py:2: error: Incompatible return value type (got "int", expected "str")  [return-value]
proof1.py:12: error: Argument 1 to "greet" has incompatible type "str"; expected "int"  [arg-type]
Found 2 errors in 1 file (checked 1 source file)
```

두 도구는 구현도, 오류 코드 이름도, 메시지 형식도 다르다(`reportReturnType` vs `[return-value]`). 하지만 둘 다 소스 코드에 적힌 타입 힌트를 읽어서, **실행 없이** 같은 버그 두 개를 찾아낸다. 이게 타입 힌트의 첫 번째 실용적 가치다. `total_price` 예제로 돌아가면, 만약 그 함수의 매개변수에 `TypedDict` 로 정확한 키를 선언해 뒀다면, 다른 개발자가 키 이름을 바꾸는 순간 — 실행하지 않아도, 코드 리뷰 전에, 커밋하기도 전에 — 에디터가 빨간 줄을 그어줬을 것이다. `TypedDict` 는 [2.5 TypedDict, NamedTuple](#/typed-containers)에서 다룬다.

### 에디터가 정확한 자동완성을 제공한다

`pyright` 는 사실 마이크로소프트가 VS Code의 Pylance를 위해 만든 엔진이다. 에디터에서 타이핑하는 동안 실시간으로 도는 것도 정적 검사기다. 매개변수 타입이 `dict` 대신 정확한 `TypedDict` 나 클래스로 선언돼 있으면, 에디터는 `order.` 까지만 쳐도 그 안에 어떤 속성·키가 있는지 목록으로 보여준다. 타입이 없으면 에디터는 그 객체가 실제로 무엇인지 알 방법이 없어서 자동완성이 무력해진다. 이건 리스트로 셀 수 없는 종류의 생산성 손실이다 — 매번 정의로 점프해서 확인하는 시간이 하루에 수십 번 쌓인다.

### 문서가 코드와 함께 산다

타입 힌트가 없으면 함수의 계약은 독스트링이나 개발자의 기억에만 존재한다. 둘 다 코드가 바뀌어도 갱신되지 않는다. 타입 힌트는 **코드 옆에** 있어서 코드가 바뀌면 눈에 띄게 어긋난다 — 그리고 검사기가 그 어긋남을 잡아준다. `def total_price(order: OrderDict) -> int:` 는 그 자체로 "이 함수는 이런 걸 받고 이런 걸 돌려준다"는 문서다. 함수 시그니처만 보고 무엇을 넘겨야 할지 알 수 있다는 건 [1.10 함수](#/functions)에서 다룬 `*args`/`**kwargs` 남용 문제와도 연결된다 — 타입이 없는 `**kwargs` 는 호출부만 봐서는 무엇이 필요한지 전혀 알 수 없다.

## 왜 코드베이스가 커지면 필수가 되는가

세 가치(정적 검사, 자동완성, 문서화)는 전부 **혼자 짜는 100줄짜리 스크립트**에서는 거의 필요 없다. 함수가 대여섯 개뿐이면 다 외운다. 문제는 규모다.

- **작성자가 여러 명**이 되면, 다른 사람이 짠 함수의 계약을 코드만 보고 알아야 한다. 타입이 그 계약이다.
- **파일이 수십 개**로 나뉘면, 한 파일을 고쳤을 때 그 영향이 다른 파일 어디까지 미치는지 사람이 추적할 수 없다. 정적 검사기는 프로젝트 전체를 훑어서 알려준다.
- **리팩터링**을 할 때 — 함수 시그니처를 바꾸거나 클래스 속성 이름을 바꿀 때 — 타입이 있으면 검사기가 "이 변경 때문에 깨지는 호출부 17곳"을 그 자리에서 나열해 준다. 타입이 없으면 그 17곳 중 몇 곳은 테스트에도 안 걸리고 운영 환경에서 발견된다.

이건 `total_price` 예제가 실제로 겪은 일이다. 딕셔너리에 아무 타입도 없었기 때문에, 키 이름이 바뀌었다는 사실이 **코드 어디에도 기록되지 않았고**, 검사기가 알 방법도 없었다. 만약 `order` 의 타입이 `OrderDict` 라는 `TypedDict` 로 선언돼 있었다면, 다른 개발자가 딕셔너리를 만드는 코드에서 키 이름을 바꾸는 순간 그 자리에서 타입 오류가 났을 것이다. 버그가 **발생 지점**(딕셔너리를 잘못 만든 곳)에서 잡히지, **폭발 지점**(엉뚱한 함수에서 `KeyError` 가 터지는 곳)에서 잡히지 않는다. 이 차이가 크다.

::: cote 코딩테스트에서는 다르다
코딩테스트 코드는 대부분 이 절에서 말한 "커지는 코드베이스"가 아니다. 한 사람이 30분~2시간 안에 짜고 버리는 코드다. 이런 코드에 꼼꼼하게 타입을 다는 건 시간 낭비인 경우가 많다. 이 책의 Part VII·VIII 예제 코드에 타입 힌트가 드물게 나오는 이유다. 타입은 **협업하거나 오래 사는 코드**에 값어치가 있다.
:::

## 타입 힌트의 한계 — 런타임엔 아무것도 없다

여기서 반드시 짚어야 할 게 있다. 지금까지 본 모든 가치는 **코드를 실행하기 전에** 일어난다. `pyright` 나 `mypy` 는 파일을 실행하지 않고 텍스트로만 판단한다. 일단 프로그램이 실제로 돌기 시작하면, 타입 힌트는 앞서 실측했듯 **완전히 사라진다.**

이게 왜 문제가 되는지 보자. 정적 검사기가 볼 수 없는 데이터가 있다 — 네트워크로 들어온 JSON, 사용자 입력, 파일에서 읽은 CSV. 이런 데이터의 실제 타입은 **컴파일 타임이 아니라 런타임에만** 알 수 있다.

```python title="검사기는 이 함수를 통과시킨다 — 그런데 안전한가?"
import json


def total_price(order: dict) -> int:
    return order["price"] * order["quantity"]


raw = '{"price": "15000", "quantity": 3}'   # price가 문자열로 왔다 — API가 잘못 보냈다
order = json.loads(raw)                      # 반환 타입은 사실 Any
print(total_price(order))
```

```pyrepl
>>> import json
>>> raw = '{"price": "15000", "quantity": 3}'
>>> order = json.loads(raw)
>>> order
{'price': '15000', 'quantity': 3}
>>> order["price"] * order["quantity"]
'150001500015000'
```

`price` 가 숫자가 아니라 문자열로 온 순간, `*` 연산은 에러조차 내지 않고 **문자열을 세 번 반복**해 버린다(`str.__mul__`). `json.loads` 의 반환 타입은 정적으로 `Any` 이기 때문에 — 실제로 그 안에 뭐가 들어 있는지는 실행해봐야 안다 — `pyright`도 `mypy`도 이 줄을 잡아내지 못한다. 타입 힌트가 검사하는 건 **코드가 선언과 일치하게 짜여 있는가**이지, **런타임에 실제로 그 타입의 값이 들어오는가**가 아니다. 이 둘은 다른 질문이다.

::: warn 타입 힌트를 검증 도구로 착각하지 마라
```python
def process(age: int):
    if age >= 18: ...

process("스물")   # 타입 검사기는 잡아도, 이 호출이 검사 없이 실행되면 TypeError조차 안 난다
```

`age >= 18` 은 `str` 과 `int` 를 비교하려 하므로 실제로는 `TypeError` 가 나긴 한다. 하지만 이건 **우연히** 그 연산자가 타입에 민감했기 때문이다. `age` 를 그냥 로그에 찍기만 했다면 아무 에러 없이 잘못된 문자열이 시스템 전체에 퍼져 나갔을 것이다. 타입 힌트는 이런 침입을 막지 않는다. 외부에서 들어오는 데이터의 형태를 **실행 중에** 확인하고 싶다면 타입 힌트로는 부족하다.
:::

이 한계를 메우려고 나온 게 **런타임 검증** 라이브러리다. `pydantic` 이 대표적이다. `pydantic` 모델은 타입 힌트와 똑같은 문법을 쓰지만, 인스턴스를 만드는 순간 **실제로 값을 검사하고, 필요하면 변환하고, 안 맞으면 예외를 던진다.** 방금 본 `order["price"]` 문제를 `pydantic` 으로 선언했다면 `json.loads` 직후 바로 `ValidationError` 가 났을 것이다 — `total_price` 함수 안까지 잘못된 값이 들어가지도 못한다. 이 라이브러리는 [2.7 attrs와 pydantic v2](#/pydantic)에서 직접 다룬다. 지금 기억해 둘 건 하나다. **정적 타입 힌트(pyright/mypy)와 런타임 검증(pydantic)은 서로 다른 문제를 푼다.** 하나가 다른 하나를 대체하지 않는다.

::: hist 왜 파이썬은 처음부터 런타임 검사를 안 넣었나
타입 힌트를 도입한 PEP 484 자체가 "런타임 타입 검사는 이 제안의 목표가 아니다"라고 명시한다. 이유는 성능과 유연성이다. 모든 함수 호출마다 인자 타입을 검사하면 그만큼 느려진다. 게다가 파이썬은 [1.15 프로토콜, ABC, 덕 타이핑](#/protocols)에서 보듯 "정확히 그 타입인가"보다 "그 동작을 할 수 있는가"를 더 중요하게 여기는 언어다. 타입을 엄격히 강제하면 이 유연성과 충돌한다. 그래서 파이썬은 검사를 **완전히 별도의 선택적 도구**로 떼어냈다. 강제하고 싶으면 `pydantic` 처럼 명시적으로 그런 도구를 골라 쓰라는 것이다.
:::

## 요약

- 파이썬은 **점진적 타이핑**을 채택했다 — 타입은 선택이고, 붙여도 실행 방식은 바뀌지 않는다.
- **실측으로 확인했다**: 틀린 타입 힌트를 단 함수도 에러 없이 실행된다. 타입 힌트는 `__annotations__` 라는 딕셔너리에 저장되는 메타데이터일 뿐이다.
- 타입 힌트를 실제로 활용하는 건 인터프리터가 아니라 **별도의 도구**다 — `pyright`/`mypy` 같은 정적 검사기, 에디터의 자동완성 엔진.
- `uvx pyright proof1.py` 와 `uvx mypy proof1.py` 는 실행 한 번 없이 함수 시그니처 위반을 정확히 찾아냈다.
- 타입은 코드베이스가 **커지고, 사람이 늘고, 오래 살수록** 값어치가 커진다. 작은 스크립트나 코딩테스트 코드에서는 부담이 더 클 수 있다.
- **한계**: 타입 힌트는 런타임에 값을 검증하지 않는다. 외부 데이터(JSON, 사용자 입력)의 실제 형태를 실행 중에 보장하려면 `pydantic` 같은 런타임 검증 도구가 필요하다.
