# 2.3 PEP 695: 새 제네릭 문법

::: lead
제네릭 함수 하나를 타입과 함께 정의하려면 3.11까지는 `TypeVar` 를 따로 만들고, `Generic` 을 상속하고, 분산까지 손으로 지정해야 했다. 3.12는 이걸 언어 문법으로 끌어올렸다 — `def f[T](...)`, `class Box[T]:`, `type Alias = ...`. 이 절은 새 문법이 무엇을 대체하는지, 그리고 겉보기엔 사소해 보이는 이 변화가 실제로 타입 검사 결과를 바꾼다는 것을 실행해서 보여준다.
:::

## 3.11까지: TypeVar를 손으로 조립해야 했다

리스트의 첫 원소를 돌려주는 함수를 제네릭하게 타이핑한다고 하자. 3.11까지 방법은 이렇다.

```python title="old_typevar.py — 3.11 이전 방식"
from typing import TypeVar

T = TypeVar("T")


def first(items: list[T]) -> T:
    return items[0]


print(first([1, 2, 3]))
print(first(["a", "b"]))
```

이게 동작은 하지만 번거롭다. `T` 라는 이름을 **모듈 레벨에 별도로 선언**해야 하고, 그 이름은 함수 시그니처와 물리적으로 떨어져 있다. 파일 하나에 제네릭 함수가 여러 개면 `T`, `K`, `V`, `T_co`, `T_contra` 가 파일 위쪽에 잔뜩 쌓인다. 게다가 `T` 는 **모듈 스코프에 실제로 존재하는 이름**이라서, 실수로 다른 곳에서 재사용하거나 이름이 충돌할 여지가 있다.

## 3.12+: 함수 시그니처 자체에 타입 매개변수를 쓴다

PEP 695가 도입한 문법은 함수 이름 바로 뒤에 대괄호를 붙인다.

```python title="new_pep695.py — 3.12+ 방식"
def first[T](items: list[T]) -> T:
    return items[0]


print(first([1, 2, 3]))
print(first(["a", "b"]))
```

두 버전 모두 같은 결과를 낸다.

```pyrepl
>>> first([1, 2, 3])
1
>>> first(["a", "b"])
'a'
```

바뀐 건 문법이지 의미가 아니다. `[T]` 는 **이 함수만의 스코프**를 갖는 타입 매개변수를 선언한다. `T` 라는 이름은 함수 밖으로 새지 않는다 — 모듈 전역에 `TypeVar` 객체를 만들 필요가 없어졌다는 뜻이다. 실제로 함수가 끝나면 `T` 는 어디에도 남지 않는다.

```pyrepl
>>> def first[T](items: list[T]) -> T:
...     return items[0]
...
>>> T
Traceback (most recent call last):
  ...
NameError: name 'T' is not defined
```

::: hist 왜 3.12에서야 문법으로 들어왔나
`TypeVar` 방식은 3.5부터 있었다. 문제는 그게 **라이브러리 수준의 관용구**였다는 것이다 — 언어가 몰라주는 걸 `typing` 모듈이 흉내 낸 것에 가깝다. 파서는 `T = TypeVar("T")` 를 평범한 대입문으로만 본다. 타입 검사기가 이 패턴을 "제네릭 선언"으로 인식하려면 `TypeVar()` 호출이라는 관례를 알아서 해석해야 했다.

PEP 695(3.12)는 제네릭을 **문법 자체**로 만들었다. `def f[T](...)`, `class C[T]:`, `type X[T] = ...` 는 파서 단계에서 타입 매개변수 스코프를 만든다. 부가 효과로 타입 검사기의 구현이 단순해지고, 뒤에서 볼 **분산 자동 추론** 같은 게 가능해졌다.
:::

## 값 제약(bound)과 여러 타입 매개변수

`T` 가 아무 타입이나 받으면 곤란할 때가 있다. `<` 비교를 하려면 최소한 비교 가능한 타입이어야 한다. **bound** 로 이걸 제한한다.

```python title="generic_class.py"
def largest[T: (int, float)](items: list[T]) -> T:
    best = items[0]
    for x in items[1:]:
        if x > best:
            best = x
    return best
```

`T: (int, float)` 는 **제약 목록**(constrained type variable)이다. `T` 는 `int` 이거나 `float` 이어야 하고, 그 외의 다른 타입은 안 된다 — `int | float` 같은 합집합이 아니라 **둘 중 하나로 고정**되는 것이다. 단일 상한만 걸고 싶으면 괄호 없이 `T: Comparable` 처럼 쓴다.

타입 매개변수는 여러 개를 쉼표로 나열한다.

```python title="여러 타입 매개변수"
def pair_up[K, V](keys: list[K], values: list[V]) -> list[tuple[K, V]]:
    return list(zip(keys, values))
```

`uvx pyright` 로 이 두 함수를 실제로 검사하면 추론된 타입을 그대로 볼 수 있다. `reveal_type()` 은 파이썬 런타임에는 없는 특수 함수다 — 타입 검사기가 그 줄에서 자신이 추론한 타입을 출력창에 찍어 주는 디버깅 훅이다.

```python title="generic_class_clean.py — reveal_type으로 추론 확인"
reveal_type(largest([3, 1, 4]))
reveal_type(pair_up([1, 2], ["a", "b"]))

largest(["a", "b"])  # bound 위반 — str은 (int, float)에 없다
```

실제 `uvx pyright generic_class_clean.py` 출력이다.

```text nolines
generic_class_clean.py:22:13 - information: Type of "largest([3, 1, 4])" is "int"
generic_class_clean.py:23:13 - information: Type of "pair_up([1, 2], ["a", "b"])" is "list[tuple[int, str]]"
generic_class_clean.py:25:9 - error: Argument of type "list[str]" cannot be assigned to
    parameter "items" of type "list[T@largest]" in function "largest"
    Type "Literal['a']" is not assignable to constrained type variable "T"
    Type "Literal['b']" is not assignable to constrained type variable "T"
1 error, 0 warnings, 2 informations
```

`largest` 는 `int` 로, `pair_up` 은 `list[tuple[int, str]]` 로 정확히 추론됐다. 그리고 마지막 줄 — `str` 리스트를 넣자 **bound 위반을 정확히 잡아낸다.** `uvx mypy generic_class_clean.py` 도 같은 지점에서 걸린다.

```text nolines
generic_class_clean.py:25: error: Value of type variable "T" of "largest" cannot be "str"  [type-var]
Found 1 error in 1 file (checked 1 source file)
```

두 검사기 모두 실제로 그렇게 말했다 — 지어낸 메시지가 아니다. (pyright 1.1.411, mypy 2.3.0 기준.)

::: warn bound가 있어도 연산 결과가 T로 좁혀지는 건 아니다
`largest` 안의 `return best` 는 `T` 를 반환하는 것처럼 보이지만, `a + b` 같은 산술 연산의 결과 타입까지 `T` 로 좁혀 주지는 않는다. 실제로 다음 함수를 pyright에 넣으면,

```python
def add[T: int](a: T, b: T) -> T:
    return a + b
```

이런 에러가 난다.

```text nolines
error: Type "int" is not assignable to return type "T@add"
    Type "int" is not assignable to type "T@add" (reportReturnType)
```

`a + b` 는 `T` 가 아니라 **`int`** 로 추론된다 (`bound=int` 라서 `+` 연산자가 정의된 건 확실하지만, 그 결과가 정확히 호출 시점의 `T` 라는 보장은 없기 때문이다). bound는 "이 타입에서 가능한 연산"을 열어 주는 것이지, 연산 결과가 자동으로 `T` 로 승격되는 게 아니다. 필요하면 명시적으로 `return best` 처럼 원래 값을 그대로 반환해야 타입이 보존된다.
:::

## 제네릭 클래스: `class Box[T]:`

클래스에도 똑같은 문법이 적용된다. 클래스 이름 뒤에 `[T]` 를 붙이면 `Generic[T]` 를 상속하는 것과 같은 효과를 낸다.

```python title="generic_class.py"
class Box[T]:
    def __init__(self, item: T) -> None:
        self.item = item

    def get(self) -> T:
        return self.item


b = Box(3.14)
print(b.get())
print(Box.__type_params__)
```

```pyrepl
>>> b.get()
3.14
>>> Box.__type_params__
(T,)
```

`Box(3.14)` 라고만 썼는데 `T` 가 `float` 로 채워졌다. 이건 [1.10 함수](#/functions)에서 본 인자 추론과 같은 메커니즘이다 — 정적 검사기가 생성자 호출의 인자 타입을 보고 타입 매개변수를 역산한다. `__type_params__` 는 3.12에서 추가된 속성으로, 이 클래스(또는 함수, 타입 별칭)가 선언한 타입 매개변수를 런타임에 확인할 수 있게 해 준다.

::: deep 자동 분산 추론 — PEP 695가 실제로 바꾼 동작
`Generic[T]` 시절에는 `T` 가 함수 인자 위치와 반환 위치 둘 다에 쓰이면 기본적으로 **불변**(invariant)이었다. `ReadOnlyBox[Dog]` 를 `ReadOnlyBox[Animal]` 이 필요한 곳에 넘기려면 `covariant=True` 를 손으로 지정해야 했다.

```python title="variance_old.py — 3.11 이전, covariant 지정 없이"
T = TypeVar("T")


class ReadOnlyBox(Generic[T]):
    def get(self) -> T:
        return self._item


def take_animal_box(box: ReadOnlyBox[Animal]) -> None: ...


dog_box: ReadOnlyBox[Dog] = ReadOnlyBox(Dog())
take_animal_box(dog_box)
```

실제 `uvx pyright variance_old.py` 결과다.

```text nolines
error: Argument of type "ReadOnlyBox[Dog]" cannot be assigned to parameter "box"
    of type "ReadOnlyBox[Animal]" in function "take_animal_box"
    Type parameter "T@ReadOnlyBox" is invariant, but "Dog" is not the same as "Animal"
```

`ReadOnlyBox` 는 `T` 를 반환만 하고 받지는 않으니 직관적으로는 공변(covariant)이어야 맞는데, 기본값이 불변이라 막힌다.

PEP 695 문법으로 똑같은 클래스를 쓰면 결과가 달라진다.

```python title="variance.py — 3.12+"
class ReadOnlyBox[T]:
    def __init__(self, item: T) -> None:
        self._item = item

    def get(self) -> T:
        return self._item
```

`uvx pyright variance.py` 는 **0 errors** 를 낸다. 3.12+ 문법에서는 타입 검사기가 클래스 본문을 분석해서 `T` 가 반환 위치에만 쓰이는 걸 확인하고 **분산을 자동으로 추론**한다. 손으로 `covariant=True` 를 지정할 필요가 없어졌다. 같은 클래스 정의인데 문법만 바꿨을 뿐인데 타입 검사 결과가 바뀐 것이다 — PEP 695가 단순 설탕 문법이 아니라는 증거다.
:::

## `type` 문: 지연 평가되는 타입 별칭

지금까지 `Alias = list[int]` 처럼 타입 별칭을 평범한 대입으로 만들어 왔다면, 3.12부터는 전용 문법 `type` 이 생겼다.

```pyrepl
>>> type IntList = list[int]
>>> IntList
IntList
>>> type(IntList)
<class 'typing.TypeAliasType'>
>>> IntList.__value__
list[int]
```

평범한 대입과 다른 점이 세 가지다.

1. **전용 타입** `TypeAliasType` 의 인스턴스가 된다. `list[int]` 그 자체가 아니라, 그걸 가리키는 별개의 객체다.
2. `.__value__` 로 실제 타입을 지연해서 얻는다.
3. 그리고 이 지연이 핵심이다 — **오른쪽 표현식이 그 줄에서 즉시 평가되지 않는다.**

```python title="type_alias.py — 아직 정의 안 된 이름을 가리켜도 통과한다"
type Later = NotYetDefined  # 이 시점엔 NotYetDefined가 없다


class NotYetDefined:
    pass


print(Later.__value__)
```

```pyrepl
>>> Later.__value__
<class '__main__.NotYetDefined'>
```

실제로 통과한다. 반면 `type` 없이 평범한 대입으로 같은 걸 시도하면 그 자리에서 바로 죽는다.

```python title="eager_fail.py — 일반 대입은 즉시 평가된다"
Later = NotYetDefined  # 즉시 평가 → 이 시점엔 이름이 없다


class NotYetDefined:
    pass
```

```text nolines
NameError: name 'NotYetDefined' is not defined
```

`type` 문의 오른쪽은 **첫 접근 시점**까지 평가를 미룬다. 이게 왜 중요하냐면, 서로를 참조하는 타입이나 아직 정의되지 않은 클래스를 가리키는 별칭을 파일 어디서든 자유롭게 선언할 수 있게 되기 때문이다. 이전까지는 문자열 리터럴로 감싼 **전방 참조**(`"NotYetDefined"`)로 이 문제를 우회했는데, 그건 타입 검사기가 문자열을 파싱해서 처리해야 하는 특수 케이스였다. `type` 문은 언어 차원에서 지연 평가를 보장하므로 그 우회가 필요 없다.

타입 별칭도 제네릭으로 만들 수 있다.

```pyrepl
>>> type Pair[T] = tuple[T, T]
>>> Pair[int]
Pair[int]
>>> Pair.__type_params__
(T,)
```

::: note 이건 어노테이션 지연 평가와는 다른 이야기다
함수·변수 어노테이션(`def f(x: SomeType)`)이 언제 평가되는지는 또 다른 주제다. 3.14는 `from __future__ import annotations` 없이도 어노테이션을 기본적으로 지연 평가하는 PEP 649를 채택했다. 이건 [2.9 런타임 타입 정보와 리플렉션](#/runtime-typing)에서 자세히 다룬다. 여기서 본 `type` 문의 지연 평가는 그것과 별개의, 타입 별칭 전용 메커니즘이다.
:::

## 타입 힌트는 런타임에 아무 영향이 없다 — 다시 한번 증명

Part I 전체를 관통하는 사실이 여기서도 그대로다. `bound` 를 걸든, 몇 겹으로 제네릭을 감싸든, **런타임 인터프리터는 타입 어노테이션을 실행 경로에서 전혀 확인하지 않는다.**

```python title="no_runtime_effect.py — 일부러 틀린 타입 힌트"
def add[T: int](a: T, b: T) -> T:
    return a + b


# bound=int 라고 선언했지만 문자열 두 개를 넣는다
print(add("파이썬", "완전정복"))
print(add([1, 2], [3, 4]))
```

이 파일을 그냥 `python` 으로 실행하면 어떤 에러도 없이 잘 돈다.

```pyrepl
>>> add("파이썬", "완전정복")
'파이썬완전정복'
>>> add([1, 2], [3, 4])
[1, 2, 3, 4]
```

`T: int` 라는 선언은 `str` 도, `list[int]` 도 완벽하게 받아들인다. 인터프리터는 `[T: int]` 를 실행 시점에 검사할 코드로 컴파일하지 않는다 — 이건 정적 검사기가 나중에 읽으라고 남겨 둔 메타데이터일 뿐이다. `a + b` 는 `str.__add__` 나 `list.__add__` 를 정상적으로 호출할 뿐, `T` 라는 이름은 런타임의 그 어떤 스택 프레임에도 존재하지 않는다.

반면 앞서 본 것처럼 `uvx pyright no_runtime_effect.py` 는 이 파일에 대해 5개의 에러를 낸다.

```text nolines
no_runtime_effect.py:2:12 - error: Type "int" is not assignable to return type "T@add"
    Type "int" is not assignable to type "T@add" (reportReturnType)
no_runtime_effect.py:6:11 - error: Argument of type "Literal['파이썬']" cannot be assigned to
    parameter "a" of type "T@add" in function "add" (reportArgumentType)
no_runtime_effect.py:6:18 - error: Argument of type "Literal['완전정복']" cannot be assigned to
    parameter "b" of type "T@add" in function "add" (reportArgumentType)
no_runtime_effect.py:7:11 - error: Argument of type "list[int]" cannot be assigned to
    parameter "a" of type "T@add" in function "add" (reportArgumentType)
no_runtime_effect.py:7:19 - error: Argument of type "list[int]" cannot be assigned to
    parameter "b" of type "T@add" in function "add" (reportArgumentType)
5 errors, 0 warnings, 0 informations
```

같은 소스, 두 개의 완전히 다른 세계다. 파이썬 인터프리터는 이 함수를 아무 문제 없이 두 번 실행했다. pyright는 같은 함수를 5개의 오류로 거부했다. **둘 다 옳다** — 서로 다른 질문에 답하고 있을 뿐이다. 인터프리터는 "이 코드가 지금 실행되는가"를 묻고, 타입 검사기는 "이 코드가 선언한 계약을 지키는가"를 묻는다. 이 구분은 [2.1 왜 타입 힌트인가](#/why-typing)의 핵심이었고, 여기서 문법이 정교해져도 변하지 않는다.

::: cote 코딩테스트에서는
타입 매개변수 문법은 코딩테스트 채점 서버에서 쓸 일이 거의 없다. 대부분 온라인 저지는 타입 검사를 돌리지 않고 인터프리터로 바로 채점하기 때문이다. 다만 **제네릭 함수를 짜 두면 여러 타입에 재사용**할 수 있어서, 템플릿 코드([8.10 실전 템플릿 모음](#/templates))를 만들 때는 유용하다. 채점에는 영향이 없지만 읽는 사람(면접관, 리뷰어)에게는 의도가 분명해진다.
:::

## 3.12 이전 코드베이스와 함께 쓰기

레거시 코드가 여전히 `TypeVar` 를 쓰고 있다면, 두 방식은 같은 타입 검사기 안에서 섞여도 문제없다. `TypeVar` 로 만든 제네릭과 `def f[T]` 로 만든 제네릭은 pyright/mypy 양쪽에서 동일하게 인식된다 — PEP 695는 기존 `typing.TypeVar` 를 폐기한 게 아니라 **더 나은 문법을 추가**한 것이다. 다만 새 프로젝트라면, 그리고 3.12+ 만 지원하면 된다면, 새 문법을 기본으로 쓰는 게 낫다. 모듈 스코프를 오염시키지 않고, 타입 매개변수의 스코프가 시그니처에 눈으로 보이고, 분산까지 자동으로 계산해 주기 때문이다.

## 요약

- 3.12 이전에는 `TypeVar` 를 모듈 레벨에 선언하고 `Generic[T]` 를 상속해야 했다. 3.12+ 는 `def f[T](...)`, `class C[T]:`, `type X[T] = ...` 문법으로 그걸 대체한다.
- `T` 의 스코프는 함수/클래스/별칭 자신으로 한정된다. 함수가 끝나면 `T` 는 남지 않는다.
- `T: (int, float)` 는 제약(constraint), `T: Comparable` 은 상한(bound)이다. bound가 있어도 연산 결과 타입이 자동으로 `T` 로 좁혀지진 않는다.
- `class Box[T]:` 는 반환 위치에만 쓰인 `T` 의 분산을 **자동으로 추론**한다. 옛 `TypeVar` 방식은 기본이 불변이라 `covariant=True` 를 손으로 지정해야 같은 효과를 냈다 — 실제로 `uvx pyright` 결과가 갈렸다.
- `type Alias = ...` 는 `TypeAliasType` 객체를 만들고, 오른쪽 표현식을 **지연 평가**한다. 그래서 아직 정의 안 된 이름을 가리켜도 된다.
- 타입 힌트는 문법이 아무리 정교해져도 **런타임에는 없는 것과 같다.** `bound=int` 를 선언한 함수에 문자열을 넣어도 파이썬은 그냥 실행한다. 계약을 지키는지는 `uvx pyright`/`uvx mypy` 가 판단할 몫이다.
