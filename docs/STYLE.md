# 집필 규범 — 『파이썬 완전 정복』

이 책의 모든 절은 이 규범을 따른다. 기준 샘플은 `content/start/how-to-read.md` 와
`content/core/objects-names.md` 다. **쓰기 전에 반드시 이 둘을 읽어라.**

---

## 1. 독자와 목표

- 독자는 **파이썬 기본 문법을 아는 사람**이다. `for`, `if`, 리스트, 딕셔너리, 함수 정의는 안다.
- 목표는 *"파이썬으로 코딩테스트를 통과하면서 ROS·ML·DL·비전·AI를 직접 만드는 개발자"* 다.
- 그러므로 **"~하는 법"이 아니라 "왜 그런가"** 를 쓴다. 사용법은 검색하면 나온다.
  검색으로 안 나오는 것을 써라: 내부 동작, 설계 이유, 함정, 실제 비용.

### 밀도의 기준

한 절은 **읽는 데 15~30분**, 본문 6,000~12,000자(코드 제외) 정도다.
짧으면 얕고, 길면 안 읽는다. 다만 **분량을 채우려고 늘리지 마라.** 할 말이 없으면 짧게 끝낸다.

---

## 2. 말투

- **평서형 반말체.** "~한다", "~이다". `~습니다`, `~해요` 금지.
- 독자를 부를 때는 "당신". 명령형은 "~하라", "~해라" 를 쓴다.
- **단정적으로 쓴다.** "~인 것 같다", "~일 수도 있다" 같은 헤지를 피한다.
  단, 모르면 모른다고 쓴다. 아는 척이 최악이다.
- 문장은 짧게. 한 문장에 한 가지만.
- **비유는 정확할 때만.** 틀린 비유는 잘못된 모델을 심는다.

### 좋은 예

> 파이썬에서 `=` 는 값을 복사하지 않는다. 이름표를 붙일 뿐이다.

> `deepcopy` 를 습관적으로 쓰지 마라. 대부분의 경우 더 나은 선택지가 있다.

### 나쁜 예

> 파이썬에서는 변수 할당 시 값이 복사되지 않을 수도 있으니 주의가 필요합니다.

> deepcopy는 유용한 함수이지만 상황에 따라 성능 이슈가 발생할 수 있습니다.

## 4. 마크다운 확장 문법

### 코드 블록

````markdown
```python title="파일명.py"
실행 가능한 완결된 코드
```

```python title="설명" {3,5-7}
3번, 5~7번 줄이 강조된다
```

```pyrepl
>>> 대화형 세션
결과            <- >>> 없는 줄은 출력으로 칠해진다
```

```bash
터미널 명령
```

```text nolines
아스키 다이어그램 (줄 번호 없음)
```
````

지원 언어: `python` `pyrepl` `bash` `json` `yaml` `toml` `cpp` `c` `sql` `js` `text`

### 표시 상자

| 문법 | 쓰는 때 |
| --- | --- |
| `::: note` | 알아두면 좋은 보충. 몰라도 진도는 나간다 |
| `::: tip` | 실무 요령 |
| `::: warn` | 틀리기 쉬운 지점 |
| `::: danger` | **버그를 만드는** 지점. 실제 장애로 이어지는 것만 |
| `::: deep` | CPython 내부, PEP, 한 단계 아래 이야기 |
| `::: perf` | 실행 속도·메모리. **반드시 측정치와 함께** |
| `::: cote` | 코딩테스트에서 바로 쓰는 지식 |
| `::: hist` | 왜 이렇게 설계됐는가 |
| `::: quiz` | 연습문제 |
| `::: answer` | 해설 (접혀서 나온다) |
| `::: lead` | 절 머리말 |

**한 절에 상자가 8~16개** 정도가 적당하다. 너무 많으면 본문이 사라진다.

`::: cote` 는 **관련 있을 때만** 넣는다. 억지로 만들지 마라.

### 기타

- 수식: `$O(n \log n)$`, `$$블록 수식$$`
- 형광펜: `==강조==`
- 인라인 태그: `<kbd>Ctrl</kbd>` 만 허용
- 링크: 같은 책 안은 `[1.6 dict](#/dict)` — **toc.json 의 id를 쓴다**

## 5. 절대 규칙

### 5.1 사실은 반드시 실행해서 확인한다

**추정한 수치를 쓰지 마라.** 벤치마크를 쓸 거면 실제로 돌려서 나온 값을 쓴다.

```markdown
(Python 3.14 / Windows 기준 실측. 절대값은 기기마다 다르지만 자릿수 차이는 어디서나 같다.)
```

REPL 출력도 마찬가지다. **`>>>` 뒤의 출력은 실제로 그렇게 나와야 한다.**
특히 조심할 것:

- `is` 비교는 **REPL과 파일에서 결과가 다르다** (코드 객체 경계 = 상수 폴딩).
- `id()` 값, 메모리 주소는 실행할 때마다 다르다 — 예시임을 알 수 있게 쓴다.
- 딕셔너리·집합의 출력 순서.
- 버전에 따라 달라지는 것.

### 5.2 아스키 다이어그램 안에는 ASCII만

한글은 고정폭 글꼴에서 **두 칸**을 차지한다. 선(`│ ─ ┌ └`)보다 **왼쪽에** 한글이 있으면 그림이 깨진다.
선보다 오른쪽의 꼬리 주석은 괜찮다.

```text nolines
   OK   ├── pyvenv.cfg      <- 설명은 한글이어도 된다
   OK   a ──▶ ┌──────────┐
   BAD  │ int 객체  │        <- 상자 안의 한글
```

`python tools/check_diagrams.py` 가 검사한다.

### 5.3 코드

- **모든 코드는 실행 가능해야 한다.** 조각이면 조각임을 밝힌다.
- 주석은 **왜**를 쓴다. 무엇을 하는지는 코드가 이미 말한다.
- 나쁜 예/좋은 예는 `# ❌` / `# ✅` 로 표시한다.
- 타입 힌트는 **필요할 때만.** 예제를 흐리면 뺀다.
- 변수명·주석은 한국어로 써도 된다. 단 식별자는 영어.

## 6. 이 책의 성격을 만드는 것

다른 책과 이 책을 가르는 지점이다. 매 절에서 최소 하나는 해라.

1. **한 층 아래로 내려간다.** "리스트는 동적 배열이다"에서 멈추지 말고,
   `sys.getsizeof` 로 실제 증폭 패턴을 보여준다.
2. **실제 비용을 말한다.** $O(1)$ 이라고만 하지 말고 상수가 얼마나 큰지,
   언제 그 상수가 문제가 되는지 쓴다.
3. **함정을 먼저 보여준다.** 독자가 실제로 당할 것을 미리 당하게 한다.
4. **설계 이유를 말한다.** "왜 파이썬은 이렇게 했나"에 답한다.
5. **뒤의 파트와 잇는다.** 이 지식이 ROS/ML/DL/코테에서 어디에 쓰이는지.
